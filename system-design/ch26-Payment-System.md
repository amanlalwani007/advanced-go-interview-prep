# Chapter 26 — Design a Payment System

## Q1: Design the core payment processing flow — from checkout button click to "Payment Successful" — covering authorization, capture, settlement, and reconciliation.

**Answer:** The payment lifecycle: (1) **Checkout initiation** — the user clicks "Pay $50.00". The client sends a request to the payment service with `{user_id, merchant_id, amount, currency, payment_method_nonce}`. (2) **Idempotency check** — the payment service checks if an idempotency key (generated by the client) has already been processed. If yes, return the existing result (prevents double-charges from retries). (3) **Risk assessment** — the payment is scored by the fraud/risk service. If high-risk, require 3D Secure authentication (SCA). (4) **Authorization** — the payment service calls the acquiring bank / processor (e.g., Stripe, Adyen) with `{card_number, expiry, cvv, amount, merchant_id}`. The gateway sends an ISO 8583 authorization message to the issuing bank. The bank blocks the amount on the user's card and returns an authorization code (e.g., `auth_123ABC`). (5) **Capture** — immediately (for digital goods) or at shipment (for physical goods), the payment service sends a capture request with the authorization code. The issuing bank completes the transfer. (6) **Settlement** — at end of day, the acquirer settles with the merchant: net amount = gross sales - interchange fees - acquirer fees. Settlement is a batch process (Fedwire, ACH, or batch file). (7) **Reconciliation** — the payment service compares its internal transaction log with the settlement report from the acquirer. Discrepancies (e.g., settled amount differs from expected) are flagged for manual review. (8) **Failure handling** — if authorization fails (insufficient funds, expired card, fraud block), the user sees the failure message and is prompted to retry with another payment method. (9) **Dual-message vs. single-message** — card-present transactions use single-message (auth+capture in one). Card-not- continued use dual-message (auth now, capture later). Digital wallets like PayPal use a single auth+token exchange.

## Q2: How do you handle the reconciliation problem — matching the payments recorded in your internal system with the settlement reports from the payment gateway, when there are millions of transactions per day?

**Answer:** Reconciliation is an automated batch process: (1) **Data sources** — (a) internal transaction log (database): `{txn_id, amount, status, gateway_txn_id, created_at}`. (b) Gateway settlement report (downloaded via SFTP or API): `{gateway_txn_id, settled_amount, fee, net_amount, settled_at}`. (2) **Matching process** — a daily batch job (Spark or scheduled task) loads both datasets and joins on `gateway_txn_id`. For each transaction, it compares: `internal.amount == gateway.settled_amount` and `internal.currency == gateway.currency`. (3) **Discrepancy handling** — (a) **Unmatched in internal** — the gateway reports a settlement but our system has no record. This could be a duplicate charge or a gateway test transaction. Flagged for manual review. (b) **Unmatched in gateway** — our system recorded a capture but the gateway doesn't show it. This is critical — the payment may have failed on the gateway side but our system thinks it succeeded. Auto-retry or freeze the order. (c) **Amount mismatch** — settled amount differs by > $0.01. Flagged for investigation. (4) **Time zones** — settlement reports are in the gateway's time zone. The reconciliation job uses UTC for all comparisons. (5) **Thresholds** — small discrepancies (< $1) are auto-resolved by adjusting the merchant's next settlement. Large discrepancies trigger a ticket. (6) **Re-reconciliation** — if the gateway sends a corrected settlement report (e.g., chargeback reverses a previous settlement), the reconciliation job processes the delta.

## Q3: Design the idempotency layer. How do you guarantee that retrying a failed payment request does not charge the user twice?

**Answer:** Idempotency is the most critical correctness property in payment systems: (1) **Idempotency key** — the client generates a unique key (UUID v4) for each payment attempt and sends it as an `Idempotency-Key` header. (2) **Idempotency store** — a fast KV store (Redis or a dedicated PostgreSQL table) stores `{idempotency_key → {status, response, created_at}}` with a TTL of 24 hours. (3) **First request** — the payment service checks the idempotency store. Key not found → proceed with payment processing. After processing, store the result: `SET idempotency:<key> '{status: "success", txn_id: "txn_123"}' EX 86400`. (4) **Retry request** — if the client retries (same idempotency key), the service finds the existing entry and returns the cached response without processing again. The client sees the same response (status, body) as the first attempt. (5) **In-flight handling** — if a request is currently being processed and a second request with the same idempotency key arrives, the service returns `409 Conflict` with a "retry after 100ms" header. (6) **Critical edge case** — what if the payment succeeds on the gateway side but the response to the client is lost (network failure)? The client retries with the same idempotency key. The service checks the idempotency store, finds the previous successful response, and returns it. Idempotency keys are unique per merchant to prevent merchant-A from replaying merchant-B's requests. Stripe's idempotency implementation is the canonical reference for this pattern.

## Q4: How do you design the 3D Secure (SCA) authentication flow? A user's bank requires additional authentication for a €100 payment. Walk through the flow.

**Answer:** 3D Secure v2 (EMVCo protocol): (1) **Step-up trigger** — the payment service sends the transaction to the issuing bank's 3DS server via the gateway. The bank evaluates the transaction risk. If SCA is required, the bank returns a `challenge_url` (or a fingerprint challenge). (2) **Fingerprinting** — the 3DS server may issue a silent challenge (device fingerprinting). The user's browser executes JavaScript from the 3DS server that collects device information (IP, user-agent, screen resolution, installed fonts). If the device matches the user's known devices, the challenge is silently authenticated (no user interaction). (3) **Challenge** — if fingerprinting is insufficient, the user is redirected to the bank's ACS (Access Control Server) page where they must provide 2FA: a one-time password sent via SMS, a biometric scan (FaceID/TouchID), or a banking app approval. (4) **Callback** — after authentication, the 3DS server sends a `cryptogram` to the payment gateway. The gateway sends the cryptogram to the acquirer, who sends it to the issuing bank as part of the authorization request. (5) **Fallback** — if the user's bank/card does not support 3DS v2, fall back to 3DS v1 (redirect to the bank's page with a password prompt). (6) **Friction minimization** — the goal is to minimize challenged transactions. Stripe's "Radar" and Adyen's "RevenueAccelerator" use machine learning to predict which transactions will authenticate without friction and only challenge the high-risk ones. The PSD2 regulation requires SCA but allows transaction risk analysis (TRA) exemptions for low-risk transactions (<€30) or transactions with trusted beneficiaries.

## Q5: Design the payment retry and dunning system for recurring billing (subscriptions). When a credit card payment fails, the system should retry intelligently and eventually notify the user.

**Answer:** The dunning (debt collection) system: (1) **Failure detection** — when a recurring payment fails (expired card, insufficient funds, fraud block), the subscription is marked `past_due`. (2) **Retry schedule** — smart retry with exponential backoff: retry after 3 hours, then 6 hours, then 12 hours, then 24 hours, then 48 hours, then 72 hours, then 5 days, then 7 days (total ~13 retries over ~20 days). The schedule is configurable per merchant. (3) **Card updater** — before retrying, check with the card network's automatic card updater service (Visa Account Updater, Mastercard Automatic Billing Updater). Many users get new cards with new expiry dates; the card updater service provides the updated card details. (4) **Payment method change request** — the day after the first failure, send an email: "Your payment failed. Update your card here." Include a link to the billing portal. (5) **Smart timing** — retries should happen at times when the bank is more likely to approve: weekday mornings (10 AM), avoid late nights and weekends when authorization success rates are lower. (6) **Escalation** — after 7 failed retries, send a final notice: "Your subscription will be cancelled in 3 days unless payment is received." After 10 days, suspend the account (downgrade to free tier). After 30 days, delete the account data (with a 30-day backup retention). (7) **Communication** — each retry attempt and failure status is communicated via email and in-app notifications. The system also handles voluntary card updates — if the user updates their card before the next retry, the retry schedule resets.

## Q6: How do you handle chargebacks and disputes? When a cardholder disputes a charge, what happens in the payment system?

**Answer:** The chargeback workflow: (1) **Notification** — the issuing bank sends a chargeback notification (ISO 8583 message 1931 or Visa's VCR file) to the acquirer. The acquirer forwards it to the payment system. (2) **Reason codes** — chargebacks have standardized reason codes: Visa 11.3 (goods not received), 13.1 (amount exceeds authorized), 30.1 (no cardholder authorization / fraud). (3) **Funds reservation** — the chargeback amount is debited from the merchant's settlement batch. If the merchant has insufficient funds, the payment system must cover it (become the "merchant of record" risk). (4) **Evidence gathering** — the payment system collects evidence: shipping confirmation (with tracking), customer communication logs, IP address logs, signed delivery receipts, proof of service delivery. (5) **Representment** — within the dispute window (e.g., 20 days for Visa), the payment system submits a representment packet with evidence to the acquirer, who forwards it to the issuing bank. (6) **Outcome** — if representment is successful, the chargeback is reversed (funds returned to the merchant). If the bank rejects the representment, the chargeback stands, and the merchant loses the funds plus a chargeback fee ($15-100). (7) **Chargeback monitoring** — the payment system tracks the merchant's chargeback ratio: total chargebacks / total transactions. If the ratio exceeds the card network's threshold (e.g., 1% for Visa), the merchant enters a monitoring program and risks being blacklisted. (8) **Prevention** — use fraud screening (AVS, CVV, 3DS), clear descriptors on the card statement, and clear refund policies to reduce chargebacks.

## Q7: Design the multi-currency and foreign exchange (FX) system. A user in Japan pays ¥10,000 to a US-based merchant who wants to receive $85.00. How does the system handle the currency conversion?

**Answer:** Multi-currency payment flow: (1) **Pricing** — the merchant sets prices in USD ($85.00). When a Japanese user views the product, the system shows: "¥10,000 (~$85.00)" using the current exchange rate from the FX service. (2) **FX rate engine** — the system maintains a stream of exchange rates from multiple providers (Reuters, Bloomberg, XE). Rates are polled every 5 seconds for major pairs and every 60 seconds for minor pairs. The system adds a markup (e.g., 0.5-2%) based on the merchant's pricing tier. (3) **Currency conversion** — when the payment is made, the system records the transaction in both currencies: `{base_currency: JPY, base_amount: 10000, settlement_currency: USD, settlement_amount: 85.00, fx_rate: 0.0085, fx_markup: 0.01}`. The authorization to the acquiring bank is in JPY (the user's local currency). (4) **Settlement** — the acquirer settles with the merchant in USD. The merchant receives $85.00 (minus processing fees). The FX conversion happens at the acquiring bank level (if the acquirer supports multi-currency) or via a dedicated FX provider like CurrencyCloud. (5) **Hedging** — if the merchant wants to lock in a rate for future payments (to avoid FX volatility), the system can offer forward contracts or dynamic conversion at the time of payment. (6) **Memory** — the fx rate used for the transaction is stored with the transaction record. This is critical for refunds: a refund must use the same fx rate as the original charge, not the current rate. (7) **Cross-currency reporting** — the merchant's dashboard shows sales in their settlement currency, but the system also reports in their base currency for tax purposes. All currency conversions are logged with timestamps for audit.

## Q8: Design the fraud detection system. How do you catch a stolen credit card being used for a high-value purchase at 3 AM from a new device?

**Answer:** Real-time fraud scoring pipeline: (1) **Rules engine** — immediate evaluation rules (executed in <10 ms): (a) Is the card on any blacklist (known stolen cards from card network alerts)? (b) Is the IP address from a high-risk country (where the merchant doesn't ship)? (c) Is the transaction amount > 5× the user's average? (d) Is the velocity check exceeded (3+ declined attempts in 5 minutes)? (2) **Device fingerprint** — collect device fingerprint via JavaScript: screen resolution, installed fonts, time zone, battery status, canvas fingerprint, WebGL fingerprint. Compare against the user's historical device fingerprints. (3) **Velocity checks** — count transactions from this card in the last hour. If > 3, flag. (4) **ML model** — an ensemble model (gradient-boosted trees or neural network) trained on millions of labeled transactions (confirmed fraud vs. legitimate). Features: transaction amount, time of day, device fingerprint match, IP geolocation vs. cardholder address distance, shipping address vs. IP distance, email domain age, account age. (5) **Real-time scoring** — the ML model runs on each transaction, outputting a fraud score (0-100). Thresholds: 0-30 → accept, 30-70 → flag for manual review (or request 3DS), 70-100 → decline. (6) **Feedback loop** — confirmed fraud transactions (chargebacks, user reports) are fed back as training data. The model is retrained daily. (7) **Adaptive friction** — if the transaction is borderline (score 60), require 3D Secure authentication (step-up). If the user passes 3DS, accept. If they fail or skip, decline. This catches the 3AM stolen-card scenario: the card is new (no history), device is new, time is unusual, amount is high → ML score > 90 → declined.

## Q9: How would you design the payment gateway API (like Stripe's API) that thousands of merchants integrate with? Focus on the API design principles and idempotency guarantees.

**Answer:** The public API design: (1) **RESTful over HTTPS** — resources: `POST /v1/charges`, `POST /v1/payment_intents`, `GET /v1/charges/:id`, `POST /v1/refunds`. Every response includes an idempotent `id` field (e.g., `ch_12345`). (2) **Idempotency key** — all mutating endpoints accept an `Idempotency-Key` header. If the key is reused within 24 hours, the same response is returned. (3) **Webhook delivery** — the gateway delivers asynchronous events (charge.succeeded, charge.failed, payment_intent.succeeded) via webhooks to the merchant's registered endpoint. Delivery is retried with exponential backoff over 3 days. (4) **Reporting** — the merchant downloads a CSV of all transactions or uses the reporting API. (5) **Versioning** — API versioning via the `Stripe-Version` header (e.g., `2019-02-19`). Breaking changes are accompanied by a new version. (6) **Rate limiting** — per-merchant rate limits (e.g., 100 req/s) using a token bucket. The response includes `X-RateLimit-Remaining` headers. (7) **Idempotency timeout** — the gateway maintains a 24-hour rolling window for idempotency keys. A merchant can safely retry within 24 hours. (8) **Partial response** — `?expand[]=customer` allows the merchant to include expanded sub-resources in a single API call, reducing N+1 query problems. (9) **Pagination** — list endpoints use cursor-based pagination: `?starting_after=ch_12345&limit=100`. Stripe's API is considered the gold standard for API design in the payments industry.

## Q10: Design the financial ledger / double-entry bookkeeping system that sits behind the payment service. Every payment must be accounted for with debits and credits.

**Answer:** A double-entry ledger is the source of truth for all money movement in a payment system: (1) **Core concept** — every transaction creates at least two entries: a debit (increase in one account) and a credit (decrease in another). Total debits must equal total credits. (2) **Chart of accounts** — predefined accounts: `ASSET:STRIPE_BALANCE`, `ASSET:MERCHANT_SETTLEMENT`, `LIABILITY:MERCHANT_PAYABLE`, `REVENUE:PROCESSING_FEE`, `EXPENSE:CHARGEBACK`. (3) **Transaction recording** — when a $100 payment is processed with a $2 fee, two journal entries are created: `Debit: STRIPE_BALANCE $98 | Credit: MERCHANT_PAYABLE $98` and `Debit: MERCHANT_PAYABLE $98 | Credit: REVENUE:PROCESSING_FEE $2`. (4) **Database schema** — `journal_entries (id, description, created_at)` and `journal_lines (entry_id, account_id, amount, currency, direction)`. All entries are immutable (append-only). (5) **Reconciliation** — the ledger is reconciled daily against gateway settlement reports and bank statements. A script compares the ledger's expected balances with the actual bank balance. (6) **Audit trail** — every financial event (charge, refund, chargeback, payout) is traced through the journal. A single payment's journey from customer to merchant can be traced by following the journal entries. (7) **Concurrency** — to prevent double-crediting, the ledger uses an optimistic locking mechanism: `UPDATE journal SET version = version + 1 WHERE id = X AND version = Y`. (8) **Settlement** — when the merchant requests a payout, the ledger creates a transfer entry: `Debit: MERCHANT_PAYABLE $98 | Credit: BANK_ACCOUNT_HOLD $98`. After the ACH transfer completes, the hold is released. (9) **System of record** — the ledger is the single source of truth. All other systems (merchant dashboard, reporting, analytics) read from the ledger. Any discrepancy between the ledger and operational data (e.g., Redis cache) is resolved by trusting the ledger.

## Q11: Design the tokenization layer for PCI DSS scope reduction. How do you ensure that raw PAN (Primary Account Number) never touches your application servers?

**Answer:** Tokenization replaces the PAN with a surrogate value (token), removing the merchant's systems from PCI DSS scope: (1) **Token generation** — the payment gateway (e.g., Stripe, Adyen) generates a token (e.g., tok_visa_1234) that represents the card. The vault maps token to {PAN, expiry, cardholder_name, issuer, last4}. The merchant's app stores only the token, NOT the PAN. (2) **PCI scope reduction** — if the merchant NEVER stores, processes, or transmits the PAN, their PCI DSS scope is reduced from SAQ D (full questionnaire, about 250 questions) to SAQ A or A-EP (about 30 questions). This saves millions in compliance costs. (3) **Tokenization at the edge** — implement a card data capture iframe hosted by the gateway (Stripe Elements, Braintree Drop-in). The iframe sends the PAN directly to the gateway's PCI-compliant infrastructure. Your server receives only the token: {token, last4, brand}. (4) **Network tokenization (Visa TMS, Mastercard MDES)** — more advanced: the card network itself issues a network token (DPAN) that replaces the PAN for online transactions. The DPAN is restricted to a specific merchant/device. Even if the DPAN is leaked, it cannot be used at another merchant. (5) **Token vault design** — (a) **Storage**: AES-256 encrypted at rest. The token to PAN mapping is stored in a dedicated HSM-backed database, isolated from the merchant's application database. (b) **Access**: only the gateway's tokenization service can decrypt. The merchant never has direct access. (c) **Token format**: single-use tokens (for one-time checkout) vs. multi-use tokens (for recurring billing). Multi-use tokens have a shorter validity and require re-authorization for large amounts. (6) **Detokenization** — when the merchant needs to run the transaction (authorization, capture), they send the token to the gateway. The gateway detokenizes to get the PAN, sends the authorization to the issuing bank, and never returns the PAN to the merchant. (7) **PCI audit evidence** — the merchant must demonstrate that PAN never enters their environment. Evidence includes: (a) Network diagrams showing PAN flows only to the gateway's domain. (b) Application logs showing no PAN in log files. (c) Source code review confirming no PAN assignment. (8) **Fallback** — if the merchant MUST handle PAN temporarily (e.g., manual phone orders), they must use a PCI-compliant virtual terminal that tokenizes immediately on entry, never storing the PAN.

## Q12: How do you optimize 3D Secure (SCA) friction for a European e-commerce platform where PSD2 requires strong authentication for most transactions, but you want to minimize checkout abandonment?

**Answer:** 3DS friction optimization is a balancing act between regulatory compliance and conversion rate: (1) **Transaction Risk Analysis (TRA) exemption** — under PSD2, transactions below EUR 30 are exempt from SCA. Use this aggressively: train a risk model to approve transactions up to EUR 100 if the risk score is low. The issuer can still challenge, but most will not for low-risk amounts. (2) **Trusted beneficiary exemption** — when a user successfully authenticates (3DS) and pays the same merchant, mark that merchant as trusted for that user. Future payments can skip SCA. The exemption lasts for the duration of the payment token (typically 12-24 months). (3) **3DS v2 — frictionless flow**: (a) The gateway sends a PayerAuthenticationRequest to the issuer's 3DS server. (b) The issuer evaluates the transaction using device fingerprinting (IP, user-agent, device ID, geolocation). (c) If the risk is low, the issuer returns authenticationValue = frictionless — no user interaction needed. (d) Goal: achieve 70%+ frictionless rate. (4) **Biometric challenge** — for transactions requiring challenge, use the 3DS v2 out-of-band method: push a notification to the user's banking app for fingerprint/FaceID approval. This is far faster than SMS OTP (3 seconds vs. 30 seconds). (5) **Adaptive 3DS** — only trigger 3DS when: (a) The transaction value exceeds the merchant's dynamic threshold (e.g., EUR 50 for new customers, EUR 500 for loyal). (b) The risk engine scores the transaction above 40 (medium risk). (c) The card is from a high-fraud region. (6) **Merchant-initiated transactions (MITs)** — recurring payments and subscriptions are exempt from SCA after the first payment (which requires SCA). Use CIT (Customer Initiated Transaction) for first payment, MIT (Merchant Initiated) for subsequent. (7) **3DS timeout handling** — if the bank's 3DS page does not load within 15 seconds, fall back to a simplified challenge (email OTP sent by the merchant, not the bank). This catches cases where the issuer's 3DS infrastructure is slow. (8) **Analytics** — track the 3DS funnel: (a) % of transactions where 3DS was triggered. (b) % frictionless. (c) % challenge success (user completed 2FA). (d) % challenge abandonment (user closed the browser). For every challenged transaction that fails, analyze why.

## Q13: Design the chargeback representment workflow. When a merchant receives a goods not received chargeback, how do you systematically gather evidence and submit a representment?

**Answer:** Representment is a structured evidence submission to the issuing bank: (1) **Chargeback receipt** — the payment system receives the chargeback notification (Visa VCR or ISO 8583 1931 message). The reason code is logged: Visa 11.3 = Goods Not Received. (2) **Evidence gathering engine** — immediately trigger an automated evidence collection workflow: (a) **Shipping evidence**: query the logistics system for tracking number, delivery confirmation (photo of package at doorstep), and delivery timestamp. If the carrier provides an API (e.g., FedEx Tracking API), pull the signed delivery receipt. (b) **Customer communication**: extract email and chat logs showing the customer confirming receipt from Zendesk/Intercom. (c) **Transaction evidence**: the authorization code, IP address of the customer at time of purchase, device fingerprint, and AVS result (address matched). (d) **Service evidence** (for digital goods): server logs showing the user's IP address downloading the file, the download timestamp, and the file version. (3) **Automated vs. manual representment**: (a) If the evidence is clear (tracking shows Delivered), auto-submit representment. (b) If the evidence is ambiguous (tracking shows Left at mailbox but customer claims not received), flag for manual review. (c) If no evidence exists (no tracking, no communication), advise the merchant to refund — it will almost certainly lose. (4) **Representment submission**: (a) Package evidence into a representment packet (PDF or structured XML per Visa's Compelling Evidence 3.0 standard). (b) Submit via the acquirer's dispute API or portal. (c) Include the Compelling Evidence 3.0 proof: at least two of (i) signed delivery, (ii) IP geo-match, (iii) previous successful transactions with same device, (iv) customer communication. (5) **Timeline management**: (a) Visa: 20 calendar days to submit representment. (b) Mastercard: 45 days. (c) The system maintains a dispute calendar. If the merchant has not responded by day 15, send an escalation email. After day 20 without action, the chargeback is automatically accepted. (6) **Second chargeback / pre-arbitration**: if the bank rejects the representment, the merchant can take it to pre-arbitration (Visa Dispute Resolution). This requires more evidence and a $500 fee. (7) **Win rate tracking** — track representment win rate per reason code, per acquirer, per shipping method.

## Q14: Design a multi-acquirer failover system. How do you route payments across multiple payment processors and fail over within seconds if the primary goes down?

**Answer:** Multi-acquirer architecture for high availability: (1) **Acquirer routing** — maintain 2-3 acquirer connections (e.g., Stripe as primary, Adyen as secondary, Worldpay as tertiary). Route traffic based on: (a) **Round-robin** for load balancing. (b) **Least-latency** — route to the acquirer with the fastest recent authorization time. (c) **Failover** — if primary returns more than 5% errors in the last 1 minute, route to secondary. (2) **Health monitoring** — every 10 seconds, send a health-check authorization (a $0.50 auth on a test card) to each acquirer. Track: success rate, latency (p50, p99), error codes. If health check fails 3 consecutive times, mark the acquirer as DEGRADED. (3) **Failover decision**: (a) Primary healthy — route 100% to primary. (b) Primary errors > 5% — route 50% to secondary. (c) Primary unavailable — route 100% to secondary. Send an alert to the ops team. (d) Secondary also unhealthy — route to tertiary. If all acquirers are down, return 503 Service Unavailable. (4) **Idempotency across acquirers** — each acquirer has a different idempotency key namespace. The payment service generates acquirer-specific keys. If a payment fails on Stripe, the retry on Adyen gets a different idempotency key = a new transaction attempt. (5) **Fallback settlement reconciliation** — when payments are split across acquirers, reconciliation becomes more complex. Each acquirer sends a separate settlement report. The reconciliation service merges them: join on (acquirer, acquirer_txn_id). (6) **Acquirer-specific features** — not all acquirers support all features. Stripe may support 3DS v2 but Worldpay may only support v1. The routing table includes capability metadata. Route traffic accordingly. (7) **Gradual failover** — do not flip 100% of traffic at once. Use a gradual shift: 10% to 25% to 50% to 100% over 2 minutes. Monitor error rates at each step. If the secondary also shows elevated errors, abort the failover. (8) **Local vs. regional failover** — if a specific data center goes down, fail over to a different region's acquirer connection. The payment service should be multi-region active-active for resilience.

## Q15: Design a payment method token vault for a platform storing 100M+ tokens. How do you handle token rotation, HSM-backed encryption, and key ceremony?

**Answer:** A token vault at scale requires HSM-grade security and careful key management: (1) **Vault schema** — (a) tokens (id UUID PRIMARY KEY, token_hash VARCHAR(64) UNIQUE, encrypted_pan BYTEA, last_four VARCHAR(4), expiry_month INT, expiry_year INT, card_brand VARCHAR(20), issuer_country VARCHAR(2), created_at TIMESTAMP, last_used_at TIMESTAMP, version INT). (b) The token_hash is SHA-256(token) for fast dedup lookup. (2) **HSM-backed encryption** — (a) Use a Hardware Security Module (AWS CloudHSM or Azure Dedicated HSM) to generate and store the master encryption key. (b) The PAN is encrypted with AES-256-GCM using a data key derived from the master key. (c) The data key is itself encrypted by the master key (envelope encryption). (d) Every decryption request goes through the HSM, which returns the decrypted PAN only to authorized services. (3) **Token rotation** — (a) If the master key is compromised, ALL tokens must be re-encrypted. Maintain key versions: current key version rotates every 6 months. (b) Background rotation job: read all tokens, decrypt with old key, re-encrypt with new key. (c) For 100M tokens at 1000/sec, this takes about 28 hours. Run during low-traffic window. (4) **Key ceremony** — (a) The master key is generated in a physical ceremony with at least 2 security officers present (split knowledge: each officer has half of the smart card PIN). (b) The key is backed up in 3 geographically separated HSMs with tamper-proof seals. (c) Key deletion requires a 2-of-3 multi-party approval and a 30-day cooling-off period. (5) **Token deduplication** — (a) When a token is created, check if the PAN already exists: compute SHA-256(PAN) and look up the token_hash column. (b) If found, return the existing token. (c) This saves storage for users who repeatedly use the same card. (6) **Token expiry** — (a) Multi-use tokens expire after 24 months of inactivity. (b) Expired tokens are soft-deleted (marked as expired) to retain audit trail. (c) Hard delete after 7 years (regulatory requirement in most jurisdictions). (7) **Access audit** — every vault access (encrypt, decrypt) is logged to an immutable audit log. (8) **Disaster recovery** — replicate the vault to a secondary region with async replication. If the primary HSM is destroyed, the secondary HSM can decrypt using the backup key. Test recovery quarterly.

## Q16: Design a real-time FX rate feed consolidation system. A global payment platform needs accurate, low-latency exchange rates from 5+ providers for 170 currencies.

**Answer:** FX rate consolidation ensures consistent, auditable rates across all transactions: (1) **Data sources** — connect to 5+ FX providers: (a) Bloomberg (B-Pipe API) — most authoritative, $2K/month. (b) Reuters (Refinitiv) — similar cost. (c) XE — lower cost, good for minor currencies. (d) Open Exchange Rates — API-based, free tier available. (e) Central bank rates (ECB, Fed) — free, updated daily. (2) **Rate ingestion** — (a) Major pairs (EUR/USD, USD/JPY, GBP/USD): poll every 1-5 seconds via WebSocket stream. (b) Minor pairs (USD/THB, EUR/ZAR): poll every 1 minute. (c) Exotic pairs (USD/IRR, EUR/VEF): poll every 1 hour. (3) **Rate quality scoring** — for each rate update, compute: (a) **Freshness**: time since last update. (b) **Bid-ask spread**: narrower = more liquid = more reliable. (c) **Provider reputation**: Bloomberg rates get weight 0.4, XE rates get 0.2. (d) **Deviation check**: if one provider's rate for EUR/USD is > 0.1% from the median, flag as outlier. (4) **Consolidated rate computation**: (a) After filtering outliers, compute the weighted median (not mean, to resist single rogue provider). (b) Apply the merchant's markup (e.g., 0.5% for standard, 1.5% for premium). (c) Store the rate in a Redis-backed rate cache with 10 second TTL. (d) The payment service reads from this cache for every transaction. (5) **Rate locking** — when a transaction is initiated, the rate is locked for 30-60 seconds (the time needed to complete the auth). If the auth takes longer, the merchant may re-quote the rate. (6) **Audit trail** — every transaction stores: fx_rate_used, fx_timestamp, provider_list, markup_applied. This is critical for: (a) Refunds (use the original rate). (b) Regulatory compliance (Europe requires disclosure of FX markup). (c) Dispute resolution. (7) **Failover** — if the primary rate provider is down, fall back to the secondary. If all paid providers are down, use the ECB's daily fixing rate (free, but only updated once daily). (8) **Hedging integration** — for merchants with large EUR/USD volume, the FX system exposes a rate stream that feeds into a hedging system (forward contracts, options). The merchant can lock in rates for future settlements.

## Q17: Optimize the dunning (retry) strategy for a subscription business with 10M paying users. How do you maximize recovery without annoying customers or triggering bank fraud blocks?

(b) **Card type**: credit cards have higher recovery than debit cards (debit has daily spend limits). (c) Randomize retry times within the hourly window. (5) **A/B testing**: run experiments to compare standard schedule vs. adaptive schedule. (6) **Grace period** — allow users to use the service for 3 days after the first failed payment. This maintains goodwill and gives them time to update their card without service interruption. (7) **Self-service dunning** — provide a billing portal where users can see card expiry warnings. Proactive card updates reduce dunning necessity by 30%. (8) **Recovery metrics** — track: (a) First retry recovery rate. (b) Cumulative recovery rate (after all retries). (c) Mean time to recovery. (d) Customer support tickets tagged payment issue. Benchmark: best-in-class dunning recovers 60-70% of failed payments.

## Q18: Design the subscription proration algorithm. When a user upgrades from a $10/month plan to a $50/month plan on day 15, how do you compute the charge?

**Answer:** Proration ensures fair billing for plan changes mid-cycle: (1) **Proration formula**: charge = (new_price - old_price) * (days_remaining / days_in_cycle). For a 30-day month, upgrade from $10 to $50 on day 15: ($50 - $10) * (15 / 30) = $40 * 0.5 = $20. The user pays $20 for the upgrade. (2) **Credit-based proration**: (a) User downgrades from $50 to $10 on day 15. (b) credit = ($50 - $10) * (15 / 30) = $20. (c) The $20 credit is applied to the next billing cycle. (d) Alternative: refund the credit immediately (more user-friendly but less revenue-retentive). (3) **Precise daily rate**: daily_rate = plan_price / days_in_billing_period. For annual plans: daily_rate = $600 / 365 = $1.6438. The charge for upgrade on day 100: (new_daily_rate - old_daily_rate) * remaining_days. (4) **Rounding**: (a) Round to the nearest cent using banker's rounding (round half to even) to avoid systematic bias. (b) Track rounding errors in a separate account: proration_rounding_account accumulates fractional cents. When the accumulated error exceeds $0.01, apply it to the next invoice. (5) **Plan change timing**: (a) **Immediate**: upgrade takes effect immediately. The user pays prorated amount now. (b) **End of billing period**: downgrade takes effect at the end of the current period. No proration charge now. (c) **Hybrid**: upgrades are immediate, downgrades are delayed. This is the most user-friendly approach. (6) **Multiple plan changes in one cycle**: (a) Track segments: segment 1 on $10 plan, segment 2 on $30 plan, segment 3 on $50 plan. (b) Compute charge per segment. (c) This can result in complex invoices. Simplify by recomputing from the beginning of the cycle using the average daily rate. (7) **Annual subscriptions**: prorate annual subscriptions differently: (a) Downgrade from annual to monthly: refund the remaining annual amount at the monthly rate, not the annual rate. (b) Example: $600 annual, user cancels after 100 days. Refund at monthly rate is more conservative and standard in the industry. (8) **Tax handling**: proration charges may trigger tax recalculation. The tax engine must recompute tax on the prorated portion.

## Q20: Design a reconciliation failure detection system. How do you detect, in real-time, that a payment was captured in your system but the gateway did not settle the funds?

**Answer:** Real-time reconciliation failure detection prevents unreconciled transactions from going unnoticed for days: (1) **Detection window** — settlement typically occurs T+1 (next business day). The reconciliation system compares yesterday's captured transactions against today's settlement report. (2) **Unreconciled transaction identification**: (a) Every captured payment has a state machine: AUTHORIZED to CAPTURED to SETTLED to RECONCILED. (b) A background job runs every hour: SELECT * FROM transactions WHERE status = CAPTURED AND captured_at < NOW() - INTERVAL 26 hours AND NOT EXISTS settlement_line. (c) Any transaction captured > 26 hours ago with no matching settlement line is flagged as UNRECONCILED. (3) **Real-time stream processing**: (a) Use Kafka Streams or Apache Flink to join the captured_events stream with the settlement_events stream. (b) Windowed join: for each captured event, wait 30 hours for a matching settlement event. If none arrives, emit an unreconciled_alert event. (4) **Alert severity**: (a) **Low**: unreconciled for 24-48 hours — log to dashboard. (b) **Medium**: unreconciled for 48-72 hours — send Slack notification. (c) **High**: unreconciled for > 72 hours — page on-call engineer and freeze merchant payouts. (5) **Auto-resolution attempts**: (a) Re-fetch the gateway's settlement report via API (maybe the report was delayed). (b) Check the gateway's transaction detail API: GET /v1/charges/{charge_id} — if the gateway shows status: succeeded, attempt manual capture again or create a support ticket. (c) If the gateway shows no record of the transaction, the payment may have been captured on the wrong gateway instance. Investigate and reverse the capture if needed. (6) **False positive reduction**: (a) If the captured timestamp is near the billing cycle boundary, adjust the window to 48 hours for weekend captures. (b) If the gateway settlement report excludes test transactions, filter those out. (c) If the merchant has a rolling reserve (settlement delayed by 30 days), exclude those from the 24-hour check. (7) **Downstream impact**: (a) If the transaction is unreconciled, the merchant cannot be paid out for that transaction. (b) The merchant's dashboard shows a Settlement Pending indicator. (8) **Historical reconciliation coverage**: run a daily batch job that reconciles ALL captured transactions with ALL settlement records. This catches: (a) Transactions that fell through the cracks in the real-time check. (b) Duplicate settlements (overpayment). (c) Chargebacks that were not previously reconciled. Coverage target: > 99.99% of transactions reconciled within 48 hours.

## Q22: How do you handle multi-currency settlement? When a merchant is in Japan, the customer is in Germany, and the payment platform is in the US, how do you convert, settle, and manage FX risk?

**Answer:** Multi-currency settlement involves three currencies: customer's payment currency, platform's settlement currency, and merchant's payout currency. (1) **Transaction flow**: (a) Customer in Germany pays €100. The payment gateway converts €100 → $108 (the platform's settlement currency) at the gateway's exchange rate + 1% markup. (b) The platform receives $108 (minus gateway fees). (c) Merchant in Japan expects ¥14,000. The platform converts $108 → ¥14,000 at the platform's exchange rate (interbank rate + 1% markup). (2) **FX risk windows**: (a) Risk window 1: between authorization (€100 → $108) and capture (settlement). If EUR/USD drops 2% in 3 days, the platform receives only $105.86 — but already promised the merchant ¥14,000. (b) Risk window 2: between capture and payout. If USD/JPY moves 3% in 7 days, the payout amount changes. (3) **Hedging strategies**: (a) Use the payment gateway's "dynamic currency conversion" — the gateway locks the exchange rate at authorization. The platform earns a DCC fee. (b) Use rolling FX forwards (1-week contracts) to lock in exchange rates for anticipated volume. (c) Natural hedging: if the platform has both inbound (payments) and outbound (payouts) in the same currency pair, they offset. (4) **Pricing strategy**: (a) Add a 1-2% FX markup to all cross-currency transactions. (b) This covers FX fluctuations and provides a profit margin. (c) For enterprise merchants with $10M+/month volume, offer interbank + 0.5% markup. (5) **Multi-currency ledger**: (a) Each merchant has a ledger per currency: `ledger (merchant_id, currency, balance)`. (b) At settlement time, the platform holds the funds in the merchant's settlement currency. (c) At payout time, convert to the merchant's payout currency at the day's rate.

## Q25: Design the 3D Secure (3DS) authentication system. When should you trigger 3DS for a transaction? How do you balance friction (user must complete a challenge) with liability shift?

**Answer:** 3D Secure adds an authentication step that shifts liability for chargebacks from the merchant to the issuing bank. (1) **Liability shift**: (a) Without 3DS: if a cardholder claims fraud, the merchant (or payment platform) is liable for the chargeback. (b) With 3DS: the issuing bank authenticated the cardholder, so the bank is liable. (2) **When to trigger 3DS**: (a) **High-risk transactions**: large amount (>€100), cross-border, new merchant, first transaction for this card, digital goods (high fraud rate). (b) **Regulatory requirement**: EU PSD2 mandates SCA (Strong Customer Authentication) for most payments. Exemptions: low-value (<€30), recurring, whitelisted merchants. (c) **Card scheme rules**: Visa may require 3DS for certain Merchant Category Codes (MCC). (3) **Friction-reduction strategies**: (a) **Exemption engine**: if the transaction qualifies for a PSD2 exemption (low value, trusted merchant), skip 3DS. This avoids the challenge dialog. (b) **Biometric authentication**: if the cardholder's bank supports biometric (fingerprint on the banking app), the challenge is a simple fingerprint scan instead of entering a password. (c) **3DS 2.0**: sends 150+ data points (device fingerprint, purchase history, IP geolocation) to the issuing bank for risk-based authentication. The bank can decide "no challenge needed" (frictionless) if the risk is low. (4) **Integration flow**: (a) Merchant submits payment with `3DS: true` flag. (b) Gateway initiates 3DS with the card network: creates a `3ds_authentication` record. (c) User is redirected to the issuing bank's challenge page (or an iframe). (d) User completes the challenge (password, OTP, biometric). (e) Bank returns `authentication_value` (CAVV/AAV). (f) Merchant submits the authorization with the CAVV. (g) If the CAVV is valid, the merchant gets liability shift. (5) **Optimization**: (a) Track challenge-to-conversion rate: if 3DS challenge drops conversion by 20%, optimize by reducing the trigger threshold. (b) A/B test: send 50% of medium-risk transactions through 3DS, 50% without. Compare chargeback rates. (c) Use 3DS only when the expected chargeback cost exceeds the conversion loss.

(b) The system immediately: (i) Updates the transaction status to `DISPUTED`. (ii) Creates a dispute record: `disputes (id, transaction_id, amount, reason_code, due_date, status)`. (iii) Notifies the merchant via email and dashboard. (2) **Reason codes**: (a) Visa reason codes: 10.4 (EMV liability shift), 11.3 (fraud), 13.1 (merchandise not received), 30 (services not provided). (b) Each reason code has a different evidence requirement and response time. Fraud disputes usually have a 20-day response window. Service disputes have 30 days. (3) **Evidence gathering**: (a) Automated evidence collection: (i) Customer information: IP address, device fingerprint, account creation date. (ii) Transaction details: amount, timestamp, currency, 3DS authentication result (CAVV). (iii) Delivery/service evidence: shipping tracking number, delivery confirmation, service fulfillment logs. (b) The merchant can supplement with additional evidence (email communication, refund policy shown at checkout). (4) **Submit to gateway**: (a) Package evidence according to the card network's spec (PDF, maximum 5MB). (b) Submit via the gateway's dispute response API. (c) Track status: `submitted`, `under_review`, `won`, `lost`. (5) **Outcome**: (a) Won: the dispute is reversed, funds returned to the merchant. (b) Lost: the funds are debited, plus a dispute fee ($15-25). (c) Pre-arbitration: if you lose but have new evidence, you can escalate to card network arbitration (at higher cost, $500-1000 fee). (6) **Merchant dashboard**: (a) Real-time dispute overview: open disputes, response deadlines (countdown timer), win/loss rate. (b) Analytics: top dispute reason codes, dispute rate % (target <0.5% of transactions). (c) Alerts: if dispute rate exceeds 1%, notify the merchant to review their business practices.

## Q30: Design the payment analytics and reporting system. How do you generate real-time and historical reports on transaction volume, approval rates, chargeback rates, and revenue for merchants and internal teams?

**Answer:** Payment analytics provides insights into business health and operational performance. (1) **Real-time metrics (dashboard)**: (a) TPS (transactions per second) — gauge chart. (b) Approval rate (authorized / attempted) — 95% target. (c) Error rate (gateway timeouts, 500s) — <0.5% target. (d) Total volume ($) today vs yesterday vs same day last week. (e) Active merchants (processed > 1 transaction in last hour). (2) **Data pipeline**: (a) Transaction events are published to Kafka: `TransactionEvent(transaction_id, status, amount, gateway, timestamp)`. (b) A streaming processor (Flink/Kafka Streams) aggregates into 1-minute windows and writes to ClickHouse. (c) ClickHouse powers the real-time dashboard (< 1 second query latency). (3) **Historical reports**: (a) Daily batch job aggregates into partitioned Parquet files on S3: `s3://analytics/date=2025-01-15/transactions.parquet`. (b) Reports generated via Athena/Trino: (i) Merchant settlement report: daily volume, fees, net payout. (ii) Gateway performance: approval rate by gateway, by card type, by MCC. (iii) Chargeback report: dispute rate by merchant, by reason code. (iv) FX report: transaction volume by currency, FX revenue earned. (4) **Merchant-facing reports**: (a) Merchants access their analytics via a dashboard: gross volume, refund rate, chargeback rate, and settlement schedule. (b) Reports are downloadable as CSV/PDF. (c) Data is scoped to the merchant only — no cross-merchant visibility. (5) **Anomaly detection**: (a) Automated alerts: if today's transaction volume deviates >20% from the 7-day rolling average, alert the operations team. (b) If approval rate drops below 90% for 5 consecutive minutes, trigger an investigation. (6) **Scheduled reports**: (a) Email daily summary to the merchant: "Yesterday you processed 152 transactions totaling $12,450. Approval rate: 94.2%." (b) Monthly business review for enterprise merchants: comprehensive PDF with trends, benchmarks, and recommendations.
