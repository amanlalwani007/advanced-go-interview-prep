# Chapter 18 — Design Google Maps

## Q1: How does Google Maps compute the shortest driving route between two points across a global road network with billions of nodes? Walk through the core algorithm and its optimizations for scale.

**Answer:** At its core, Google Maps uses a weighted shortest-path algorithm on a directed graph where intersections are nodes and road segments are edges weighted by travel time (not distance). Running A* or Dijkstra on the full global graph is infeasible. Key optimizations: (1) **Hierarchical routing** — roads are categorized into levels (local, arterial, highway, interstate). Route computation starts with A* on the local graph near the source and destination, then stitches to higher-level road graphs for the long-distance portion. This reduces the search space from billions to millions of nodes. (2) **Contraction Hierarchies (CH)** — a preprocessing step that "contracts" low-importance nodes by adding shortcut edges, creating a hierarchy. At query time, bidirectional Dijkstra on the CH graph runs in milliseconds. (3) **Landmark-based A*** — precompute distances from a set of landmarks to all nodes, using these as an admissible heuristic to prune the search. (4) **Real-time traffic** — travel times are dynamically adjusted based on live traffic data, which invalidates precomputed weights. Google recomputes routes using the current traffic-weighted graph, leveraging the fact that CH can be rebuilt incrementally.

## Q2: How does Google Maps handle real-time traffic data? Describe the data pipeline from raw GPS pings to congestion overlay on the map.

**Answer:** The pipeline: (1) **Data collection** — anonymous GPS pings from Android devices, Waze, and fleet tracking. Each ping is `{lat, lng, heading, speed, timestamp}`. (2) **Map matching** — raw GPS points are noisy. A Hidden Markov Model (HMM) snaps each ping to the most likely road segment, considering heading and speed. (3) **Speed aggregation** — for each road segment (identified by OpenStreetMap way ID or Google's internal ID), the system aggregates speeds from all matched pings within a 2-5 minute window. Outliers are removed. (4) **Traffic model** — the aggregated speed is compared to the historical baseline speed for that segment at this day-of-week and time-of-day. The ratio (current / baseline) determines the traffic level (green/yellow/red). (5) **Rendering** — the traffic overlay is generated by coloring road segments based on traffic level and served as vector tiles via the map rendering pipeline. The entire pipeline from ping to visible traffic is typically 2-5 minutes.

## Q3: Describe the map rendering pipeline — how does Google Maps serve detailed vector tiles for any zoom level across the entire planet while keeping download sizes under 100 KB per tile?

**Answer:** The world is divided into a quadtree of tiles. At zoom level 0, one tile covers the entire planet. At zoom level 20, each tile covers ~1 m². (1) **Pre-rendering** — map data (roads, buildings, terrain, labels) is rendered into vector tiles offline. Each tile contains geometry data (points, lines, polygons) with attributes, not pixels. (2) **Simplification** — the Douglas-Peucker algorithm reduces the number of points in a polyline while preserving shape. At lower zoom levels, roads are heavily simplified. (3) **Encoding** — geometry is encoded using protobuf or the Mapbox Vector Tile (MVT) format with delta + zigzag encoding for integers, compressing coordinates efficiently. (4) **Serving** — tiles are cached on CDNs. The client requests tiles for the current viewport: `https://maps.googleapis.com/maps/vt?pb=...&z=15&x=5234&y=12678`. (5) **Client rendering** — the browser renders the vector geometry using WebGL (for performance) or Canvas 2D. Vector tiles are smaller than raster tiles (typically 20-80 KB vs 200-500 KB) and scale cleanly to any device resolution.

## Q4: Design the geocoding service — converting "1600 Amphitheatre Parkway, Mountain View, CA" to (lat, lng). How does it handle ambiguous addresses and typos?

**Answer:** Geocoding uses a multi-stage pipeline: (1) **Parsing** — the input string is tokenized and classified into address components using a CRF (Conditional Random Field) model trained on labeled address data. (2) **Normalization** — abbreviations (St→Street, Ave→Avenue, CA→California) are expanded using a dictionary. (3) **Matching** — the parsed address is matched against the authoritative address database using a trie-based index on street names and an inverted index on locality + region. (4) **Ranking** — multiple candidate matches are ranked by a scoring function that considers: component match quality, distance from the center of the locality, and historical correction data. (5) **Fallback** — if no exact match, use fuzzy matching (edit distance < 3 on street name) and the closest matching ZIP code centroid. For typos, a spell-check model corrects common errors before matching. The entire lookup typically completes in <50 ms using an in-memory index sharded by country.

## Q5: How does Google Maps handle the "E-tag" or cache invalidation for map tiles when a new road opens or a building is constructed?

**Answer:** Each tile has a version identifier (timestamp or content hash). When a road is updated in the map database, the tile-rendering pipeline regenerates affected tiles with a new version. The CDN caches tiles using the URL as key. Invalidation strategies: (1) **URL-based versioning** — the tile request includes a version parameter: `?v=20240701`. When data changes, the version is bumped, and new requests get the new tile from the origin. (2) **Tiled cache invalidation** — when a road changes, the system can determine exactly which tiles contain that road (via spatial mapping) and purge those tiles from the CDN cache using the CDN's API. (3) **Stale-while-revalidate** — the client can display a cached tile while fetching the updated version in the background, ensuring the map never shows a blank region. For highly dynamic data (traffic), tiles have a TTL of 1-5 minutes. For static basemap data, tiles are cached for weeks and only invalidated on explicit map data updates.

## Q6: How does Google Maps compute Estimated Time of Arrival (ETA) accounting for traffic, traffic light patterns, turns, and historical patterns?

**Answer:** The ETA model is a machine learning system, not a simple distance/speed calculation. Features: (1) **Road segment travel time** — from real-time GPS pings, with adjustments for time-of-day and day-of-week. (2) **Intersection delay** — ML model trained on historical slowdown patterns at specific intersections. (3) **Traffic signal timing** — known traffic light cycles at major intersections (collected via fleet data). The model predicts whether you'll hit a red or green based on your arrival time at the intersection. (4) **Turn penalties** — left turns cost more than right turns; unprotected left turns cost more than protected ones. (5) **Road characteristics** — number of lanes, speed limit, road surface quality. The ML model (gradient-boosted trees or neural network) is trained on historical trip data: actual route taken, actual travel time, and the feature set. At query time, the model predicts travel time for each candidate route and selects the one with minimum ETA. Google uses an ensemble of models for different road types and traffic regimes.

## Q7: Design the route-diversion detection system. How does Google Maps know you've deviated from the suggested route and need a reroute?

**Answer:** The navigation client sends periodic location updates (every 5-15 seconds) to the routing service. The server checks if the snapped-to-road location still lies on the suggested route. If the off-route distance exceeds a threshold (e.g., 50 meters for 2 consecutive pings), the system triggers a reroute. To minimize unnecessary reroutes: (1) Use a hysteresis filter — don't reroute until the user has been off-route for at least 200 meters or 10 seconds, reducing reroutes due to GPS noise. (2) Use map matching — snap the GPS point to the nearest road before comparing. (3) Allow short off-route segments for gas stations or driveways. The reroute itself is a new route computation using the same algorithm but from the current location. To keep latency low (<500 ms), the routing engine precomputes alternative branches for the first few miles of the route (a "reroute tree"), so most diversions can be handled without a full recomputation.

## Q8: How does Google Maps handle indoor mapping (airports, malls, transit stations) including floor-level routing?

**Answer:** Indoor maps use a separate spatial layer overlaid on the base map. Each indoor venue has its own coordinate system (aligned to the building footprint) with floor-level metadata. (1) **Data model** — each floor is a separate layer. Points of interest (shops, gates, restrooms) are positioned with `{x, y, floor}` coordinates relative to the building anchor. (2) **Routing** — indoor routing uses a graph of walkable paths within each floor plus vertical transitions (stairs, elevators, escalators) between floors. The routing algorithm (A* with Manhattan distance heuristic) finds the shortest path across floors. (3) **Localization** — GPS is unreliable indoors. Google uses Wi-Fi fingerprinting, Bluetooth beacons, and cellular triangulation. The device scans visible Wi-Fi APs and compares their BSSIDs and signal strengths against a known database. (4) **Rendering** — indoor tiles are served at zoom levels 17-20 and only visible when the user zooms in sufficiently. The client determines which indoor venue (if any) the user is in based on location and displays the appropriate floor plan.

## Q9: How does Google Maps handle live transit data (bus/train delays) from thousands of transit agencies worldwide, each with different data formats?

**Answer:** Google uses the General Transit Feed Specification (GTFS) and GTFS-Realtime. (1) **Static data (GTFS)** — agencies publish static schedule data in a standard format (ZIP file with CSV tables: stops, routes, trips, stop_times). Google ingests these daily. (2) **Real-time data (GTFS-RT)** — agencies provide real-time updates (delay, cancellation, trip updates) via HTTP endpoints in Protocol Buffer format. Google's pipeline polls thousands of endpoints at configurable intervals (30 seconds to 5 minutes depending on agency). (3) **Transit routing** — the route planner treats transit as a time-expanded graph. Each transit trip is a series of departure/arrival events at specific times. A modified Dijkstra algorithm considers: walking to stop, waiting for transit, the transit ride, transfers. The algorithm accounts for real-time delays by adjusting timestamps in the graph. (4) **Fallback** — if an agency's real-time feed is down, Google falls back to the static schedule. For agencies without GTFS, Google provides tools and partnerships to help them convert.

## Q10: What data structures and storage systems back the reverse geocoding service (lat, lng → street address)?

**Answer:** Reverse geocoding uses a spatial index over the address database. The address database is stored as a set of polygons (building footprints, parcel boundaries, city boundaries) and linestrings (road centerlines). Storage: (1) **Spatial index** — a QuadTree or R-Tree on the address geometry, stored in Bigtable. (2) **Lookup** — query point (lat, lng) → traverse the spatial index to find the containing polygon or nearest road. (3) **Address resolution** — if the point falls within a building polygon, return that building's address. If on a road segment, interpolate the street number based on the point's distance along the segment and the known address ranges on that block. (4) **Fallback** — for points that don't match any known address (e.g., wilderness), return the nearest city or locality. (5) **Quality** — reverse geocoding has multiple granularities: street address, neighborhood, city, state, country. The system returns the finest-granularity match it can reliably determine. The entire lookup completes in <10 ms using an in-memory spatial index on the API server.
