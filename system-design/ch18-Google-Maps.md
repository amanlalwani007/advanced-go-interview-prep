# Chapter 18 — Design Google Maps

## Q1: How does Google Maps compute the shortest driving route between two points across a global road network with billions of nodes? Walk through the core algorithm and its optimizations for scale.

**Answer:** At its core, Google Maps uses a weighted shortest-path algorithm on a directed graph where intersections are nodes and road segments are edges weighted by travel time (not distance). Running A* or Dijkstra on the full global graph is infeasible. Key optimizations: (1) **Hierarchical routing** — roads are categorized into levels (local, arterial, highway, interstate). Route computation starts with A* on the local graph near the source and destination, then stitches to higher-level road graphs for the long-distance portion. This reduces the search space from billions to millions of nodes. (2) **Contraction Hierarchies (CH)** — a preprocessing step that "contracts" low-importance nodes by adding shortcut edges, creating a hierarchy. At query time, bidirectional Dijkstra on the CH graph runs in milliseconds. (3) **Landmark-based A*** — precompute distances from a set of landmarks to all nodes, using these as an admissible heuristic to prune the search. (4) **Real-time traffic** — travel times are dynamically adjusted based on live traffic data, which invalidates precomputed weights. Google recomputes routes using the current traffic-weighted graph, leveraging the fact that CH can be rebuilt incrementally.

## Q2: How does Google Maps handle real-time traffic data? Describe the data pipeline from raw GPS pings to congestion overlay on the map.

**Answer:** The pipeline: (1) **Data collection** — anonymous GPS pings from Android devices, Waze, and fleet tracking. Each ping is `{lat, lng, heading, speed, timestamp}`. (2) **Map matching** — raw GPS points are noisy. A Hidden Markov Model (HMM) snaps each ping to the most likely road segment, considering heading and speed. (3) **Speed aggregation** — for each road segment (identified by OpenStreetMap way ID or Google's internal ID), the system aggregates speeds from all matched pings within a 2-5 minute window. Outliers are removed. (4) **Traffic model** — the aggregated speed is compared to the historical baseline speed for that segment at this day-of-week and time-of-day. The ratio (current / baseline) determines the traffic level (green/yellow/red). (5) **Rendering** — the traffic overlay is generated by coloring road segments based on traffic level and served as vector tiles via the map rendering pipeline. The entire pipeline from ping to visible traffic is typically 2-5 minutes.

## Q3: Describe the map rendering pipeline — how does Google Maps serve detailed vector tiles for any zoom level across the entire planet while keeping download sizes under 100 KB per tile?

**Answer:** The world is divided into a quadtree of tiles. At zoom level 0, one tile covers the entire planet. At zoom level 20, each tile covers ~1 m². (1) **Pre-rendering** — map data (roads, buildings, terrain, labels) is rendered into vector tiles offline. Each tile contains geometry data (points, lines, polygons) with attributes, not pixels. (2) **Simplification** — the Douglas-Peucker algorithm reduces the number of points in a polyline while preserving shape. At lower zoom levels, roads are heavily simplified. (3) **Encoding** — geometry is encoded using protobuf or the Mapbox Vector Tile (MVT) format with delta + zigzag encoding for integers, compressing coordinates efficiently. (4) **Serving** — tiles are cached on CDNs. The client requests tiles for the current viewport: `https://maps.googleapis.com/maps/vt?pb=...&z=15&x=5234&y=12678`. (5) **Client rendering** — the browser renders the vector geometry using WebGL (for performance) or Canvas 2D. Vector tiles are smaller than raster tiles (typically 20-80 KB vs 200-500 KB) and scale cleanly to any device resolution.

## Q4: Design the geocoding service — converting "1600 Amphitheatre Parkway, Mountain View, CA" to (lat, lng). How does it handle ambiguous addresses and typos?

**Answer:** Geocoding uses a multi-stage pipeline: (1) **Parsing** — the input string is tokenized and classified into address components using a CRF (Conditional Random Field) model trained on labeled address data. (2) **Normalization** — abbreviations (St→Street, Ave→Avenue, CA→California) are expanded using a dictionary. (3) **Matching** — the parsed address is matched against the authoritative address database using a trie-based index on street names and an inverted index on locality + region. (4) **Ranking** — multiple candidate matches are ranked by a scoring function that considers: component match quality, distance from the center of the locality, and historical correction data. (5) **Fallback** — if no exact match, use fuzzy matching (edit distance < 3 on street name) and the closest matching ZIP code centroid. For typos, a spell-check model corrects common errors before matching. The entire lookup typically completes in <50 ms using an in-memory index sharded by country.

## Q5: How does Google Maps handle the "E-tag" or cache invalidation for map tiles when a new road opens or a building is constructed?

**Answer:** Each tile has a version identifier (timestamp or content hash). When a road is updated in the map database, the tile-rendering pipeline regenerates affected tiles with a new version. The CDN caches tiles using the URL as key. Invalidation strategies: (1) **URL-based versioning** — the tile request includes a version parameter: `?v=20240701`. When data changes, the version is bumped, and new requests get the new tile from the origin. (2) **Tiled cache invalidation** — when a road changes, the system can determine exactly which tiles contain that road (via spatial mapping) and purge those tiles from the CDN cache using the CDN's API. (3) **Stale-while-revalidate** — the client can display a cached tile while fetching the updated version in the background, ensuring the map never shows a blank region. For highly dynamic data (traffic), tiles have a TTL of 1-5 minutes. For static basemap data, tiles are cached for weeks and only invalidated on explicit map data updates.

## Q6: How does Google Maps compute Estimated Time of Arrival (ETA) accounting for traffic, traffic light patterns, turns, and historical patterns?

**Answer:** The ETA model is a machine learning system, not a simple distance/speed calculation. Features: (1) **Road segment travel time** — from real-time GPS pings, with adjustments for time-of-day and day-of-week. (2) **Intersection delay** — ML model trained on historical slowdown patterns at specific intersections. (3) **Traffic signal timing** — known traffic light cycles at major intersections (collected via fleet data). The model predicts whether you'll hit a red or green based on your arrival time at the intersection. (4) **Turn penalties** — left turns cost more than right turns; unprotected left turns cost more than protected ones. (5) **Road characteristics** — number of lanes, speed limit, road surface quality. The ML model (gradient-boosted trees or neural network) is trained on historical trip data: actual route taken, actual travel time, and the feature set. At query time, the model predicts travel time for each candidate route and selects the one with minimum ETA. Google uses an ensemble of models for different road types and traffic regimes.

## Q7: Design the route-diversion detection system. How does Google Maps know you've deviated from the suggested route and need a reroute?

**Answer:** The navigation client sends periodic location updates (every 5-15 seconds) to the routing service. The server checks if the snapped-to-road location still lies on the suggested route. If the off-route distance exceeds a threshold (e.g., 50 meters for 2 consecutive pings), the system triggers a reroute. To minimize unnecessary reroutes: (1) Use a hysteresis filter — don't reroute until the user has been off-route for at least 200 meters or 10 seconds, reducing reroutes due to GPS noise. (2) Use map matching — snap the GPS point to the nearest road before comparing. (3) Allow short off-route segments for gas stations or driveways. The reroute itself is a new route computation using the same algorithm but from the current location. To keep latency low (<500 ms), the routing engine precomputes alternative branches for the first few miles of the route (a "reroute tree"), so most diversions can be handled without a full recomputation.

## Q8: How does Google Maps handle indoor mapping (airports, malls, transit stations) including floor-level routing?

**Answer:** Indoor maps use a separate spatial layer overlaid on the base map. Each indoor venue has its own coordinate system (aligned to the building footprint) with floor-level metadata. (1) **Data model** — each floor is a separate layer. Points of interest (shops, gates, restrooms) are positioned with `{x, y, floor}` coordinates relative to the building anchor. (2) **Routing** — indoor routing uses a graph of walkable paths within each floor plus vertical transitions (stairs, elevators, escalators) between floors. The routing algorithm (A* with Manhattan distance heuristic) finds the shortest path across floors. (3) **Localization** — GPS is unreliable indoors. Google uses Wi-Fi fingerprinting, Bluetooth beacons, and cellular triangulation. The device scans visible Wi-Fi APs and compares their BSSIDs and signal strengths against a known database. (4) **Rendering** — indoor tiles are served at zoom levels 17-20 and only visible when the user zooms in sufficiently. The client determines which indoor venue (if any) the user is in based on location and displays the appropriate floor plan.

## Q9: How does Google Maps handle live transit data (bus/train delays) from thousands of transit agencies worldwide, each with different data formats?

**Answer:** Google uses the General Transit Feed Specification (GTFS) and GTFS-Realtime. (1) **Static data (GTFS)** — agencies publish static schedule data in a standard format (ZIP file with CSV tables: stops, routes, trips, stop_times). Google ingests these daily. (2) **Real-time data (GTFS-RT)** — agencies provide real-time updates (delay, cancellation, trip updates) via HTTP endpoints in Protocol Buffer format. Google's pipeline polls thousands of endpoints at configurable intervals (30 seconds to 5 minutes depending on agency). (3) **Transit routing** — the route planner treats transit as a time-expanded graph. Each transit trip is a series of departure/arrival events at specific times. A modified Dijkstra algorithm considers: walking to stop, waiting for transit, the transit ride, transfers. The algorithm accounts for real-time delays by adjusting timestamps in the graph. (4) **Fallback** — if an agency's real-time feed is down, Google falls back to the static schedule. For agencies without GTFS, Google provides tools and partnerships to help them convert.

## Q10: What data structures and storage systems back the reverse geocoding service (lat, lng → street address)?

**Answer:** Reverse geocoding uses a spatial index over the address database. The address database is stored as a set of polygons (building footprints, parcel boundaries, city boundaries) and linestrings (road centerlines). Storage: (1) **Spatial index** — a QuadTree or R-Tree on the address geometry, stored in Bigtable. (2) **Lookup** — query point (lat, lng) → traverse the spatial index to find the containing polygon or nearest road. (3) **Address resolution** — if the point falls within a building polygon, return that building's address. If on a road segment, interpolate the street number based on the point's distance along the segment and the known address ranges on that block. (4) **Fallback** — for points that don't match any known address (e.g., wilderness), return the nearest city or locality. (5) **Quality** — reverse geocoding has multiple granularities: street address, neighborhood, city, state, country. The system returns the finest-granularity match it can reliably determine. The entire lookup completes in <10 ms using an in-memory spatial index on the API server.

## Q11: How does Google Maps extract building footprints from satellite imagery at global scale, and how do you handle the ambiguity of irregular structures, shadows, and occlusions?

**Answer:** Building footprint extraction uses a semantic segmentation pipeline on high-resolution satellite imagery (30-50 cm/pixel). The pipeline: (1) **Data preprocessing** — raw satellite tiles are orthorectified (corrected for terrain and sensor distortion) and split into 512×512 or 1024×1024 pixel chips. Multiple spectral bands (RGB, near-infrared, panchromatic) are stacked as input channels. (2) **Model architecture** — a U-Net or Mask R-CNN with a ResNet-50/101 encoder backbone, trained on labeled building footprints (OpenStreetMap, government cadastral data). Google uses a custom variant called "SpaceNet" architecture that adds an additional "direction" head to predict roof orientation for better edge delineation. (3) **Handling ambiguity** — shadows are addressed by using multi-date imagery (buildings cast shadows in different directions at different times of day) and training the model on labeled shadow examples with a "shadow-aware" loss function. Occlusions (trees overhanging buildings) are handled with a "completeness" head that predicts the probability a building continues under the occlusion — the model inpaints the occluded region using a GAN-based completion module. (4) **Post-processing** — raw model outputs (probability masks) are converted to vector polygons using marching squares, then simplified with the Douglas-Peucker algorithm. Adjacent buildings that the model predicted as separate but share a wall (row houses) are merged using a wall-sharing heuristic based on edge alignment. (5) **Quality assurance** — extracted footprints are cross-validated against authoritative datasets (where available) and against human-validated samples. Buildings below a confidence threshold (e.g., 0.75 IoU) are flagged for manual review. The full pipeline processes 10+ million km² of imagery annually and achieves ~85% mean IoU on urban buildings and ~70% on rural buildings. The key engineering challenge: training data scarcity in developing countries — Google addresses this with semi-supervised learning, using a teacher model trained on US/European data to generate pseudo-labels for other regions, then fine-tuning with a small set of locally-labeled examples.

## Q12: Design an address validation and standardization pipeline that takes a raw, unstructured address string and returns a canonical, deliverable address with confidence score. How do you handle addresses from 200+ countries with wildly different formats?

**Answer:** Address validation is a multi-model pipeline that must account for 200+ national addressing schemes. The pipeline: (1) **Country detection** — use a lightweight NER model or regex-based classifier trained on country names, postal code patterns, and common locality names. This determines which address parser to invoke. (2) **Country-specific parsing** — each country has one of four address families: Western (number + street + city + postcode: US/UK/DE), Asian (prefecture + city + district + block: JP/KR/TW), Latin American (street + number + neighborhood + city + state: BR/MX/AR), and "descriptive" (landmark-based: many African and rural Asian addresses). For descriptive addresses (e.g., "near the red mosque, behind the petrol station"), the parser uses a geocoding pass — a text-to-coordinates ML model that identifies landmarks and spatial relationships, then snaps to the nearest known point. (3) **Component normalization** — each component (street name, locality, administrative area) is spell-checked against a reference dictionary using a weighted edit distance that accounts for common transliteration errors (e.g., "Mumbai" vs "Bombay"). Abbreviations are expanded (St → Street, Rd → Road, Apt → Apartment). (4) **Postal code validation** — postal codes are validated against a per-country format regex and cross-referenced against an authoritative postal code database. Mismatches between the city and postal code are flagged with a confidence penalty. (5) **Delivery point verification** — the validated address is geocoded to coordinates, and the coordinates are checked against the building footprint database. If the geocoded point falls outside the expected building polygon for that address, the confidence score drops. (6) **Confidence scoring** — the system outputs a score from 0-100 based on: exact match in reference database (100), all components validated (80-99), partial match with one fuzzy component (50-79), city+postcode only (20-49), no match (0-19). Addresses below a configurable threshold (e.g., 70) are flagged for manual verification. The pipeline handles 10,000+ addresses per second per instance and is the backbone of Google Maps Platform's Address Validation API. The hardest part: "descriptive" addresses common in Africa, rural India, and parts of Southeast Asia — Google has invested heavily in Google+, a plus-code-based addressing system that assigns a short alphanumeric code to any location, which acts as a structured fallback for unstructured addresses.

## Q13: Describe the turn-by-turn navigation voice generation pipeline. How does Google Maps decide when to announce a turn, and how does it generate natural-sounding instructions like "In 200 meters, at the traffic circle, take the second exit onto Broadway"?

**Answer:** The voice generation pipeline has three stages: instruction planning, language generation, and speech synthesis. (1) **Instruction planning** — the routing engine outputs a sequence of maneuvers (turn left, merge, exit roundabout). A planning module decides when to announce each maneuver based on distance thresholds that vary by road type: highway exits at 2 miles and 1 mile; city turns at 400m and 100m; roundabout exits at 200m. The planner also de-duplicates — if a left turn is followed by another left turn within 200m, it merges them into "take two lefts." (2) **Landmark augmentation** — the system has a database of landmarks (stores, gas stations, parks) geocoded with their relation to road segments. The instruction generator augments standard maneuvers with landmarks: "Turn left at the Shell station" instead of "Turn left on Oak Street." This is driven by a salience score — a landmark is mentioned only if it's visible from the road and relevant (not every passing store). (3) **Language generation** — the system uses a template-based NLG engine with context-dependent templates. Templates include distance qualifiers ("in 200 meters"), road name ("onto Broadway"), direction modifiers ("slight left", "sharp right"), and spatial prepositions ("through the roundabout"). The NLG has rules for junction types: roundabouts use ordinal ("second exit"), forked roads use direction ("keep right"), and multi-lane roads use lane guidance ("use the left two lanes"). (4) **TTS synthesis** — Google uses WaveNet-based neural TTS (now moved to a more efficient Tacotron 2 + WaveRNN hybrid). The TTS system takes the text instruction and a prosody markup — emphasis on the direction word ("TURN left"), pause before the road name, and higher pitch for urgency. The prosody is generated by a separate model that learns from recorded human navigators. (5) **Latency optimization** — the entire pipeline from route calculation to spoken instruction must complete in <200ms. Voice instructions are pre-generated for the entire route at navigation start (not on-the-fly), then cached on the client. Only reroute instructions are generated in real-time, and even those use a pre-warmed TTS cache for common phrases ("Recalculating..."). The pipeline generates instructions in 50+ languages, each with locale-specific templates. For languages with complex morphology (German compound words, Arabic right-to-left with embedded numbers), the NLG uses a morphology-aware generator that handles noun declension and verb conjugation.

## Q14: Design the isochrone map service — given a point and a travel time (e.g., "30 minutes by car"), return the area reachable within that time. What algorithms and data structures back this, and how do you compute it at query time without precomputing for every possible point?

**Answer:** Isochrone computation is a time-constrained shortest-path frontier expansion. The core algorithm is a variant of Dijkstra that runs backward from the origin point, expanding edges until the cumulative travel time exceeds the threshold. The result is a set of polygon vertices — not just a radius circle, because road networks are irregular. Algorithm: (1) **Contraction Hierarchies for time** — precompute a CH graph where edge weights are travel times (not distance). At query time, run bidirectional Dijkstra on the CH graph but with a twist: instead of searching for a single destination, expand all edges until the time budget is exhausted. This produces a set of reachable nodes. (2) **Polygon construction** — the reachable nodes form a "frontier." Interpolate along the edges connected to frontier nodes at the exact time-budget boundary (e.g., if a 60-second edge is partially reachable within a 30-second budget, only include the fraction of the edge that's within budget). Connect these interpolation points to form a closed polygon. This is a computational geometry problem — use a concave hull algorithm (alpha shapes) rather than convex hull because isochrones are typically star-shaped but not convex (rivers, park boundaries create indentations). (3) **Multi-modal expansion** — for "30 minutes by transit," the graph is a time-expanded graph with walking edges (to/from stops), waiting edges (at stops), and transit edges (on vehicles). The frontier expands through multiple modes, weighted by their respective speeds. The Mueller algorithm handles multi-modal isochrones by dividing the time budget into three phases: access (walk to transit), transit ride, and egress (walk to destination). (4) **Caching** — precompute isochrones for high-demand locations (city centers, transit hubs) on a schedule. For arbitrary query points, use a cache of precomputed isochrones and interpolate between them. If the query point is within 100m of a cached point, return the cached isochrone (it's unlikely to differ significantly). (5) **Resolution control** — the number of polygon vertices is adaptive: at a 5-minute threshold, high resolution (100+ vertices to capture street-level detail); at a 60-minute threshold, low resolution (30 vertices, smoothed). This keeps both computation and rendering fast. The system must handle 1000+ isochrone queries per second — a single query with a 60-minute car budget takes ~50ms on the CH-enhanced graph with optimized C++ routing engine. The output is a GeoJSON polygon that's displayed as a semi-transparent overlay on the map. For "depart at a specific time" isochrones (factoring in traffic), the time-dependent CH graph is used with time-dependent edge weights, but this increases computation time by ~3×.

## Q15: How does Google Maps use machine learning to predict real-time ETAs, and how do you handle the cold-start problem for new roads or areas with sparse historical data?

**Answer:** Real-time ETA prediction uses a gradient-boosted decision tree (GBDT) or a lightweight neural network that ingests features from three categories: (1) **Static features** — road class (highway vs local), speed limit, number of lanes, intersection density, historical baseline travel time (50th percentile for this road segment at this day-hour). (2) **Dynamic features** — current real-time speed on the segment (from GPS pings), trend (accelerating/decelerating), adjacent segment speeds, event data (construction, accidents, road closures from Waze/Google crowdsourcing). (3) **Contextual features** — weather (rain/snow reduces speed by 5-15%), time of day (school zones during drop-off/pick-up), holiday calendar, sporting event schedules (game days at stadiums). The model outputs a predicted travel time for each road segment, and the total ETA is the sum across all segments on the route. For cold-start segments (new roads, or roads with <10 recent pings), the system falls back to a hierarchy: (1) If no real-time data exists, use the historical baseline for that segment at this day-hour. (2) If the segment has no history (<1 month of data), use the road-class average speed (e.g., "US highways average 65 mph at this hour"). (3) If the entire region is cold-start (e.g., a new subdivision), use the speed limit adjusted by a road-class-specific congestion factor (e.g., local roads are typically at 60% of the speed limit during peak hours). The cold-start heuristic is gradually replaced by real data as pings accumulate — a Bayesian update approach: start with prior = speed-limit-based estimate, then update the posterior with each observed ping. Google also uses "transfer learning" — a model trained on well-covered regions (US/Europe) is fine-tuned for sparsely-covered regions by adjusting only the road-class coefficients while freezing the intersection-delay and traffic-signal layers. This reduces the cold-start period from months to weeks. The model is retrained weekly on global trip data and deployed via a model-serving infrastructure that can swap model versions without downtime. ETA predictions are evaluated against actual trip completion times — Google reports a P50 error of <2 minutes and a P90 error of <8 minutes for typical urban trips.

## Q16: How do you optimize map tile rendering for low-end Android devices with 1 GB RAM and limited GPU capabilities, while still providing a smooth pan/zoom experience?

**Answer:** Rendering on low-end devices requires a tiered rendering strategy: (1) **Tile resolution scaling** — serve downsampled tiles to low-end devices. The server detects device capability via the `User-Agent` header or a client-reported capability flag (RAM, GPU model, screen DPI). For low-end devices, serve tiles at 1× resolution (instead of 2× for Retina/high-DPI) and at a lower zoom level offset (e.g., request zoom level 14 tiles instead of 15, then let the client nearest-neighbor upscale). This reduces tile size from ~50 KB to ~15 KB. (2) **Geometry simplification** — the vector tile pipeline has a "simplicity profile" for low-end devices: increase the Douglas-Peucker simplification tolerance by 2×, omit building footprints beyond zoom 16 (only show at zoom 17+), reduce the maximum label count per tile from 200 to 50. (3) **Texture atlas management** — the WebGL renderer on low-end devices is constrained by the maximum texture size (often 2048×2048, not 4096×4096). The tile cache on the GPU must be aggressively managed: use a partial LRU eviction scheme that keeps only tiles within the current viewport + 1 tile margin, discarding pre-fetched tiles for the next zoom level. (4) **Canvas fallback** — if WebGL is not available or fails to initialize (common on older devices), fall back to Canvas 2D rendering. Canvas 2D is slower for complex geometry but handles basic tile rendering acceptably. The client detects WebGL availability at startup and chooses the renderer. (5) **Frame budget** — the rendering loop has a 16ms frame budget (60fps). If a frame exceeds this (e.g., during map rotation at zoom 18 with building footprints), the renderer drops to 30fps by skipping every other frame. The animation system uses `requestAnimationFrame` with a frame budget tracker — if three consecutive frames exceed 16ms, the device's rendering profile is downgraded for the remainder of the session. (6) **Memory budget** — on 1 GB RAM devices, the map library is limited to 200 MB of heap. The client reduces the precaching radius: instead of pre-loading tiles for 2 screens in each direction, pre-load only 0.5 screens. Tile textures are stored as compressed WebP (lossy, quality 70) instead of PNG. The tile cache on disk is limited to 50 MB (vs 200 MB on high-end devices). Google Maps' "Lite Mode" (introduced for Android Go) implements all of these optimizations and reduces the Maps SDK size from 25 MB to 8 MB while maintaining acceptable rendering quality.

## Q17: Design an indoor positioning system using Wi-Fi Round-Trip Time (RTT) (IEEE 802.11mc). How do you achieve sub-meter accuracy in a crowded mall, and how do you handle the fingerprint database maintenance problem?

**Answer:** Wi-Fi RTT measures the time-of-flight between the device and an AP, giving a distance estimate (resolution ~1-2 meters). Multi-lateration from 3+ APs gives a 2D position. The system: (1) **Ranging** — the device sends an RTT burst to each visible Wi-Fi AP that supports 802.11mc (now common on Wi-Fi 6/6E APs). The round-trip time (typically 100-500 ns) times the speed of light gives distance. Each range measurement has an error of ±2m due to multipath. The system takes 20+ measurements per AP and uses a median filter to reject outliers. (2) **Multi-lateration** — with 3+ AP distances, solve for the device position using weighted nonlinear least squares. The weights are inversely proportional to the reported RTT variance. If only 2 APs are visible, use the intersection of two circles (two possible positions) and resolve ambiguity with the previous known position (assume device didn't teleport). (3) **Fingerprint augmentation** — RTT alone gives 2-5m accuracy; sub-meter requires combining RTT with a fingerprint of all visible APs (BSSID + RSSI). Build a Gaussian Process regression model that learns the spatial error pattern of RTT measurements in a specific venue. The model is trained on calibration walks where ground-truth position is known (from a floorplan-aligned path). (4) **Database maintenance problem** — Wi-Fi APs are moved, replaced, or fail. The fingerprint database must be kept fresh. Solution: **crowdsourced SLAM (Simultaneous Localization and Mapping)** — as users walk through the venue, their phones collect RTT measurements and accelerometer/gyroscope data (pedestrian dead reckoning, PDR). A particle filter jointly estimates the user's trajectory and updates the AP positions. When Google's indoor positioning service detects that an AP's estimated position has drifted by more than 5 meters from its database entry, it flags the AP for recalibration. If the drift persists for 30+ days across multiple user trajectories, the database is updated automatically. (5) **Zone of confidence** — the system outputs a position with a confidence ellipse (not a single point). The ellipse is computed from the covariance matrix of the multi-lateration solution. If the ellipse's major axis exceeds 10m, the system reports "within zone" (e.g., "you are in the food court") rather than a specific coordinate. This is critical for usability — showing a dot jumping around a 10m area is worse than showing a zone label. Google Maps uses this for mall maps, airport gate finding, and transit station navigation. The system supports 10,000+ venues globally, and the fingerprint database for each venue is maintained through a combination of Google's Street View Wi-Fi collection vans (which map APs as they drive by on nearby streets) and crowdsourced updates from Google Maps users who opt in to "Wi-Fi scanning" for improved location accuracy.

## Q18: How does Google Maps predict traffic conditions using Graph Neural Networks (GNNs) on the road network graph, and how does this approach improve over traditional time-series methods?

**Answer:** Traditional traffic prediction uses per-segment time-series models (ARIMA, LSTM) that treat each road segment independently. GNNs model the road network as a graph where nodes are road segments and edges capture spatial adjacency (intersections) and temporal dependencies. Google's approach (based on published papers like "Traffic Prediction with Graph Neural Networks" and "GSP") works as follows: (1) **Graph construction** — each road segment is a node. Edges are created for: physical connectivity (segments that meet at an intersection, directed by allowed turns), spatial proximity (segments within 200m that don't physically connect but influence each other — e.g., parallel streets where congestion on one diverts traffic to the other), and functional similarity (all segments of the same road class in a region). (2) **Node features** — each node's input is a time-series of speed readings over the last 60 minutes, at 5-minute intervals (12 time steps). Additional static features: road class, number of lanes, speed limit, length. (3) **GNN architecture** — a Spatio-Temporal Graph Convolutional Network (ST-GCN) with: a graph convolution layer that aggregates neighbor features via a learned adjacency matrix (the model learns which neighbors are most predictive, not relying solely on physical connectivity), a temporal convolution layer (1D dilated CNN) that captures patterns across the 12 time steps, and a gated fusion mechanism that combines spatial and temporal representations. The output is a prediction of the speed on each segment for the next 15, 30, and 60 minutes. (4) **Training** — the model is trained on a year of historical GPS ping data (speed per segment per 5-minute interval) using a mean absolute error (MAE) loss. The training graph covers an entire metropolitan area (e.g., 100K nodes for the LA metro area). Training is done on TPU pods with graph partitioning across devices. (5) **Improvement over time-series** — GNNs capture spillover effects: if a highway exit is blocked, the GNN predicts congestion on the parallel surface street because the graph connects them. Traditional LSTMs can't model this without explicit feature engineering. In Google's published benchmarks, GNNs reduce prediction MAE by 15-25% compared to per-segment LSTMs, especially during incident events (accidents, road closures, sporting events). The tradeoff: GNN inference is 5-10× slower than a simple LSTM per segment (due to the graph convolution), but still well within the 5-minute update cycle. Google also uses a "simulation-informed" GNN that takes output from a traditional traffic simulation (like Google's own simulation engine) as an additional input feature, creating a hybrid ML-simulation system that combines the pattern recognition of ML with the physical constraints of simulation.

## Q19: Design the API rate limiting and quota management system for Google Maps Platform. How do you enforce per-user, per-method, per-minute, and per-day quotas across thousands of API methods for millions of developers, while minimizing latency overhead?

**Answer:** The quota system uses a distributed token bucket architecture with local caching for low latency. Architecture: (1) **Token buckets per dimension** — each API key has multiple token buckets: per-minute (e.g., 300 requests/min for Routes API), per-day (e.g., 10,000 requests/day for Geocoding API), per-user-per-minute (e.g., 100 requests/min from a single IP for a single API key), and per-method-per-key (e.g., 50 requests/min for Place Details with field masks). (2) **Distributed refill** — tokens are stored as Lua scripts in a Redis cluster sharded by API key. The token bucket state is `{tokens: N, last_refill: timestamp}`. On each request, the API gateway executes a Redis `EVAL` script that: checks if `last_refill + refill_interval` has passed (refill tokens if so), decrements tokens, returns `remaining_tokens` and `reset_timestamp`. The Lua script is atomic and completes in <1ms on a local Redis node. (3) **Local cache hot path** — to avoid Redis roundtrip on every request, each API gateway instance caches token bucket state locally for 1 second. A request first checks the local cache; if the cached `remaining_tokens > 0`, it serves the request without hitting Redis. If the local cache is empty or shows exhausted, it falls through to Redis. The local cache TTL of 1 second means a burst of up to (gateway instances × 1 second of traffic) can exceed the quota before being caught — this is acceptable because the per-minute quota is a rolling average, not a hard wall. (4) **Quota enforcement response** — when quota is exceeded, the API returns HTTP 429 with headers: `X-RateLimit-Limit` (total quota), `X-RateLimit-Remaining` (0), `X-RateLimit-Reset` (Unix timestamp when the bucket refills) and a `Retry-After` header in seconds. The response body includes a `QuotaExceeded` error with details about which quota dimension was hit. (5) **Hierarchical quota** — Google Maps Platform has a three-tier quota system: project-level (all keys for a GCP project share a global limit), key-level (per API key), and IP-level (per requesting IP). The project-level limit is enforced at a separate Redis cluster (coarser synchronization, 10-second cache TTL). This prevents a developer with 100 API keys from bypassing the project limit. (6) **Burstable quota** — most APIs have a "burst" allowance: 2× the per-minute rate for up to 60 seconds, drawing from a separate burst bucket that refills slowly. This handles short-term spikes (e.g., a fleet of delivery vehicles all using Geocoding at the top of the hour). (7) **Monitoring and alerting** — each quota enforcement point emits a metric to Google's monitoring system (Cloud Monitoring). Developers can set up alerts (email, webhook) when usage reaches 80% of the daily quota or when the error rate due to quota exceeds 1%. The entire enforcement pipeline adds <2ms latency overhead (local cache hit: <0.1ms, Redis hit: <2ms) and handles 10M+ requests/second globally. The hardest part: coordinating quota across 30+ global regions with sub-second consistency — Google uses a combination of Redis CRDTs for eventual consistency and a periodically reconciled authoritative counter in Spanner.

## Q20: Describe the Street View imagery pipeline — from capture vehicle to user-facing panoramic viewer. How does Google handle the massive storage (petabytes of imagery), the stitching/blending of 8+ camera feeds into a seamless 360-degree panorama, and the licensing/face-blurring compliance?

**Answer:** Street View's pipeline is one of the largest image-processing systems ever built. The pipeline: (1) **Capture** — the Street View car has a rig with 8+ 12-24 MP cameras (with Sony sensors) mounted in a ring on the roof, plus 3 LIDAR units for depth. The car drives at posted speed limits, capturing a full 360° frame every 10-15 meters (at 30 km/h, that's about 1 frame per second). Each frame is 8 images, 12 MP each → 96 MP per frame, ~100 MB raw per frame. A typical city capture run generates 10-50 TB per day. (2) **Stitching** — the 8 camera images overlap by ~30%. Stitching uses: feature matching (SIFT or ORB) to find corresponding points in overlapping regions, bundle adjustment to correct for camera pose errors (the car bumps, causing misalignment), and multi-band blending to hide exposure differences between cameras (one side may be in sun, the other in shadow). The output is a 360° equirectangular image (26,000 × 13,000 pixels for Street View's "super-high resolution" mode). (3) **Face and license plate blurring** — a legal requirement in most countries. Google uses a two-stage detector: the first stage is a fast YOLO-based face/plate detector that runs on the raw frames onboard the capture vehicle (edge processing), flagging regions of interest. The second stage is a more accurate Mask R-CNN that runs in the cloud on the flagged regions. Blurring is done with a Gaussian kernel (radius proportional to the detected object size) to prevent deblurring via AI restoration. The blur is applied to the equirectangular image before tiling. (4) **Tiling and encoding** — the final equirectangular image is too large to serve as a single image. It's tiled into a 3-level zoom pyramid (similar to map tiles): at zoom 0, the full equirectangular is 1024×512 pixels (for thumbnail preview); at zoom 1, it's 4096×2048 (8 tiles); at zoom 2, it's 16384×8192 (32 tiles). Tiles are encoded as JPEG (for photorealistic quality) or WebP (for bandwidth savings). Each panorama is ~5-15 MB total. (5) **Storage and serving** — panoramas are stored in Google Cloud Storage (GCS) in an object hierarchy: `streetview/panos/<country>/<city>/<pano_id>/zoom{z}_{x}_{y}.jpg`. The CDN edge caches tiles aggressively — popular panoramas (Times Square, Eiffel Tower) are cached for months. Less-visited panoramas have a shorter TTL but are always available from GCS origin with <200ms latency. (6) **Licensing compliance** — Street View has varying legal status across countries (Germany banned it for years, India has restrictions). The pipeline includes a geo-fencing layer: for each country, the legal status is encoded in per-country metadata (allowed, restricted with blurring, or blocked). The capture planning system does not schedule drives in blocked countries. For restricted countries, the blurring pipeline uses a stricter blurring threshold and an additional "architectural blur" for government buildings (detected via a building facade classifier). The total Street View imagery archive is estimated at 20+ petabytes, growing by 100+ TB per month. Google processes it all through a massive Apache Beam pipeline running on thousands of workers, with the stitching and blurring stages being the most computationally intensive (GPU-accelerated for the ML detection stages).

## Q24: Design the isochrone map service — "30 minutes by car from this point, show reachable area." Compute at query time without precomputing for every point.

**Answer:** Time-constrained shortest-path frontier expansion: (1) **Algorithm**: variant of Dijkstra that expands all edges until cumulative travel time exceeds the threshold. Uses Contraction Hierarchies (CH) where edge weights are travel times. (2) **Polygon construction**: reachable nodes form a "frontier." Interpolate along edges at exact time-budget boundary. Connect interpolation points using concave hull (alpha shapes) — isochrones are star-shaped but not convex due to rivers, parks. (3) **Multi-modal**: for transit, use time-expanded graph with walking edges, waiting edges, transit edges. Mueller algorithm divides time budget into access, transit, egress phases. (4) **Caching**: precompute for high-demand locations (city centers). For arbitrary points, cache similar precomputed isochrones — if query point is within 100m of a cached point, return cached result with a 95% confidence interval.

## Q25: How does Google Maps use ML for real-time ETA, and how do you handle the cold-start problem for new roads with sparse data?

**Answer:** GBDT or lightweight neural network with three feature categories: (1) **Static**: road class, speed limit, lanes, historical baseline. (2) **Dynamic**: real-time speed, trend, adjacent segment speeds, event data (construction, accidents). ## Q28: How does Google Maps use Graph Neural Networks for traffic prediction? How does GNN improve over traditional time-series methods?

**Answer:** (1) **Graph construction**: nodes = road segments. Edges for: physical connectivity, spatial proximity (<200m parallel streets), functional similarity. (2) **Node features**: speed time-series over 60 minutes (12 steps at 5-min intervals) + static features (road class, lanes, speed limit). (3) **ST-GCN architecture**: graph convolution aggregates neighbor features via learned adjacency, temporal 1D dilated CNN captures patterns across time steps, gated fusion combines spatial+ temporal. Outputs speed prediction for next 15/30/60 minutes. (4) **Advantage over time-series**: GNN captures spillover effects — congestion on a highway off-ramp predicting slowdown on the highway itself (correlated but not physically connected). Explains 15-20% more variance than per-segment LSTM.
