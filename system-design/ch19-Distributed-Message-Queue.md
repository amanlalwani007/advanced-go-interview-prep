# Chapter 19 — Design a Distributed Message Queue

## Q1: Compare Apache Kafka, RabbitMQ, and AWS SQS at the architectural level. Why might you choose one over the other for a high-throughput (1M+ msgs/sec) use case?

**Answer:** Kafka: persistent commit log on disk, partitioned, pull-based consumers, naturally supports replay and multiple consumer groups. Handles 1M+ msgs/sec on a modest cluster. Best for: event streaming, data pipelines, log aggregation, where you need message persistence and replayability. RabbitMQ: broker-based, push model with exchanges and bindings, supports complex routing (direct, topic, fanout, headers). Messages are deleted after acknowledgment. Best for: transactional messaging, task queues with worker pools, where you need flexible routing but throughput is moderate (100K msgs/sec). SQS: fully managed, at-least-once delivery, simple API. Best for: serverless architectures, AWS-native applications, where you want zero ops overhead. Key trade-offs: Kafka has higher throughput and persistence but higher operational complexity. RabbitMQ has richer routing but doesn't scale horizontally as well as Kafka. SQS has the simplest operations but no ordering guarantees (unless using FIFO queues, which are limited to 300 TPS). For 1M+ msgs/sec, Kafka is the standard choice.

## Q2: Kafka guarantees at-least-once delivery by default. How do you achieve exactly-once semantics (EOS) in Kafka, and what are the performance costs?

**Answer:** Kafka EOS is achieved through: (1) **Idempotent producers** — the producer attaches a unique ID and sequence number to each message. The broker deduplicates duplicates within a session. (2) **Transactional writes** — the producer writes to multiple partitions atomically using a transaction coordinator. The transaction is committed only if all writes succeed. (3) **Read-process-write exactly-once** — a consumer reads from a source topic, processes, and writes to a sink topic within the same transaction. The consumer offset is committed as part of the transaction. Performance costs: (1) Producer throughput drops 10-20% due to the transaction coordinator overhead and the additional protocol roundtrips for commit/abort. (2) Latency increases because the producer must wait for the transaction coordinator acknowledgment. (3) Only works within a single Kafka cluster — cross-cluster exactly-once requires external idempotency logic. For most use cases, idempotent producer (without transactions) is sufficient and cheaper.

## Q3: Design a consumer rebalancing protocol for a partitioned queue. How does Kafka's "cooperative sticky" rebalancing differ from the older "eager" protocol?

**Answer:** **Eager rebalancing (protocol v0)**: on any consumer join/leave, ALL consumers in the group revoke ALL their partitions, stop consuming, and the group coordinator reassigns the entire partition set. This causes a "stop-the-world" event lasting seconds for large groups. **Cooperative sticky rebalancing (protocol v2)**: consumers only revoke the specific partitions they must give up to achieve balance. The rebalance happens in multiple rounds — in each round, consumers revoke a subset of partitions, and the coordinator reassigns only those partitions. This allows consumers to continue processing unaffected partitions during the rebalance. Sticky means the assigner tries to keep partitions on the same consumer across rebalances to minimize cache churn. The change is significant: in a 100-consumer group with 1000 partitions, a new consumer joining under eager rebalancing causes all 1000 partitions to stop; under cooperative sticky, only ~10 partitions are revoked.

## Q4: How would you design a low-latency (sub-5ms) message queue for financial tick data where ordering within a symbol must be preserved but symbols are independent?

**Answer:** Partition by symbol — each symbol gets its own partition (or shard). The producer ensures all messages for the same symbol go to the same partition (partition key = symbol). The broker stores each partition as a sequential log on the filesystem, using a consumer-grade SSD with direct I/O for lower latency. For sub-5ms end-to-end latency: (1) Use a single-threaded event loop per partition on the broker to avoid context switching. (2) Minimize batching — Kafka batches by default (linger.ms), but for latency you flush immediately (linger.ms=0) at the cost of lower throughput. (3) Use zero-copy transfers (`sendfile` syscall) to move data from page cache to network socket without copying through userspace. (4) Place brokers and consumers on the same rack / same availability zone to minimize network latency. (5) Use a kernel-bypass networking stack (DPDK or io_uring) for further microsecond-level improvements. At this latency regime, avoid cloud providers and use bare-metal with tuned Linux kernel.

## Q5: How do you handle message backpressure in a distributed queue when a consumer falls behind? Design a system that degrades gracefully.

**Answer:** Multiple strategies: (1) **Consumer-side backpressure** — the consumer library pauses message fetching when its processing queue exceeds a configurable high-water mark, resuming when it drops below a low-water mark. This prevents the consumer from buffering millions of messages in memory. (2) **Broker-side flow control** — Kafka has `replica.fetch.max.bytes` and `fetch.max.bytes` to limit the data pushed to a slow consumer. (3) **Dead letter queue (DLQ)** — messages that fail processing after N retries are routed to a DLQ for later analysis, preventing the consumer from being stuck on poison pills. (4) **Auto-scaling** — if consumer lag exceeds a threshold, auto-scale the consumer group (e.g., add more consumer instances). (5) **Shedding** — if the system is severely overloaded, the producer can be asked to slow down via a response header or control channel (backpressure signal). For Kafka, this is achieved by the producer's `max.in.flight.requests.per.connection` — if the broker is overloaded, it responds slower, and the producer naturally throttles.

## Q6: Compare pull-based (Kafka) vs push-based (RabbitMQ) message delivery. Which is better for a mobile push notification service serving 100M devices?

**Answer:** For push notifications, the push-based model is more natural — the broker pushes the message to the consumer as soon as it arrives. However, at 100M scale, pull-based has advantages: (1) Pull allows the consumer to control the rate at which it fetches messages, preventing overwhelm. (2) Pull supports batching — the consumer can fetch 500 messages at once, amortizing the RPC overhead. (3) Push-based systems with 100M consumers would overwhelm the broker with connection management overhead. The practical answer: use a hybrid. A push notification gateway maintains persistent TCP connections (APNS/FCM connections) to the platform's push notification service. Behind this gateway, messages are consumed from Kafka (pull) in batches, then pushed to the respective platform. The pull-based consumption from Kafka handles the scale; the push is done at the edge (gateway), not the broker.

## Q7: Design a reliable exactly-once delivery mechanism for a payment system using a message queue. How do you handle idempotency on the consumer side?

**Answer:** Exactly-once in a queue is impossible without consumer cooperation — at-least-once with idempotent processing is the practical solution. Design: (1) Each payment event carries a unique `event_id` (UUID). (2) The consumer processes the event transactionally: begin DB transaction, check if `event_id` has been processed (look up in `processed_events` table with a unique constraint), if not, apply the payment logic and insert `event_id` into the `processed_events` table, commit transaction. (3) If the consumer crashes after processing but before committing the offset, it re-fetches the same event on restart, processes it again, but the `event_id` check makes it a no-op. (4) Use a two-phase commit (2PC) between the queue offset commit and the DB transaction, or use the transactional outbox pattern: write the event to the DB as part of the business transaction, then a separate process publishes it. This is the standard approach used by payment systems (Stripe, Adyen) to ensure idempotency.

## Q8: How does Kafka achieve its high throughput (2 GB/s per broker)? What specific OS and hardware optimizations are involved?

**Answer:** Kafka's throughput comes from several design choices: (1) **Sequential disk I/O** — writing to a log (append-only) is comparable to network speed on modern hardware (600 MB/s+). No random seeks. (2) **Zero-copy** — when serving data to consumers, Kafka uses `sendfile()` to transfer data from the page cache to the socket without copying through the application's memory space. (3) **Page cache** — Kafka relies heavily on the OS page cache rather than its own cache. Writes go to page cache immediately (fast), and the OS flushes to disk asynchronously. (4) **Batching** — producers batch messages into larger network packets (default 16 KB-1 MB), reducing the per-message overhead. (5) **Efficient binary protocol** — no XML/JSON overhead, just a compact binary format. (6) **Partition parallelism** — each partition is a separate log, and multiple partitions can be served by different threads/cores. Hardware: fast SSDs (NVMe), multiple 10/25 GbE NICs, large RAM (128 GB+) for page cache, modern CPUs with multiple cores (each core handles different partitions).

## Q9: How do you handle message ordering when a consumer crashes and messages are re-processed? What guarantees can you actually provide?

**Answer:** Strict ordering with exactly-once reprocessing is contradictory under failures. Kafka provides **partition-level ordering**: messages within a partition are ordered, and a consumer reads them in order. If the consumer crashes and restarts, it resumes from the last committed offset. However: (1) If the consumer auto-commits offsets, a crash between processing a message and committing the offset causes re-processing of the last batch — those messages will be re-read **in order**, but some may be duplicates. (2) If you manually commit after each message (bad for throughput), ordering is preserved but throughput suffers. (3) For strict ordering with at-least-once, the consumer must be idempotent (so duplicates don't matter). (4) If you need ordering across partitions, you need a single partition (limiting parallelism) or a global sequencer. For financial applications, use a single partition per symbol, and make consumers idempotent. Real-world guarantee: **ordered at-least-once within a partition**.

## Q10: Design a message queue for IoT devices with limited connectivity. Devices come online sporadically, send bursts of data, then go offline. How does the queue handle this? 

**Answer:** This is the "disconnected client" pattern. (1) **Buffering at the edge** — the IoT device itself should have a local persistent queue (on flash) to buffer messages when offline and flush when connected. (2) **Server-side buffering** — the server-side queue retains messages for each device for a configurable retention period (e.g., 7 days). Kafka's log-based storage is naturally suited — messages persist regardless of whether the consumer is online. (3) **Connection burst handling** — when 100K devices come online simultaneously (e.g., after a power outage), they all attempt to connect within seconds. The connection pool must handle this: rate-limit new connections, use exponential backoff on the device side. (4) **Graceful capacity** — the queue must size its storage for the worst-case buffering scenario: if devices are offline for 7 days and produce 1 MB/hour each, 100K devices produce 100 GB × 24 × 7 = 16.8 TB of buffered data. The queue cluster must have sufficient disk space. (5) **Smart polling** — instead of constant polling, devices use long-polling to wait for new messages, reducing the connection overhead.
