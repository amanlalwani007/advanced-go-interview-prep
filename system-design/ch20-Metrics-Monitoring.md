# Chapter 20 — Design a Metrics Monitoring & Alerting System

## Q1: Design a time-series database (TSDB) like Prometheus or TimescaleDB from scratch. How do you store billions of metric data points per day efficiently?

**Answer:** Key design decisions: (1) **Storage layout** — store metrics in a columnar format, not row-oriented. A metric with labels `{host=web01, endpoint=/api, status=200}` and a float value at a timestamp becomes one row. Row-oriented storage is inefficient for time-series because queries typically read all values for a specific metric and time range. (2) **Compaction** — use Gorilla compression (Facebook's paper): timestamps are delta-of-delta encoded, values use XOR-of-previous with leading/trailing zero compression. This achieves ~1.3 bytes per data point on average. (3) **Partitioning** — partition by time (e.g., 2-hour blocks). Each block contains all data points for that time range across all series. Store blocks as immutable files on disk. (4) **Indexing** — build an inverted index mapping label pairs (e.g., `host=web01`) to the set of series IDs that match. Queries like `{host=~"web.*", status="200"}` are resolved by intersecting series ID sets from the inverted index. (5) **Write path** — incoming points are written to a Write-Ahead Log (WAL) for durability, then batched into a memory-mapped block. When the block reaches a threshold, it's flushed to disk as a compressed block. (6) **Read path** — query determines the time range, loads the relevant disk blocks, decompresses them, and filters by label using the inverted index. This is essentially how Prometheus's TSDB (now used by Thanos, Mimir, and Grafana Cloud) works internally.

## Q2: How does Prometheus handle the "cardinality explosion" problem — what happens when a label like `user_id` is accidentally added to a metric?

**Answer:** Cardinality explosion is the most common Prometheus anti-pattern. Each unique combination of label values creates a new time series. If an `http_requests_total` metric adds `user_id` (100M users), it creates 100M × (endpoint, method, status) combinations, potentially billions of series. This crashes the TSDB (memory exhaustion, compaction OOM). Mitigations: (1) Prometheus has a `--storage.tsdb.max-series` flag that panics if exceeded. (2) Recording rules aggregate high-cardinality data before ingestion. (3) Service-level enforcement — use a label validator that rejects metrics with labels that shouldn't exist (e.g., reject `user_id` at the agent or gateway level). (4) Use exemplars or separate logging for high-cardinality dimensions instead of metrics. In a distributed monitoring system (like Grafana Mimir or Thanos), cardinality limits are enforced per-tenant, and the system drops metrics if the tenant exceeds their allowed series count.

## Q3: Design the alerting pipeline — from metric threshold breach to notification delivery (PagerDuty, Slack, email). How do you avoid alert fatigue and false positives?

**Answer:** The pipeline: (1) **Evaluation** — the alert manager evaluates PromQL rules at a configurable interval (e.g., every 30 seconds). A rule like `avg:cpu_usage > 0.9 for 5m` requires the condition to hold for 5 minutes before firing. This is the "for" duration — the single most important tool against false positives (transient spikes don't trigger alerts). (2) **Deduplication** — if 10 replicas of the same service all breach CPU, the alert manager groups them into a single notification: `[FIRING] 10 instances of web-svc have high CPU`. Grouping by alert name and severity. (3) **Silencing** — known issues can be silenced with a comment and TTL. (4) **Inhibition** — if a node is down, all alerts from its child processes are inhibited (silenced) because the node down alert is the root cause. (5) **Escalation** — if alert is not acknowledged within 15 minutes, escalate to secondary on-call. (6) **Alert fatigue reduction** — use severity levels (P0=critical, P1=warning, P2=info). P0 alerts should aim for <5 per month per team. Implement hysteresis (thresholds for firing and recovery are different, e.g., fire at >90%, recover at <80%).

## Q4: How would you design the metrics ingestion pipeline to handle 10 million data points per second (10M DPS) from 100K hosts?

**Answer:** At 10M DPS, the write path is the bottleneck. Architecture: (1) **Local agent** — Prometheus Node Exporter or Grafana Agent runs on each host, collects metrics, and buffers locally in case of remote endpoint failure. (2) **Load-balanced ingesters** — a fleet of stateless ingester services behind a load balancer receive metrics via remote write (protobuf over HTTP). The ingesters batch incoming samples into larger blocks before writing to the TSDB. (3) **Shard by metric** — use consistent hashing on the metric + labels hash to assign series to specific ingesters. This allows the TSDB to keep all data for a series on one node (index locality). (4) **Distributed TSDB** — use Thanos or Grafana Mimir, which store data in object storage (S3/GCS) and use ingesters as a hot write buffer. (5) **Write amplification** — each ingested data point generates 3 replicas (write to 3 ingesters), then asynchronously compacts into object storage. This reduces per-point write amplification compared to full replication. (6) **Scaling** — at 10M DPS, you need roughly 10 ingesters (each handling 1M DPS) with fast SSDs for the write buffer and 10 GbE networking.

## Q5: PromQL's `rate()` function is commonly misunderstood. Explain exactly how it works, including the extrapolation behavior, and when you should use `irate()` instead.

**Answer:** `rate(v range-vector)` calculates the per-second average rate of increase of a counter over a specified time range. How it works: (1) It selects two data points within the window `[now - window, now]`: the first and last point that have a value. (2) It computes `(last_value - first_value) / (last_timestamp - first_timestamp)`. (3) If the range window (e.g., 5m) doesn't align exactly with the data point timestamps, Prometheus extrapolates the rate to the full window boundary — this can cause rate inflation for slow-moving counters. (4) The window must cover at least two samples (scrape intervals × 2 + jitter). `irate()` (instant rate) computes the rate using only the last two data points: `(last - second_last) / (t_last - t_second_last)`. Use `irate()` for short-lived, spiky metrics (like CPU) where you want to see instantaneous changes. Use `rate()` for slow-moving counters (like total requests) where you want a smooth average. A common mistake is using `rate()` with too short a window (like `[30s]`) when the scrape interval is 15s — you'll get noisy results.

## Q6: Design a distributed tracing system (like Jaeger or Zipkin) that integrates with a metrics monitoring system. How do you sample traces without losing the rare errors?

**Answer:** Distributed tracing collects individual request spans across services. At high volume, you cannot sample 100% of traces. Sampling strategies: (1) **Head-based sampling** — at the root span, decide randomly (e.g., 1% sample rate) whether to trace. Simple but rare errors are likely missed. (2) **Tail-based sampling** — collect all spans temporarily (buffered), then after the request completes, decide whether to retain based on criteria (e.g., error status, latency > p99, or a specific user ID). This requires a buffer (Jaeger uses Kafka + a "sampling strategy" service). (3) **Adaptive sampling** — automatically increase the sampling rate for services or endpoints with low traffic and decrease for high-traffic ones. (4) **Integration with metrics** — when an alert fires (high error rate), the alert manager triggers a signal to the tracing system to increase the sampling rate for the affected service for the next 10 minutes, so engineers can investigate with full traces. The traces themselves are stored in a columnar store (Cassandra or Elasticsearch) indexed by `service`, `operation`, `tags`, and `duration`. Jaeger's architecture uses gRPC for span collection, Kafka for buffering, and a Spark/Flink job for trace aggregation.

## Q7: How would you design a dashboard system (like Grafana) that can render 10,000 panels from a TSDB in under 5 seconds?

**Answer:** Rendering 10K panels in 5 seconds means each panel has 500 µs budget — impossible to fetch fresh data per panel. Strategies: (1) **Rule-based dashboard** — define what queries the dashboard needs upfront. A backend service pre-fetches all required queries in parallel (as a single multi-query to the TSDB) and returns a unified dataset. This avoids N individual HTTP connections. (2) **Query caching** — if multiple users view the same dashboard, the query results are cached in Redis with a 30-second TTL. (3) **Downsampling** — dashboards showing 30 days of data don't need raw 15-second scrape resolution. Pre-aggregate data into 5-minute or 1-hour rollups and query those. (4) **Lazy loading** — load visible panels first, then background-load panels below the fold. (5) **Prometheus / Grafana plugin** — use the "Prometheus Streaming" capability where the TSDB pushes data incrementally to the panel as it's read. (6) **Graphite-style** — use a TSDB that supports batch query endpoints (e.g., Graphite's `render` API that accepts multiple targets in one request). At scale, Grafana's caching proxy (Grafana Caching Service) sits between the UI and the TSDB.

## Q8: Design a system for anomaly detection on metrics — detecting unusual spikes or dips automatically without manual threshold configuration.

**Answer:** Statistical anomaly detection pipeline: (1) **Baseline model** — for each metric, maintain a time-of-day (hh:mm mapped to weekly) baseline using a moving average or Holt-Winters exponential smoothing. The model captures daily/weekly seasonality. (2) **Error residual** — the difference between the actual observed value and the predicted value. (3) **Threshold computation** — compute the standard deviation of the residuals. Flag any point where the residual exceeds 3σ (or a configurable multiple). (4) **Multi-seasonal decomposition** — use STL (Seasonal-Trend decomposition using LOESS) to separate the metric into seasonal + trend + residual components. Alert on the residual. (5) **Sliding window** — for metrics without seasonality (e.g., newly deployed), use a rolling window (last 2 hours) to compute mean/std and flag deviations. (6) **False positive reduction** — require the anomaly to persist for at least 2 consecutive data points. Use a Bayesian change-point detection algorithm (e.g., the PELT algorithm) to detect sustained shifts rather than single outliers. This approach is implemented by Facebook's Prophet and used in Datadog's Watchdog.

## Q9: How does Prometheus's pull model differ from a push-based monitoring system (e.g., Graphite, Datadog)? What are the trade-offs at scale?

**Answer:** Prometheus pull: the server scrapes each target's `/metrics` endpoint on a schedule. Push: the application sends (pushes) metrics to a central collector. Trade-offs: (1) **Discovery** — pull requires a service discovery mechanism (Kubernetes API, Consul, DNS SRV records). Push requires the application to know the collector address. (2) **Health signaling** — if Prometheus can't scrape a target, it knows the target is down. In push systems, absence of data could mean the app is down OR the collector is temporarily slow. (3) **Scalability** — pull with many targets creates a thundering herd (all scrapes fire at :00 of the interval). Push distributes the load across the interval naturally. (4) **Firewalls** — pull requires Prometheus to reach each target (network connectivity). Push works through firewalls because the target initiates the connection. (5) **Load balancing** — push naturally load-balances across collector instances. Pull requires a consistent hash ring or a proxy. Prometheus at very large scale (>100K targets) uses push via Prometheus Agent / Grafana Agent (which scrape locally and push to a central remote-write endpoint), combining both models.

## Q10: How do you design the SLO (Service Level Objective) burn-rate alerting system that Google uses (as described in the SRE workbook)?

**Answer:** Burn-rate alerting measures how fast your error budget is being consumed. Design: (1) Define SLO: e.g., 99.9% of requests succeed over a 30-day rolling window. Error budget = 0.1% of total requests = 0.001 × total_requests over 30 days. (2) Compute burn rate = current error rate / SLO error rate (e.g., if your current error rate is 1% and your SLO permits 0.1%, your burn rate is 10x). (3) Alerting thresholds: a burn rate of 10x for 1 hour consumes as much error budget as 1x for 10 hours. Google recommends: page (P0) when burn rate is 14x over 1 hour (consumes 2% of budget) or 6x over 6 hours (consumes 5% of budget). (4) Implementation: two alerting rules — `burn_rate_1h` (fast burn) and `burn_rate_6h` (slow burn). This catches both sudden spikes (high burn rate, short window) and gradual degradation (lower burn rate, longer window). (5) Multi-window, multi-burst — use multiple time windows to prevent alert fatigue. The multi-window approach ensures the alert only fires if BOTH a short-term high burn rate AND a longer-term sustained burn rate are observed. This is the gold-standard alerting approach used by Google's SRE teams.
