# Chapter 20 — Design a Metrics Monitoring & Alerting System

## Q1: Design a time-series database (TSDB) like Prometheus or TimescaleDB from scratch. How do you store billions of metric data points per day efficiently?

**Answer:** Key design decisions: (1) **Storage layout** — store metrics in a columnar format, not row-oriented. A metric with labels `{host=web01, endpoint=/api, status=200}` and a float value at a timestamp becomes one row. Row-oriented storage is inefficient for time-series because queries typically read all values for a specific metric and time range. (2) **Compaction** — use Gorilla compression (Facebook's paper): timestamps are delta-of-delta encoded, values use XOR-of-previous with leading/trailing zero compression. This achieves ~1.3 bytes per data point on average. (3) **Partitioning** — partition by time (e.g., 2-hour blocks). Each block contains all data points for that time range across all series. Store blocks as immutable files on disk. (4) **Indexing** — build an inverted index mapping label pairs (e.g., `host=web01`) to the set of series IDs that match. Queries like `{host=~"web.*", status="200"}` are resolved by intersecting series ID sets from the inverted index. (5) **Write path** — incoming points are written to a Write-Ahead Log (WAL) for durability, then batched into a memory-mapped block. When the block reaches a threshold, it's flushed to disk as a compressed block. (6) **Read path** — query determines the time range, loads the relevant disk blocks, decompresses them, and filters by label using the inverted index. This is essentially how Prometheus's TSDB (now used by Thanos, Mimir, and Grafana Cloud) works internally.

## Q2: How does Prometheus handle the "cardinality explosion" problem — what happens when a label like `user_id` is accidentally added to a metric?

**Answer:** Cardinality explosion is the most common Prometheus anti-pattern. Each unique combination of label values creates a new time series. If an `http_requests_total` metric adds `user_id` (100M users), it creates 100M × (endpoint, method, status) combinations, potentially billions of series. This crashes the TSDB (memory exhaustion, compaction OOM). Mitigations: (1) Prometheus has a `--storage.tsdb.max-series` flag that panics if exceeded. (2) Recording rules aggregate high-cardinality data before ingestion. (3) Service-level enforcement — use a label validator that rejects metrics with labels that shouldn't exist (e.g., reject `user_id` at the agent or gateway level). (4) Use exemplars or separate logging for high-cardinality dimensions instead of metrics. In a distributed monitoring system (like Grafana Mimir or Thanos), cardinality limits are enforced per-tenant, and the system drops metrics if the tenant exceeds their allowed series count.

## Q3: Design the alerting pipeline — from metric threshold breach to notification delivery (PagerDuty, Slack, email). How do you avoid alert fatigue and false positives?

**Answer:** The pipeline: (1) **Evaluation** — the alert manager evaluates PromQL rules at a configurable interval (e.g., every 30 seconds). A rule like `avg:cpu_usage > 0.9 for 5m` requires the condition to hold for 5 minutes before firing. This is the "for" duration — the single most important tool against false positives (transient spikes don't trigger alerts). (2) **Deduplication** — if 10 replicas of the same service all breach CPU, the alert manager groups them into a single notification: `[FIRING] 10 instances of web-svc have high CPU`. Grouping by alert name and severity. (3) **Silencing** — known issues can be silenced with a comment and TTL. (4) **Inhibition** — if a node is down, all alerts from its child processes are inhibited (silenced) because the node down alert is the root cause. (5) **Escalation** — if alert is not acknowledged within 15 minutes, escalate to secondary on-call. (6) **Alert fatigue reduction** — use severity levels (P0=critical, P1=warning, P2=info). P0 alerts should aim for <5 per month per team. Implement hysteresis (thresholds for firing and recovery are different, e.g., fire at >90%, recover at <80%).

## Q4: How would you design the metrics ingestion pipeline to handle 10 million data points per second (10M DPS) from 100K hosts?

**Answer:** At 10M DPS, the write path is the bottleneck. Architecture: (1) **Local agent** — Prometheus Node Exporter or Grafana Agent runs on each host, collects metrics, and buffers locally in case of remote endpoint failure. (2) **Load-balanced ingesters** — a fleet of stateless ingester services behind a load balancer receive metrics via remote write (protobuf over HTTP). The ingesters batch incoming samples into larger blocks before writing to the TSDB. (3) **Shard by metric** — use consistent hashing on the metric + labels hash to assign series to specific ingesters. This allows the TSDB to keep all data for a series on one node (index locality). (4) **Distributed TSDB** — use Thanos or Grafana Mimir, which store data in object storage (S3/GCS) and use ingesters as a hot write buffer. (5) **Write amplification** — each ingested data point generates 3 replicas (write to 3 ingesters), then asynchronously compacts into object storage. This reduces per-point write amplification compared to full replication. (6) **Scaling** — at 10M DPS, you need roughly 10 ingesters (each handling 1M DPS) with fast SSDs for the write buffer and 10 GbE networking.

## Q5: PromQL's `rate()` function is commonly misunderstood. Explain exactly how it works, including the extrapolation behavior, and when you should use `irate()` instead.

**Answer:** `rate(v range-vector)` calculates the per-second average rate of increase of a counter over a specified time range. How it works: (1) It selects two data points within the window `[now - window, now]`: the first and last point that have a value. (2) It computes `(last_value - first_value) / (last_timestamp - first_timestamp)`. (3) If the range window (e.g., 5m) doesn't align exactly with the data point timestamps, Prometheus extrapolates the rate to the full window boundary — this can cause rate inflation for slow-moving counters. (4) The window must cover at least two samples (scrape intervals × 2 + jitter). `irate()` (instant rate) computes the rate using only the last two data points: `(last - second_last) / (t_last - t_second_last)`. Use `irate()` for short-lived, spiky metrics (like CPU) where you want to see instantaneous changes. Use `rate()` for slow-moving counters (like total requests) where you want a smooth average. A common mistake is using `rate()` with too short a window (like `[30s]`) when the scrape interval is 15s — you'll get noisy results.

## Q6: Design a distributed tracing system (like Jaeger or Zipkin) that integrates with a metrics monitoring system. How do you sample traces without losing the rare errors?

**Answer:** Distributed tracing collects individual request spans across services. At high volume, you cannot sample 100% of traces. Sampling strategies: (1) **Head-based sampling** — at the root span, decide randomly (e.g., 1% sample rate) whether to trace. Simple but rare errors are likely missed. (2) **Tail-based sampling** — collect all spans temporarily (buffered), then after the request completes, decide whether to retain based on criteria (e.g., error status, latency > p99, or a specific user ID). This requires a buffer (Jaeger uses Kafka + a "sampling strategy" service). (3) **Adaptive sampling** — automatically increase the sampling rate for services or endpoints with low traffic and decrease for high-traffic ones. (4) **Integration with metrics** — when an alert fires (high error rate), the alert manager triggers a signal to the tracing system to increase the sampling rate for the affected service for the next 10 minutes, so engineers can investigate with full traces. The traces themselves are stored in a columnar store (Cassandra or Elasticsearch) indexed by `service`, `operation`, `tags`, and `duration`. Jaeger's architecture uses gRPC for span collection, Kafka for buffering, and a Spark/Flink job for trace aggregation.

## Q7: How would you design a dashboard system (like Grafana) that can render 10,000 panels from a TSDB in under 5 seconds?

**Answer:** Rendering 10K panels in 5 seconds means each panel has 500 µs budget — impossible to fetch fresh data per panel. Strategies: (1) **Rule-based dashboard** — define what queries the dashboard needs upfront. A backend service pre-fetches all required queries in parallel (as a single multi-query to the TSDB) and returns a unified dataset. This avoids N individual HTTP connections. (2) **Query caching** — if multiple users view the same dashboard, the query results are cached in Redis with a 30-second TTL. (3) **Downsampling** — dashboards showing 30 days of data don't need raw 15-second scrape resolution. Pre-aggregate data into 5-minute or 1-hour rollups and query those. (4) **Lazy loading** — load visible panels first, then background-load panels below the fold. (5) **Prometheus / Grafana plugin** — use the "Prometheus Streaming" capability where the TSDB pushes data incrementally to the panel as it's read. (6) **Graphite-style** — use a TSDB that supports batch query endpoints (e.g., Graphite's `render` API that accepts multiple targets in one request). At scale, Grafana's caching proxy (Grafana Caching Service) sits between the UI and the TSDB.

## Q8: Design a system for anomaly detection on metrics — detecting unusual spikes or dips automatically without manual threshold configuration.

**Answer:** Statistical anomaly detection pipeline: (1) **Baseline model** — for each metric, maintain a time-of-day (hh:mm mapped to weekly) baseline using a moving average or Holt-Winters exponential smoothing. The model captures daily/weekly seasonality. (2) **Error residual** — the difference between the actual observed value and the predicted value. (3) **Threshold computation** — compute the standard deviation of the residuals. Flag any point where the residual exceeds 3σ (or a configurable multiple). (4) **Multi-seasonal decomposition** — use STL (Seasonal-Trend decomposition using LOESS) to separate the metric into seasonal + trend + residual components. Alert on the residual. (5) **Sliding window** — for metrics without seasonality (e.g., newly deployed), use a rolling window (last 2 hours) to compute mean/std and flag deviations. (6) **False positive reduction** — require the anomaly to persist for at least 2 consecutive data points. Use a Bayesian change-point detection algorithm (e.g., the PELT algorithm) to detect sustained shifts rather than single outliers. This approach is implemented by Facebook's Prophet and used in Datadog's Watchdog.

## Q9: How does Prometheus's pull model differ from a push-based monitoring system (e.g., Graphite, Datadog)? What are the trade-offs at scale?

**Answer:** Prometheus pull: the server scrapes each target's `/metrics` endpoint on a schedule. Push: the application sends (pushes) metrics to a central collector. Trade-offs: (1) **Discovery** — pull requires a service discovery mechanism (Kubernetes API, Consul, DNS SRV records). Push requires the application to know the collector address. (2) **Health signaling** — if Prometheus can't scrape a target, it knows the target is down. In push systems, absence of data could mean the app is down OR the collector is temporarily slow. (3) **Scalability** — pull with many targets creates a thundering herd (all scrapes fire at :00 of the interval). Push distributes the load across the interval naturally. (4) **Firewalls** — pull requires Prometheus to reach each target (network connectivity). Push works through firewalls because the target initiates the connection. (5) **Load balancing** — push naturally load-balances across collector instances. Pull requires a consistent hash ring or a proxy. Prometheus at very large scale (>100K targets) uses push via Prometheus Agent / Grafana Agent (which scrape locally and push to a central remote-write endpoint), combining both models.

## Q10: How do you design the SLO (Service Level Objective) burn-rate alerting system that Google uses (as described in the SRE workbook)?

**Answer:** Burn-rate alerting measures how fast your error budget is being consumed. Design: (1) Define SLO: e.g., 99.9% of requests succeed over a 30-day rolling window. Error budget = 0.1% of total requests = 0.001 × total_requests over 30 days. (2) Compute burn rate = current error rate / SLO error rate (e.g., if your current error rate is 1% and your SLO permits 0.1%, your burn rate is 10x). (3) Alerting thresholds: a burn rate of 10x for 1 hour consumes as much error budget as 1x for 10 hours. Google recommends: page (P0) when burn rate is 14x over 1 hour (consumes 2% of budget) or 6x over 6 hours (consumes 5% of budget). (4) Implementation: two alerting rules — `burn_rate_1h` (fast burn) and `burn_rate_6h` (slow burn). This catches both sudden spikes (high burn rate, short window) and gradual degradation (lower burn rate, longer window). (5) Multi-window, multi-burst — use multiple time windows to prevent alert fatigue. The multi-window approach ensures the alert only fires if BOTH a short-term high burn rate AND a longer-term sustained burn rate are observed. This is the gold-standard alerting approach used by Google's SRE teams.

## Q11: How do you correlate logs, metrics, and traces in a unified observability pipeline? What are the practical challenges of implementing "observability signals correlation" at scale?

**Answer:** Correlation is the holy grail of observability — being able to go from "CPU is high" (metric) → "which request is causing it" (trace) → "what log lines was that request emitting" (logs). The implementation approaches: (1) **Common identifiers** — inject a `trace_id` and `span_id` into every log line, metric exemplar, and trace span. The Prometheus exemplar storage (introduced in Prometheus v2.40+) allows attaching a trace_id to a metric data point as metadata. Grafana Tempo can then link from a metric spike directly to the related traces. (2) **OpenTelemetry Collector** — deploy the OTel Collector as an agent on each host. It receives logs (via OTLP or file tailing), metrics (via Prometheus receiver or OTLP), and traces (via OTLP). It attaches resource attributes (`host`, `service.name`, `k8s.pod_id`) to all signals, then fans them out to dedicated backends (Mimir for metrics, Tempo for traces, Loki for logs). (3) **Practical challenges** — (a) **Sampling mismatch**: traces are sampled (e.g., 1%), logs are usually unsampled. A metric spike in the 99% unsampled population won't have associated traces. You need tail-based sampling driven by metric anomaly signals. (b) **Cardinality explosion on trace_id**: attaching a unique trace_id to every metric data point creates millions of unique series — Prometheus cannot index this. Exemplars store only the last N per series, which limits correlation depth. (c) **Storage cost**: storing full logs + traces + high-res metrics for the same retention window triples storage costs. Most orgs tier retention: metrics 12 months, traces 7 days, logs 30 days. (d) **Tool sprawl**: Grafana + Tempo + Loki + Mimir works in one ecosystem, but many orgs mix Datadog (metrics) + Splunk (logs) + Honeycomb (traces), making correlation manual.

## Q12: Design an OpenTelemetry Collector deployment strategy for a multi-region Kubernetes cluster with 10,000 pods. How do you handle load shedding, retry, and backpressure?

**Answer:** The OTel Collector is the critical data plane — its failure means total observability loss. Deployment architecture: (1) **Two-tier deployment**: Tier 1 — **Agent Collector** (DaemonSet, one per node, ~50 pods/node). Handles scraping Prometheus metrics from pods on the same node, tailing container logs, and receiving OTLP traces via gRPC from application sidecars. Agent collectors perform lightweight processing: add resource attributes, batch, compress, and forward. Tier 2 — **Gateway Collector** (Deployment, sized to 3-5 replicas per region). Receives from all agents, performs heavy processing: attribute enrichment, sampling decisions, fan-out to backends. (2) **Load shedding**: Each collector has a `memory_limiter` processor configured to 75% of available RAM. When exceeded, the collector drops the least-priority data (logs first, then traces, then metrics — because metrics are smallest and most important for alerting). Config: `ballast_size_mib: 4096` for Go GC optimization. (3) **Backpressure**: The `batch` processor accumulates data into 5 MB batches or 1-second windows. If the downstream backend (e.g., Mimir) returns 429 (rate limited) or 503 (unavailable), the collector buffers up to a configurable queue (e.g., 5000 items). If the queue fills, it blocks the receiver — applies backpressure to the application. This prevents the collector from being an infinite sponge. (4) **Retry**: The `exporter` uses exponential backoff (500 ms initial, 30 s max, 0.2 jitter) for transient failures. After 10 retries, data is dropped to avoid cascading failure. (5) **Security**: mTLS between agent and gateway collectors. Each agent has a Kubernetes service account token that the gateway validates. (6) **Multi-region**: Each region's gateway collectors forward to the region-local backend (e.g., Mimir in us-east-1, Mimir in eu-west-1). A global Thanos querier performs cross-region query federation. If the local backend is down, the gateway falls back to a secondary region's backend with a different routing header.

## Q13: How would you design a custom Prometheus exporter for a legacy application that exposes only a proprietary binary protocol? Walk through the instrumentation decisions, caching strategy, and performance considerations.

**Answer:** Building a Prometheus exporter for a legacy system requires bridging two worlds: (1) **Protocol parsing**: If the legacy app exposes a binary protocol (e.g., a custom TCP-based stats dump), the exporter must implement a parser in Go. Use a `bufio.Scanner` with custom split function for the protocol framing. Parse into internal structs like `type MetricPoint struct { Name string; Labels map[string]string; Value float64; Timestamp int64 }`. (2) **Scrape lifecycle**: Prometheus scrapes the exporter's `/metrics` endpoint. The exporter should NOT fetch from the legacy app on every scrape. Instead, maintain a background goroutine that polls the legacy app every `N` seconds (e.g., 15s, matching Prometheus's scrape interval). Results are cached in a `sync.Map` of `metric_family_name -> []MetricPoint`. The `/metrics` handler renders from this cache. This decouples scrape timing from collection timing and prevents thundering herd if Prometheus scrapes aggressively. (3) **Caching considerations**: The cache TTL should be ~2x the poll interval. If the legacy app is slow (takes 5 seconds to respond), the poll interval must account for this: `poll_interval = max(15s, legacy_response_time * 3)`. Use a `context.WithTimeout` on the poll to prevent a hung legacy app from starving the exporter. (4) **Performance**: The exporter must handle 1000+ concurrent scrapes (from multiple Prometheus servers). Use `prometheus.NewRegistry()` (not the global registry) to avoid lock contention. Register metrics with `prometheus.NewCounterVec`, `NewGaugeVec` etc. The cost of rendering `/metrics` should be <10 ms for 10K metric families. Use `gzip` compression if the client supports it. (5) **Error handling**: If the legacy app is unreachable, the exporter should expose a `up{target="legacy-app"} 0` gauge. Prometheus can alert on this. DO NOT return HTTP 500 from the exporter — return stale data with a warning metric instead. (6) **Resource profiling**: Instrument the exporter itself with `go_memstats_*` and `promhttp_*` metrics. A common bug is memory leaks in the cache (goroutine not releasing old entries). Use `prometheus.NewBuildInfoCollector` for version tracking.

## Q14: Design a long-term metrics storage solution using Thanos or Cortex/Mimir that handles 10,000 active series per second (aSPS) with 2-year retention. Compare the cost and architectural trade-offs.

**Answer:** At 10K aSPS (active series per second — the rate of new series being created), over 2 years you accumulate ~630B data points. Solutions: (1) **Thanos** (sidecar pattern on top of Prometheus): Each Prometheus server runs a Thanos Sidecar that uploads TSDB blocks to object storage (S3/GCS) every 2 hours. Thanos Store Gateway serves queries from object storage. Thanos Compactor downsamples older data (60s → 5m → 1h resolution) and applies dedup. Trade-off: Thanos keeps Prometheus's single-node architecture per shard — a single Prometheus cannot handle >2M active series. You must shard by namespace/team, each with its own Prometheus + Sidecar. Querying across shards requires Thanos Querier with the `--endpoint` flag pointing to all Store Gateways. (2) **Cortex/Mimir** (truly distributed TSDB): Ingesters receive data via remote-write, replicate to 3 ingesters for durability, and flush to object storage in ~2-hour blocks. The ingesters are stateless — if one dies, the hash ring rebalances. Mimir supports faster series churn (10K aSPS is comfortable — Mimir can handle 100K+ aSPS). Query: Mimir uses a query-frontend with parallelization and caching. (3) **Cost breakdown** (for 10K aSPS, 2 years): Prometheus alone: requires ~50 servers (each handling 200 series/second) × $200/month = $120K/year. Thanos: same Prometheus compute + S3 storage (~$0.023/GB/month). With downsampling to 1-hour resolution after 6 months, cold storage costs ~$500/month for the full 2-year dataset. Mimir: fewer servers (10 ingesters × $400/month = $48K/year) + S3 storage. Mimir's query cost is lower because queries don't need to scan all shards. (4) **Winner**: Mimir if you have >10K aSPS and want operational simplicity. Thanos if you already have Prometheus deployed and want to add long-term storage without migrating the ingestion layer.

## Q15: Explain the difference between Prometheus recording rules and alerting rules in depth. When should you use each, and what are the common misconfigurations that cause production incidents?

**Answer:** This subtle distinction has caused many cascading failures. (1) **Recording rules** — pre-compute expensive PromQL expressions and store them as new time series. Example: `rule:node_cpu_avg_5m = avg by (instance) (rate(node_cpu_seconds_total[5m]))`. Evaluation runs on a schedule (default every 15s for recording rules). They are critical for: (a) reducing query latency on dashboards (complex joins pre-computed), (b) aggregating high-cardinality metrics into low-cardinality ones (e.g., aggregate `http_request_duration_seconds_bucket` (high cardinality) into `http_request_duration_p99` per service), (c) building building blocks for alerting rules. (2) **Alerting rules** — also evaluate a PromQL expression, but they generate an alert if the expression is true for a `for` duration. Example: `alert:HighCPU expr:node_cpu_avg_5m > 0.9 for: 5m`. Common misconfigurations: (a) **Alerting on recording rules that change**: If you update a recording rule's expression, the existing series stop and new series start — your alerting rule that references `node_cpu_avg_5m` will have no data for a few minutes, potentially causing false negatives. Solution: use `or` with `absent()` to handle gaps, or avoid alerting directly on recording rules that change often. (b) **Too many recording rules**: Each recording rule creates a new time series. If you have 1000 recording rules each producing 10 series, you've added 10K series to your TSDB. This consumes memory and slows down queries. (c) **For duration too short**: Alerting on `rate(http_errors[1m]) > 0.01 for: 30s` burns error budget too fast. Short `for` durations on noisy metrics cause pager fatigue. Minimum recommended: 5 minutes for warning, 2-5 minutes for critical. (d) **Aggregation lost labels**: `sum by (instance) (rate(...))` loses all other labels. If you later need to route the alert to a specific team based on `namespace` label, you'll find it's gone. Solution: include all needed labels in the `by ()` clause.

## Q16: How do you define an SLI (Service Level Indicator) for a complex multi-step operation (e.g., a checkout flow across 5 microservices) and compute an SLO that accounts for both availability and latency? Walk through the algebra.

**Answer:** For a multi-step checkout flow, a single "is the checkout working" SLI must model the dependencies: (1) **SLI definition**: Define two SLIs — `SLI_good` = requests that completed successfully in <2 seconds, `SLI_total` = all checkout requests initiated. The ratio is the "good request" rate. But each checkout call involves 5 services (cart, payment, inventory, shipping, notification). A failure in any one service fails the entire checkout. (2) **Dependency algebra**: If each service has availability `A_i` and latency distribution `L_i`, the end-to-end availability is `A_total = ∏ A_i` (product). For 5 services each at 99.9%, A_total = 0.999^5 = 99.5%. This means your composite SLO cannot exceed 99.5% even if each service is at 99.9%. (3) **Latency composition**: End-to-end latency is the sum of individual service latencies (assuming sequential calls). If each service has p99 = 200 ms, the end-to-end p99 could be 200 ms × 5 = 1000 ms. However, latency distributions are NOT additive at the tail — the p99 of a sum is higher than the sum of p99s. Use the convolution of distributions or measure the actual end-to-end latency distribution from a single distributed trace. (4) **SLO computation**: Target: 99% of checkouts complete in <2 seconds over a 30-day rolling window. Error budget = 0.01 × total_checkouts_30d. Burn rate alerting: fire P0 if error rate exceeds 10× SLO error rate for 1 hour (fast burn) or 4× for 6 hours (slow burn). (5) **Composite SLO trade-off**: If cart service degrades from 99.9% to 99% availability, the composite drops to 99.4% (checkout). You can't determine which service caused the SLO burn from the composite alone. Decompose: track SLIs per service AND SLIs per end-user flow. Use the "dependency graph" to attribute SLO burn to the root cause service: `burn_{service_i} = (1 - A_i_{current}) / (1 - A_i_{target}) × weight_i`. (6) **Implementation**: Use Google's `slo-generator` or a custom Prometheus rule: `(sum(rate(checkout_failed_total[1h])) / sum(rate(checkout_total[1h]))) > 0.01`. Window this over 1h and 6h for multi-window burn rate alerting.

## Q17: Design a multi-cluster observability architecture for 10 Kubernetes clusters across 3 data centers. How do you federate metrics, logs, and traces while maintaining tenant isolation and single-pane-of-glass querying?

**Answer:** Multi-cluster observability is a hard problem — naive federation creates a single point of failure and massive data duplication. Architecture: (1) **Per-cluster stack**: Each Kubernetes cluster runs its own lightweight stack: Mimir (or Thanos) for metrics, Loki for logs, Tempo for traces. Each stack is independently available — if the central query layer is down, per-cluster teams can still look at their own data by port-forwarding to the local stack. (2) **Global querier**: A central Grafana instance queries all clusters via an intermediate "global querier" service. This service fans out queries to each cluster's store gateway. For metrics: use Thanos Querier with a list of all Store API endpoints across clusters (via DNS SRV records). For logs: the global querier fans out to each cluster's Loki querier. For traces: a cross-cluster trace lookup requires a global trace ID index (e.g., Cassandra table keyed by trace_id → cluster location). (3) **Tenant isolation**: Each cluster is a separate "tenant" in Mimir/Loki. Tenants are isolated at the storage layer (different object storage prefixes). A central auth proxy (e.g., OAuth2-proxy + OPA) enforces that Team A can only see Tenant A's data. This prevents cross-tenant data leakage in the central Grafana. (4) **Cross-cluster federation**: Some metrics need to be global (e.g., "total requests across all clusters"). Use a "global recording rule" — a Prometheus ruler in the global query layer that periodically queries all clusters and computes the global aggregate. Store this as a synthetic series in the global tenant. (5) **Latency challenges**: Fan-out queries to 10 clusters means the slowest cluster determines query latency. Set a sensible timeout (e.g., 10 seconds). Mimir supports partial results: if 8/10 clusters respond in time, return results with a warning "partial results: 2 clusters timed out". (6) **Network costs**: Sending all logs, metrics, and traces from 10 clusters to a central store burns bandwidth and egress cost (~$0.09/GB on AWS). Solution: keep 90% of observability data local; only send aggregated data (recording rules, aggregated logs) centrally. Full raw data is queried directly from the cluster-local store when needed for deep debugging.

## Q18: How do you design a metrics cardinality management system that prevents label explosions at the ingestion layer without losing useful debugging dimensions?

**Answer:** Cardinality management is a permission model — you must prevent bad actors (or bad code) from creating millions of series while allowing legitimate high-cardinality use cases like per-request latency with `user_id`. Architecture: (1) **Cardinality budget per tenant**: Each team/service gets a series quota — e.g., 10K active series per service. Track usage in real-time: an in-memory counting blooms filter counts unique label combinations per metric per tenant. When a tenant exceeds 80% of their budget, emit a warning alert. At 100%, the ingress gateway (e.g., Grafana Agent or OTel Collector) rejects new series for that tenant with HTTP 429. (2) **Label allowlist/blocklist**: Maintain a global registry of allowed metric names and label keys. `user_id` is not an allowed label on `http_requests_total` — it must be an exemplar or a separate `request_latency_seconds` metric with `user_id` explicitly approved. Block list: `request_id`, `trace_id`, `session_id`, `timestamp` (these are guaranteed to be unique per request). Implementation: a Mutating Webhook in the agent that drops blocked labels. (3) **Label value length limit**: A label value like `error_message="something went wrong with user alice in transaction 0x1234..."` is a cardinality bomb if every error has a unique message. Enforce a max label value length (e.g., 256 bytes). Truncate longer values and emit a warning metric. (4) **Active series tracking**: Prometheus exposes `prometheus_tsdb_head_series`. Alert when this exceeds 80% of the per-instance max. For Mimir, the distributor tracks per-tenant active series rate and exposes it as `cortex_distributor_active_series_per_tenant`. (5) **Retroactive cleanup**: If a cardinality explosion has already happened, you need a "series pruning" tool — Thanos Tools' `thanos tools bucket cleanup` or Mimir's `mimirtool backfill` can remove specific series by label matcher. In extreme cases, you must delete the affected TSDB blocks. (6) **Audit trail**: Log every rejected metric with the reason: `series=user_request_latency{user_id=alice, endpoint=/api} rejected: user_id is not in allowlist for this metric`. This prevents the developer from being surprised and helps them fix the instrumentation.

## Q19: Compare and contrast synthetic monitoring (pingdom-style "is it up") with real-user monitoring (RUM — actual browser performance data). When would you use each, and how do you combine them to form a complete picture?

**Answer:** Synthetic monitoring and RUM measure fundamentally different things and have different failure modes. (1) **Synthetic monitoring**: Probes from known data center locations hit your endpoints at regular intervals (e.g., every 60 seconds). It measures: availability (is the HTTP response 200?), DNS resolution time, TLS handshake time, first byte time. Pros: deterministic, no sampling error, works even when real traffic is low (pre-production environments), catches "site is completely down" within seconds. Cons: probes from data centers, not real user devices (last-mile latency invisible); probes are whitelisted (CDN caches may serve stale content to probes but not to real users); cannot measure client-side performance (JavaScript execution, DOM rendering). (2) **RUM**: A JavaScript snippet injected into all pages sends performance data (Navigation Timing API, First Contentful Paint, Largest Contentful Paint) from actual user browsers. Pros: measures actual user experience across devices, networks, and geographies; detects regional slowdowns (e.g., "users in India see 5-second load times while US users see 1-second"). Cons: depends on users visiting the site (no data in off-hours or for unvisited pages); clients can block the RUM script (ad blockers); sampling — you only see a fraction of traffic. (3) **Combined picture**: Synthetic is the "canary" — it tells you the site is up. RUM is the "reality" — it tells you how the site performs for real users. Use synthetic for alerting (P0 alert if all probes fail for 2 minutes). Use RUM for SLO measurement (your SLO should be based on real-user data, not synthetic). Correlate: when RUM shows degradation but synthetic is green, you have a client-side issue (e.g., a third-party script is blocking rendering). When synthetic shows red but RUM is green, you have a probe issue or CDN whitelist effect. (4) **Implementation**: Synthetic — use Grafana Synthetic Monitoring (k6) or Checkly. Run probes from 10+ locations globally. Configure multi-step API check (login → get data → logout) plus browser check (full page load with JavaScript execution). RUM — use OpenTelemetry JavaScript instrumentation or a hosted RUM service (Datadog RUM, New Relic Browser, Grafana Faro). Set up a session replay feature for debugging user-reported issues. (5) **Cost**: Synthetic monitoring costs are proportional to number of probes × frequency. RUM costs are proportional to page views × sampling rate. At scale, RUM is cheaper per data point than synthetic (because you're piggybacking on real traffic).

## Q20: Design an incident response runbook automation system that integrates with your monitoring and alerting pipeline. When an alert fires, what automated actions should be triggered before a human even looks at the page?

**Answer:** Automation before human attention is the key to reducing MTTR (Mean Time To Resolve). The system: (1) **Alert → Runbook mapper**: Every alert rule has a `runbook_url` annotation. When an alert fires, the Alertmanager sends a webhook to a "runbook automation engine" (e.g., a Kubernetes-native operator or a workflow engine like Temporal). The engine looks up the runbook for this alert type. (2) **Automated diagnostics** (run before paging): The engine executes a sequence of diagnostic actions: (a) Logs query: "fetch the last 5 minutes of error logs from the affected service" → summarize and attach to the incident. (b) Trace sampling: "increase trace sampling rate for this service to 100% for 10 minutes" → so when the engineer looks at the incident, they see full trace data. (c) Configuration check: "diff the current configmap against the last known good version" → detect recent deployments as potential cause. (d) Dependency check: "are downstream services (database, cache, queue) healthy?" → ping them and report status. (3) **Automated mitigation** (only if diagnostic confidence is high): For well-understood failure modes, automated mitigation can fire: (a) If a deployment just happened and error rate spiked, automatically rollback. (b) If the database connection pool is exhausted, kill idle connections or increase pool size. (c) If a specific pod is unhealthy, restart it (but only if the runbook explicitly allows this — and log the action). (4) **Human handoff**: After automated diagnostics and any safe mitigation, the system pages the on-call engineer with a pre-populated incident report containing: alert details, automated diagnostic results, timeline of changes (deployments, config changes), and the runbook link. The engineer acknowledges and decides next steps. (5) **Post-incident feedback**: After resolution, the runbook automation engine logs which automated actions helped and which were noise. If an automated action didn't help in 3 consecutive incidents of the same type, the engine suggests updating the runbook to remove or change it. (6) **Implementation pattern**: Use Alertmanager's `webhook_configs` to call a "RunbookExecutor" service (Go service, stateless). The executor submits jobs to a Temporal/P99 workflow queue. Each workflow step is a Kubernetes Job or a Lambda function. The side effect: must not use `--disable` or `--enable` flags on the prometheus operator — instead, use a separate reconciler that watches alert events. This prevents the runbook system from being a single point of failure for the monitoring itself.

## Q21: How do you correlate logs, metrics, and traces in a unified observability pipeline? What are the practical challenges at scale?

**Answer:** Correlation via common identifiers: inject trace_id and span_id into every log line, metric exemplar, and trace span. OpenTelemetry Collector receives all three signals, attaches resource attributes (host, service.name, k8s.pod_id), fans out to Mimir (metrics), Tempo (traces), Loki (logs). Challenges: (1) **Sampling mismatch**: traces sampled 1%, logs unsampled — metric spike in 99% unsampled population lacks traces. Need tail-based sampling driven by metric anomaly. (2) **Cardinality explosion**: attaching unique trace_id to every data point creates millions of series. Exemplars store only the last N per series. (3) **Storage cost**: full logs + traces + high-res metrics triples cost. Tier retention: metrics 12 months, traces 7 days, logs 30 days. (4) **Tool sprawl**: mixing Datadog (metrics) + Splunk (logs) + Honeycomb (traces) makes correlation manual — prefer unified stack.

## Q22: Design an OpenTelemetry Collector deployment for multi-region Kubernetes with 10,000 pods. Handle load shedding, retry, and backpressure.

**Answer:** Two-tier deployment: (1) **Agent Collector** (DaemonSet per node): scrapes pod metrics, tails logs, receives OTLP traces. Lightweight: add resource attributes, batch, compress, forward. (2) **Gateway Collector** (3-5 replicas per region): receives from agents, performs heavy processing (attribute enrichment, sampling, fan-out). (3) **Load shedding**: memory_limiter processor at 75% RAM — drops logs first, then traces, then metrics (smallest, most important). (4) **Backpressure**: batch processor accumulates 5 MB / 1s windows. If backend returns 429/503, buffer up to 5000 items, then block receiver — applies backpressure to application. (5) **Retry**: exponential backoff (500ms initial, 30s max, 0.2 jitter). Drop after 10 retries. (6) **Multi-region**: each region's gateway forwards to local backend. Global Thanos querier performs cross-region federation. If local backend down, fall back to secondary region's endpoint.

## Q23: How would you design a custom Prometheus exporter for a legacy app with proprietary binary protocol? Walk through instrumentation, caching, and performance.

**Answer:** (1) **Protocol parsing**: implement parser in Go with bufio.Scanner and custom split function. Parse into MetricPoint structs. (2) **Scrape lifecycle**: background goroutine polls legacy app every 15s, caches results in sync.Map. The /metrics handler renders from cache, not live. Decouples scrape timing from Prometheus scrape interval. (3) **Caching**: cache TTL = 2x poll interval. Use context.WithTimeout on poll to prevent hung app from starving exporter. (4) **Performance**: use prometheus.NewRegistry() (not global) to avoid lock contention. /metrics render should be <10ms for 10K metric families. Use gzip if client supports. (5) **Error handling**: expose up{target="legacy-app"} 0 gauge on failure. Return stale data + warning metric, not HTTP 500. (6) **Self-instrumentation**: expose exporter's own scrape_duration_seconds, last_success_timestamp. Prometheus alerts on exporter health.

## Q24: Design long-term metrics storage with Thanos vs. Cortex/Mimir for 10K aSPS, 2-year retention. Compare cost and architecture.

**Answer:** At 10K active series/sec, 2 years = ~630B data points. (1) **Thanos**: Prometheus sidecar uploads TSDB blocks to S3 every 2h. Thanos Store Gateway serves queries from object storage. Compactor downsamples (60s → 5m → 1h) and dedups. Single Prometheus maxes at ~2M active series — you must shard by namespace/team. (2) **Mimir**: ingesters receive via remote-write, replicate 3x, flush every 2h. Stateless ingesters — if one dies, hash ring rebalances. Mimir handles 100K+ aSPS. Query: query-frontend with parallelization and caching. (3) **Cost**: Prometheus alone: ~50 servers × $200 = $120K/yr. Thanos: same compute + S3 (~$500/mo after downsampling). Mimir: 10 ingesters × $400 = $48K/yr + S3. Mimir query cost lower — no scatter-gather across shards. Winner: Mimir for >10K aSPS and operational simplicity. Thanos if you already have Prometheus.

## Q25: Explain recording vs. alerting rules in depth. Common misconfigurations that cause incidents.

**Answer:** (1) **Recording rules**: pre-compute expensive PromQL, store as new series. Critical for reducing query latency, aggregating high-cardinality into low, building blocks for alerts. (2) **Alerting rules**: evaluate PromQL, fire if true for `for` duration. (3) **Common misconfigurations**: (a) Alerting on recording rules that change — updating expression restarts series, causing false negatives. Use `or absent()` for gap handling. (b) Too many recording rules — each rule creates new series. 1000 rules × 10 series = 10K added series. (c) For duration too short — `rate(http_errors[1m]) > 0.01 for: 30s` burns error budget too fast. Minimum: 5min warning, 2-5min critical. (d) Label loss in aggregation — `sum by (instance)` strips all other labels needed for routing. Include all needed labels in `by()` clause.

## Q26: Define an SLI for a complex multi-step checkout flow across 5 services and compute an SLO accounting for availability and latency algebra.

**Answer:** (1) **SLIs**: SLI_good = requests completed in <2s, SLI_total = all checkout requests. (2) **Dependency algebra**: A_total = A₁ × A₂ × ... × A₅ (product). For 5 services at 99.9%, checkout availability = 99.5%. Composite SLO cannot exceed 99.5%. (3) **Latency composition**: end-to-end latency is the sum of service latencies. The p99 of a sum is higher than the sum of p99s — use convolution of distributions from distributed traces. (4) **SLO computation**: target 99% in 2s over 30-day rolling window. Error budget = 0.01 × total_checkouts_30d. Burn rate alerting: 10x for 1h (fast burn), 4x for 6h (slow burn). (5) **Decomposition**: track per-service SLIs + flow SLIs. Attribute SLO burn to root cause: `burn_i = (1 - A_i_current) / (1 - A_i_target) × weight_i`.

## Q27: Design multi-cluster observability for 10 K8s clusters across 3 data centers. Maintain tenant isolation and single-pane-of-glass querying.

**Answer:** (1) **Per-cluster stack**: each K8s cluster runs Mimir (metrics), Loki (logs), Tempo (traces). Independently available — if central query layer down, teams can port-forward to local stack. (2) **Global querier**: central Grafana fans out queries to each cluster's store gateway. For metrics: Thanos Querier with all Store API endpoints (DNS SRV discovery). For logs: global querier fans out to each Loki. For traces: global trace ID index (Cassandra keyed by trace_id → cluster location). (3) **Tenant isolation**: each cluster is a Mimir tenant with separate S3 prefix. Auth proxy (OAuth2 + OPA) enforces Team A sees only Tenant A. (4) **Global metrics**: use "global recording rules" — Prometheus ruler queries all clusters and computes aggregates. (5) **Latency**: slowest cluster determines query latency. Set 10s timeout. Mimir supports partial results — return results with "partial: 2 clusters timed out" warning.

## Q28: Design a cardinality management system that prevents label explosions without losing useful debugging dimensions.

**Answer:** (1) **Cardinality budget per tenant**: 10K active series per service. In-memory counting Bloom filter tracks unique label combos per metric. At 80% → warning alert. At 100% → ingress gateway rejects new series with 429. (2) **Label allowlist/blocklist**: global registry of allowed metric names and label keys. user_id is NOT allowed on http_requests_total — must be exemplar. Block: request_id, session_id, timestamp. Mutating webhook in agent drops blocked labels. (3) **Label value length limit**: max 256 bytes. Truncate longer values with warning. (4) **Active series tracking**: alert when prometheus_tsdb_head_series exceeds 80% of per-instance max. (5) **Retroactive cleanup**: Thanos tools bucket cleanup or mimirtool backfill can remove series by label matcher. (6) **Audit trail**: log every rejected or truncated label for compliance.
