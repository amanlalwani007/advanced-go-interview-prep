# Chapter 23 — Design a Distributed Email Service (Gmail)

## Q1: Design the email storage system. How do you store billions of emails per user and support fast search, threading, and label-based queries?

**Answer:** Email storage at Gmail scale requires a custom storage layer — Gmail uses Bigtable (Google's wide-column store). (1) **Core table** — each email is a row keyed by `(user_id, message_id)`. Columns include: raw message (MIME blob), parsed metadata (sender, recipients, subject, date), labels (Inbox, Sent, Starred, custom labels), and read/unread state. (2) **Threading** — GMail groups messages into conversations using a thread ID. A thread is identified by the normalized subject line (after stripping Re:/Fwd:) and participants. Messages are stored indexed by `(user_id, thread_id)` for efficient conversation loading. (3) **Label/tag model** — Gmail doesn't use folders; every message has a set of labels. A message can be in multiple "folders" simultaneously (e.g., both "Inbox" and "Travel"). This is stored as a bitmap or a set column in Bigtable. Querying by label is an index lookup: `SELECT * FROM messages WHERE label CONTAINS 'Inbox'`. (4) **Storage optimization** — attachments are deduplicated across users. If 1M users receive the same attachment (a popular PDF), it's stored once in a blob store and referenced by hash. The email metadata table stores just the content hash. (5) **Caching** — the most recent 50 emails per user are cached in memory on the API server. This covers 90%+ of "check inbox" requests.

## Q2: How does Gmail handle full-text search over billions of emails with sub-second latency? Describe the indexing pipeline.

**Answer:** The search index is separate from the primary mail store. (1) **Index pipeline** — when an email arrives (or is modified), a background worker parses the text content, extracts terms, and writes to a per-user inverted index stored in a custom search engine (Google's internal search technology, similar to what powers Google Search). (2) **Index structure** — the inverted index maps `(term, user_id) → list of message_ids`. Terms are tokenized using: Unicode normalization, stemming (e.g., "running" → "run"), stop word removal (a, an, the), and phrase indexing (quoted phrases). (3) **Sharding** — the search index is sharded by `user_id` (hash range). Each shard is a separate Bigtable range, collocated with the user's mail data. This ensures search queries are single-shard operations (no scatter-gather). (4) **Query execution** — a search query `from:alice subject:meeting after:2024/01/01` is parsed into: `author_alice AND subject_meeting AND date >= 2024-01-01`. The system intersects the postings lists for each term, returning matching message IDs in ranking order. (5) **Ranking** — results are ranked by: recency (most recent first), relevance (TF-IDF on query terms), and user signals (how often the user interacts with messages from this sender). (6) **Latency** — the search index is optimized for sub-second queries. The system returns the top 10 results first (to populate the UI) while computing the full result set asynchronously.

## Q3: Design the email sending pipeline. How does Gmail handle sending 100M+ emails per day while ensuring deliverability (avoiding spam folders) and respecting rate limits?

**Answer:** The outbound pipeline: (1) **Outbound queue** — when a user clicks "Send", the message is placed in a per-user outbound queue (Kafka topic partitioned by sender domain). (2) **Rate limiting** — per-user: max 500 emails per day (Gmail's limit for free accounts), per domain: limits set by the receiving server (MX server). The rate limiter uses a sliding window counter in Redis. (3) **Spam classification** — before delivery, the outbound service runs the message through a spam classifier. Messages that match spam patterns (too many links, suspicious attachments, unusual sending patterns) are flagged. If flagged, the user sees a warning: "This message looks suspicious. Are you sure?" (4) **DKIM/SPF/DMARC signing** — the outbound service signs the message with DKIM (DomainKeys Identified Mail) using the sender domain's private key. The sending IP is checked against SPF (Sender Policy Framework) records. DMARC policy (published by the sender's domain) is respected. These three protocols determine whether the receiving server classifies the email as legitimate or spam. (5) **Delivery retry** — if the receiving server returns a transient error (450/451), the system retries with exponential backoff over 48 hours. Permanent errors (550/551) generate a bounce notification to the sender. (6) **Feedback loops** — Gmail subscribes to FBLs (Feedback Loops) from major ISPs. When a recipient marks an email as spam, the FBL informs Gmail, and the sender's sending reputation is adjusted. Reputation is per-sender-domain, per-IP.

## Q4: How does Gmail design the highly-available IMAP/POP3 frontend layer that handles 1B+ active connections?

**Answer:** The frontend layer uses a connection management architecture: (1) **Load balancer** — a global anycast IP (Google's GFE) routes each TCP connection to the nearest frontend server. (2) **Frontend server** — each frontend can handle 100K+ concurrent connections using an event-driven, non-blocking I/O model (epoll on Linux). Connections are HTTP/2 or custom long-lived TCP connections (not traditional IMAP/TCP per connection, to avoid per-connection overhead). (3) **Session state** — the frontend stores per-user connection state (which folders are open, which messages are visible, sync state) in a local in-memory cache. If the frontend fails, the client reconnects and state is reconstructed from the backend. (4) **Backend query** — the frontend proxies the request to the backend mail store (Bigtable). For IMAP IDLE (push notifications), the frontend subscribes to a Redis pub/sub channel for the user's email ID. When new mail arrives, the backend publishes a notification, the frontend receives it, and pushes it to the client over the open IMAP connection. (5) **Scaling** — connection-sticky load balancing ensures a user's persistent connection stays on the same frontend server. This is done via a consistent hash (client IP or session cookie) at the load balancer.

## Q5: How does Gmail's Smart Inbox (automatic categorization: Primary, Social, Promotions, Updates, Forums) work under the hood?

**Answer:** The categorization is an ML model: (1) **Feature extraction** — for each incoming message, extract features: sender domain (newsletter@mail.example.com vs. john@gmail.com), message headers (List-Unsubscribe presence → likely promotion/unsubscribe is present → promotional), email content (presence of "unsubscribe" link, HTML-to-text ratio, image-to-text ratio), user interaction history (has the user ever clicked links in this sender's emails?). (2) **Classification model** — a gradient-boosted decision tree or neural network trained on millions of labeled examples. Classes: Primary (personal, no bulk), Social (Facebook, Twitter, LinkedIn), Promotions (marketing, deals), Updates (receipts, shipping confirmations, bills), Forums (mailing lists, discussion groups). (3) **Training data** — users can manually drag messages between tabs. Each manual reclassification is a training signal. Google also uses crowd-sourced labeling (anonymized) to bootstrap the model. (4) **Inference** — the model runs on every incoming message, outputting a probability distribution over categories. The message is placed in the highest-probability category. Inference latency must be <50 ms to not delay email delivery. (5) **Personalization** — the model is fine-tuned per user based on their reclassification history. If a user always moves a specific sender's emails from Promotions to Primary, the model learns this and starts routing them to Primary directly.

## Q6: Design the attachment storage and preview system. How does Google handle 50 MB attachments and previewing images, PDFs, and docs inline without downloading the whole file?

**Answer:** (1) **Upload** — attachments are uploaded directly to GCS (Google Cloud Storage) via resumable upload sessions. The email metadata stores the content hash and GCS object path. (2) **Virus scanning** — each upload triggers a ClamAV scan (or Google's internal virus scanning service). Infected attachments are quarantined and the user is alerted. (3) **Preview generation** — when an email with an attachment arrives, a preview pipeline is triggered: images are compressed to a thumbnail (using ImageMagick or a custom WebP transcoder); PDFs/Docs are converted to HTML (libreoffice or Google Docs conversion engine); videos have a preview thumbnail extracted. (4) **Storage tiers** — raw attachments go to nearline storage (GCS Nearline for attachments < 30 days old) and archive storage (GCS Archive for older). Previews are cached in a CDN (edge cache). (5) **Inline viewing** — the Gmail client displays previews by loading the converted HTML or the image thumbnail from the CDN, not the original 50 MB attachment. The original file is only downloaded if the user clicks "Download" or accesses it offline. (6) **Size limits** — Gmail's max attachment size is 25 MB (for incoming) and 25 MB for outgoing (via Google Drive links for larger files). The 25 MB limit is an SMTP constraint; larger files are uploaded to Drive and a link is sent.

## Q7: How does Gmail detect and prevent phishing attacks in real-time?

**Answer:** Multi-layered phishing detection: (1) **URL scanning** — every link in an incoming email is rewritten through Google's Safe Browsing proxy server. When clicked, the user's browser checks the URL against Google's constantly updated list of known phishing/malware sites. If the site is dangerous, the user sees a full-screen red warning. (2) **ML classifier** — each email is scored by a phishing ML model that looks for: display-name spoofing (the sender is "PayPal" but the email is from attacker@gmail.com), urgency language ("Your account will be closed in 24 hours"), mismatched reply-to addresses, and known phishing domain patterns. (3) **DMARC enforcement** — for high-value targets (executives, finance teams), Gmail enforces the strictest DMARC policy: if an email fails SPF and DKIM authentication and the sender domain has a `p=reject` DMARC policy, the email is rejected entirely (not even sent to spam). (4) **User reporting** — if a user clicks "Report phishing", the email is forwarded to Gmail's abuse analysis team. This feedback trains the ML model and the suspicious URLs are added to Safe Browsing. (5) **Tenant isolation** — Google Workspace administrators can set up email authentication controls: only allow emails from specific IP ranges, block emails with specific file types (e.g., executables), and require TLS encryption for incoming traffic from specific domains.

## Q8: Design offline access in Gmail. How does the Gmail offline Chrome app cache emails locally and sync changes when connectivity returns?

**Answer:** The Gmail Offline Chrome app uses: (1) **Local storage** — uses IndexedDB (in-browser NoSQL store) to cache email metadata and recent email bodies. The cache is encrypted at rest using the user's Google Account credentials. (2) **Sync state** — the local store tracks the state of each message (read/unread, starred, label assignments) with a `local_version` and `server_version`. (3) **Outgoing queue** — emails composed offline are stored in a local outbox. They are submitted as pending with a local timestamp. (4) **Online sync** — when connectivity is restored, the client sends the server its last known sync cursor. The server responds with the delta of changes (new messages, label changes, deletions) since that cursor. The client applies these changes to the local IndexedDB. Then, the client pushes any changes made offline (composed emails, star toggles, label changes). Conflicts (e.g., a message was deleted on the server while the user starred it offline) are resolved: server changes take precedence for deletions; local changes for labels are applied if the message still exists. (5) **Sync strategy** — full resync is expensive. Gmail uses incremental sync with pagination. The first sync downloads only headers and snippet (not full message bodies). Full message bodies are lazily downloaded when the user opens a message.

## Q9: How does Gmail handle the migration from the legacy "Inbox by Gmail" feature set (bundles, highlights, snooze) into the main Gmail app?

**Answer:** Feature migration at Gmail's scale requires backward compatibility and gradual rollout: (1) **Feature flagging** — every feature is behind a flag. An A/A/B test approach: start with 1% of users, verify stability and UX metrics, ramp to 10%, then 50%, then 100%. (2) **Backward compatibility** — all new features (snooze, smart replies, nudges) are implemented as additions to the existing Gmail data model. Snooze: a `snooze_until` timestamp field is added to the message metadata. The existing "unread" and "inbox" labels remain unchanged. If a user snoozes an email, it's removed from Inbox and a cron job returns it when `snooze_until` elapses. (3) **Storage migration** — adding new fields (snooze timestamp, bundle ID, highlight tags) to a Bigtable schema with billions of rows is a non-trivial migration. Google's approach: use a "column family" addition in Bigtable — new writes get the new field; old rows are backfilled asynchronously (a MapReduce job that touches every row over weeks). Queries handle missing fields gracefully (default values). (4) **Client migration** — the Gmail web client is a single-page application (Angular/Dart compiled to JS). Features are loaded as modules. The new UI (Inbox features) is a different set of React components toggled by the feature flag. Users can switch between "New Gmail" and "Classic Gmail" during the transition period.

## Q10: Design the spam filter. How does Gmail achieve >99.9% spam detection accuracy with <0.05% false positive rate?

**Answer:** Gmail's spam filter is a Bayesian classifier augmented with deep learning: (1) **Content analysis** — extract features from email body and headers: word frequencies, character n-grams (catch obfuscated words like "V1@gra"), HTML structure, image-to-text ratio. (2) **Sender reputation** — each sending domain and IP has a reputation score based on: historical spam reports, volume of email sent, authentication (SPF/DKIM/DMARC) pass rate, and complaints from FBLs. Unknown senders start with neutral reputation and earn trust (or distrust) over time. (3) **User filters** — users can create custom filters and whitelist/blacklist senders. Whitelist is absolute (never go to spam). (4) **Deep learning model** — Google uses a neural network (BERT-based or similar) trained on billions of emails. The model understands context, not just keywords. It can detect sophisticated phishing emails that use proper grammar and brand logos but have a malicious intent. (5) **False positive protection** — Gmail automatically moves legitimate mail out of spam if the user interacts with it (opens, replies, clicks links). An email classified as spam but that matches a user's contact list or has been in the user's previous inbox (sender has emailed before) is given a lower spam score. (6) **Feedback loop** — when users mark emails as "Not spam" or "Report spam", these signals are fed back into the model training pipeline. Google aggregates these signals across all users (anonymized) to continuously improve the global model. (7) **Adversarial robustness** — spammers constantly evolve. Google's model is adversarially trained on known spammer techniques (e.g., replacing characters with Unicode lookalikes, embedding text in images). The system also uses Gmail's domain-wide reputation: if a domain suddenly starts sending email after being dormant for years, it's treated with suspicion.

## Q11: Design an SMTP outbound delivery retry engine that handles 10M+ emails per hour. What queue architecture, backoff strategy, and bounce classification do you use?

**Answer:** At 10M emails/hour (~2,778/sec), the retry engine must handle transient failures without losing messages or getting blacklisted by ISPs. Architecture: (1) **Queue architecture**: Use a multi-tiered queue. Tier 1 — **Priority queue** (Redis sorted sets): immediate retries for soft bounces (4xx errors). Messages here expire in 15 minutes. Tier 2 — **Delayed queue** (Kafka with per-partition delivery timers): deferred retries for throttled messages (e.g., "too many connections" from Yahoo). Each message has a `next_attempt_at` timestamp. Tier 3 — **Dead letter queue** (DLQ): messages that have exhausted all retry attempts (48 hours). (2) **Backoff strategy**: Exponential backoff with jitter. First retry: 30 seconds. Then: 2 min, 5 min, 15 min, 30 min, 1 hour, 2 hours, 4 hours, 8 hours, 12 hours, 24 hours. After 24 hours: retry every 4 hours up to 48 hours total. The jitter (random ±25% of interval) prevents the "thundering herd" — when a major ISP (e.g., Gmail) recovers after an outage, you don't want all senders retrying simultaneously. (3) **Bounce classification**: Each SMTP response is classified: (a) **Hard bounce** (5xx permanent): `550 5.1.1 User unknown` — remove from list immediately, no retry. (b) **Soft bounce** (4xx transient): `451 4.3.0 Temporary server error` — retry with backoff. (c) **Throttle** (4xx with rate limit indication): `421 4.2.1 Too many messages` — retry with longer backoff AND reduce sending rate to that domain. (d) **Content rejection**: `550 5.7.1 Message rejected due to spam` — this is often a false positive. Retry once after 24 hours (the ISP's reputation algorithm may have re-evaluated). If it bounces again, classify as hard bounce. (4) **Per-domain rate limiting**: Track sending rate per recipient domain using a sliding window counter in Redis. For each domain MX (e.g., yahoo.com), max ~10-20 emails per connection. If the MX returns a throttle response, reduce the per-domain rate by 50% for 1 hour. The rate limiter prevents IP blacklisting. (5) **Connection pooling**: Maintain persistent SMTP connections to each MX server to avoid TCP handshake latency. A connection pool (per domain) with min 2, max 20 connections. If a connection returns a 5xx error, close it. If 4xx, keep it warm but reduce its usage. (6) **Multi-IP reputation management**: Sending from multiple IPs spreads reputation risk. A dedicated "sending IP" pool per email stream (transactional vs. marketing). If one IP gets blacklisted (monitored via DNSBL lookups), quarantine that IP and route all traffic through healthy IPs until the blacklist clears.

## Q12: Walk through the MX server discovery and prioritization logic. How does a distributed email system select which MX server to connect to, and how does it handle failures and priority weightings?

**Answer:** MX server discovery is defined by RFC 5321, but at production scale the naive implementation fails. (1) **DNS MX record resolution**: Query DNS for the recipient domain's MX records. Example: `gmail.com` MX records: `gmail-smtp-in.l.google.com. 10`, `alt1.gmail-smtp-in.l.google.com. 20`, `alt2.gmail-smtp-in.l.google.com. 30`, `alt3.gmail-smtp-in.l.google.com. 40`, `alt4.gmail-smtp-in.l.google.com. 50`. Lower priority number = higher preference. The sending MTA should try priority 10 first. (2) **RFC-compliant selection vs. production reality**: RFC says: sort by priority, try the lowest, if it fails, try the next. Production: (a) Round-robin across equal-priority servers (Gmail has 5 servers at priority 10 but they all go to the same IP pool — connecting to any works). (b) Health-check: before sending, probe each MX server with a quick TCP connect to port 25. Cache the health status per MX for 30 seconds. Avoid connecting to a known-dead MX. (c) Latency-aware routing: if `gmail-smtp-in.l.google.com` has 200 ms latency from your us-east-1 datacenter, but `alt1` has 50 ms latency (geo-routed to nearby POP), prefer the lower-latency server even if its priority is higher (less favorable). (3) **Fallback logic**: After 3 failed attempts to the first-priority MX (timeout or connection refused), try the next priority. If all MX servers fail for a domain, the message goes to the delayed queue and retries in 15 minutes. Track "domain reachability" — if a domain has been unreachable for >1 hour, stop attempting deliveries to that domain and batch retries to avoid wasting connections. (4) **Connection caching**: Reuse SMTP connections across messages to the same domain. A connection pool per MX with keepalive is critical — establishing a TCP+TLS SMTP connection takes 2-3 round trips (~200-600 ms). For 10M emails/hour, connection reuse reduces latency by 10×. (5) **IPv6 handling**: Check if the sending and receiving servers support IPv6. MX records may return both AAAA (IPv6) and A (IPv4) records. Happy Eyeballs (RFC 8305): try IPv6 first, fall back to IPv4 in 250 ms if IPv6 connection fails. Many ISPs have IPv6 blacklists that are less sophisticated than IPv4 — using IPv6 can improve deliverability. (6) **Monitoring**: Alert if the average MX resolution time exceeds 500 ms, or if any domain's MX fails health check for >5 minutes (could be a DNS propagation issue or a DDoS on the receiving side).

## Q13: Explain the full chain of email authentication (SPF, DKIM, DMARC) and what happens at the receiving server when each check fails. How do you design a system that doesn't drop legitimate email due to misconfigured authentication?

**Answer:** The three authentication protocols work together to prove an email is legitimate. (1) **SPF (Sender Policy Framework)**: The sending domain publishes a TXT record listing IPs authorized to send mail for that domain. Example: `v=spf1 ip4:192.0.2.0/24 include:_spf.google.com ~all`. At delivery: receiving server looks up SPF record for `envelope-from` domain, checks if the connecting IP is in the authorized list. Result: Pass (IP matches `all`), SoftFail (`~all`, suspicious but not reject), Fail (`-all`, reject), Neutral (`?all`, no policy), None (no SPF record). (2) **DKIM (DomainKeys Identified Mail)**: The sending server signs the email with a private key. The signature is added as a `DKIM-Signature` header. The receiving server looks up the public key from the domain's DNS at `<selector>._domainkey.example.com`. It verifies the signature against the email body and select headers (From, Subject, Date). If the body was modified in transit (e.g., a mailing list added a footer), the signature breaks. Solution: ARC (Authenticated Received Chain) preserves authentication results across forwarding hops. (3) **DMARC (Domain-based Message Authentication, Reporting, and Conformance)**: The sending domain publishes a policy: `v=DMARC1; p=quarantine; rua=mailto:dmarc@example.com`. The receiving server checks SPF alignment (domain in `From:` header matches SPF-authenticated domain) and DKIM alignment (domain in `From:` header matches DKIM `d=` domain). If both fail DMARC: action based on `p=none` (monitor, don't change delivery), `p=quarantine` (send to spam), `p=reject` (reject outright). (4) **Failure handling at the receiving server**: (a) SPF fail but DKIM pass -> DMARC may still pass if DKIM domain aligns with From domain. If DMARC passes, deliver normally. If DMARC fails, apply policy. (b) DKIM fail (signature broken by a mailing list) but SPF pass -> SPF alignment probably fails (mailing list rewrites envelope sender). DMARC likely fails. Solution: ARC (Authenticated Received Chain) preserves the original authentication status. If ARC is valid, the server may accept the email even though current SPF/DKIM fail. (c) All three fail -> spam score increases significantly. (5) **Preventing false positives for misconfigured senders**: (a) **DMARC `p=none`** — the sender domain is in "monitor" mode. The receiving server should report failures to the `rua` address but deliver normally. (b) **SPF neutral/softfail** — these are not hard failures. The server should still accept but may add a spam penalty. (c) **DNS caching challenges**: SPF and DKIM records have TTLs of 24-48 hours. If a sender changes their infrastructure (IPs, selectors), old records persist in cache for up to the TTL. The receiving server should cache SPF/DKIM results for at most 1 hour, not the DNS TTL, otherwise legitimate emails from new IPs get rejected. (6) **Feedback loop**: DMARC aggregate reports (rua) are sent to the domain owner daily, showing: total messages, SPF/DKIM pass rates, and IPs sending on behalf of the domain. The domain owner uses this to detect unauthorized senders (spoofing) and misconfigured legitimate senders.

## Q14: Design an attachment scanning pipeline that processes 10K attachments per second. How do you handle evasive malware, encrypted files, and file-type mismatch attacks?

**Answer:** Attachment scanning at 10K/sec requires a multi-engine pipeline with dynamic analysis. (1) **Pipeline stages**: Stage 1 — **File type detection** (magic bytes, not extension). Mismatch detection: a file claiming to be `.jpg` but with magic bytes `MZ` (Windows executable) is blocked. Stage 2 — **Static analysis**: (a) Hash matching against known malware hash databases (ClamAV, YARA rules). (b) Entropy analysis — packed executables have high entropy → sandbox. (c) String extraction — search for suspicious patterns (PowerShell commands, URLs to known C2 servers). Stage 3 — **Dynamic analysis (sandbox)**: execute the file in an isolated sandbox (Firejail or Cuckoo) for 5 seconds. Monitor: file system writes, network connections, process spawning. If the sample opens a network connection to a known malicious IP, it's malware. (2) **Encrypted files**: Password-protected ZIPs and encrypted PDFs are opaque — the scanner cannot inspect content. Policy options: (a) Block all encrypted attachments (most common — Gmail blocks encrypted ZIPs). (b) Require the password to be sent via a separate channel (SMS). (c) Allow with a warning: "This attachment is encrypted — we cannot verify it's safe." Implementation: if the file type is ZIP and the central directory indicates encryption (`bit 1` of general purpose bit flag is set), reject. (3) **File-type mismatch attacks**: A `.doc` file that is actually a JavaScript dropper (magic bytes `<?xml` for Office Open XML, but the file executes a script). Detection: (a) Parse the file according to its declared format — if it fails to parse and contains executable code, it's suspicious. (b) If a file declares as a PDF but the PDF parser encounters JavaScript (PDFs can embed JS), flag it and sandbox. (c) Use `file` command + MIME type detection — if they disagree, increase suspicion score. (4) **Performance at 10K/sec**: (a) Static analysis (hash + YARA) takes ~100 µs per file — can handle 10K/sec on a single server. (b) Sandbox is expensive (~2-5 seconds per file). Only sandbox if static analysis yields a score above a threshold. With 1% of files being sandboxed, you need ~100 sandbox instances (5 sec / 0.01 × 10K/sec = 2,000 concurrent sandboxes — expensive). Optimize: use machine learning to predict which files are likely malicious: if the model says "benign with 99.9% confidence", skip the sandbox. (5) **Dead-letter handling**: Files flagged as malicious are quarantined (stored in an encrypted bucket, not delivered). A notification is sent to the recipient: "An attachment was removed from this message because it appears unsafe." The sender can appeal by clicking a link (generates a support ticket for manual review by a security analyst). (6) **Continuous model retraining**: New malware variants appear daily. The scanner updates YARA rules every hour from threat intelligence feeds (VirusTotal, AlienVault OTX). The sandbox output (network connections, file writes) is fed back into the ML model training pipeline, which retrains daily.

## Q15: Explain the JMAP email threading algorithm (RFC 8621) in detail. How does a distributed email service group messages into conversations when messages arrive out of order or across different threads with the same subject?

**Answer:** Email threading is surprisingly complex because the data (Subject headers, References/In-Reply-To headers) is unreliable. (1) **RFC 8621 threading model**: JMAP defines a `Thread` object — a set of `Email` objects that belong to the same conversation. Two messages are in the same thread if one is a reply to the other (via `In-Reply-To` or `References` headers), or they share a common ancestor. (2) **Header-based linking**: The primary signal is the `In-Reply-To` header (the Message-ID of the email being replied to) and the `References` header (a list of Message-IDs forming the chain). Algorithm: maintain a map `message_id -> thread_id`. When a new message arrives with `In-Reply-To: <abc@example.com>`: look up `<abc@example.com>` in the map. If found, assign the same `thread_id`. If not found (message arrived out of order, or the parent is in a different folder), create a new thread_id and merge later when the parent arrives. (3) **Subject-based fallback**: When headers are missing (e.g., the client didn't include `In-Reply-To`), fall back to subject normalization: strip `Re:`, `Fwd:`, `Aw:` (and localized equivalents) from the subject, lower-case, remove whitespace. If the normalized subject matches an existing thread's subject, and the message is from/near the same participants, add to that thread. This handles email clients that strip threading headers. (4) **Out-of-order delivery**: Emails can arrive with a future `References` (parent not yet delivered). Solution: (a) Lazy threading — assign a provisional thread_id on arrival. When the parent eventually arrives, merge the two threads. (b) Thread merge is a heavy operation (must update all messages in both threads). Implement thread merge as an async background job (Kafka consumer) with a 5-second debounce — if multiple merges for the same thread arrive in quick succession, batch them. (c) The merge must also update the user's search index — re-index all messages in the merged threads. (5) **Thread trimming**: Very long threads (100+ messages) are expensive to load. Gmail trims threads: only show the last 50 messages and a "View full thread" link. The thread representation in the storage layer stores the first and last message timestamps, message count, and a list of participant emails (deduplicated). (6) **Cross-folder threads**: A thread can span multiple labels/folders (e.g., some messages in Inbox, some in Archived). The storage model stores thread_id as a first-class field in the message metadata, separate from labels. When querying by label, the system returns all messages matching the label, not the entire thread. The client reassembles the thread by querying all messages with the same thread_id in the same user account.

## Q16: Design a push notification delivery system for a modern email service. Compare IMAP IDLE vs. WebPush vs. proprietary push protocols. How do you handle 1B+ persistent connections?

**Answer:** Email push notifications are critical for mobile — users expect sub-second notification when a new email arrives. (1) **IMAP IDLE** (RFC 2177): The client opens a TCP connection to the IMAP server and sends `IDLE`. The server holds the connection open. When new mail arrives for that user, the server sends an `EXISTS` response, the client exits IDLE and fetches the new message. Pros: Standard protocol, supported by all email clients. Cons: (a) Each IDLE connection consumes 1 goroutine/thread on the server — at 1B users, you'd need 1B concurrent threads which is impossible. (b) TCP keepalives burn bandwidth. (c) IDLE only works for one folder (usually INBOX). (2) **WebPush** (RFC 8030): The email client (web or mobile) subscribes to push via the browser's Push API or Firebase Cloud Messaging (Android) / APNs (iOS). The subscription is registered with the email service. When a new email arrives, the server sends a push notification via the platform push service. Pros: (a) Scales trivially — the push platform handles 1B connections (Apple/Google already do this). (b) Battery efficient — no persistent TCP to email server. Cons: (a) Cannot deliver email content — push messages are limited to ~4KB (just the subject line). (b) Delivery is best-effort (no guarantee). (c) Platform push services may throttle or delay notifications. (3) **Proprietary persistent connection** (Google's approach): Google uses a custom binary protocol over a persistent TCP/TLS connection (part of Google Play Services on Android). Each connection handles multiple services (Gmail, Calendar, Drive) over the same socket. Key optimizations: (a) Multiplexing — one connection per device carries push for all Google services. (b) Compression — protobuf-based, small wire footprint. (c) Reconnection — automatic reconnect with exponential backoff, state sync on reconnect (don't lose notifications). (4) **At 1B+ connections**: You cannot have a traditional threaded server. Use an event-loop architecture (epoll in C++ or Go's netpoller). Each connection is a file descriptor in the epoll set, not a thread. Google's push infrastructure runs on thousands of servers, each handling ~500K connections. (5) **Notification routing**: When a new email arrives for user X: (a) Look up user X's active push channels (could be multiple: mobile app + web app + desktop client). (b) Fan out the notification to all channels. (c) Each channel server maintains a local map of `user_id -> connection`. Write the notification to each connection's send buffer. (d) The epoll loop writes the notification to the TCP socket. (6) **Delivery guarantees**: Push is "fire and forget" for real-time. If the device is offline, the push platform queues the notification for a limited time (e.g., Android: 28 days max storage, prioritizes newest). The email service must also fetch new mail when the app comes to foreground (the "sync on open" pattern) to avoid missing notifications.

## Q17: Design a quarantine and remediation workflow for a corporate email service. When an email is flagged as malicious, what automated remediation actions can the system take without user intervention?

**Answer:** When an email is flagged as a confirmed threat (e.g., phishing, malware), the clock is ticking — every minute it stays in the user's inbox increases the chance the user clicks the link. Architecture: (1) **Quarantine trigger**: The flag comes from: (a) post-delivery analysis (a sandbox finishes 30 seconds after delivery and identifies malware); (b) user report ("Report phishing"); (c) threat intelligence feed (a URL in the email is now known as a phishing site). All triggers converge on a "remediation queue" (Kafka topic). (2) **Automated remediation actions** (no human approval needed): (a) **Move to quarantine folder** — use the storage layer to change the message's label from "INBOX" to "QUARANTINE". The user is notified: "This message was moved to quarantine because it contained a suspicious link." (b) **Hard delete from all folders** — for confirmed malware (virus detected), delete the message from all labels (Inbox, Archive, All Mail). Add a tombstone so the message cannot be resurrected via undo. (c) **URL blocking** — extract all URLs from the malicious email and add them to the corporate URL blocklist. Future clicks on any of these URLs (even from different emails) will show a warning page. (d) **Sender block** — if the sender's IP/domain is confirmed malicious, add to the blocklist for all users. Future emails from this sender are rejected at SMTP time (550 error). (3) **Bulk remediation**: If the threat affects 1M+ users (e.g., a common phishing template sent to all employees of a company), the system cannot process each email individually. Use a bulk label operation: "remove `INBOX` label from all messages matching `sender_domain=evil.com AND received_time > 2024-01-01T00:00:00Z`". This runs as a distributed MapReduce job across the Bigtable rows (thousands of tablets scanned in parallel). (4) **Undo capability**: If the flag was a false positive, the administrator can "release from quarantine" — the system restores the original labels (snapshot taken when quarantine action was applied). The URL blocklist entry is removed. The sender blocklist may be downgraded to a watchlist. (5) **Audit trail**: Every quarantine action is logged: `{message_id, user_id, trigger (sandbox/user-report/feed), action (quarantine/delete/block), timestamp, admin_id (if manually released)}`. This log must be immutable and retained for 1+ years. (6) **Rate-limited escalation**: If the same user receives >3 quarantine alerts in 1 hour, there's a coordinated attack. Escalate to the security operations team (automated ticket in Jira). The system pauses automated deletion for that user and requires human review.

## Q18: How does a distributed email service handle email aliases and plus-addressing (e.g., user+tag@gmail.com)? Design the storage, routing, and filtering implications.

**Answer:** Email aliases and plus-addressing allow a single mailbox to receive mail under multiple addresses. (1) **Plus-addressing** (RFC 5233): Gmail and many providers treat `user+tag@gmail.com` as delivering to `user@gmail.com`. The `+tag` part is passed in the `Delivered-To` header but the recipient is the base user. Implementation: (a) During SMTP reception, the receiving MTA checks if the local part contains `+`. If so, it strips the tag, looks up the base user, and delivers to their mailbox. (b) The tag is stored as a label in the message metadata: `delivered_to = "user+tag@gmail.com"`. This allows filters to match on the specific alias. (2) **Custom domain aliases** (Google Workspace): A user can have `alice@company.com` and `alice.sales@company.com` both delivered to the same inbox. Storage: a mapping table `alias @ domain → canonical_user_id`. When an email arrives for `alice.sales@company.com`, the storage layer writes the message under `canonical_user_id` and also indexes the alias so replies appear to come from the alias. (3) **Distribution list aliases** (`team@company.com` → 20 users): The MTA must expand the alias — look up the list membership, split the message into 20 copies, deliver to each. Performance: expansion must happen before the SMTP `250 OK` response — for a list of 500 users, this can take seconds. Solution: (a) For small lists (<100), expand inline. (b) For large lists, accept the message, queue it, and expand asynchronously. The SMTP session returns `250 OK` immediately. The expanded copies are delivered in the background. (4) **Filtering and routing**: Users can set up filters based on the `delivered_to` address. A user who uses `user+shopping@domain.com` can create a filter: "if delivered_to contains `+shopping`, auto-label `Shopping` and skip Inbox." This filter must be evaluated after the alias resolution but before the final label assignment. Implementation: a per-user Sieve filter (RFC 5228) engine that processes each incoming message with the `delivered_to` address as a variable. (5) **Reply consistency**: When replying to an email sent to `user+tag@gmail.com`, the `From:` header should show `user+tag@gmail.com` (not `user@gmail.com`). This is handled by storing the `reply_to` address separately: the message metadata includes `original_delivered_to = "user+tag@gmail.com"`. The compose UI reads this field and uses it as the `From:` address. (6) **Anti-abuse**: Plus-addressing can be abused for spamming — spammers guess `common.name+amazon`, `common.name+bank`, etc. The service should not silently drop unknown aliases. If the alias doesn't exist (e.g., `user+randomstring@gmail.com` where the user has no filter for that alias), deliver normally to Inbox. The user can create a filter to auto-delete unknown aliases if they get spam.

## Q19: Design a migration engine that moves 100M users (500M messages each) from a legacy IMAP mail store to a modern API-based email platform with zero downtime and no data loss.

**Answer:** Migrating 100M users with 500M messages each (~50 exabytes total) is a multi-year project. Strategy: (1) **Dual-write + backfill** (the only approach that works at scale): Phase 1 — enable dual-write: all new email delivery is written to both the legacy store and the new store. This ensures new data exists in both systems. Phase 2 — backfill historical data: copy messages from legacy to new in the background. Phase 3 — cutover: when backfill is complete, route reads to the new store. (2) **Backfill pipeline**: A distributed scanner (Spark or custom Go workers) reads messages from the legacy IMAP store in batches (1000 messages per batch, sorted by timestamp). Each batch is written to the new API store (Bigtable or similar). The pipeline runs continuously, processing ~1M users per day (adjusting parallelism). Progress: each user has a migration status: `pending`, `in_progress`, `completed`, `failed`. (3) **Consistency verification**: After backfill, verify that each user's message count, label assignments, and last 10 messages match between legacy and new store. Run a spot-check: fetch a random sample of 100 messages per user and compare checksums (SHA256 of the raw MIME blob). If any discrepancy, re-migrate that user's batch. (4) **Zero-downtime cutover**: (a) The API gateway has a rule: "if user is in cohort A, route to new backend; if cohort B, route to legacy." (b) Migrate users in batches of 10K (small enough to revert if issues). (c) Before cutover: verify dual-write is working and backfill is complete for the batch. (d) At cutover: update the API gateway routing table for the batch. Existing users' sessions must be drained gracefully — no active connection to legacy store for this user is forcibly killed; instead, session tokens are set to expire in 5 minutes, and new sessions go to new store. (e) After cutover: monitor latency, error rate, and user complaints for 24 hours. If error rate > 0.1%, revert the batch (route back to legacy). (5) **Label/folder mapping**: IMAP folders map to Gmail-style labels. The mapping must be lossless. Challenges: (a) IMAP allows sub-folders (e.g., `INBOX/Work/Important`). The new system must flatten this into labels (`Work/Important`). (b) IMAP has no "Archive" concept — messages not in Inbox but still existing must be preserved. Map `[Gmail]/All Mail` to the `All` label in the new system. (c) IMAP flags (Seen, Flagged, Draft) map to read/unread, starred, and draft labels. (6) **Rollback strategy**: If the new store has a catastrophic failure, the system must fail over to the legacy store. The API gateway maintains a circuit breaker: if the new store's error rate exceeds 5% for 1 minute, route ALL traffic back to legacy. The dual-write must continue during rollback (both stores receiving writes) to prevent data loss during the fallback period.

## Q20: Design the anti-spam ML model retraining pipeline that runs continuously to adapt to evolving spammer techniques while maintaining <0.05% false positive rate.

**Answer:** Spammers evolve daily — a static model becomes stale within weeks. A continuous retraining pipeline is essential. (1) **Data collection and labeling pipeline**: (a) Incoming emails are sampled at 1% (representative across all users) and stored in a raw event store (Kafka → Parquet in S3). (b) Labels come from: user actions (mark as spam, move to inbox, report phishing); automated filters (DMARC failures, known malicious URLs, sandbox results); and manual review (a team of analysts reviews a random sample of 0.1% of emails daily). (c) Label conflicts: if user moves an email to inbox but the automated filter flagged it as spam, the user's action takes precedence (the user is always right for their own mailbox). (2) **Feature engineering pipeline**: Features are extracted from the raw email and stored as feature vectors in a columnar format (TFRecord or Parquet). Features: (a) Headers: authentication results (SPF/DKIM/DMARC pass/fail, alignment), sender reputation score (from a separate service), number of recipients, X-Mailer, received path (number of hops, IP reputations). (b) Body: word frequency (bag-of-words with TF-IDF), n-gram presence, HTML-to-text ratio, presence of forms (password inputs), URL domains and their reputations, image entropy (text embedded in images to evade text-based filters). (c) Behavioral: sending IP's sending velocity (emails/hour), sending domain's age (newly registered domains are suspicious), recipient's interaction history with sender (has the user ever emailed this sender before?). (d) Temporal: time since user created account, time of day (spam often arrives at specific hours from botnets). (3) **Model architecture and training**: (a) Base model: a transformer-based model (like BERTweet or a custom lightweight transformer) that reads the email body as token sequences. This captures sophisticated phishing language patterns. (b) Ensemble with gradient-boosted trees (LightGBM) on hand-crafted features (sender reputation, authentication results, behavioral features). The final prediction is a weighted average of transformer score and tree score. (c) Training: weekly full retrain (30 days of data, ~30M labeled emails on a large ML cluster with GPUs). Incremental update daily: the tree model is updated with new samples (incremental training via warm start); the transformer model is fine-tuned once a week (full retrain is expensive). (d) Threshold tuning: after each retrain, run a validation set (1 week of held-out data) and pick the threshold that achieves <0.05% false positive rate while maximizing recall. The threshold is usually between 0.7-0.9 for spam probability. (4) **Adversarial robustness**: (a) Spammers try to evade the model by slightly modifying their patterns. Use adversarial training: synthetically generate spam emails with common evasion techniques (character substitution, homoglyphs, text in images) and include them in the training set. (b) Monitor for "model stealing" — if a specific sender's emails are always classified as spam (100% time), they may be reverse-engineering the model. Vary the model's behavior by adding random noise (±0.01 to the output score for 1% of emails) so the model behaves as a black box. (5) **A/B testing for model updates**: A new model version is first evaluated offline against the held-out test set. Then deployed as a shadow model: score emails in parallel with the production model but don't use its decisions. If the shadow model's false positive rate is lower and recall is higher for 24 hours, it becomes the new production model. (6) **Drift detection**: Monitor the distribution of model scores over time. If the average spam probability for all emails shifts by more than 5% in a single day, trigger an alert — a new spam campaign may be evading detection or the email landscape has changed. This triggers an out-of-band retraining cycle.

## Q21: Design the attachment handling system for a large-scale email service. How do you store, scan, thumbnail, and serve attachments with billions of files, while handling deduplication, size limits, and format conversion?

**Answer:** Email attachment storage at Gmail scale is essentially an object store with processing pipeline. (1) **Storage architecture**: (a) Attachments are stored separately from email metadata in a blob store (GCS/S3). (b) Key: `SHA256(content)`. This provides content-addressable storage and automatic deduplication — if 1M users receive the same kitten photo, it's stored once. (c) Metadata table: `attachments (hash, size, mime_type, upload_timestamp, virus_scan_status, thumbnail_paths)`. (2) **Upload pipeline**: (a) When a user sends an email with attachments, the MTA splits the MIME multipart message and extracts each attachment. (b) The attachment is streamed to the blob store in 8MB chunks (S3 multipart upload). (c) After upload, a pipeline triggers: virus scan → thumbnail generation → content classification → dedup (if another email with same hash exists, reference existing). (d) The email body references the attachment by hash: `email_attachments (email_id, attachment_hash, filename, content_type, disposition)`. (3) **Virus scanning**: (a) Each attachment is scanned by ClamAV (or a commercial engine like McAfee). (b) Scanning runs in a sandboxed container with 30-second timeout. (c) Results: CLEAN, INFECTED, UNSCANNABLE (encrypted/password-protected). INFECTED attachments are blocked — the email is delivered with a warning: "Attachment blocked — potential virus." (4) **Thumbnail generation**: (a) For images: generate thumbnails at 3 sizes: small (100×100), medium (400×400), large (1200×1200). (b) For PDFs/documents: convert first page to PNG thumbnail using LibreOffice headless or ImageMagick. (c) For videos: extract a frame at 5 seconds using ffmpeg. (d) Thumbnails are stored in the same blob store with key: `{attachment_hash}/thumb_{size}.jpg`. (5) **Serving attachments**: (a) Generate a signed URL (pre-signed S3 URL or GCS signed URL) with a short TTL (5 minutes). (b) The email client redirects to the signed URL for download. (c) This prevents the email service from being a bandwidth bottleneck and allows CDN caching. (6) **Size limits**: (a) Gmail allows 25MB total per email (including encoded body). The MTA rejects emails exceeding the limit with a bounce message. (b) For larger files, Gmail provides a Google Drive integration (insert Drive link instead of attachment). (7) **File type restrictions**: (a) Executables (.exe, .bat, .vbs, .js) are blocked at the MTA level. (b) Archives (.zip, .rar) are scanned recursively. Password-protected archives are blocked (unscannable).

## Q23: Design the SMTP email delivery pipeline (outbound mail). How do you handle sending billions of emails per day with high deliverability, DKIM/SPF/DMARC signing, bounce handling, and reputation management?

**Answer:** Outbound email delivery at Gmail scale is a complex pipeline managing sender reputation. (1) **Outbound pipeline stages**: (a) User clicks Send → email is stored in the sent mail spool (Bigtable). (b) The delivery scheduler picks up the email and enqueues a delivery task per recipient. (c) SMTP connection pool: maintain persistent SMTP connections to recipient MX servers. (d) Message transfer: send the email over SMTP with STARTTLS. (2) **Email authentication**: (a) DKIM signing: sign the email with the domain's private DKIM key. (3) **Bounce handling**: (a) Hard bounces (invalid recipient): update the sender's address book with `bounce_marked = true`. (4) **Reputation management**: (a) Sending IP pools: separate IPs for transactional (password reset) vs. marketing (promotions). Transactional has higher reputation. (b) Warm up new IPs gradually: start with 100 emails/day, increase by 50% daily. (c) Monitor blacklists (Spamhaus, Barracuda). If an IP gets blacklisted, immediately stop sending from that IP and request delisting. (5) **Rate limiting per domain**: (a) Each recipient domain has a rate limit (Gmail sends to Yahoo at max 100 emails/second). (b) The delivery scheduler maintains per-domain token buckets. (c) If a domain returns "too many connections" (450 4.7.28), back off and reduce rate. (6) **Tracking**: (a) Embed a tracking pixel (1×1 transparent GIF) for open tracking. (b) Rewrite links for click tracking: `https://mail.example.com/redirect?url=...&sig=...`. (c) Complaints: ISPs provide FBL. Track complaint rate — if >0.1%, investigate and improve targeting.

## Q26: Design the email forwarding system. How do you allow users to automatically forward emails to another address, handle loop detection, and maintain deliverability (SPF/DKIM/DMARC compliance for forwarded mail)?

**Answer:** Email forwarding is deceptively complex because SPF/DKIM/DMARC break on forwarding. (1) **Forwarding setup**: (a) User configures forward-to address in settings. (b) Options: forward all mail, forward only mail from specific senders, forward mail matching filters. (2) **Forwarding pipeline**: (a) On email delivery, evaluate forwarding rules before placing in inbox. (b) If match, enqueue a forwarding task. (c) Forwarding task: rewrite the email envelope (MAIL FROM: the original sender, RCPT TO: the forwarding address). (d) The forwarder must re-sign with DKIM (using the forwarding domain's key, not the original sender's key) because the original DKIM signature will break when headers change (the forwarder adds `Received:` headers). (3) **SPF/DKIM/DMARC breakage**: (a) SPF fails because the forwarding server's IP is not listed in the original sender's SPF record. (b) DKIM fails if any header is modified (the forwarder adds `Received:` headers, breaking the body hash). (c) DMARC policy may be `p=reject` — the forwarding server should not reject, but the final recipient might. (4) **Solutions — SRS (Sender Rewriting Scheme)**: (a) Rewrite the envelope sender: `MAIL FROM: SRS0=xxxx=yy=original_domain=original_user@forwarding_domain.com`. (b) The forwarding server signs the rewritten address. (c) Bounces go back to the forwarding server, which can map back to the original sender and relay the bounce. (d) Without SRS, SPF fails on the forward. (5) **ARC (Authenticated Received Chain)**: (a) ARC preserves the original authentication results across forwarding hops. (b) The forwarding server adds an ARC-Seal header that chains the original DKIM/SPF results with the forwarding server's validation. (c) The final recipient can see: "original email passed DKIM, and it was forwarded by a trusted forwarder." (d) Gmail, Yahoo, and Outlook all support ARC as of 2025. (6) **Loop detection for forwarding**: (a) If user A forwards to user B, and user B forwards to user A, each email triggers a loop. (b) Solution: the MTA checks `Received` header count. If an email has been received more than 3 times, reject. (c) Additionally, check for forwarding loops: if the current server's domain appears in the `Received` chain, reject. (7) **Spam forwarding protection**: (a) If the forwarded email is spam, the forwarding server's reputation suffers. (b) Scan forwarded emails for spam. If spam score > threshold, drop the forwarding and file the email in the user's spam folder instead.

## Q27: Design the unsubscribe and mailing list management system. How do you handle one-click unsubscribe (RFC 8058), list-unsubscribe headers, and block-list management for bulk senders?

**Answer:** Email services must honor unsubscribe requests to maintain deliverability and comply with CAN-SPAM and GDPR. (1) **List-Unsubscribe header**: (a) Incoming bulk emails should include `List-Unsubscribe: <mailto:unsubscribe@example.com?subject=...>, <https://example.com/unsubscribe/...>`. (b) Gmail parses this header and shows an "Unsubscribe" link next to the sender name. (c) One-click unsubscribe (RFC 8058): `List-Unsubscribe-Post: List-Unsubscribe=One-Click`. Gmail sends an HTTP POST to the HTTPS URL automatically when the user clicks Unsubscribe. (2) **Unsubscribe action**: (a) When user clicks Unsubscribe, the system: (i) Parses the List-Unsubscribe header. (ii) Sends a POST to the HTTPS URL (one-click) or generates a mailto: (legacy). (iii) Marks the sender in the user's block list: `blocked_senders (user_id, sender_domain, blocked_at)`. (3) **Block list management**: (a) Users can manually add senders to their block list. (b) Blocked senders' emails are automatically filed to Spam (not deleted — user can review). (c) Block list applies at the domain level: blocking "newsletter@company.com" also blocks "promo@company.com" (same domain). (4) **Mailing list filtering**: (a) Detect mailing list emails via: `List-Id`, `List-Help`, `List-Post`, `List-Subscribe` headers. (b) Separate tab in Gmail UI: "Promotions" tab uses this classification. (c) User can choose: "Always mark as read" or "Always send to Spam" for specific lists. (5) **Bulk sender compliance**: (a) Gmail requires bulk senders (>5,000 emails/day to Gmail addresses) to authenticate with SPF/DKIM/DMARC and enable one-click unsubscribe. (b) Non-compliant senders see higher spam rates and eventual delivery blocks. (c) The system enforces this via a sender reputation score: if a sender's emails receive >0.3% spam complaints, future emails are throttled. (6) **GDPR right to object**: (a) Direct marketing emails: users can object to processing for direct marketing (GDPR Article 21). (b) The unsubscribe action also triggers a GDPR suppression record: `marketing_suppressions (email_hash, channel, timestamp)`. (c) The suppression list is shared with the sender via a feedback loop.

## Q28: Design the email categorization system (Primary, Social, Promotions tabs in Gmail). How do you classify incoming emails into categories with high accuracy while allowing users to override and train the system?

**Answer:** Gmail's tabbed inbox uses ML classification with user feedback. (1) **Categories**: Primary (personal correspondence), Social (notifications from social networks), Promotions (marketing/offers), Updates (receipts, bills, statements), Forums (discussion groups, mailing lists). (2) **ML classification model**: (a) Features: sender domain reputation, email content (n-grams, TF-IDF), headers (List-Id, Precedence, X-Mailer), user interaction patterns (does the user typically read this sender's email immediately?). (b) Model: lightweight neural network or gradient-boosted trees. Trained on user actions (move-to-tab events are labeled training data). (c) Inference: <5ms per email, evaluated at delivery time before placing in inbox. (3) **Rule-based pre-classification**: before the ML model, apply rules: (a) If email has `List-Id: <*>` → likely Forums or Promotions. (b) If email is from a known social domain (facebook.com, twitter.com, linkedin.com) → Social. (c) If email contains "receipt", "invoice", "statement" → Updates. (d) If sender is in user's contacts → Primary. (4) **User override and training**: (a) User drags an email from Promotions to Primary → this is a training signal. Store: `tab_override (user_id, sender_domain, from_tab, to_tab, timestamp)`. (b) After 3 overrides for the same sender-move pair, the system learns: "user@example.com's emails go to Primary for this user." (c) Personalization: each user has a per-sender tab assignment stored in a user-sender mapping table. (5) **Cold start**: for new users with no history, use global model defaults. (6) **Tab hallucination prevention**: some emails could fit multiple categories (a purchase receipt from Amazon is both Promotions and Updates). The system picks the best match and allows the user to move it. The override trains the model for future emails from the same sender. (7) **Evaluation**: accuracy measured by override rate — if <5% of emails are overridden, the model is performing well. Dashboard tracks override rate per category.

## Q29: Design the email threading algorithm. How do you group related emails (replies, forwards, bounces) into a single conversation, even when participants change or subjects are modified?

**Answer:** Email threading at Gmail scale must handle imperfect subject lines and changing participants. (1) **Threading by normalized subject**: (a) Strip Re:/Fwd:/AW:/VS: prefixes (in all languages — 心 for Japanese reply, etc.). (b) Normalize whitespace and lowercase. (c) Group messages with the same normalized subject within a time window (default: 7 days). (2) **References and In-Reply-To headers**: (a) The `References` header contains a list of Message-IDs that this email replies to. (b) The `In-Reply-To` header contains the immediate parent Message-ID. (c) These headers provide the ground truth for threading. Algorithm: build a tree structure: Message-ID → list of child Message-IDs. The thread is the tree rooted at the earliest message. (3) **Fallback when headers are missing**: (a) Many email clients don't set References/In-Reply-To correctly. (b) Fallback heuristic: (i) Same normalized subject. (ii) Sender of one email is recipient of another (reply pattern). (iii) Temporal proximity: emails within 3 days are more likely to belong together. (c) Clustering algorithm: similarity score based on subject similarity (Levenshtein), participant overlap (Jaccard index), and time distance. (4) **Gmail's conversation view**: (a) Gmail doesn't show the tree structure — it shows a flat, chronologically sorted list of messages in the conversation. (b) The "most recent message" is at the top (or bottom, depending on settings). (c) Quoted text is collapsed by default — only the new content is shown. (5) **Thread splitting and merging**: (a) If a user changes the subject entirely (not just a Re: prefix), the message may start a new thread. (b) Gmail allows manual thread splitting: "Split this conversation into two." (c) Thread merging: user selects two threads and merges them. Both actions update the thread metadata: `thread_merges (thread_id_A, thread_id_B, merged_into_thread_id)`. (6) **Edge cases**: (a) Bounce messages: delivery failure notifications are grouped with the original sent message — they share the same Message-ID (the original email's ID in the bounce report). (b) Drafts: draft replies are shown as part of the thread but marked as "Draft." (c) Sent messages: a sent email appears in the Sent folder AND in the recipient's inbox — threading works because the Sent copy has the same Message-ID.

## Q30: Design the internationalization (i18n) system for a global email service. How do you handle email composition and display in 100+ languages, including RTL languages, character encoding, date/number formatting, and translated UI?

**Answer:** Internationalization for email requires handling composition, rendering, and storage across diverse scripts. (1) **Character encoding**: (a) All storage uses UTF-8 internally. (b) For outbound email, use the appropriate charset in MIME headers: `Content-Type: text/plain; charset=UTF-8` (or ISO-2022-JP for Japanese legacy compatibility). (c) Encode non-ASCII in headers using encoded words: `Subject: =?UTF-8?B?5q2j5Zyo5rKz5Yy6?=`. (2) **Composition — input method support**: (a) The web UI integrates with the browser's IME (Input Method Editor) for CJK (Chinese, Japanese, Korean) languages. (b) For mobile, the OS keyboard handles IME. (c) Spell-check: use language-specific dictionaries. (3) **Rendering — RTL support**: (a) Arabic, Hebrew, Urdu, and Farsi are right-to-left. (b) The email composer must handle mixed-direction text (an RTL sentence with an English URL embedded). Use Unicode bidi (bidirectional) algorithm. (c) The HTML renderer sets `dir="auto"` on the email body container to auto-detect direction. (4) **Date and number formatting**: (a) Dates: format according to the user's locale. US: "Jul 12, 2026". UK: "12 July 2026". Japan: "2026年7月12日". (b) Time: 12-hour vs 24-hour based on locale. (c) Numbers: thousands separator (1,000 vs 1.000) and decimal separator (1.5 vs 1,5). (5) **Translated UI**: (a) All UI strings are externalized in locale files (JSON/PO files). (b) Translation management: use a platform like Crowdin or Transifex for community translations. (c) Fallback: if a string is not translated for the user's locale, fall back to English. (d) Locale detection: prioritize user setting > browser Accept-Language header > IP geolocation. (6) **Locale-specific features**: (a) Japan: emphasis on 「」 (corner brackets) for quoting. Korea: use 《》. (b) CJK: no whitespace between words — line-breaking rules differ (CJK allows break after any character). (c) Some locales prefer `様` (sama) honorific after recipient name in Japan. (7) **Email templates**: (a) Transactional emails (password reset, verification) are rendered per locale. (b) Template engine (Handlebars/Mustache) loads locale-specific strings: `{{\`email.verify.subject\`}}`. (c) Images in emails: avoid text in images (untranslatable). Use CSS-styled text instead.
