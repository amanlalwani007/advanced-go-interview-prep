# Chapter 23 — Design a Distributed Email Service (Gmail)

## Q1: Design the email storage system. How do you store billions of emails per user and support fast search, threading, and label-based queries?

**Answer:** Email storage at Gmail scale requires a custom storage layer — Gmail uses Bigtable (Google's wide-column store). (1) **Core table** — each email is a row keyed by `(user_id, message_id)`. Columns include: raw message (MIME blob), parsed metadata (sender, recipients, subject, date), labels (Inbox, Sent, Starred, custom labels), and read/unread state. (2) **Threading** — GMail groups messages into conversations using a thread ID. A thread is identified by the normalized subject line (after stripping Re:/Fwd:) and participants. Messages are stored indexed by `(user_id, thread_id)` for efficient conversation loading. (3) **Label/tag model** — Gmail doesn't use folders; every message has a set of labels. A message can be in multiple "folders" simultaneously (e.g., both "Inbox" and "Travel"). This is stored as a bitmap or a set column in Bigtable. Querying by label is an index lookup: `SELECT * FROM messages WHERE label CONTAINS 'Inbox'`. (4) **Storage optimization** — attachments are deduplicated across users. If 1M users receive the same attachment (a popular PDF), it's stored once in a blob store and referenced by hash. The email metadata table stores just the content hash. (5) **Caching** — the most recent 50 emails per user are cached in memory on the API server. This covers 90%+ of "check inbox" requests.

## Q2: How does Gmail handle full-text search over billions of emails with sub-second latency? Describe the indexing pipeline.

**Answer:** The search index is separate from the primary mail store. (1) **Index pipeline** — when an email arrives (or is modified), a background worker parses the text content, extracts terms, and writes to a per-user inverted index stored in a custom search engine (Google's internal search technology, similar to what powers Google Search). (2) **Index structure** — the inverted index maps `(term, user_id) → list of message_ids`. Terms are tokenized using: Unicode normalization, stemming (e.g., "running" → "run"), stop word removal (a, an, the), and phrase indexing (quoted phrases). (3) **Sharding** — the search index is sharded by `user_id` (hash range). Each shard is a separate Bigtable range, collocated with the user's mail data. This ensures search queries are single-shard operations (no scatter-gather). (4) **Query execution** — a search query `from:alice subject:meeting after:2024/01/01` is parsed into: `author_alice AND subject_meeting AND date >= 2024-01-01`. The system intersects the postings lists for each term, returning matching message IDs in ranking order. (5) **Ranking** — results are ranked by: recency (most recent first), relevance (TF-IDF on query terms), and user signals (how often the user interacts with messages from this sender). (6) **Latency** — the search index is optimized for sub-second queries. The system returns the top 10 results first (to populate the UI) while computing the full result set asynchronously.

## Q3: Design the email sending pipeline. How does Gmail handle sending 100M+ emails per day while ensuring deliverability (avoiding spam folders) and respecting rate limits?

**Answer:** The outbound pipeline: (1) **Outbound queue** — when a user clicks "Send", the message is placed in a per-user outbound queue (Kafka topic partitioned by sender domain). (2) **Rate limiting** — per-user: max 500 emails per day (Gmail's limit for free accounts), per domain: limits set by the receiving server (MX server). The rate limiter uses a sliding window counter in Redis. (3) **Spam classification** — before delivery, the outbound service runs the message through a spam classifier. Messages that match spam patterns (too many links, suspicious attachments, unusual sending patterns) are flagged. If flagged, the user sees a warning: "This message looks suspicious. Are you sure?" (4) **DKIM/SPF/DMARC signing** — the outbound service signs the message with DKIM (DomainKeys Identified Mail) using the sender domain's private key. The sending IP is checked against SPF (Sender Policy Framework) records. DMARC policy (published by the sender's domain) is respected. These three protocols determine whether the receiving server classifies the email as legitimate or spam. (5) **Delivery retry** — if the receiving server returns a transient error (450/451), the system retries with exponential backoff over 48 hours. Permanent errors (550/551) generate a bounce notification to the sender. (6) **Feedback loops** — Gmail subscribes to FBLs (Feedback Loops) from major ISPs. When a recipient marks an email as spam, the FBL informs Gmail, and the sender's sending reputation is adjusted. Reputation is per-sender-domain, per-IP.

## Q4: How does Gmail design the highly-available IMAP/POP3 frontend layer that handles 1B+ active connections?

**Answer:** The frontend layer uses a connection management architecture: (1) **Load balancer** — a global anycast IP (Google's GFE) routes each TCP connection to the nearest frontend server. (2) **Frontend server** — each frontend can handle 100K+ concurrent connections using an event-driven, non-blocking I/O model (epoll on Linux). Connections are HTTP/2 or custom long-lived TCP connections (not traditional IMAP/TCP per connection, to avoid per-connection overhead). (3) **Session state** — the frontend stores per-user connection state (which folders are open, which messages are visible, sync state) in a local in-memory cache. If the frontend fails, the client reconnects and state is reconstructed from the backend. (4) **Backend query** — the frontend proxies the request to the backend mail store (Bigtable). For IMAP IDLE (push notifications), the frontend subscribes to a Redis pub/sub channel for the user's email ID. When new mail arrives, the backend publishes a notification, the frontend receives it, and pushes it to the client over the open IMAP connection. (5) **Scaling** — connection-sticky load balancing ensures a user's persistent connection stays on the same frontend server. This is done via a consistent hash (client IP or session cookie) at the load balancer.

## Q5: How does Gmail's Smart Inbox (automatic categorization: Primary, Social, Promotions, Updates, Forums) work under the hood?

**Answer:** The categorization is an ML model: (1) **Feature extraction** — for each incoming message, extract features: sender domain (newsletter@mail.example.com vs. john@gmail.com), message headers (List-Unsubscribe presence → likely promotion/unsubscribe is present → promotional), email content (presence of "unsubscribe" link, HTML-to-text ratio, image-to-text ratio), user interaction history (has the user ever clicked links in this sender's emails?). (2) **Classification model** — a gradient-boosted decision tree or neural network trained on millions of labeled examples. Classes: Primary (personal, no bulk), Social (Facebook, Twitter, LinkedIn), Promotions (marketing, deals), Updates (receipts, shipping confirmations, bills), Forums (mailing lists, discussion groups). (3) **Training data** — users can manually drag messages between tabs. Each manual reclassification is a training signal. Google also uses crowd-sourced labeling (anonymized) to bootstrap the model. (4) **Inference** — the model runs on every incoming message, outputting a probability distribution over categories. The message is placed in the highest-probability category. Inference latency must be <50 ms to not delay email delivery. (5) **Personalization** — the model is fine-tuned per user based on their reclassification history. If a user always moves a specific sender's emails from Promotions to Primary, the model learns this and starts routing them to Primary directly.

## Q6: Design the attachment storage and preview system. How does Google handle 50 MB attachments and previewing images, PDFs, and docs inline without downloading the whole file?

**Answer:** (1) **Upload** — attachments are uploaded directly to GCS (Google Cloud Storage) via resumable upload sessions. The email metadata stores the content hash and GCS object path. (2) **Virus scanning** — each upload triggers a ClamAV scan (or Google's internal virus scanning service). Infected attachments are quarantined and the user is alerted. (3) **Preview generation** — when an email with an attachment arrives, a preview pipeline is triggered: images are compressed to a thumbnail (using ImageMagick or a custom WebP transcoder); PDFs/Docs are converted to HTML (libreoffice or Google Docs conversion engine); videos have a preview thumbnail extracted. (4) **Storage tiers** — raw attachments go to nearline storage (GCS Nearline for attachments < 30 days old) and archive storage (GCS Archive for older). Previews are cached in a CDN (edge cache). (5) **Inline viewing** — the Gmail client displays previews by loading the converted HTML or the image thumbnail from the CDN, not the original 50 MB attachment. The original file is only downloaded if the user clicks "Download" or accesses it offline. (6) **Size limits** — Gmail's max attachment size is 25 MB (for incoming) and 25 MB for outgoing (via Google Drive links for larger files). The 25 MB limit is an SMTP constraint; larger files are uploaded to Drive and a link is sent.

## Q7: How does Gmail detect and prevent phishing attacks in real-time?

**Answer:** Multi-layered phishing detection: (1) **URL scanning** — every link in an incoming email is rewritten through Google's Safe Browsing proxy server. When clicked, the user's browser checks the URL against Google's constantly updated list of known phishing/malware sites. If the site is dangerous, the user sees a full-screen red warning. (2) **ML classifier** — each email is scored by a phishing ML model that looks for: display-name spoofing (the sender is "PayPal" but the email is from attacker@gmail.com), urgency language ("Your account will be closed in 24 hours"), mismatched reply-to addresses, and known phishing domain patterns. (3) **DMARC enforcement** — for high-value targets (executives, finance teams), Gmail enforces the strictest DMARC policy: if an email fails SPF and DKIM authentication and the sender domain has a `p=reject` DMARC policy, the email is rejected entirely (not even sent to spam). (4) **User reporting** — if a user clicks "Report phishing", the email is forwarded to Gmail's abuse analysis team. This feedback trains the ML model and the suspicious URLs are added to Safe Browsing. (5) **Tenant isolation** — Google Workspace administrators can set up email authentication controls: only allow emails from specific IP ranges, block emails with specific file types (e.g., executables), and require TLS encryption for incoming traffic from specific domains.

## Q8: Design offline access in Gmail. How does the Gmail offline Chrome app cache emails locally and sync changes when connectivity returns?

**Answer:** The Gmail Offline Chrome app uses: (1) **Local storage** — uses IndexedDB (in-browser NoSQL store) to cache email metadata and recent email bodies. The cache is encrypted at rest using the user's Google Account credentials. (2) **Sync state** — the local store tracks the state of each message (read/unread, starred, label assignments) with a `local_version` and `server_version`. (3) **Outgoing queue** — emails composed offline are stored in a local outbox. They are submitted as pending with a local timestamp. (4) **Online sync** — when connectivity is restored, the client sends the server its last known sync cursor. The server responds with the delta of changes (new messages, label changes, deletions) since that cursor. The client applies these changes to the local IndexedDB. Then, the client pushes any changes made offline (composed emails, star toggles, label changes). Conflicts (e.g., a message was deleted on the server while the user starred it offline) are resolved: server changes take precedence for deletions; local changes for labels are applied if the message still exists. (5) **Sync strategy** — full resync is expensive. Gmail uses incremental sync with pagination. The first sync downloads only headers and snippet (not full message bodies). Full message bodies are lazily downloaded when the user opens a message.

## Q9: How does Gmail handle the migration from the legacy "Inbox by Gmail" feature set (bundles, highlights, snooze) into the main Gmail app?

**Answer:** Feature migration at Gmail's scale requires backward compatibility and gradual rollout: (1) **Feature flagging** — every feature is behind a flag. An A/A/B test approach: start with 1% of users, verify stability and UX metrics, ramp to 10%, then 50%, then 100%. (2) **Backward compatibility** — all new features (snooze, smart replies, nudges) are implemented as additions to the existing Gmail data model. Snooze: a `snooze_until` timestamp field is added to the message metadata. The existing "unread" and "inbox" labels remain unchanged. If a user snoozes an email, it's removed from Inbox and a cron job returns it when `snooze_until` elapses. (3) **Storage migration** — adding new fields (snooze timestamp, bundle ID, highlight tags) to a Bigtable schema with billions of rows is a non-trivial migration. Google's approach: use a "column family" addition in Bigtable — new writes get the new field; old rows are backfilled asynchronously (a MapReduce job that touches every row over weeks). Queries handle missing fields gracefully (default values). (4) **Client migration** — the Gmail web client is a single-page application (Angular/Dart compiled to JS). Features are loaded as modules. The new UI (Inbox features) is a different set of React components toggled by the feature flag. Users can switch between "New Gmail" and "Classic Gmail" during the transition period.

## Q10: Design the spam filter. How does Gmail achieve >99.9% spam detection accuracy with <0.05% false positive rate?

**Answer:** Gmail's spam filter is a Bayesian classifier augmented with deep learning: (1) **Content analysis** — extract features from email body and headers: word frequencies, character n-grams (catch obfuscated words like "V1@gra"), HTML structure, image-to-text ratio. (2) **Sender reputation** — each sending domain and IP has a reputation score based on: historical spam reports, volume of email sent, authentication (SPF/DKIM/DMARC) pass rate, and complaints from FBLs. Unknown senders start with neutral reputation and earn trust (or distrust) over time. (3) **User filters** — users can create custom filters and whitelist/blacklist senders. Whitelist is absolute (never go to spam). (4) **Deep learning model** — Google uses a neural network (BERT-based or similar) trained on billions of emails. The model understands context, not just keywords. It can detect sophisticated phishing emails that use proper grammar and brand logos but have a malicious intent. (5) **False positive protection** — Gmail automatically moves legitimate mail out of spam if the user interacts with it (opens, replies, clicks links). An email classified as spam but that matches a user's contact list or has been in the user's previous inbox (sender has emailed before) is given a lower spam score. (6) **Feedback loop** — when users mark emails as "Not spam" or "Report spam", these signals are fed back into the model training pipeline. Google aggregates these signals across all users (anonymized) to continuously improve the global model. (7) **Adversarial robustness** — spammers constantly evolve. Google's model is adversarially trained on known spammer techniques (e.g., replacing characters with Unicode lookalikes, embedding text in images). The system also uses Gmail's domain-wide reputation: if a domain suddenly starts sending email after being dormant for years, it's treated with suspicion.
