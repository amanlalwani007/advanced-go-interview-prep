# Chapter 24 — Design S3-like Object Storage

## Q1: Describe the write path in S3. When a client uploads a 100 MB file, what happens inside the S3 system before the client gets a success response?

**Answer:** The S3 write path: (1) **Frontend (API gateway)** — the client sends a `PUT /bucket/key` request to the S3 API gateway (AWS's globally distributed Request Router). The gateway authenticates the request (Signature V4), validates the bucket/key, and checks permission (IAM policy + bucket policy). (2) **Partition placement** — the gateway consults the placement service to determine which storage partition (shard) owns this object. S3 partitions keys by the hash of `(bucket, key)`. (3) **Storage node (WAL write)** — the request is forwarded to the primary storage node for that partition. The node writes the 100 MB payload to its local disk first as a Write-Ahead Log (WAL) and immediately replicates to N-1 secondary nodes (synchronous replication within the same data center). (4) **Acknowledgment** — once the WAL(s) are written and N replicas confirmed, the primary responds with "200 OK" to the client. (5) **Async compaction** — after acknowledgment, the storage node asynchronously converts the WAL entry into the final storage format (S3 indexes objects by key prefix for efficient range scans). During this time, the object is still readable because the WAL entry is indexed. (6) **Metadata update** — the metadata store (keyed by `(bucket, key_hash)`) is updated with the object's storage location, size, ETag (MD5), and timestamp. This metadata is replicated across three data centers for durability. The client sees success after step (4) — typically within 200-500 ms for a 100 MB file on a fast connection.

## Q2: S3 supports strong read-after-write consistency for PUT of new objects. How does it achieve this at global scale without sacrificing availability?

**Answer:** S3's strong consistency for PUTs (new objects) is achieved through: (1) **Synchronously replicated metadata** — the metadata store for a given key is replicated synchronously across three AZs within the region before the PUT returns success. Any subsequent GET in that region will see the latest metadata. (2) **Quorum-based reads** — a GET request reads from the metadata store quorum (majority of replicas), ensuring it sees the latest committed write. (3) **Partition-level consistency** — each partition (key range) has a single leader that serializes writes. All reads go to the leader or to a read replica that has replicated the latest metadata via a consistent replication stream. (4) **Key difference from eventually-consistent systems (e.g., DynamoDB)**: S3 does NOT use last-writer-wins (LWW) or conflict resolution. Instead, it uses a single leader per partition for writes, and all replicas apply writes in the same order. This is a more expensive approach (lower write throughput per partition) but provides stronger guarantees. S3's trade-off: it can "only" handle millions of PUTs per second per prefix (vs. DynamoDB's tens of millions) but guarantees read-after-write consistency.

## Q3: How does S3 support prefix-based listing (e.g., `ListObjects` with prefix `photos/2024/`)? What data structure enables efficient range queries over billions of keys?

**Answer:** S3 stores object metadata in an ordered key-value store (ordered by `(bucket, key)`). The data structure is a **B-Tree or LSM-Tree** that maintains keys in sorted order: (1) **Key ordering** — objects within a bucket are stored sorted by UTF-8 binary order of the key. All keys starting with `photos/2024/` are contiguous in this ordering. (2) **Partitioning** — S3 partitions the key space into ranges. Each partition covers a contiguous range of keys (e.g., `photos/a` to `photos/m`). The partition boundaries are managed by the placement service. (3) **List request** — when the client calls `ListObjects?prefix=photos/2024/&delimiter=/`, the system finds the first partition that could contain keys with that prefix using a binary search on the partition metadata. It then scans the partition's local index from the first key matching the prefix, reading sequentially until the prefix no longer matches or the page size limit is reached. (4) **Delimiter-based grouping** — the `/` delimiter groups keys into common prefixes (pseudo-folders). The scan detects key boundaries at the delimiter and returns them as `CommonPrefixes` entries without actually scanning all sub-keys. (5) **Performance** — a list operation on a prefix with 1000 keys reads exactly 1000 index entries. A list operation on a prefix with 1M keys reads only the requested page (default 1000) because the B-Tree efficiently seeks to the correct starting position.

## Q4: Explain how S3 handles multi-part upload for large objects, including the life cycle of upload parts and what happens if the upload is aborted.

**Answer:** Multi-part upload is designed to overcome single-stream throughput limitations and network unreliability: (1) **Initiation** — client sends `POST /bucket/key?uploads`. S3 creates an `upload_id` and returns it. No storage is allocated yet. (2) **Part upload** — client uploads each part independently: `PUT /bucket/key?uploadId=<id>&partNumber=1`. Each part (1 MB to 5 GB) is stored as a separate object behind the scenes. The server returns an ETag for that part. (3) **Completion** — client sends `POST /bucket/key?uploadId=<id>` with the list of (partNumber, ETag) in order. S3 assembles the final object by concatenating the parts in order. The multi-part upload ID is closed; the assembled object is now referenced as the single object. (4) **Abort** — if the client sends `DELETE /bucket/key?uploadId=<id>`, or the upload exceeds the timeout (typically 7 days for incomplete parts), all uploaded parts are garbage-collected by a background reaper process. (5) **Retry behavior** — if a single part upload fails (network timeout), the client retries only that part, not the entire object. The uploaded parts are independent, so retried part uploads with the same part number simply overwrite the old part. (6) **Optimal part size** — for a 100 GB file, use 100 parts of 1 GB each. This allows parallel upload of 10-20 parts simultaneously, saturating the client's bandwidth. (7) **Backend assembly** — the completion request creates the final object manifest: `{object_key, parts: [{part_number: 1, storage_loc: <path>, size: 1GB}, ...]}`. The object's data is not copied — the parts remain in their original locations and the manifest references them.

## Q5: How does S3 implement versioning? When a user deletes a versioned object, what actually happens?

**Answer:** S3 versioning creates a new object version for each write (PUT, POST, COPY) to the same key. (1) **Version ID** — each object version is assigned a unique version ID (a monotonically increasing number or a UUID). The object's key + version ID uniquely identifies a specific version. (2) **Storage** — different versions of the same key are stored as separate objects in the underlying storage. They share no data unless they have identical content (in which case S3's dedup may point to the same underlying blocks). (3) **Delete marker** — when the client sends `DELETE /bucket/key` on a versioned bucket, S3 does NOT remove the object. Instead, it inserts a "delete marker" — a zero-byte object version with a special flag. A subsequent GET request without a version ID sees no object (the delete marker is the latest version). But a GET with the specific version ID still retrieves that version. (4) **Permanent deletion** — `DELETE /bucket/key?versionId=<id>` permanently deletes that specific version. This is an irreversible, idempotent operation. (5) **Storage management** — versioning multiplies storage costs because old versions are retained indefinitely. S3 Lifecycle Policies can automatically expire non-current versions after a configurable period. (6) **MFA Delete** — for critical buckets, S3 supports MFA (Multi-Factor Authentication) Delete — a DELETE request is rejected unless it includes a valid MFA token. This prevents accidental or malicious deletion of versioned objects.

## Q6: Design the S3 pre-signed URL mechanism. How does it work, and how do you prevent a pre-signed URL from being used to upload more data than intended?

**Answer:** Pre-signed URLs delegate access to a specific object for a limited time, without embedding AWS credentials in the client app. (1) **Generation** — the server creates a URL: `GET /bucket/key?AWSAccessKeyId=<access_key>&Expires=<expiry_timestamp>&Signature=<HMAC-SHA256(secret_key, string_to_sign)>`. The `string_to_sign` includes HTTP method, expiry, and the resource path. (2) **Verification** — when the client sends a request with the pre-signed URL, S3 recomputes the signature using the signer's secret key (stored securely on S3's side). If the signature matches and the current time < Expires, the request is authorized. (3) **Preventing over-upload** — pre-signed PUT URLs allow uploading any content up to the object's full size. To limit the upload size, the signing server includes a `content-length` range in the `string_to_sign`, or uses a custom header check on the server side (not supported by S3 directly — you would need a proxy). Alternative: use a multi-part upload with pre-signed URLs for each part, and the completion request is only sent after the server validates total size. (4) **Security considerations** — pre-signed URLs expire. A leaked pre-signed URL grants access to the specific object for the duration of its validity. For sensitive data, use short expiry (60 seconds) and HTTPS. Pre-signed URLs are tied to a specific HTTP method — a pre-signed GET URL cannot be used to PUT or DELETE. (5) **Bandwidth control** — pre-signed URLs commit to no bandwidth guarantee. At extreme scale, the requester might experience throttling (HTTP 503). For guaranteed throughput, use S3 Transfer Acceleration or AWS PrivateLink.

## Q7: How does S3 implement cross-region replication (CRR) with a lag of typically under 15 minutes?

**Answer:** CRR asynchronously replicates objects from a source bucket to a destination bucket in a different region. (1) **Replication trigger** — when an object is PUT (or DELETE, if enabled) into the source bucket, S3 records the event in an internal replication log. This log is stored in a highly durable, ordered queue (similar to Kafka). (2) **Replication worker** — a fleet of cross-region replication workers read from the replication log, fetch the object from the source region's storage, and PUT it into the destination bucket. PUTs to the destination bucket are standard S3 PUTs (including full lifecycle processing). (3) **Batching** — the workers batch multiple objects into a single replication request. If 1000 objects are uploaded to the source bucket within seconds, they are replicated as a batch to the destination, amortizing the per-request overhead. (4) **Lag sources** — typical lag is 5-15 minutes. Sources of lag: cross-region network latency (50-200 ms), destination bucket PUT rate limits (3,500 PUT/s per prefix), worker autoscaling delays. (5) **Consistency** — CRR provides eventual consistency. If the source bucket has strong consistency for PUTs, the destination bucket may lag. If you read from the destination immediately after a PUT to the source, you may not see the object. (6) **Retry** — if the destination region is unavailable, the replication log queues the events and retries indefinitely until successful. Objects are not lost during destination region outages. (7) **Replication time control (RTC)** — S3 offers RTC which provides a 99.99% SLA on replication within 15 minutes, by dedicating more resources to the replication pipeline (at additional cost).

## Q8: How does S3 handle the "eventual consistency" of LIST operations? Why does LIST not have strong consistency even though individual GETs do?

**Answer:** S3's consistency model differentiates between reading a specific key (GET) and reading a list of keys (LIST). (1) **GET consistency** — strong read-after-write consistency for individual keys. This is possible because the metadata is replicated synchronously within a region and read from a quorum. (2) **LIST consistency** — S3 offers "list consistency" which is read-after-write for NEW objects in most cases, but not guaranteed for overwrites/deletes. Reason: LIST operations scan a range of keys across potentially multiple partitions. When a new object is written, the partition index is updated. However, S3 may add new partitions dynamically as a bucket grows. A LIST operation that starts reading from one partition might miss objects written to a newly created partition if the partition metadata cache hasn't been updated yet. (3) **Practical impact** — for most use cases, S3's LIST is consistent for new objects after a short delay (typically <1 second). For overwrite/deletes, the old listing may persist for a brief period because the index update may lag. (4) **Partition splitting** — when a partition grows too large, it's split into two partitions. During the split, the new partition map is propagated asynchronously. A LIST operation that cached the old partition map may not see objects in the new partition. S3 has since improved this and now offers stronger list consistency guarantees, but it's not as strong as GET consistency.

## Q9: Design S3 Glacier's retrieval mechanism. How does an "expedited" retrieval return data within 1-5 minutes for archival storage?

**Answer:** Glacier archival storage uses a fundamentally different storage medium than S3 standard. Data is stored on tape, optical media, or deep HDD spindles. (1) **Storage format** — data is stored in large shards (hundreds of GBs) with heavy erasure coding (e.g., 24-of-36) across multiple devices. This minimizes storage cost at the expense of read latency. (2) **Retrieval request** — when the client initiates an expedited retrieval, the request is queued at high priority. The system locates the shards containing the requested data from the index (a metadata DB mapping object → shard location and erasure coding layout). (3) **Data retrieval** — the system pre-allocates a cache of short-term storage (SSD scratch space). The shard location is passed to the robotic retrieval queue (if tape) or directly to the disk infrastructure (if HDD). For expedited retrievals, the system may maintain a hot cache of recently accessed objects. (4) **Reconstruction** — the data (erasure-coded shards) is read and decoded back into the original object. This reconstruction is CPU-intensive but parallelizable — the system can fan out the read and decoding across many nodes. (5) **Delivery** — the reconstructed data is written to the SSD scratch space and served to the client via the standard S3 GET API. The entire process completes in 1-5 minutes for expedited, 3-5 hours for standard, and 5-12 hours for bulk retrieval. (6) **Cost** — expedited retrievals require pre-provisioned capacity or have higher per-GB costs. The system must maintain enough SSD scratch space and compute capacity to handle the peak expedited retrieval rate.

## Q10: How would you design S3's event notification system (S3 → SQS/Lambda/SNS)? How does it guarantee at-least-once delivery without duplication?

**Answer:** S3 event notifications use an internal change log: (1) **Event capture** — every object write, delete, or restore operation is recorded in an internal event log (the same log used for CRR). (2) **Notification service** — a notification service reads from the event log and fans out to configured destinations (SQS, SNS, Lambda). (3) **Batching** — events are batched (e.g., up to 10 events or 10 KB per batch) to reduce the per-event overhead. The batch is written to the destination queue/topic as a single message. (4) **Delivery guarantees** — S3 provides "at-least-once" delivery. The notification service tracks delivery state. If the destination is unavailable or returns an error, the notification is retried with exponential backoff (up to several days). (5) **Deduplication** — destinations should be idempotent. S3 does not guarantee exactly-once delivery of notifications. If the notification service crashes after sending but before acknowledging the event log offset, the event will be re-sent. The destination (e.g., Lambda) should handle duplicates using the object's `version_id` or `event_time` as a dedup key. (6) **Event types** — `s3:ObjectCreated:*`, `s3:ObjectRemoved:*`, `s3:ObjectRestore:*`, `s3:Replication:*`. Filters for prefix/suffix reduce the notification volume by only sending events that match a specific prefix/filter rule (e.g., only `prefix=incoming/`). (7) **Scaling** — each S3 bucket can have up to 100 notification configurations. The system processes events from the event log continuously, with throughput scaling based on the number of shards/workers consuming from the log.

## Q11: Compare erasure coding vs. replication for S3-like object storage. Under what conditions does erasure coding become cheaper, and what latency trade-offs does it introduce?

**Answer:** Erasure coding (EC) and replication represent a fundamental cost-vs-latency trade-off: (1) **Replication** — store N full copies (e.g., 3x replication). Storage overhead = N×. Each read hits any replica (low latency). Write must update all N replicas before acknowledging. (2) **Erasure coding** — split data into k data fragments and create m parity fragments (e.g., k=6, m=3). Storage overhead = (k+m)/k = 1.5x for 6+3. Write requires writing k+m fragments, but only a subset (e.g., any k fragments) suffices for reads. (3) **Cost crossover** — at small object sizes (< 1 MB), replication wins because the metadata overhead of tracking fragments dominates. At large object sizes (>= 128 MB), EC wins because the 1.5x overhead of 6+3 EC beats 3x replication. AWS S3 Glacier uses 24+16 EC (1.67x overhead) on multi-TB shards. (4) **Latency penalty** — EC writes must compute parity (CPU cost, about 1 microsecond per MB for XOR, about 50 microseconds per MB for Reed-Solomon). EC reads must reconstruct from k fragments if one is lost, incurring network read of k×1/k = 1x data (plus decode). In healthy state, EC reads are fast because all k fragments are available. (5) **Degraded reads** — when a fragment is unavailable, the system must read k fragments and reconstruct. This adds decode latency (about 10-100 microseconds per MB for Reed-Solomon via Intel ISA-L SIMD). (6) **LRC (Local Reconstruction Codes)** — Azure's LRC and Facebook's f4 use a hybrid: within a 16+2 EC group, add local parities so degraded reads need only read from a subset of nodes. LRC trades slightly more storage overhead for much faster degraded reads. (7) **Operational complexity** — EC requires a background repair daemon that constantly scans for degraded fragments and rebuilds them. Replication is simpler: just detect a missing replica and copy from another.

## Q12: Design the S3 Batch Operations architecture. How does it enable processing billions of objects with a single API call (e.g., restore Glacier objects or set tags)?

**Answer:** S3 Batch Operations is a managed job orchestration system: (1) **Job manifest** — the user provides a CSV or S3 Inventory report listing object keys + version IDs to process (e.g., 2 billion objects). The manifest is stored in S3 and referenced by the Batch Operations API. (2) **Job creation** — POST /jobs with {manifest, operation, priority, report_config}. Returns a job ID. (3) **Task generation** — the Batch service splits the manifest into shards (e.g., 1 million objects per shard). Each shard becomes a task object stored in a task queue (backed by SQS or a distributed work queue). (4) **Worker fleet** — a pool of EC2 or Fargate containers (AWS-managed) pull tasks from the queue, executing the operation on each object in the shard. Workers batch API calls to the target S3 bucket (e.g., RestoreObject for 1000 objects per batch call). (5) **Progress tracking** — each worker reports progress (succeeded, failed, skipped) to a progress table (DynamoDB). The job controller aggregates shard progress. (6) **Retry and error handling** — failed operations (e.g., network error on restore) are retried up to 3 times. Objects that fail permanently are recorded in the failure report. (7) **Completion report** — when all tasks complete, the job generates a completion report (CSV in S3) listing succeeded, failed, and skipped objects. The user receives a CloudWatch event or SNS notification. (8) **Throttling** — the Batch service respects the target bucket's request rate limits. It uses adaptive throttling: if the bucket returns 503 SlowDown, the worker backs off exponentially. (9) **Scaling** — the worker fleet auto-scales based on the task queue depth. For 2 billion objects at 1000 ops/s per worker, the job completes in about 2B / (1000 x worker_count) seconds. With 500 workers, this is about 1.1 hours.

## Q13: How does S3 Object Lock implement WORM (Write-Once-Read-Many) compliance? Walk through the retention modes and the bypass mechanism for legal holds.

**Answer:** S3 Object Lock enforces WORM at the object level: (1) **Modes** — (a) **Governance mode**: an object version can be overwritten or deleted only by users with the s3:BypassGovernanceRetention permission (typically admins). This is the softer WORM — it prevents accidental deletion but allows compliance overrides. (b) **Compliance mode**: once set, an object version CANNOT be overwritten or deleted by ANY user, including the root account. This is true WORM — even AWS support cannot delete it. (2) **Retention period** — set via PUT /bucket/key?retention specifying {Mode: COMPLIANCE, RetainUntilDate: 2026-01-01T00:00:00Z}. The object is locked until the retain-until-date. After expiry, it is unlocked and can be deleted (unless another retention rule applies). (3) **Default policy** — bucket-level default settings apply retention periods to new objects: PUT /bucket?object-lock with {DefaultRetention: {Mode: GOVERNANCE, Days: 365}}. (4) **Legal hold** — separate from retention periods. PUT /bucket/key?legal-hold with {Status: ON}. A legal hold prevents deletion indefinitely (no expiry). It is removed only by setting Status OFF. Legal holds are always bypassable by users with s3:PutObjectLegalHold permission. (5) **Bypass mechanisms** — for Governance mode, the user must include x-amz-bypass-governance-retention: true in the DELETE request, AND have the IAM permission. For Legal Hold, the user must have s3:PutObjectLegalHold. Compliance mode has NO bypass — this is enforced by the storage backend refusing to deallocate the object's storage blocks even if the metadata says deleted. (6) **Compliance certification** — S3 Object Lock with Compliance mode meets SEC Rule 17a-4(f) and FINRA 4511(c) for electronic record retention. An independent audit (Cohasset Associates) certifies this.

## Q14: Design S3 Access Points for a multi-tenant SaaS platform where each customer needs their own namespace with custom policies.

**Answer:** Access Points provide per-application/per-tenant entry points into a shared bucket: (1) **Access Point creation** — the SaaS platform creates one Access Point per tenant: arn:aws:s3:us-east-1:123456789012:accesspoint/tenant-acme-corp. The Access Point has its own policy document and network controls. (2) **Policy isolation** — the Access Point policy is evaluated INSTEAD of the bucket policy for requests routed through the Access Point. Example tenant policy: {Effect: Allow, Action: s3:GetObject, Resource: arn:aws:s3:us-east-1:123456789012:accesspoint/tenant-acme-corp/object/*, Condition: {IpAddress: {aws:SourceIp: 203.0.113.0/24}}}. This ensures tenant A can only see their own objects. (3) **Prefix strategy** — each tenant's objects are stored under a unique prefix: s3://shared-bucket/tenant-acme-corp-uuid/. The Access Point's policy restricts access to that prefix. The Access Point ARN implicitly scopes to that prefix. (4) **Network controls** — Access Points support VPC origin, Internet origin, or both. A tenant's Access Point can be restricted to their VPC: {Effect: Deny, Principal: *, Action: *, Condition: {StringNotEquals: {aws:SourceVpc: vpc-12345}}}. No internet access to that tenant's data. (5) **Multi-region** — Access Points are regional. For multi-region SaaS, create Access Points in each region. (6) **Auditing** — CloudTrail logs show the Access Point ARN in the requestParameters. The platform can attribute every S3 operation to a specific tenant via the Access Point. (7) **Scalability** — up to 1000 Access Points per account per region. For 10,000+ tenants, the platform must share Access Points or use a different isolation mechanism (separate buckets per tenant, combined with IAM role per tenant).

## Q15: How does S3 Object Lambda work? Describe a real use case where you transform data on-the-fly during GET, without storing multiple copies.

**Answer:** S3 Object Lambda intercepts GET, HEAD, and LIST requests and invokes a Lambda function to transform the data before returning it to the caller: (1) **Architecture** — the user creates an S3 Object Lambda Access Point, pointing to a standard S3 Access Point + a Lambda function ARN. When a client calls GET on the Object Lambda Access Point, S3 transparently: (a) fetches the original object from the supporting Access Point, (b) passes it as an event to the Lambda function, (c) returns the Lambda's response to the client. (2) **Transformation types** — (a) **Redaction**: strip PII from CSV/JSON files on read. The Lambda parses the file, removes columns containing sensitive data, and returns the redacted version. (b) **Format conversion**: convert CSV to Parquet on the fly for analytics clients. The Lambda uses pandas or awswrangler to read CSV and stream Parquet bytes. (c) **Watermarking**: overlay text on images during read. The Lambda fetches the original image from S3, adds text via Pillow, and returns the watermarked JPEG. (d) **Compression**: transparently GZip the response if the client does not natively support it. (3) **Caching** — Object Lambda responses are NOT cached by CloudFront or S3 by default. Each GET invokes the Lambda. For hot objects, this is expensive. Mitigation: use Lambda@Edge in front of CloudFront for caching transformed responses. (4) **Pricing trade-off** — Lambda transforms cost about $0.20 per GB processed (for 1 sec execution). If 1 TB of data is read with transforms per month, that is about $200 extra vs. $23 for standard S3 GET. (5) **Real use case** — a healthcare analytics platform stores full patient records (including SSN) in S3. Researchers query via Athena. Instead of two copies (full + redacted), they use a single full records bucket and an Object Lambda Access Point that strips PII columns. Researchers read through the Lambda AP to get redacted data. Compliance auditors read directly from the main Access Point to get full data. One copy, two access patterns.

## Q16: Contrast a data lake architecture (single centralized storage) with a storage silo pattern (per-team buckets) in an S3 ecosystem. When would you advocate for each?

**Answer:** The choice hinges on governance complexity vs. autonomy: (1) **Data lake (single bucket)** — all data lives under one bucket with prefix-based partitioning: s3://data-lake/raw/sales/2025/, s3://data-lake/curated/marketing/. Pros: (a) unified governance — one bucket policy, one encryption key (KMS), one lifecycle rule. (b) Cross-team joins are trivial — queries join data across prefixes without cross-bucket access setup. (c) Centralized cost tracking — S3 Storage Lens inventory shows cost per prefix. Cons: (a) blast radius — a single misconfigured bucket policy exposes ALL data. (b) Prefix hot-spotting — a write-heavy prefix (e.g., ingesting IoT streams) may throttle reads on adjacent prefixes due to partition limits. (c) Resource limits — up to 100 notification configurations per bucket. (2) **Storage silos (per-team buckets)** — each team owns a bucket: s3://team-sales-data, s3://team-marketing-data. Pros: (a) isolation — a policy misconfig in team-sales does not affect team-marketing. (b) autonomy — each team sets their own lifecycle, encryption, replication rules. (c) performance isolation — write-heavy teams do not impact read-heavy teams. Cons: (a) cross-team queries require cross-account IAM roles or bucket policies (complex). (b) Governance is fragmented — no single view of all data. (c) Cost multiplication — each bucket has minimum 1 notification config, 1 inventory config. (3) **Recommendation** — use a data lake when: the organization has a dedicated data platform team, cross-team data joins are frequent, and strict governance is required (e.g., financial services regulatory reporting). Use silos when: teams are fully autonomous (startup culture), data joins across teams are rare, and each team wants full control over encryption key rotation and replication. The most mature pattern is a HYBRID: a data mesh where each domain owns a bucket, and a central catalog (AWS Glue Catalog) provides a unified query layer via Lake Formation — teams control their buckets but data is discoverable and joinable through the catalog.

## Q17: Optimize S3 Glacier retrieval for a petabyte-scale archive where users need sporadically bursty access. How do you choose between expedited, standard, and bulk retrieval tiers?

**Answer:** Glacier retrieval optimization is a capacity planning and cost minimization problem: (1) **Tier characteristics** — (a) **Expedited** (1-5 min): about $0.03/GB + $0.01 per request. Requires pre-provisioned capacity (100 MB/s per provisioned unit). (b) **Standard** (3-5 hrs): about $0.01/GB + $0.0004 per request. No pre-provisioning needed; available capacity shared across all requests. (c) **Bulk** (5-12 hrs): about $0.0025/GB + $0.000025 per request. Lowest priority. (2) **Provisioned capacity** — for predictable burst needs, purchase expedited provisioned capacity: 100 MB/s per unit, about $100/unit/month. With 10 units, you can retrieve 1 GB/s sustained. (3) **Smart tier selection** — implement a retrieval priority queue: (a) user-facing retrievals (customer support needs a file NOW) — expedited. (b) Scheduled analytics (nightly batch processing) — standard. (c) Deep archival restores (legal hold review, data migration) — bulk. (4) **Expedited cost containment** — at petabyte scale, expedited retrieval of even 1% of data costs $300/TB. Solution: (a) **Hot cache**: maintain an SSD cache (S3 Standard-IA) of the most frequently accessed 1% of archive objects. Cache hit — S3 Standard GET ($0.0004/GB). Cache miss — expedited restore + cache the result. (b) **Pre-warming**: for known upcoming bursts (e.g., audit starting next week), trigger standard retrievals in advance. (5) **Parallel restoration** — for large datasets, split restoration across multiple retrieval requests. Glacier retrieves are per-object; requesting 10,000 objects simultaneously fans out across the storage infrastructure. (6) **Monitoring** — CloudWatch metrics for RetrievalQueueDepth (pending requests) and RetrievalSlotUtilization. Alert when queue depth exceeds provisioned capacity — this prevents expedited requests degrading to standard speed. (7) **Vault Lock** — if the archive is for compliance (e.g., SEC 17a-4), the objects are in Glacier Vault Lock (not S3 Glacier). Retrieval requires a different API. Plan accordingly — vault lock objects cannot be expedited.

## Q18: What are the most critical security best practices for pre-signed URLs beyond short expiry? How would you design a system that generates millions of pre-signed URLs per minute?

**Answer:** Beyond basic expiry settings, pre-signed URLs require careful security architecture: (1) **Scoped permissions** — the IAM user/role used to sign pre-signed URLs should have minimal permissions. Create a dedicated signing role with exactly: s3:GetObject on arn:aws:s3:::bucket/* (not s3:*). Never use the root account key for signing. (2) **Signing key rotation** — rotate the signing IAM user's access keys every 90 days. Use AWS KMS or a dedicated key management service for HMAC keys if you are not using IAM-based signing. (3) **IP restrictions** — pre-signed URLs generated in S3's native scheme CANNOT include IP restrictions in the policy. Workaround: use a CloudFront signed URL (which supports IP, date, and path restrictions) or a custom proxy that validates IP before forwarding to S3. (4) **Bandwidth theft prevention** — a pre-signed URL for a large file ($5 movie) can be shared, and every download costs YOU. Mitigation: (a) **CloudFront signed cookies** — single sign-on generates a cookie valid for all content. (b) **Referrer validation** — CloudFront checks the Referer header of the client requesting the S3 object. Only requests from your domain are allowed. (c) **Token-based auth** — generate a short-lived auth token that the client exchanges for a pre-signed URL. The server validates the client's identity (e.g., session cookie) before issuing the URL. (5) **Replay attack prevention** — pre-signed URLs are vulnerable to replay within their validity window. Mitigation: (a) use very short expiry (30 seconds). (b) **One-time URLs**: embed a nonce in the URL, and the server tracks used nonces. This adds state to a stateless scheme. Rarely used because it requires a DB lookup at S3-side, which S3 does not support natively. (6) **High-volume generation system** — for millions of URLs/min: (a) Use a pool of signing workers, each with their own IAM credentials (to avoid rate limiting on a single access key). (b) Pre-sign URLs in Go/Rust for performance: s3Client.PutObjectRequest(bucket, key).Presign(15*time.Minute) in the AWS SDK. (c) Cache signed URLs for frequently accessed objects (with short TTL). (d) Use S3 Batch Operations to pre-sign URLs for batch jobs. (7) **Auditing** — CloudTrail logs the aws:SourceIp of the request using the pre-signed URL. If a URL is shared widely, you will see requests from many IPs — this is your detection signal.

## Q19: How does S3 handle cross-region replication (CRR) conflict resolution when the same key is written concurrently in both the source and destination regions?

**Answer:** CRR is one-way (source to destination). Conflicts arise in bi-directional replication scenarios: (1) **Unidirectional replication (standard CRR)** — conflicts do not occur because writes only happen in the source region. The destination region is read-only (for replicated data). If a user writes directly to the destination bucket, that write is NOT replicated back to the source. This means: if both region A and region B accept writes, the last writer wins based on the last-modified timestamp. (2) **Bidirectional replication (S3 Replication Time Control + custom logic)** — for active-active multi-region setups, S3 provides last-writer-wins (LWW) conflict resolution: (a) Each object version carries a Last-Modified timestamp. (b) When the replication service sees a conflict (two writes to the same key in different regions), it compares the timestamps. The write with the later timestamp wins — the earlier write is either overwritten or becomes a non-current version. (c) **Problem**: clock skew between regions. If region A's clock is 10 ms ahead, region A writes always win over region B. (3) **Solution — version vector / vector clock**: the object's metadata includes a version_vector: {region: logical_clock}. When a write happens in region A, the version vector is {A: 2, B: 1} (A's local clock advanced). When a concurrent write in region B has {A: 1, B: 2}, the conflict is detected by comparing the vector clocks: neither dominates the other (A's clock does not include B's update and vice versa). (4) **Conflict resolution strategy** — (a) **LWW**: accept the write with the higher x-amz-meta-last-modified timestamp. (b) **CRDT (Conflict-Free Replicated Data Type)**: for state that can be merged (e.g., a JSON document with per-field resolution), use a custom CRDT layer on top of S3. (c) **Manual resolution**: flag conflicting versions and let a human resolve them. (5) **S3's native behavior** — S3 does NOT natively support CRDT or manual conflict resolution for CRR. It uses simple LWW based on the object's Last-Modified field. For applications needing stronger guarantees, implement a conflict resolution service that re-reads both versions on conflict detection and writes the merged result. (6) **Delete conflict** — if delete markers are replicated, a DELETE in the source region creates a delete marker that replicates to the destination, effectively deleting the object at the destination even if the destination has a concurrent write. Mitigation: disable delete marker replication unless you understand the implications.

## Q20: Discuss S3 event notification ordering guarantees. How would you design exactly-once, ordered processing of s3:ObjectCreated events when a single key gets 1000 rapid overwrites?

**Answer:** S3 event notifications have limited ordering guarantees and are at-least-once: (1) **Native guarantee** — S3 sends events in the order they occur for a single key ONLY if the events are processed by a single consumer. However, if the notification service batches events or if the downstream (SQS, Lambda) processes them concurrently, ordering is not guaranteed. (2) **1000 rapid overwrites scenario**: if key data.csv is written 1000 times in 1 second, S3 sends 1000 notifications (one per PUT). These may arrive at SQS in any order due to: (a) internal batching in the notification pipeline, (b) network jitter, (c) SQS FIFO queue limitations (batching within a group). (3) **Solution architecture for ordered processing**: (a) Use SQS FIFO queue with MessageGroupId = bucket/key. All events for the same key go to the same group. SQS FIFO delivers messages within a group in order, one at a time. (b) However, S3 does NOT natively send events to SQS FIFO with the key as the group ID. You must use Lambda as an intermediary: S3 goes to Lambda (with S3 event source) then Lambda writes to SQS FIFO with the correct GroupId. (c) Lambda processes events in batches, de-duplicates by s3:object:versionId, and writes to SQS FIFO in order of eventTime. This re-establishes ordering. (4) **Exactly-once processing**: S3 is at-least-once (duplicates are possible). To achieve exactly-once: (a) The consumer maintains a dedup table (DynamoDB) keyed by (bucket, key, versionId). Before processing an event, query DDB. If the versionId already processed, skip. (b) Use the sequencer field in the S3 event record (a hex string that increases monotonically per key): eventRecord.s3.object.sequencer. Ignore events with a sequencer value less than or equal to last processed value for that key. (5) **Thinning / dedup in transit** — if the consumer only cares about the LATEST state (not every intermediate state), implement event thinning: maintain a last known sequencer for each key in a fast cache (Redis). When a new event arrives, check if its sequencer is greater than cached. If not, drop. Only the latest event for each key is processed. (6) **Alternative: Bucket notifications with S3 EventBridge** — EventBridge pipes offer filtering and enrichment. You can configure a filter to only pass events where the object size changed above a threshold, reducing volume. But ordering is still best-effort.

## Q21: Design the multi-part upload system for large objects in S3. How do you handle parallel uploads, abort incomplete uploads, and ensure data integrity?

**Answer:** Multi-part upload enables uploading large objects (up to 5TB) in parts of up to 5GB each (minimum 5MB per part except last). (1) **Upload flow**: (a) Client calls CreateMultipartUpload which returns an UploadId. (b) Client splits the file into parts and uploads each part in parallel (typically 10-50 concurrent parts). Each part upload returns an ETag (MD5 of the part). (c) After all parts are uploaded, client calls CompleteMultipartUpload with the list of part numbers and ETags. S3 assembles the object in order. (2) **Error handling**: (a) Failed part upload: retry with the same part number. Only the last successful upload of that part is kept. (b) AbortMultipartUpload: cancels the upload and deletes all uploaded parts. S3 automatically aborts incomplete uploads after 7 days of inactivity (configurable via lifecycle policy). (3) **Integrity**: (a) Each part is verified by MD5 (S3 computes ETag = MD5 of the part content). (b) For additional integrity, use Content-MD5 header or checksum algorithms (SHA256, CRC32) via the trailing checksum header. (c) On CompleteMultipartUpload, S3 computes the composite ETag = MD5 of concatenated part MD5s. (4) **Optimal part size**: aim for <10,000 parts (max allowed). For a 5TB file, use 500MB parts (10,000 parts exactly). For smaller files, use 50MB parts for good parallelism. (5) **Pause/resume**: store upload state (UploadId, parts uploaded) in a local DB. On resume, skip already-uploaded parts and continue.

## Q22: How do you design S3 lifecycle policies to automatically transition objects between storage classes and expire them? What are the cost implications and minimum storage durations?

**Answer:** S3 lifecycle policies automate data movement between tiers to optimize cost. (1) **Transition rules**: (a) Standard (frequently accessed) → Standard-IA (infrequent access, 30-day minimum) after 30 days. (b) Standard-IA → Glacier Instant Retrieval (millisecond retrieval, 90-day minimum) after 90 days. (c) Glacier Instant Retrieval → Glacier Deep Archive (12-hour retrieval, 180-day minimum) after 180 days. (d) Glacier Deep Archive → Expire (delete) after 365 days. (2) **Minimum storage durations**: (a) Standard-IA: 30 days. If you transition on day 5, you pay 30 days of storage. (b) Glacier Instant Retrieval: 90 days. (c) Glacier Flexible Retrieval: 90 days. (d) Glacier Deep Archive: 180 days. These are pro-rated — early deletion incurs the minimum charge. (3) **Cost optimization**: (a) Infrequently accessed data (>30 days without access) should leave Standard. (b) Use S3 Analytics to generate storage class recommendations based on access patterns. (c) Archive data that hasn't been accessed in 90 days to Glacier. (d) Deep Archive for compliance data that must be kept for 7 years. (4) **Expiration actions**: (a) Delete expired object versions (versioning-enabled buckets). (b) Delete incomplete multi-part uploads older than 7 days. (c) Expire current object versions at end-of-life. (5) **Lifecycle filtering**: apply rules by prefix (logs/), tag (environment=prod), or object size (only archive objects > 128KB — smaller objects are not cost-effective to archive due to per-object overhead). (6) **Monitoring**: Lifecycle transitions are billed per 1000 requests. Monitor transition costs via Cost Explorer. If an object is retrieved shortly after transitioning to Glacier, the retrieval cost + early deletion fee may exceed the storage savings — this is called "lifecycle thrashing."

## Q23: Design cross-region replication (CRR) for S3. How do you replicate objects across regions with low RPO, handle delete markers, and deal with replication errors?

**Answer:** CRR provides automatic asynchronous replication across AWS regions. (1) **Setup**: (a) Enable versioning on both source and destination buckets. (b) Configure IAM role with permissions to read from source and write to destination. (c) Define replication rules: replicate all objects or by prefix/tag. (2) **Replication flow**: (a) When an object is PUT to the source, S3 generates a replication event. (b) S3 reads the object and metadata from source, writes to destination using the same key, metadata, and storage class. (c) The destination object retains the source's last-modified timestamp. (3) **RPO (Recovery Point Objective)**: (a) CRR typically completes within 15 minutes for most objects (p99 < 15 min). (b) For compliance-critical data, use S3 Replication Time Control (RTC) which guarantees replication within 15 minutes (99.99% of objects). (4) **Delete markers**: (a) CRR does NOT replicate delete markers by default. This prevents a malicious or accidental delete in the source from deleting the replica. (b) If you want to replicate delete markers (e.g., for compliance), enable "Replicate delete markers" — use with caution. (c) Versioned deletes (DELETE with a specific VersionId) are not replicated. (5) **Replication errors**: (a) Failed replications (permissions, destination bucket full, KMS key issues) are reported in CloudWatch Metrics and S3 Inventory reports. (b) The Replication Metrics dashboard shows: pending replication count, replication latency, and replication error rate. (c) Use S3 Batch Operations to retry failed replications. (6) **Chaining**: CRR destinations can be in different regions. For multi-region active-active, use bi-directional CRR (source → dest and dest → source). But beware of replication loops — S3 detects and breaks loops by not replicating objects that were themselves replicated.

## Q24: How do you design S3 Access Points to manage data access for thousands of applications sharing a single bucket? What are the permission models and VPC integration options?

**Answer:** S3 Access Points provide granular, per-application access control to shared buckets. (1) **Access Point as an entry point**: each application gets its own Access Point with a unique DNS name (e.g., `app1-bucket.s3-accesspoint.us-east-1.amazonaws.com`). (4) **Scaling**: a single bucket can have up to 10,000 Access Points. Each Access Point acts as its own namespace with independent policy and audit trail. (5) **Auditing**: CloudTrail logs include the Access Point ARN (arn:aws:s3:us-east-1:account:accesspoint/app1). This allows per-application audit trails. (6) **Multi-account access**: Access Points can be shared across accounts using AWS Resource Access Manager (RAM). An application in Account B can use an Access Point created in Account A.

## Q25: Design a system using S3 Select and Athena for querying data directly on S3 without loading into a database. What are the performance characteristics, file format requirements, and cost trade-offs?

**Answer:** S3 Select and Athena provide serverless querying of data on S3. (1) **S3 Select**: (a) Works on a single object. Client sends a SQL query and S3 filters the object server-side, returning only matching rows. (b) Supported formats: CSV, JSON, Parquet. (c) Performance: 10x faster and 80% cheaper than downloading the full object and filtering client-side. (d) Use case: extract specific columns from large compressed CSV files. (2) **Athena**: (a) Presto/Trino-based engine that queries across multiple S3 objects. (b) Schema-on-read: define table schemas in the Glue Data Catalog pointing to S3 locations. (c) Performance: query speed depends on data format and partitioning. (3) **File format impact**: (a) CSV: slowest — Athena must scan all rows. (c) Parquet: columnar, compressed, with min/max statistics per row group. Athena can skip row groups that don't match the query filter. 10-100x faster than CSV for typical queries. (d) ORC: similar to Parquet. (4) **Partitioning**: (a) Partition data by commonly filtered columns: `s3://bucket/dt=2025-01-01/hr=12/`. (b) Athena uses partition pruning — if you filter by `dt = '2025-01-01'`, only that partition's files are scanned. (c) Avoid too many partitions (>10K leads to slow metadata operations). (5) **Cost**: (a) Athena charges per byte scanned ($5/TB). Costs can balloon on unoptimized queries. (b) Use columnar formats + partitioning to minimize bytes scanned. (c) Set query cost limits (max bytes scanned per query) to prevent runaway costs. (d) S3 Select is cheaper per-byte-filtered than Athena for single-object queries. (6) **Performance recommendations**: (a) Convert data to Parquet with snappy compression. (b) Partition by date/hour. (c) Use `OPTIMIZE` or `VACUUM` (Iceberg/Delta Lake) for ACID transactions. (d) Convert to Iceberg/Delta Lake format for upserts and time-travel queries.

## Q30: Design a disaster recovery strategy for S3-based storage. How do you handle accidental deletion, region failure, and ransomware with versioning, replication, and Object Lock?

**Answer:** S3 disaster recovery requires defense in depth across multiple layers. (1) **Layer 1 — Versioning**: (a) Enable versioning on all buckets. When an object is deleted, a DeleteMarker is created, but previous versions remain. (b) MFA Delete: require multi-factor authentication to permanently delete versions. This prevents an attacker with console access from wiping versions. (c) Lifecycle policy: expire old noncurrent versions after N days (e.g., keep last 90 days of versions). (2) **Layer 2 — S3 Object Lock**: (a) **Retention mode**: COMPLIANCE — object versions cannot be deleted or overwritten by any user, including root, until the retention period expires. (b) **Legal hold**: prevent deletion indefinitely (for litigation). (c) Use Object Lock for WORM (Write Once Read Many) compliance — financial records, medical data. (3) **Layer 3 — Cross-Region Replication (CRR)**: (a) Replicate all objects to a bucket in another AWS region. (b) If a region fails, promote the replica bucket: redirect DNS or use Route53 failover to the alternate region. (c) RPO: 15 minutes (or 1 minute with RTC). RTO: minutes (DNS change propagation). (4) **Layer 4 — Backup with AWS Backup**: (a) AWS Backup supports S3, creating periodic backups with retention policies. (b) Backup vault enforces backups to be immutable (prevent deletion of backups even with admin access). (c) Cross-account backup: store backups in a different AWS account to prevent account-level compromise. (5) **Layer 5 — Ransomware detection**: (a) Monitor for mass DELETE/ PUT operations via CloudTrail and CloudWatch Anomaly Detection. (b) Alert if: delete operations spike >10x normal, or S3 Object Lock bypass attempts. (c) Automated response: apply a legal hold to all objects in the bucket, lock the bucket policy. (6) **Recovery procedure**: (a) Accidental delete: use S3 Versioning to restore the latest non-deleted version. (b) Ransomware (objects encrypted): restore from unencrypted previous versions. (c) Region failure: promote CRR replica bucket, update DNS/application config to point to the replica region. (d) Account compromise: use backup in a different account, restore to a new bucket. (7) **Testing**: Perform an annual DR drill: simulate a region failure, and verify that you can fail over to the replica within the RTO. Simulate a ransomware event and verify version restoration works.
