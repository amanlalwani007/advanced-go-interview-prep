# Chapter 24 — Design S3-like Object Storage

## Q1: Describe the write path in S3. When a client uploads a 100 MB file, what happens inside the S3 system before the client gets a success response?

**Answer:** The S3 write path: (1) **Frontend (API gateway)** — the client sends a `PUT /bucket/key` request to the S3 API gateway (AWS's globally distributed Request Router). The gateway authenticates the request (Signature V4), validates the bucket/key, and checks permission (IAM policy + bucket policy). (2) **Partition placement** — the gateway consults the placement service to determine which storage partition (shard) owns this object. S3 partitions keys by the hash of `(bucket, key)`. (3) **Storage node (WAL write)** — the request is forwarded to the primary storage node for that partition. The node writes the 100 MB payload to its local disk first as a Write-Ahead Log (WAL) and immediately replicates to N-1 secondary nodes (synchronous replication within the same data center). (4) **Acknowledgment** — once the WAL(s) are written and N replicas confirmed, the primary responds with "200 OK" to the client. (5) **Async compaction** — after acknowledgment, the storage node asynchronously converts the WAL entry into the final storage format (S3 indexes objects by key prefix for efficient range scans). During this time, the object is still readable because the WAL entry is indexed. (6) **Metadata update** — the metadata store (keyed by `(bucket, key_hash)`) is updated with the object's storage location, size, ETag (MD5), and timestamp. This metadata is replicated across three data centers for durability. The client sees success after step (4) — typically within 200-500 ms for a 100 MB file on a fast connection.

## Q2: S3 supports strong read-after-write consistency for PUT of new objects. How does it achieve this at global scale without sacrificing availability?

**Answer:** S3's strong consistency for PUTs (new objects) is achieved through: (1) **Synchronously replicated metadata** — the metadata store for a given key is replicated synchronously across three AZs within the region before the PUT returns success. Any subsequent GET in that region will see the latest metadata. (2) **Quorum-based reads** — a GET request reads from the metadata store quorum (majority of replicas), ensuring it sees the latest committed write. (3) **Partition-level consistency** — each partition (key range) has a single leader that serializes writes. All reads go to the leader or to a read replica that has replicated the latest metadata via a consistent replication stream. (4) **Key difference from eventually-consistent systems (e.g., DynamoDB)**: S3 does NOT use last-writer-wins (LWW) or conflict resolution. Instead, it uses a single leader per partition for writes, and all replicas apply writes in the same order. This is a more expensive approach (lower write throughput per partition) but provides stronger guarantees. S3's trade-off: it can "only" handle millions of PUTs per second per prefix (vs. DynamoDB's tens of millions) but guarantees read-after-write consistency.

## Q3: How does S3 support prefix-based listing (e.g., `ListObjects` with prefix `photos/2024/`)? What data structure enables efficient range queries over billions of keys?

**Answer:** S3 stores object metadata in an ordered key-value store (ordered by `(bucket, key)`). The data structure is a **B-Tree or LSM-Tree** that maintains keys in sorted order: (1) **Key ordering** — objects within a bucket are stored sorted by UTF-8 binary order of the key. All keys starting with `photos/2024/` are contiguous in this ordering. (2) **Partitioning** — S3 partitions the key space into ranges. Each partition covers a contiguous range of keys (e.g., `photos/a` to `photos/m`). The partition boundaries are managed by the placement service. (3) **List request** — when the client calls `ListObjects?prefix=photos/2024/&delimiter=/`, the system finds the first partition that could contain keys with that prefix using a binary search on the partition metadata. It then scans the partition's local index from the first key matching the prefix, reading sequentially until the prefix no longer matches or the page size limit is reached. (4) **Delimiter-based grouping** — the `/` delimiter groups keys into common prefixes (pseudo-folders). The scan detects key boundaries at the delimiter and returns them as `CommonPrefixes` entries without actually scanning all sub-keys. (5) **Performance** — a list operation on a prefix with 1000 keys reads exactly 1000 index entries. A list operation on a prefix with 1M keys reads only the requested page (default 1000) because the B-Tree efficiently seeks to the correct starting position.

## Q4: Explain how S3 handles multi-part upload for large objects, including the life cycle of upload parts and what happens if the upload is aborted.

**Answer:** Multi-part upload is designed to overcome single-stream throughput limitations and network unreliability: (1) **Initiation** — client sends `POST /bucket/key?uploads`. S3 creates an `upload_id` and returns it. No storage is allocated yet. (2) **Part upload** — client uploads each part independently: `PUT /bucket/key?uploadId=<id>&partNumber=1`. Each part (1 MB to 5 GB) is stored as a separate object behind the scenes. The server returns an ETag for that part. (3) **Completion** — client sends `POST /bucket/key?uploadId=<id>` with the list of (partNumber, ETag) in order. S3 assembles the final object by concatenating the parts in order. The multi-part upload ID is closed; the assembled object is now referenced as the single object. (4) **Abort** — if the client sends `DELETE /bucket/key?uploadId=<id>`, or the upload exceeds the timeout (typically 7 days for incomplete parts), all uploaded parts are garbage-collected by a background reaper process. (5) **Retry behavior** — if a single part upload fails (network timeout), the client retries only that part, not the entire object. The uploaded parts are independent, so retried part uploads with the same part number simply overwrite the old part. (6) **Optimal part size** — for a 100 GB file, use 100 parts of 1 GB each. This allows parallel upload of 10-20 parts simultaneously, saturating the client's bandwidth. (7) **Backend assembly** — the completion request creates the final object manifest: `{object_key, parts: [{part_number: 1, storage_loc: <path>, size: 1GB}, ...]}`. The object's data is not copied — the parts remain in their original locations and the manifest references them.

## Q5: How does S3 implement versioning? When a user deletes a versioned object, what actually happens?

**Answer:** S3 versioning creates a new object version for each write (PUT, POST, COPY) to the same key. (1) **Version ID** — each object version is assigned a unique version ID (a monotonically increasing number or a UUID). The object's key + version ID uniquely identifies a specific version. (2) **Storage** — different versions of the same key are stored as separate objects in the underlying storage. They share no data unless they have identical content (in which case S3's dedup may point to the same underlying blocks). (3) **Delete marker** — when the client sends `DELETE /bucket/key` on a versioned bucket, S3 does NOT remove the object. Instead, it inserts a "delete marker" — a zero-byte object version with a special flag. A subsequent GET request without a version ID sees no object (the delete marker is the latest version). But a GET with the specific version ID still retrieves that version. (4) **Permanent deletion** — `DELETE /bucket/key?versionId=<id>` permanently deletes that specific version. This is an irreversible, idempotent operation. (5) **Storage management** — versioning multiplies storage costs because old versions are retained indefinitely. S3 Lifecycle Policies can automatically expire non-current versions after a configurable period. (6) **MFA Delete** — for critical buckets, S3 supports MFA (Multi-Factor Authentication) Delete — a DELETE request is rejected unless it includes a valid MFA token. This prevents accidental or malicious deletion of versioned objects.

## Q6: Design the S3 pre-signed URL mechanism. How does it work, and how do you prevent a pre-signed URL from being used to upload more data than intended?

**Answer:** Pre-signed URLs delegate access to a specific object for a limited time, without embedding AWS credentials in the client app. (1) **Generation** — the server creates a URL: `GET /bucket/key?AWSAccessKeyId=<access_key>&Expires=<expiry_timestamp>&Signature=<HMAC-SHA256(secret_key, string_to_sign)>`. The `string_to_sign` includes HTTP method, expiry, and the resource path. (2) **Verification** — when the client sends a request with the pre-signed URL, S3 recomputes the signature using the signer's secret key (stored securely on S3's side). If the signature matches and the current time < Expires, the request is authorized. (3) **Preventing over-upload** — pre-signed PUT URLs allow uploading any content up to the object's full size. To limit the upload size, the signing server includes a `content-length` range in the `string_to_sign`, or uses a custom header check on the server side (not supported by S3 directly — you would need a proxy). Alternative: use a multi-part upload with pre-signed URLs for each part, and the completion request is only sent after the server validates total size. (4) **Security considerations** — pre-signed URLs expire. A leaked pre-signed URL grants access to the specific object for the duration of its validity. For sensitive data, use short expiry (60 seconds) and HTTPS. Pre-signed URLs are tied to a specific HTTP method — a pre-signed GET URL cannot be used to PUT or DELETE. (5) **Bandwidth control** — pre-signed URLs commit to no bandwidth guarantee. At extreme scale, the requester might experience throttling (HTTP 503). For guaranteed throughput, use S3 Transfer Acceleration or AWS PrivateLink.

## Q7: How does S3 implement cross-region replication (CRR) with a lag of typically under 15 minutes?

**Answer:** CRR asynchronously replicates objects from a source bucket to a destination bucket in a different region. (1) **Replication trigger** — when an object is PUT (or DELETE, if enabled) into the source bucket, S3 records the event in an internal replication log. This log is stored in a highly durable, ordered queue (similar to Kafka). (2) **Replication worker** — a fleet of cross-region replication workers read from the replication log, fetch the object from the source region's storage, and PUT it into the destination bucket. PUTs to the destination bucket are standard S3 PUTs (including full lifecycle processing). (3) **Batching** — the workers batch multiple objects into a single replication request. If 1000 objects are uploaded to the source bucket within seconds, they are replicated as a batch to the destination, amortizing the per-request overhead. (4) **Lag sources** — typical lag is 5-15 minutes. Sources of lag: cross-region network latency (50-200 ms), destination bucket PUT rate limits (3,500 PUT/s per prefix), worker autoscaling delays. (5) **Consistency** — CRR provides eventual consistency. If the source bucket has strong consistency for PUTs, the destination bucket may lag. If you read from the destination immediately after a PUT to the source, you may not see the object. (6) **Retry** — if the destination region is unavailable, the replication log queues the events and retries indefinitely until successful. Objects are not lost during destination region outages. (7) **Replication time control (RTC)** — S3 offers RTC which provides a 99.99% SLA on replication within 15 minutes, by dedicating more resources to the replication pipeline (at additional cost).

## Q8: How does S3 handle the "eventual consistency" of LIST operations? Why does LIST not have strong consistency even though individual GETs do?

**Answer:** S3's consistency model differentiates between reading a specific key (GET) and reading a list of keys (LIST). (1) **GET consistency** — strong read-after-write consistency for individual keys. This is possible because the metadata is replicated synchronously within a region and read from a quorum. (2) **LIST consistency** — S3 offers "list consistency" which is read-after-write for NEW objects in most cases, but not guaranteed for overwrites/deletes. Reason: LIST operations scan a range of keys across potentially multiple partitions. When a new object is written, the partition index is updated. However, S3 may add new partitions dynamically as a bucket grows. A LIST operation that starts reading from one partition might miss objects written to a newly created partition if the partition metadata cache hasn't been updated yet. (3) **Practical impact** — for most use cases, S3's LIST is consistent for new objects after a short delay (typically <1 second). For overwrite/deletes, the old listing may persist for a brief period because the index update may lag. (4) **Partition splitting** — when a partition grows too large, it's split into two partitions. During the split, the new partition map is propagated asynchronously. A LIST operation that cached the old partition map may not see objects in the new partition. S3 has since improved this and now offers stronger list consistency guarantees, but it's not as strong as GET consistency.

## Q9: Design S3 Glacier's retrieval mechanism. How does an "expedited" retrieval return data within 1-5 minutes for archival storage?

**Answer:** Glacier archival storage uses a fundamentally different storage medium than S3 standard. Data is stored on tape, optical media, or deep HDD spindles. (1) **Storage format** — data is stored in large shards (hundreds of GBs) with heavy erasure coding (e.g., 24-of-36) across multiple devices. This minimizes storage cost at the expense of read latency. (2) **Retrieval request** — when the client initiates an expedited retrieval, the request is queued at high priority. The system locates the shards containing the requested data from the index (a metadata DB mapping object → shard location and erasure coding layout). (3) **Data retrieval** — the system pre-allocates a cache of short-term storage (SSD scratch space). The shard location is passed to the robotic retrieval queue (if tape) or directly to the disk infrastructure (if HDD). For expedited retrievals, the system may maintain a hot cache of recently accessed objects. (4) **Reconstruction** — the data (erasure-coded shards) is read and decoded back into the original object. This reconstruction is CPU-intensive but parallelizable — the system can fan out the read and decoding across many nodes. (5) **Delivery** — the reconstructed data is written to the SSD scratch space and served to the client via the standard S3 GET API. The entire process completes in 1-5 minutes for expedited, 3-5 hours for standard, and 5-12 hours for bulk retrieval. (6) **Cost** — expedited retrievals require pre-provisioned capacity or have higher per-GB costs. The system must maintain enough SSD scratch space and compute capacity to handle the peak expedited retrieval rate.

## Q10: How would you design S3's event notification system (S3 → SQS/Lambda/SNS)? How does it guarantee at-least-once delivery without duplication?

**Answer:** S3 event notifications use an internal change log: (1) **Event capture** — every object write, delete, or restore operation is recorded in an internal event log (the same log used for CRR). (2) **Notification service** — a notification service reads from the event log and fans out to configured destinations (SQS, SNS, Lambda). (3) **Batching** — events are batched (e.g., up to 10 events or 10 KB per batch) to reduce the per-event overhead. The batch is written to the destination queue/topic as a single message. (4) **Delivery guarantees** — S3 provides "at-least-once" delivery. The notification service tracks delivery state. If the destination is unavailable or returns an error, the notification is retried with exponential backoff (up to several days). (5) **Deduplication** — destinations should be idempotent. S3 does not guarantee exactly-once delivery of notifications. If the notification service crashes after sending but before acknowledging the event log offset, the event will be re-sent. The destination (e.g., Lambda) should handle duplicates using the object's `version_id` or `event_time` as a dedup key. (6) **Event types** — `s3:ObjectCreated:*`, `s3:ObjectRemoved:*`, `s3:ObjectRestore:*`, `s3:Replication:*`. Filters for prefix/suffix reduce the notification volume by only sending events that match a specific prefix/filter rule (e.g., only `prefix=incoming/`). (7) **Scaling** — each S3 bucket can have up to 100 notification configurations. The system processes events from the event log continuously, with throughput scaling based on the number of shards/workers consuming from the log.
