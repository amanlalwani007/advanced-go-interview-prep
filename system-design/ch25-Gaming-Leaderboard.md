# Chapter 25 — Design a Real-time Gaming Leaderboard

## Q1: Design a real-time leaderboard for an online game with 50 million daily active users. The leaderboard must show the top 100 players globally and the current user's rank, updating within seconds of a score change.

**Answer:** The leaderboard data model uses Redis sorted sets with a sharded architecture: (1) **Score submission** — when a player finishes a game, their new score is sent to the API: `{user_id, score, game_mode}`. (2) **Redis sorted set** — the global leaderboard per game mode is stored as a Redis sorted set: `ZADD leaderboard:global:<mode> <new_score> <user_id>`. ZADD is O(log N) per insertion. For 50M players, the sorted set has 50M members, and each ZADD takes ~100 µs. (3) **Top 100 query** — `ZREVRANGE leaderboard:global:<mode> 0 99 WITHSCORES`. Returns the top 100 players in O(log N + 100) time. (4) **User rank query** — `ZREVRANK leaderboard:global:<mode> <user_id>`. Returns the user's rank as O(log N). (5) **Scaling bottleneck** — a single Redis sorted set with 50M members requires significant RAM (~2 GB for 50M user IDs + scores). Writes: ZADD throughput is ~50K-100K ops/sec on a single Redis node. For 50M DAU with peak concurrent players (~2M), the write rate could exceed 100K/sec. Solution: **shard by score range** — partition players into brackets by score (e.g., 0-1000, 1000-5000, 5000-20000, 20000+). The top N global leaderboard is computed by merging the top K from each bracket (a bounded merge, O(brackets × K)). (6) **Alternative** — use a dedicated leaderboard service (like AWS GameLift or a custom service built on DynamoDB) instead of Redis for persistence and auto-scaling.

## Q2: The leaderboard must handle ties — if two players have the same score, the one who achieved it first should rank higher. How do you implement this in Redis?

**Answer:** Redis sorted sets allow a "double-sort" by encoding the timestamp into the score using a fractional part: (1) **Score encoding** — encode the score as `actual_score + (1 - tiebreaker / max_tiebreaker)`. The tiebreaker is `(max_possible_timestamp - current_timestamp) / max_possible_timestamp`, so an earlier timestamp gets a higher fractional part. E.g., for score 1000 achieved at time 1000 and score 1000 at time 2000: `1000 + (1 - 1000/1e12) ≈ 1000.999999999` and `1000 + (1 - 2000/1e12) ≈ 1000.999999998`. The first player has a slightly higher score and ranks above. (2) **Retrieval** — when retrieving, the leaderboard service strips the fractional part: `display_score = floor(actual_score)`. (3) **Alternative with two sorted sets** — store scores in one sorted set (`score:leaderboard`) and timestamps in another (`time:leaderboard`). When querying, if scores are equal, break ties by looking up the timestamp. However, this requires custom logic client-side and is less efficient. (4) **Redis rankscore notation** — if using a custom leaderboard service, the service can adjust ranks after retrieval by comparing timestamps for equal-score groups, but this is O(K log K) for the top K. The first approach (encoding tiebreaker into score) is the most efficient and is the industry standard.

## Q3: How do you design the leaderboard's anti-cheat system — detecting players with impossibly high scores that suggest hacking?

**Answer:** A pipeline of validation checks: (1) **Score distribution monitoring** — maintain a histogram of scores per game mode. If a player's score falls >6 standard deviations from the mean, flag for review. (2) **Rate-of-change detection** — if a player goes from 0 to the top 10 in 5 minutes, it's likely cheating. The system tracks per-player score increase per minute and flags anomalous rates. (3) **Input validation** — scores must be signed by the game client. The game client sends `{user_id, score, nonce, signature}` where `signature = HMAC(secret_key, user_id || score || nonce)`. The server verifies the signature. (4) **Gameplay metrics correlation** — send telemetry alongside the score: play time, actions per minute, accuracy, etc. If the score is high but gameplay metrics are too low (e.g., 100% accuracy on a shooter game is suspicious), flag it. (5) **ML-based detection** — a model trained on known cheater behavior detects subtle patterns: impossible reaction times, pixel-perfect aim, identical play patterns across accounts. (6) **Consequence system** — flagged scores are held in a "pending review" state and not shown on the leaderboard. After manual or automated review, the score is either confirmed or rejected. Rejected scores are removed, and the account may be suspended. (7) **Ghosting** — the anti-cheat system should not reveal its detection criteria publicly to avoid helping cheaters adapt.

## Q4: Design the "friends leaderboard" — a user should see their rank among their friends, not just globally. How does this scale when a user has 500 friends?

**Answer:** The friends leaderboard is computed per-user on read (not precomputed): (1) **Friend list storage** — the social graph is stored in a separate service (e.g., a Neo4j graph DB or Redis sets: `SMEMBERS friends:<user_id>` → list of 500 friend IDs). (2) **Leaderboard query** — when a user opens the friends leaderboard, the API server: (a) fetches the user's friend list from the social graph (O(1) lookup). (b) For each friend ID, fetches their score from the Redis sorted set: `ZSCORE leaderboard:global <friend_id>`. This is O(F) where F = number of friends (typically ≤ 500). (c) Computes the rank by calling `ZREVRANK` for each friend OR by fetching the top 500 scores and sorting locally. (3) **Optimization** — use a Lua script that does a batch fetch in one Redis call: `local scores = {}; for i, id in ipairs(friend_ids) do scores[i] = redis.call('ZSCORE', KEYS[1], id) end`. This reduces network round-trips from 500 to 1. (4) **Caching** — friend leaderboard results are cached in a local cache (e.g., in-process LRU with 10-second TTL) keyed by `leaderboard:friend:<user_id>`. When a friend submits a new score, the invalidation is scoped: publish `INVALIDATE friend_leaderboard:{user_id}` to a Redis pub/sub channel consumed by API server instances. (5) **Alternative** — maintain a separate sorted set per user for their friends' scores: `ZADD leaderboard:friend:<user_id> <score> <friend_id>`. On each friend's score update, update all their friends' sorted sets. This is O(FriendsCount) per score update — expensive for users with many friends but makes reads ultra-fast. This is the approach used by Facebook for their "Friends Leaderboard" in games.

## Q5: How would you design a weekly/monthly leaderboard that resets? What happens to the old scores?

**Answer:** Leaderboard reset with time partitioning: (1) **Temporal key structure** — store leaderboards with a time period qualifier: `leaderboard:global:weekly:2024-W27`, `leaderboard:global:monthly:2024-07`. When the period ends, the system switches to a new key. (2) **Archival** — the old leaderboard's sorted set is renamed (e.g., `RENAME leaderboard:global:weekly:2024-W27 leaderboard:archive:weekly:2024-W27`). This preserves the data for historical queries without halting writes. (3) **Reset moment** — at the period boundary (e.g., Monday 00:00 UTC), a cron job or scheduled function atomically switches the "active" key. The old key is archived to Redis (or exported to a long-term store like S3 as a CSV/Parquet for historical analysis). (4) **Rolling retention** — Redis stores the last 4 weeks and last 12 months of leaderboards. Older data is exported to S3 and the Redis keys are deleted. (5) **Awards processing** — at period end, a Spark/MapReduce job processes the finalized leaderboard and distributes rewards (badges, in-game currency, titles) to top players based on their final rank. (6) **Consistency** — during the final minutes of a period, there may be a rush of last-minute score submissions. The finalized leaderboard is captured at exactly the period boundary. Writes that arrive after the boundary go to the new period, not the old one. This is enforced by the leaderboard service checking the current time period.

## Q6: Design the "live" leaderboard that updates in real-time on the client. How does the client get notified of rank changes without polling?

**Answer:** Push-based updates via WebSocket: (1) **WebSocket connection** — when the player opens the leaderboard UI, the mobile/web client establishes a persistent WebSocket connection to the notification service. (2) **Subscriptions** — the client subscribes to events for: (a) the player's own rank change, (b) the top 100 list changes, (c) friend list rank changes (if on friends tab). (3) **Event flow** — when a score is submitted: (a) the leaderboard service updates Redis. (b) If the new score affects the top 100 (the new score is higher than the current 100th place), the leaderboard service publishes a `top100_changed` event to a Redis pub/sub channel `leaderboard:events:global`. (c) If the user's rank changed (moved up/down), publish a `rank_changed:{user_id}` event. (4) **Fan-out** — the notification service consumes from the Redis pub/sub and fans out to WebSocket connections. For the top 100, it wants to notify all ~50K players currently viewing the leaderboard. This is done with a shared connection pool: each notification server holds WebSocket connections for 10K clients. The server broadcasts the event to all connected clients by iterating the connection map (O(10K) per event, which is fast in memory). (5) **Throttling** — rate-limit events to at most 1 update per second per leaderboard. If multiple scores arrive within a second, the system sends only the latest state (not every intermediate state). This prevents overwhelming the client with 100 updates/second. (6) **Fallback** — if WebSocket is unavailable (poor connection), the client falls back to polling the leaderboard API every 10 seconds.

## Q7: How do you handle the "hot spot" problem where millions of players are all looking at the same leaderboard and scores are changing rapidly?

**Answer:** The hot spot problem manifests as high read traffic on the same Redis key (the top 100 sorted set). Mitigations: (1) **Caching at the API layer** — the top 100 leaderboard is cached in the API server's local memory with a 1-second TTL. 1M reads per second → API serves 999K from local memory, 1K from Redis. This reduces Redis read load by 1000×. (2) **Read replicas** — use Redis read replicas (Redis Cluster with replicas). Write goes to the primary; reads go to any replica. This scales read throughput linearly with the number of replicas. (3) **Stale-while-revalidate** — serve the cached top 100 (up to 5 seconds old) and asynchronously refresh the cache. Admits slightly stale data but guarantees the client never waits. (4) **Write coalescing** — multiple score updates within a 100 ms window are batched. Instead of 100 separate ZADD commands, the leaderboard service coalesces them into one ZADD with multiple score-member pairs: `ZADD leaderboard 1000 user1 1500 user2 2000 user3`. This reduces write overhead. (5) **Partitioned reads** — instead of reading the top 100 from the global set directly, maintain a separate sorted set just for the top 1000 candidates: when a score is submitted, check if it's in the top 1000 range; if so, add it to a "hot" set. Readers read from the hot set, which has at most 1000 members (vs. 50M in the full set). This is called a "hot shard" pattern.

## Q8: How would you design a "nearby" leaderboard showing top players in the same geographical region or the same skill level bracket?

**Answer:** Multi-dimensional leaderboard with server-side grouping: (1) **Region leaderboard** — use the user's region (determined by IP geo-lookup at login) to assign a region tag. Maintain per-region sorted sets: `leaderboard:region:us_east`, `leaderboard:region:europe_west`. On score submission, the server tags the user's region and updates both the global and the regional sorted sets. (2) **Skill bracket** — assign players to skill brackets (Bronze, Silver, Gold, Platinum, Diamond) based on their current score or a separate MMR (Match Making Rating) system. `leaderboard:bracket:diamond`, `leaderboard:bracket:gold`. (3) **Update complexity** — on each score submission, the system potentially updates 3-4 sorted sets: global, region, skill bracket, friends. Each ZADD is O(log N). If the player's score crosses a bracket boundary (e.g., from Gold to Platinum), the system moves them: ZREM from Gold, ZADD to Platinum. (4) **Region + bracket combined** — for the most specific view (`top players in Europe in Diamond rank`), use a combined key: `leaderboard:region:europe:bracket:diamond`. This is precomputed on score submission, not on read. Trade-off: more write amplification (each score update writes to combined keys) but faster reads. (5) **Bracket boundary management** — bracket thresholds are dynamic (e.g., top 20% of players are Diamond). A background job recalculates thresholds every hour based on the score distribution across all players.

## Q9: Design the persistence and disaster recovery for the leaderboard. What happens if the Redis cluster goes down?

**Answer:** A leaderboard must survive Redis failures: (1) **Redis persistence** — enable Redis AOF (Append-Only File) with fsync every second. In the event of a crash, at most 1 second of data is lost. (2) **Database backup** — every 5 minutes, the Redis leaderboard is snapshotted to a relational database (PostgreSQL) by reading the entire sorted set and writing to a `leaderboard_snapshots` table: `INSERT INTO leaderboard_snapshot (period, rank, user_id, score, timestamp) VALUES ...`. This is an async batch job. (3) **WAL-based capture** — the leaderboard service writes every score change to a WAL (Write-Ahead Log) in Kafka (append-only). The WAL is consumed by a backup service that writes to S3 (Parquet format). If Redis is completely lost, the leaderboard can be rebuilt from the WAL (replay all events from Kafka to rebuild the Redis sorted set). (4) **Warm standby** — a second Redis cluster (in a different AZ) is kept in sync via async replication (Redis replication). If the primary crashes, DNS-based failover switches to the secondary within 30 seconds. (5) **Graceful degradation** — if both Redis clusters are down, serve the leaderboard from the PostgreSQL snapshot (cached in local API server memory). The data may be up to 5 minutes stale. Show a "Last updated: X minutes ago" banner on the leaderboard UI.

## Q10: How would you design a weighted leaderboard where scores are a function of multiple factors (e.g., kills × accuracy × win_streak) and the scoring formula changes each season?

**Answer:** (1) **Score computation as a service** — score is computed server-side, not client-side. The game client submits raw events: `{user_id, kills: 15, deaths: 3, assists: 5, match_duration: 300}`. The server runs the scoring engine: `score = kills × 100 + accuracy_bonus(kills, deaths) + win_streak_bonus(user_id)`. This prevents client-side score manipulation. (2) **Configurable formula** — the scoring formula is defined in a config file or DSL that can be updated at season boundaries without redeploying: `formula = """score = kills * 100 + (kills / GREATEST(deaths, 1)) * 50 + (SELECT win_streak FROM users WHERE id = user_id) * 200"""`. (3) **Season change** — at the start of a new season, the leaderboard is reset. However, the formula may change. The system loads the new formula and starts applying it to all new score submissions. (4) **Historical comparison** — old season leaderboards are preserved as-is (with the old formula) and are read-only. New season uses the new formula. Users' seasonal rankings are stored per-season. (5) **Real-time score recomputation** — the scoring engine must be fast (<5 ms per event). It uses a rules engine (e.g., a caching expression evaluator) that compiles the formula to bytecode. For complex scoring that depends on historical data (like win streaks), the engine fetches the user's current streak from Redis: `GET user:{user_id}:win_streak`. (6) **Testing** — before a new season, the formula is tested on historical data to ensure score distributions are reasonable (no exploits, no scores exceeding uint64 limits). The old and new scores for a sample of users are compared. If the new formula produces wildly different rankings, it's adjusted before rollout.

## Q11: Design a multi-dimensional leaderboard that shows a player their rank filtered by region, skill bracket, and friends simultaneously. How do you compute this without precomputing every combination?

**Answer:** Multi-dimensional leaderboards must balance write amplification against read latency: (1) **Dimension hierarchy** — not all dimensions are combined at write time. Instead, store base dimensions: leaderboard:global, leaderboard:region:{region_id}, leaderboard:bracket:{bracket_id}. On score update, write to all relevant base sets (O(dimensions) ZADDs). (2) **Intersection at read time** — when a user requests region=US_East + bracket=Diamond + friends, the API server: (a) fetches the user's friend list (500 ids). (b) Fetches the user's region (from session). (c) Fetches the user's bracket (from their current score). (d) Calls ZINTERSTORE or ZUNIONSTORE with WEIGHTS to compute the intersection of leaderboard:region:us_east and leaderboard:bracket:diamond, then filters for friend IDs. This is O(N + M + F) where N and M are set sizes (Diamond bracket may have 10K players, region may have 1M). (3) **Optimization — precomputed hot intersections**: precompute the most commonly requested dimension combinations (e.g., region+global) in a background job every minute. For long-tail combinations (e.g., region+skill+friends), compute on read with a read-through cache (10s TTL). (4) **Bloom-filter pre-filter** — before computing a friends-based leaderboard, check each friend against a Bloom filter for the region+bracket set. If the friend is definitely NOT in the bracket, skip their score fetch. Reduces ZSCORE calls from 500 to about 50 (only friends in the same bracket). (5) **Fallback for extreme scale** — for games with 100M+ players, precompute the top-N combinations: the top 100 globally, top 50 per region, top 50 per bracket. The friends leaderboard is computed by: fetch friends, get their global ranks, sort. This requires no intersection but shows global rank not within-bracket rank. (6) **Caching strategy** — dimension combination results are cached in a distributed cache (Redis or Memcached) with TTL of 5-30 seconds depending on how live the leaderboard needs to be. A score update invalidates the cache for the affected dimensions.

## Q12: Design a real-time score update fan-out system. When a player's score changes, it must update global, regional, bracket, and friends leaderboards within 500ms.

**Answer:** Fan-out updates must be fast and reliable: (1) **Event-driven update** — the game server emits a ScoreChanged event to a Redis pub/sub channel or Kafka topic: {user_id, new_score, old_score, region, bracket, timestamp}. (2) **Fan-out handler** — a dedicated score-updater service subscribes to these events. For each event, it updates leaderboard sorted sets in parallel using Redis pipelining: ZADD leaderboard:global, ZADD leaderboard:region:{region}, ZADD leaderboard:bracket:{bracket}. (3) **Bracket movement detection** — if old_score was in bracket X and new_score is in bracket Y (e.g., crossed from Gold to Platinum), the handler must ZREM from the old bracket and ZADD to the new bracket. (4) **Friends notification** — the handler publishes a FriendScoreChanged event to a separate pub/sub channel per user: leaderboard:notification:{user_id}:friends. The notification service (WebSocket) subscribes to friends' channels for each connected user. (5) **Batched processing** — to avoid N ZADDs per score update (where N = number of dimensions), batch events within a 50ms window: group all ScoreChanged events by user_id within the window, apply only the latest score per user. (6) **Latency budget** — (a) Event publication: < 5ms (Redis pub/sub in same DC). (b) Fan-out handler processing: each ZADD is about 0.1ms in Redis. For 5 dimensions, total about 0.5ms per update. (c) Pipelining: send all ZADDs in one round-trip: about 0.5ms network. Total: about 6ms well under 500ms. (7) **Failure handling** — if the handler crashes mid-update (e.g., updated global but not region), the system will have inconsistent leaderboards. Solution: (a) use Redis transactions (MULTI/EXEC) per user update — all ZADDs are atomic. Or (b) use a WAL (Kafka log) and replay any failed updates. (8) **Throttling per user** — a user spamming games (100 scores/sec) should only have the latest score reflected. Implement a per-user rate limiter: if more than 10 updates/sec for the same user, only process the latest.

## Q13: Design an anti-cheat system that detects impossible scores. Walk through the heuristics, data pipeline, and escalation process.

**Answer:** Anti-cheat at leaderboard scale requires layered heuristics: (1) **Impossible score heuristics**: (a) **Score magnitude cap** — per game mode, define a theoretical maximum score (e.g., a perfect game = 999,999 points). Any score above the cap is instantly rejected. (b) **Score delta cap** — if a player jumps from 500 to 50,000 in one match, flag. The cap is dynamic: max_increase = max(1000, previous_score x 10). (c) **Play time anomaly** — if a score is submitted 2 seconds after the previous match ended, but the minimum match duration is 5 minutes, the client is likely sending fake scores. Reject. (d) **Score distribution outlier** — maintain a per-game-mode histogram (e.g., 100 buckets). A score that falls in a bucket with less than 0.1% of total players is flagged. (2) **Data pipeline**: (a) The game client sends telemetry alongside the score: {score, match_duration, actions_per_minute, accuracy, headshot_ratio, movement_pattern}. (b) A real-time stream processor (Kafka Streams or Flink) evaluates heuristic rules on each score event. Each rule outputs a suspicion score (0-100). (c) An aggregator combines all rule scores into a cheat_probability using a weighted formula. (3) **ML model**: train a gradient-boosted classifier on labeled data (known cheaters from manual review, known legitimate players). Features: APM, accuracy, score_rate, match_duration, device_fingerprint_match, account_age. The model runs in less than 10ms per event (ONNX runtime). (4) **Escalation**: (a) Score rejected (cheat_prob > 95%): return a score not accepted error to client. Log for ban review. (b) Score flagged (70-95%): accept the score but mark it status=PENDING_REVIEW. Show it on the leaderboard with a pending indicator. A manual review team investigates within 24 hours. If confirmed hacker, remove score and ban account. (c) Score passed (less than 70%): accept and display. (5) **Player feedback loop** — if a player consistently gets flagged but manual review finds they are legitimate, add them to an allowlist (e.g., pro esports players). Their scores skip heuristic checks (but still go through ML). (6) **Concealment** — never expose the exact cheat detection thresholds. Use obfuscated error codes.

## Q14: Design the seasonal reset and data archival system. When a season ends, how do you archive petabytes of historical score data and ensure the new season starts with zero latency?

**Answer:** Seasonal reset with zero-downtime transition: (1) **Double-buffer key structure** — use time-partitioned keys: leaderboard:global:season:2025-S1 (active), leaderboard:global:season:2025-S2 (pre-created). The leaderboard service reads from the active season based on current time. (2) **Pre-creation** — 1 hour before season end, pre-create all leaderboard keys for the new season (global, per-region, per-bracket). This avoids slow ZADD on first write of the new season. (3) **Atomic switch** — at the exact season boundary (e.g., Sunday 23:59:59 UTC), a cron job or scheduler updates a distributed config key: SET active_season 2025-S2. The leaderboard service checks this config on every read/write. No restart required. (4) **Archival pipeline**: (a) A Spark job reads the entire sorted set for the old season: ZRANGE leaderboard:global:season:2025-S1 0 -1 WITHSCORES. (b) Writes to Parquet files in S3 partitioned by score range: s3://leaderboard-archive/season=2025-S1/range=0-1000/part-00001.parquet. (c) The Parquet files are registered in a query engine (Athena or Presto) so historical queries can be run via SQL. (d) A Redis UNLINK (asynchronous delete) removes the old sorted set from memory. (5) **Incremental snapshotting** — instead of reading the entire sorted set at season end (which could take hours for 50M members), take incremental snapshots every day during the season. At season end, only the final day's changes need to be merged. (6) **Data retention** — hot archive (S3 Standard): last 5 seasons for real-time historical comparison. Warm archive (S3 Glacier-IA): seasons 6-20. Cold archive (Glacier Deep Archive): seasons 21+. Retrieval takes 12 hours but costs $0.001/GB/month. (7) **User-facing historical view** — if a user wants to see their rank trend over 10 seasons, the API reads from S3 Parquet via Athena on-demand (5-10 second latency). Cache results per user for 1 day. (8) **Awards at season end** — the Spark job also computes top-1000 rankings and triggers reward distribution (badges, titles, tokens). This is a separate job from archival so it can complete before the next season starts.

## Q15: How do you cache the leaderboard to handle a viral event (e.g., a streamer tells 500K viewers to check the leaderboard)? Design a cache hierarchy that survives a 1000x traffic spike.

**Answer:** A viral event can drive 500K concurrent views of the top-100 leaderboard. The cache hierarchy must absorb this: (1) **CDN caching** — cache the top-100 leaderboard HTML/JSON at the CDN edge (CloudFront, Cloudflare). Set TTL to 10 seconds with stale-while-revalidate=30s. A viewer from Tokyo gets the cached response from the Tokyo edge, not the origin. This absorbs 99% of read traffic. (2) **API gateway caching** — if the CDN misses, the API gateway (e.g., AWS API Gateway) caches the response for 5 seconds. This catches duplicate requests from the same edge when CDN cache expires. (3) **Application cache (Redis)** — the application server reads the top-100 from a Redis read replica. Redis can handle 100K reads/sec on a single replica. During a spike, scale to 5-10 read replicas. (4) **Stale-while-revalidate** — if the leaderboard is stale by up to 5 seconds, serve the stale version immediately and trigger an async refresh. The user sees instant response, even during cache stampede. (5) **Local in-memory cache** — each API server instance holds a sync.Map of the top-100 with a 2-second TTL. For a fleet of 100 servers, each server serves 5K req/s from local memory. This is the fastest layer (sub-microsecond). (6) **Cache stampede prevention** — when the top-100 cache expires and 1000 requests arrive simultaneously, only ONE request should recompute the top-100. Use a mutex (Redis SET NX lock with 1s TTL): the first request acquires the lock and recomputes; others wait for the lock to release and get the cached value. (7) **Traffic shaping** — if the traffic exceeds a threshold (e.g., 1M req/s), the API gateway returns 429 Too Many Requests with a Retry-After: 1 header. Legitimate clients retry after 1 second, the stampede passes. (8) **Monitoring** — set CloudWatch alarms: if cache hit ratio drops below 95% across all layers, auto-scale read replicas and shorten CDN TTL. (9) **Warm-up** — if the viral event is known in advance (scheduled stream), pre-warm the CDN and cache by making requests from geo-distributed locations before the stream starts.

## Q16: How do you compute percentile rank (e.g., you are in the top 5%) instead of just absolute rank? Design this for a leaderboard with 100M players.

**Answer:** Percentile rank requires knowing the total player count and the player's position: (1) **Formula**: percentile = (total_players - rank) / total_players * 100. For rank 1 out of 100M: 99.999999th percentile. For rank 5M: 95th percentile. (2) **Total player count** — maintain a separate counter: SET leaderboard:global:player_count. Increment on first score ever, decrement on account deletion. Use INCR/DECR for atomic updates. (3) **Score distribution histogram** — for faster percentile queries without ZREVRANK overhead: maintain an array of score brackets (bucket size = 100 points). The histogram is a Redis sorted set or a separate list: leaderboard:histogram where each member is a score bucket with score = count. Example: ZINCRBY leaderboard:histogram 1 5000 (one more player in the 5000-5099 bucket). (4) **Percentile from histogram**: (a) Sum all players in buckets higher than the player's bucket. (b) Add the player's position within their bucket (estimated by assuming uniform distribution within the bucket). (c) Divide by total players. (5) **Precision vs. speed trade-off**: (a) Fine-grained histogram (bucket size 10): high precision but 100M/10 = 10M buckets — memory intensive (~100 MB). (b) Coarse histogram (bucket size 1000): low precision but 100M/1000 = 100K buckets — about 1 MB. (c) **Two-tier**: coarse histogram for fast approximate percentile (shown by default), fine-grained for exact percentile (computed on demand via ZREVRANK + count). (6) **Real-time updates** — the histogram is updated on every score change: ZINCRBY on the old bucket to decrement, ZINCRBY on the new bucket to increment. This is done transactionally with the score update. (7) **Display**: You are in the top 3.2% (rank 3,200,477 of 100,000,000 players). The rank is exact (from ZREVRANK), the percentile is computed from the histogram for speed. (8) **Batch vs. real-time** — for 100M players, ZREVRANK is O(log N) = about 27 steps on a 100M-member sorted set. This is fast enough (< 1ms) for on-demand computation. The histogram is useful for aggregate views.

## Q17: Design the live leaderboard WebSocket push infrastructure. How do you broadcast rank changes to 1M concurrent viewers with less than 500ms latency?

**Answer:** WebSocket push at 1M concurrent requires careful fan-out architecture: (1) **Connection management** — use a WebSocket gateway cluster (e.g., AWS API Gateway WebSockets, or a custom Go server using gorilla/websocket). Each gateway server holds 10K-50K connections. (2) **Client subscription** — when a client opens the leaderboard, it sends a subscription message: {type: subscribe, channels: [top100, rank:user_123]}. The gateway registers the connection in an in-memory channel map: map[channel] to set of connection IDs. (3) **Event flow** — when a score update changes the top-100: (a) The leaderboard service publishes to Redis pub/sub. (b) Each gateway server subscribes to the top100 event via Redis pub/sub (single subscriber per gateway). (c) When a gateway receives the event, it iterates its local channels top100 connection set and writes a WebSocket message to each. (4) **Scaling to 1M connections**: (a) With 40 gateway servers (25K connections each), each gateway sends to 25K connections when a top100 event fires. Writing 25K WebSocket messages takes about 50-100ms (the bottleneck is the kernel's send() syscall). (b) **Batching**: use WriteBatch or coalesce messages: if multiple top100 events arrive within 50ms, send only the latest one. (c) **Goroutine per connection** (in Go): each connection has a write goroutine with a buffered channel (size 16). When the gateway receives a top100 event, it pushes to the channel. The goroutine writes to the TCP socket. If the channel is full (client too slow), the connection is closed. (5) **Rate limiting per connection** — cap message rate at 10 msg/sec per WebSocket. If the leaderboard is updating rapidly (e.g., last minute of a tournament), send a state snapshot every 100ms instead of every individual change. (6) **Graceful degradation** — if a client's connection is congested (TCP backpressure), close the connection gracefully. The client reconnects and receives the latest state from the REST API. (7) **Fallback to SSE** — if WebSocket fails (corporate firewall blocking WS), fall back to Server-Sent Events (SSE) over HTTP, or long-polling every 5 seconds. (8) **Monitoring** — track connection_count, messages_pushed_per_sec, avg_push_latency, channel_depth. Alert if push latency exceeds 1 second.

## Q18: Integrate an Elo/MMR rating system into the leaderboard. When a player wins a match, the leaderboard should reflect their new MMR, not just their raw score.

**Answer:** Elo/MMR-based leaderboards replace raw scores with skill ratings: (1) **Elo formula** — after each match, compute expected score: E_A = 1 / (1 + 10^((R_B - R_A)/400)) where R_A, R_B are players' ratings. New rating: R_A_new = R_A_old + K * (S_A - E_A), where S_A = 1 for win, 0 for loss, and K = adjustment factor (32 for new players, 16 for established). (2) **Leaderboard sorted set** — the Redis sorted set stores MMR as the score: ZADD leaderboard:global:mmr. Higher MMR = higher rank. (3) **Match outcome processing**: (a) Game server sends match result: {winner_id, loser_id, winner_mmr, loser_mmr, match_timestamp}. (b) The MMR service computes new ratings: winner_new = winner_old + 32 * (1 - E_win), loser_new = loser_old + 32 * (0 - E_lose). (c) The leaderboard service updates both players. (4) **Provisional rating** — new players start with a provisional period (e.g., first 10 matches). During this period: (a) K-factor = 64 (faster adjustment). (b) The player's MMR is shown as Unranked on the leaderboard until the provisional period ends. (c) This prevents smurf accounts from appearing at the top after 1 win. (5) **Rating decay** — if a player does not play for 30 days, their MMR decays by 5% per week. A background job scans the last_match_timestamp for all players and decreases MMR if stale. (6) **Season reset with MMR** — at season start, apply a soft reset: new_mmr = (old_mmr - 1500) * 0.5 + 1500. This compresses ratings toward the mean but preserves relative ordering. Prevents top players from having an insurmountable lead. (7) **Leaderboard display** — show MMR alongside rank. The leaderboard also shows win rate and games played. (8) **Glicko-2 vs Elo** — for higher accuracy, use Glicko-2 (used by Chess.com, League of Legends). It tracks rating deviation (RD) — uncertainty in the rating. High RD = few games, so rating changes more. Low RD = many games, rating is stable.

## Q19: Design leaderboard sharding by score brackets. How do you maintain the top-N global leaderboard when scores are partitioned across 100 Redis shards?

**Answer:** Sharding by score brackets allows horizontal scaling: (1) **Bracket definition** — define non-overlapping score ranges: bracket1 (0-1000), bracket2 (1001-5000), bracket3 (5001-20000), bracket4 (20001-100000), bracket5 (100001+). Each bracket is stored in a separate Redis sorted set on a separate shard (or cluster node). (2) **Score routing** — the leaderboard service determines which bracket a score belongs to and writes to the corresponding shard: ZADD leaderboard:bracket:3. (3) **Top-N global query** — to compute the global top 100: (a) Query the top 100 from each bracket: ZREVRANGE. (b) Merge all bracket tops into a combined list. (c) Sort by score descending. (d) Return the top 100 from the merged list. (4) **Bracket boundary crossing**: when a player's score increases past a bracket boundary (e.g., from 4800 to 5200), the system: ZREM from bracket2, ZADD to bracket3. This is done atomically with a Lua script. (5) **Dynamic bracket boundaries** — as the game evolves, score distribution shifts. A background job recalculates bracket boundaries weekly: (a) Sample the score distribution. (b) Set boundaries so each bracket has roughly equal player count (e.g., 5 brackets of 10M players each). (c) Move all players to new brackets (this is a heavy operation; do during maintenance window). (6) **Read load balancing** — bracket queries go to the shard hosting that bracket. Hot brackets (the top bracket) are accessed most frequently for top-N queries. Mitigation: (a) Replicate the top bracket to more shards. (b) Cache the top-100 merged result aggressively (CDN + API cache + app cache). (7) **Comparison to consistent hashing** — this is NOT consistent hashing. Score-bracket sharding is intentional (scores near each other are in the same shard), while consistent hashing scatters keys randomly. The trade-off: writes are evenly distributed (random scores go to random brackets), but reads for top-N hit the top bracket shard heavily (hot shard problem). The hot shard is acceptable because the top bracket has the fewest members (highest scores are rare). (8) **Write distribution** — since score distribution follows a bell curve (most players in the middle brackets), the middle shards get the most writes. Monitor write load per shard and split the middle bracket if needed.

## Q21: Design the real-time leaderboard update system. When a player's score changes, how do you update the leaderboard for millions of concurrent players with sub-second latency?

**Answer:** Real-time leaderboard updates require incremental rank recalculation, not full rebuilds. (1) **Redis Sorted Sets**: (a) `ZADD leaderboard:global <player_id> <new_score>` updates the score. (b) Redis updates the sorted set in O(log N) time. (c) `ZREVRANK leaderboard:global <player_id>` returns the player's rank. (d) For 10M players, a ZADD takes ~1ms. (2) **Batched updates**: (a) Players don't need instant rank updates for every minor score change. (b) Buffer score changes for 1 second (or 10 changes) before flushing to Redis. Use a local buffer per game server, then batch ZADD to Redis. (c) This reduces Redis write load by 10-100x. (3) **Atomic leaderboard operations**: (a) Use Redis transactions or Lua scripts for atomic update: `local newScore = redis.call('ZINCRBY', 'leaderboard:global', player_id, delta); local rank = redis.call('ZREVRANK', 'leaderboard:global', player_id); return {newScore, rank}`. (b) This ensures the rank is always consistent with the score. (4) **Caching the top-K**: (a) Most requests are for the top 100 players. Cache this list: `ZREVRANGE leaderboard:global 0 99 WITHSCORES`. Refresh every 1 second via a background job. (b) The API returns the cached top 100 in <1ms without hitting Redis. (5) **Region-aware updates**: for global leaderboards, replicate scores across regions. Each region's game server writes to its local Redis. A cross-region Kafka stream aggregates scores into the global Redis leaderboard with eventual consistency (< 2 second lag).

## Q22: How do you handle tie-breaking when multiple players have the same score? Should they share the same rank or be ordered by a secondary criterion?

**Answer:** Tie-breaking rules significantly impact player psychology and competition. (1) **Option A — Same rank (tie)**: (a) All players with score X get rank R. The next score band starts at R + count(tied_players). (b) Redis implements this via ZREVRANK: players with the same score get different ranks (Redis orders by score, then by member lexicographically). (c) To implement ties: after ZREVRANK, group by score and assign the minimum rank per score group. (2) **Option B — Secondary criteria**: (a) Order by score desc, then by time (earliest to achieve the score ranks higher). (b) Encode as a composite score: `redis_score = score * 10^12 + (MAX_TIMESTAMP - achievement_timestamp)`. (c) Example: score=1500, timestamp=2025-01-15T10:00:00Z (epoch=1736920800). `redis_score = 1500 * 10^12 + (10^18 - 1736920800) = 1500000000000000000 + 998263079200`. (d) This gives deterministic tie-breaking by time while preserving the primary sort by score. (3) **Option C — Rank sharing with display variance**: (a) At the API layer, if scores are equal, assign the same rank but show different "display rank" for UI purposes (e.g., "Gold Tier" for top 1%). (b) This avoids players feeling unfairly ordered while still showing individual rank numbers. (4) **Recommendation**: For competitive games, use Option B (composite score). Players accept time-based tie-breaking as fair ("they got there first"). For casual games, use Option A (shared rank) to avoid demotivating players.

## Q24: Design an anti-cheat system for leaderboards. How do you detect and block score manipulation, bot farming, and exploit-based rank inflation?

**Answer:** Leaderboard integrity requires detection of anomalous score patterns. (1) **Rate-based detection**: (a) Track score submission velocity per player: if a player submits 1000 scores in 1 minute (where max human rate is 10/min), flag as bot. (b) Implement as a sliding window counter: Redis `INCR player:scores:{player_id}:{current_minute}` with 60-second TTL. If count > threshold, reject the score submission. (2) **Score trajectory analysis**: (a) Legitimate players improve gradually. A player jumping from rank 1M to rank 100 in 5 minutes is suspicious. (b) Maintain a per-player score history (last 100 scores). Train a simple anomaly detection model: if the score delta exceeds 5 standard deviations from the player's historical improvement rate, flag for review. (3) **Client-side validation**: (a) The game client signs each score submission with a secret embedded in the binary (obfuscated). The server verifies the HMAC signature. (b) Even better: use a session-based HMAC key exchanged at login. The server sends a per-session key; the client signs scores with this key. An attacker must extract the key from memory (harder than from the binary). (4) **Server-authoritative scoring**: (a) The game server (not the client) should be the source of truth for scores. The client sends game events (enemy killed, level complete), and the server computes the score. (b) This prevents the client from lying about the score. The trade-off is higher server load. (5) **Manual review queue**: (a) Flagged players are placed in a review queue. (b) An automated review bot replays the player's last 10 game sessions (recorded as deterministic replays) to verify score legitimacy. (c) If confirmed cheat: ban the player, reset their score to 0, and remove from the leaderboard. (6) **Appeal process**: (a) Banned players can appeal via a support ticket. (b) On appeal, a human reviewer examines the evidence and either confirms or reverses the ban. (c) False positive rate target: <0.01% of all bans. (7) **Leaderboard cleansing**: after a ban wave, recalculate ranks for affected players. This is done by deleting the banned player's entry from the sorted set and re-running rank calculation for remaining players.

(c) This reduces storage: only 10 entries per player instead of all friends. (5) **Global + friend combined display**: (a) Show: "You're #1,234 globally, #5 among friends." (b) Global rank from the global leaderboard sorted set. (c) Friend rank from the friend leaderboard sorted set. (d) Both fetched in a single API call (parallel Redis queries). ## Q27: Design a leaderboard sharding strategy for 100M+ players. How do you distribute the sorted set across multiple Redis nodes while still supporting global rank queries?

**Answer:** Sharding a global sorted set is challenging because rank requires the complete ordered view. (1) **Option A — Score-range sharding**: (a) Assign each Redis node a score range: Node 0: scores 0-999, Node 1: scores 1000-1999, etc. (b) ZADD routes to the correct node based on the score. (c) ZREVRANK: sum counts from all nodes with higher score ranges + rank within the player's node. (d) COUNT is fast (ZCARD on each node, O(1)). (e) Issue: score ranges can become skewed (most players cluster at low scores). Use dynamic range splitting. (2) **Option B — Consistent hashing by player ID**: (a) Shard by player_id: Node = hash(player_id) % N. (b) Score writes are evenly distributed. (c) Global rank query: fan-out to all N nodes, get each node's players with scores > player's score. Sum counts + 1. (d) Issue: O(N) fan-out per rank query. For N=100 nodes, 100ms extra latency per query. (3) **Option C — Two-tier architecture** (recommended): (a) **Tier 1**: per-shard sorted sets. Shard by player_id. Each shard maintains its local top-N list. (b) **Tier 2**: a global top-K sorted set maintained by a separate service. It subscribes to score change events and only keeps the top 10,000 scores globally. (c) Query logic: most players are NOT in the top 10K. For them, estimate rank: `estimated_rank = ZCOUNT(tier1_shard, score+1, +inf) + sum(ZCARD(other_shards))`. (d) For top 10K players, exact rank from tier 2: `ZREVRANK(tier2_global, player_id)`. (e) Accuracy: 99% of players get rank within 0.1% of exact. Top players get exact rank. (4) **Option D — CockroachDB/TiDB with global index**: (a) Use a distributed SQL database with a global secondary index on score. (b) Query: `SELECT COUNT(*) + 1 FROM leaderboard WHERE score > :my_score`. (c) The database distributes the count across nodes transparently. (d) Latency: 50-100ms for 100M rows. (5) **Recommendation**: Use Option C (two-tier) for the best balance of write performance and rank accuracy. Exact global rank for top players, estimated for everyone else.
