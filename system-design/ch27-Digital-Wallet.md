# Chapter 27 — Design a Digital Wallet

## Q1: Design the core wallet architecture. How does the wallet maintain a real-time balance while supporting thousands of concurrent transactions per user?

**Answer:** The wallet architecture uses a dual-accounting model: (1) **Balance table** — `wallets (user_id, currency, balance, version, updated_at)`. The `balance` is the current available balance. The `version` is used for optimistic locking. (2) **Transaction log** — `wallet_transactions (id, user_id, currency, amount, type, reference_id, status, created_at)`. Types: DEPOSIT, WITHDRAWAL, TRANSFER, REFUND, FEE. This is an append-only immutable log. (3) **Balance query** — the displayed balance is NOT computed from the transaction log (too expensive). Instead, it's stored in the wallet row and updated atomically with each transaction. (4) **Concurrent transactions** — use `UPDATE wallets SET balance = balance - 100, version = version + 1 WHERE user_id = X AND currency = 'USD' AND balance >= 100 AND version = V`. If two concurrent withdrawals occur, only one succeeds (the first update); the second fails with a version mismatch or insufficient balance. The application retries the failed transaction with a fresh balance read. (5) **Hot wallet vs. cold wallet** — the online wallet (`hot wallet`) holds only the amount needed for daily operations. The majority of funds are in a `cold wallet` (offline or with withdrawal delay) for security. (6) **Ledger integration** — every wallet transaction creates a double-entry journal entry (see chapter 26). The wallet's balance is the cached state of the ledger's running total for that user account.

## Q2: How does a wallet handle the "double-spend" problem — a user initiates two withdrawals simultaneously and the system must ensure only one succeeds?

**Answer:** Double-spend prevention is enforced at the database level: (1) **Pessimistic locking** — `BEGIN; SELECT balance FROM wallets WHERE user_id = X FOR UPDATE; -- check balance; UPDATE wallets SET balance = balance - 100 WHERE user_id = X; INSERT INTO wallet_transactions ...; COMMIT;`. The `FOR UPDATE` lock serializes concurrent transactions for the same wallet. (2) **Optimistic locking (version-based)** — as described in Q1: `UPDATE wallets SET balance = balance - 100, version = version + 1 WHERE user_id = X AND version = V AND balance >= 100`. If the affected row count is 0, the transaction was already modified by another request → the operation fails and must be retried. This is more scalable than pessimistic locking because it allows higher concurrency. (3) **Idempotency key** — each withdrawal request has a unique `idempotency_key`. The system checks: `SELECT 1 FROM processed_idempotency_keys WHERE key = X`. If the key exists, return the existing result without processing again. This prevents double-spending from network retries (user's phone sends the withdrawal request twice). (4) **Queue-based serialization** — for extremely competitive wallets (e.g., trading platforms), serialize all transactions for a given wallet through a single Kafka partition (keyed by `user_id`). A single consumer processes transactions sequentially, eliminating race conditions entirely. This is the approach used by modern banking systems (e.g., Stripe's ledger, Square's wallet).

## Q3: Design the wallet's integration with external payment rails (ACH, Wire, SEPA, UPI). How do you handle the multi-day settlement delay of ACH transfers?

**Answer:** External payment rail integration requires careful state management: (1) **ACH lifecycle** — (a) User initiates ACH debit: `{routing_number, account_number, amount}`. (b) The wallet records a `PENDING` deposit. (c) The ACH file is generated and sent to the ODFI (Originating Depository Financial Institution) via secure SFTP. (d) NACHA rules: the funds are available to the user on day 2 (if prenote was done). The ODFI reports settlement on day 2-3. (e) If the ACH returns (NSF — insufficient funds, or account closed), the deposit is reversed on day 3-5. (2) **Availability vs. finality** — the wallet credits the user's available balance immediately (day 1) but has a "good funds" flag set to false. The user can spend against the pending balance. If the ACH bounces, the wallet deducts the amount back (essentially a negative balance if the user has already spent it). This is the "available vs. collected" balance distinction. (3) **Loss protection** — merchants with high ACH return rates (>2%) have collection hold periods extended to 5 business days. The risk engine adjusts the hold period dynamically. (4) **Cross-border** — for international wires (SWIFT), the lifecycle is similar but takes 1-5 business days and involves correspondent banks. SWIFT ISO 20022 messages are the standard for interbank communication. (5) **UPI (India)** — UPI transactions settle in real-time (immediate finality). The wallet integration is simpler: `immediate_balance_change` with no settlement delay. (6) **Reconciliation** — a daily reconciliation job matches the wallet's pending transactions against the settlement report from the clearing house (FedACH, EBA Clearing). Discrepancies are resolved manually.

## Q4: Design a micro-wallet for an app that allows users to earn "coins" by watching ads and spend them on premium features. How does the virtual currency ledger differ from real-money wallets?

**Answer:** Virtual currency wallets are simpler (no KYC, no bank integrations) but have unique challenges: (1) **Data model** — identical to real-money wallets: `user_id, currency (COINS), balance, version`. But the "value" is not real money — it's a in-app currency that the app controls. (2) **Earning rules** — coins are minted when a user: watches an ad (+10 coins), completes a survey (+50 coins), wins a game (+25 coins). These are "system credit" transactions with no real-money backing. (3) **Spending rules** — coins are burned when a user: unlocks a premium feature (-200 coins), buys a virtual good (-100 coins). (4) **Inflation control** — the app must control the coin economy to prevent inflation. Common techniques: coins have expiry (use them in 30 days), time-gated earnings (max 100 coins/hour), or variable pricing for popular items. (5) **Anti-fraud** — since coins have no real-money cost, there's a higher incentive to cheat. Implement rate limits: max 100 ad views per day, device fingerprinting to prevent bot farms, and server-side validation of ad completion events. (6) **Export restrictions** — virtual currencies cannot be converted back to real money (gambling regulations). The system must enforce one-way flow: real money can buy coins, but coins cannot be withdrawn as cash. This avoids regulatory classification as a money transmitter.

## Q5: Design the wallet's security architecture. How do you protect against account takeover (ATO) and unauthorized transfers?

**Answer:** Layered security: (1) **Authentication** — strong customer authentication (SCA) for all sensitive operations: login (2FA/TOTP), sending money (biometric approval on mobile), changing the linked bank account (email + SMS OTP). (2) **Transaction signing** — every outbound transfer request includes a digital signature from the approved device. The server verifies: `HMAC(device_secret, amount || recipient || timestamp)`. (3) **Device binding** — the wallet is bound to known devices. If a login comes from a new device, a cooling period (24 hours) is enforced for new recipient transfers. (4) **Velocity limits** — per-user: max 10 transfers per day, max $10,000 cumulative per day, max $50,000 cumulative per month. Per-beneficiary: max $5,000 per day to a new recipient (24-hour clock). (5) **Beneficiary whitelist** — new beneficiaries require a verification step: the wallet deposits a small amount (e.g., $0.12) to the beneficiary account and asks the user to confirm the amount (micro-deposit verification). This is standard in the US for ACH verification. (6) **Anomaly detection** — an ML model monitors for: login from unusual location, transfers to unusual destinations, rapid device changes, and check-cashing patterns. Anomalies trigger Transaction Hold: the transfer is queued for manual review. (7) **Session management** — all sensitive operations require a fresh authentication session. Idle timeout: 5 minutes for mobile. (8) **Encryption** — PII and financial data is encrypted at rest (AES-256) and in transit (TLS 1.3). The wallet balance column is encrypted using application-level encryption (envelope encryption with AWS KMS).

## Q6: How does the wallet handle international remittances — sending money from a user in the US to a recipient in the Philippines? Design the currency conversion and local payout rail integration.

**Answer:** Cross-border remittance: (1) **Origination** — the US user initiates a transfer of $200 to a Philippine recipient. The wallet debits $200 from the user's USD balance (plus a remittance fee, e.g., $2.99). (2) **FX conversion** — the wallet uses its FX provider (e.g., CurrencyCloud, Wise) to convert $200 USD to PHP. Rate: $1 USD = PHP 56.50 → PHP 11,300. The FX rate includes a spread (0.5-1% margin for the wallet). (3) **Local payout rail** — in the Philippines, the funds are paid out to the recipient via: GCash (mobile wallet), PayMaya, or bank transfer (PESONet or InstaPay). PESONet: batch settlement, 1-day delay. InstaPay: real-time 24/7. The wallet selects the fastest available rail that the recipient is registered for. (4) **Recipient notification** — the recipient receives an SMS or email: "You received PHP 11,300 from [sender]. Click here to claim." If the recipient is already a wallet user, the funds arrive instantly in their wallet. (5) **Regulatory compliance** — each jurisdiction has specific requirements. Philippines: BSP (Bangko Sentral ng Pilipinas) regulations require KYC verification for recipients if the amount exceeds PHP 500,000. US: FinCEN (Financial Crimes Enforcement Network) requires reporting of remittances over $1,000. The wallet logs all remittances and files regulatory reports. (6) **Cost breakdown** — to the sender: $202.99. The wallet's cost: $0.50 FX fee to CurrencyCloud + $0.10 GCash payout fee. Profit: $2.39 per remittance. (7) **Tracking** — the sender can track the transfer status: "FX converted" → "Sent to Philippines" → "Recipient credited." Tracking uses the partner API's status webhooks.

## Q7: Design the wallet notification system. How does the user get real-time push notifications for every transaction (deposit, withdrawal, transfer)?

**Answer:** The notification pipeline: (1) **Event sourcing** — every wallet transaction is an event published to Kafka: `wallet_transaction.{user_id}.{type}`. (2) **Notification service** — a consumer reads from the event stream and constructs the notification message: "You received $50.00 from John Doe." / "Your withdrawal of $200.00 to Bank Account ending in 1234 was successful." (3) **Delivery channels** — (a) Push notification (Firebase Cloud Messaging for Android, APNS for iOS) — delivered within seconds. (b) SMS — for high-value transactions (> $10,000) or security alerts (login from new device). (c) Email — weekly digest or large deposits. (d) In-app notification — real-time banner in the wallet app. (4) **User preferences** — users can configure notification thresholds: "Notify me for all transactions" vs. "Only notify me for transactions over $100". Preferences are stored in a user profile service and evaluated at notification time. (5) **Channel prioritization** — push is the default for daily transactions. SMS is used only for security-critical events (password change, withdrawal to a new account). (6) **Batch notifications** — if the user makes 20 micro-deposits in a minute (e.g., from a payment splitter), the notification system batches them: "You received 20 payments totaling $134.50." (7) **Delivery guarantees** — the notification system guarantees at-least-once delivery. If the push notification service is unavailable (FCM/APNS downtime), the notification is queued and retried. (8) **Opt-out** — users can opt out of marketing notifications but not transaction notifications (regulatory requirement for financial apps).

## Q8: How do you design the wallet's P2P (peer-to-peer) transfer system — "Send money to a friend by phone number or username"?

**Answer:** P2P transfer flow: (1) **Recipient lookup** — sender enters the recipient's phone number, email, or username. The wallet's directory service resolves this to a `recipient_user_id` using an inverted index: `phone_index (phone_hash → user_id)`. (2) **Initiation** — sender enters amount ($50) and hits "Send". (3) **Authorization** — the sender authorizes the transfer: biometric (FaceID) or PIN. The server checks: `balance >= amount + fee`. (4) **Transaction execution** — the system debits the sender's wallet and credits the recipient's wallet in a single database transaction: `UPDATE wallets SET balance = balance - 50 WHERE user_id = sender AND balance >= 50; UPDATE wallets SET balance = balance + 50 WHERE user_id = recipient; INSERT INTO wallet_transactions ...`. (5) **Recipient notification** — the recipient gets a push notification: "You received $50.00 from [Sender Name]." (6) **Scheduled transfers** — support "Send $50 to John on the 1st of every month." Implemented as a cron job that checks scheduled_transfers where `next_run_date <= now()`. (7) **Request money** — recipient can send a payment request: "John requests $25.00 from you." The sender approves or declines. (8) **Limits** — unverified users: max $500 per day P2P. Verified (KYC completed): max $10,000 per day. Limits are configurable and affect the transfer at authorization time. (9) **Dispute resolution** — if the sender claims they didn't authorize a P2P transfer, the system flags it for review. If the transfer was to a new recipient from a new device, the burden of proof is on the recipient. This is the standard "unauthorized transaction" process for P2P.

## Q9: How does the wallet handle currency conversion when sending money to a user in a different currency zone?

**Answer:** Multi-currency P2P: (1) **Scenario** — sender (USD) sends $50 to recipient in Europe who holds EUR. (2) **Conversion decision** — does the sender pay in USD (and the system converts) or does the recipient receive USD and convert themselves? The system shows both options with estimated amounts. (3) **FX rate quote** — the FX service provides a real-time quote: `GET /fx/quote?from=USD&to=EUR&amount=50`. The quote is valid for 30 seconds (the user sees "Rate expires in 15 seconds"). (4) **Execution** — when the sender confirms, the system: locks the rate, converts $50 USD → €46.22 EUR using the quoted rate, credits the recipient's EUR wallet with €46.22. The rate lock ensures the user gets the quoted amount even if market rates change during the 5-second confirmation flow. (5) **Spread** — the wallet adds a spread (e.g., 0.5%) to the mid-market rate. Mid-market: 1 USD = 0.93 EUR. With spread: 1 USD = 0.924 EUR. The spread is the wallet's revenue for providing the conversion service. (6) **Recipient choice** — the recipient can choose to receive in USD (keep the funds in a USD wallet) or convert to EUR. This choice affects the FX rate and any conversion fees. (7) **Multi-currency wallets** — the wallet system maintains per-user, per-currency balances. User A may hold both USD and EUR wallets. A P2P transfer from User A (USD) to User B (EUR) is: debit User A's USD wallet, credit User B's EUR wallet (with FX conversion). This is a single atomic operation in the ledger.

## Q10: Design the wallet's regulatory compliance system — how do you enforce Know Your Customer (KYC), Anti-Money Laundering (AML), and sanctions screening at scale?

**Answer:** Compliance is non-negotiable for a wallet operating globally: (1) **KYC tiers** — (a) Tier 0: email + phone only — max balance $100, max monthly volume $500. (b) Tier 1: Tier 0 + identity verification (government ID scan + selfie) — max balance $10,000, max monthly volume $25,000. (c) Tier 2: Tier 1 + proof of address (utility bill) — unlimited. The KYC verification uses a third-party provider (Jumio, Onfido, Stripe Identity) for document verification and liveness detection. (2) **AML monitoring** — all transactions are scored by an AML engine (e.g., Chainalysis for crypto, or a custom rules engine for fiat). Rules: (a) Structuring detection — multiple deposits under $10,000 within 24 hours (CTR threshold in the US). (b) Rapid movement — funds deposited and withdrawn to a different account within minutes. (c) High-risk jurisdictions — transactions to/from OFAC-sanctioned countries are blocked. (3) **Sanctions screening** — every transaction participant (sender, recipient, bank account holder) is screened against: OFAC SDN list, EU sanctions list, UN sanctions list, and local regulatory lists. Screening is done at signup and on every transaction using a fuzzy name-matching service (e.g., World-Check, LexisNexis). (4) **Suspicious Activity Report (SAR)** — transactions flagged by AML rules are escalated to a compliance team. If confirmed suspicious, a SAR is filed with FinCEN (in the US) or the relevant FIU. (5) **Travel Rule** — for transfers over $3,000 (FinCEN) or €1,000 (EU), the wallet must share sender/recipient information with the counterparty wallet (name, address, account number). This is implemented as a secure message exchange between compliant wallet providers (using the TRISA protocol or similar). (6) **Audit trail** — all compliance decisions (KYC approvals, AML flags, SAR filings) are logged immutably for regulatory audit. Retention: 5 years after account closure (per BSA/AML requirements).
