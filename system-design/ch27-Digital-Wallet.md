# Chapter 27 — Design a Digital Wallet

## Q1: Design the core wallet architecture. How does the wallet maintain a real-time balance while supporting thousands of concurrent transactions per user?

**Answer:** The wallet architecture uses a dual-accounting model: (1) **Balance table** — `wallets (user_id, currency, balance, version, updated_at)`. The `balance` is the current available balance. The `version` is used for optimistic locking. (2) **Transaction log** — `wallet_transactions (id, user_id, currency, amount, type, reference_id, status, created_at)`. Types: DEPOSIT, WITHDRAWAL, TRANSFER, REFUND, FEE. This is an append-only immutable log. (3) **Balance query** — the displayed balance is NOT computed from the transaction log (too expensive). Instead, it's stored in the wallet row and updated atomically with each transaction. (4) **Concurrent transactions** — use `UPDATE wallets SET balance = balance - 100, version = version + 1 WHERE user_id = X AND currency = 'USD' AND balance >= 100 AND version = V`. If two concurrent withdrawals occur, only one succeeds (the first update); the second fails with a version mismatch or insufficient balance. The application retries the failed transaction with a fresh balance read. (5) **Hot wallet vs. cold wallet** — the online wallet (`hot wallet`) holds only the amount needed for daily operations. The majority of funds are in a `cold wallet` (offline or with withdrawal delay) for security. (6) **Ledger integration** — every wallet transaction creates a double-entry journal entry (see chapter 26). The wallet's balance is the cached state of the ledger's running total for that user account.

## Q2: How does a wallet handle the "double-spend" problem — a user initiates two withdrawals simultaneously and the system must ensure only one succeeds?

**Answer:** Double-spend prevention is enforced at the database level: (1) **Pessimistic locking** — `BEGIN; SELECT balance FROM wallets WHERE user_id = X FOR UPDATE; -- check balance; UPDATE wallets SET balance = balance - 100 WHERE user_id = X; INSERT INTO wallet_transactions ...; COMMIT;`. The `FOR UPDATE` lock serializes concurrent transactions for the same wallet. (2) **Optimistic locking (version-based)** — as described in Q1: `UPDATE wallets SET balance = balance - 100, version = version + 1 WHERE user_id = X AND version = V AND balance >= 100`. If the affected row count is 0, the transaction was already modified by another request → the operation fails and must be retried. This is more scalable than pessimistic locking because it allows higher concurrency. (3) **Idempotency key** — each withdrawal request has a unique `idempotency_key`. The system checks: `SELECT 1 FROM processed_idempotency_keys WHERE key = X`. If the key exists, return the existing result without processing again. This prevents double-spending from network retries (user's phone sends the withdrawal request twice). (4) **Queue-based serialization** — for extremely competitive wallets (e.g., trading platforms), serialize all transactions for a given wallet through a single Kafka partition (keyed by `user_id`). A single consumer processes transactions sequentially, eliminating race conditions entirely. This is the approach used by modern banking systems (e.g., Stripe's ledger, Square's wallet).

## Q3: Design the wallet's integration with external payment rails (ACH, Wire, SEPA, UPI). How do you handle the multi-day settlement delay of ACH transfers?

**Answer:** External payment rail integration requires careful state management: (1) **ACH lifecycle** — (a) User initiates ACH debit: `{routing_number, account_number, amount}`. (b) The wallet records a `PENDING` deposit. (c) The ACH file is generated and sent to the ODFI (Originating Depository Financial Institution) via secure SFTP. (d) NACHA rules: the funds are available to the user on day 2 (if prenote was done). The ODFI reports settlement on day 2-3. (e) If the ACH returns (NSF — insufficient funds, or account closed), the deposit is reversed on day 3-5. (2) **Availability vs. finality** — the wallet credits the user's available balance immediately (day 1) but has a "good funds" flag set to false. The user can spend against the pending balance. If the ACH bounces, the wallet deducts the amount back (essentially a negative balance if the user has already spent it). This is the "available vs. collected" balance distinction. (3) **Loss protection** — merchants with high ACH return rates (>2%) have collection hold periods extended to 5 business days. The risk engine adjusts the hold period dynamically. (4) **Cross-border** — for international wires (SWIFT), the lifecycle is similar but takes 1-5 business days and involves correspondent banks. SWIFT ISO 20022 messages are the standard for interbank communication. (5) **UPI (India)** — UPI transactions settle in real-time (immediate finality). The wallet integration is simpler: `immediate_balance_change` with no settlement delay. (6) **Reconciliation** — a daily reconciliation job matches the wallet's pending transactions against the settlement report from the clearing house (FedACH, EBA Clearing). Discrepancies are resolved manually.

## Q4: Design a micro-wallet for an app that allows users to earn "coins" by watching ads and spend them on premium features. How does the virtual currency ledger differ from real-money wallets?

**Answer:** Virtual currency wallets are simpler (no KYC, no bank integrations) but have unique challenges: (1) **Data model** — identical to real-money wallets: `user_id, currency (COINS), balance, version`. But the "value" is not real money — it's a in-app currency that the app controls. (2) **Earning rules** — coins are minted when a user: watches an ad (+10 coins), completes a survey (+50 coins), wins a game (+25 coins). These are "system credit" transactions with no real-money backing. (3) **Spending rules** — coins are burned when a user: unlocks a premium feature (-200 coins), buys a virtual good (-100 coins). (4) **Inflation control** — the app must control the coin economy to prevent inflation. Common techniques: coins have expiry (use them in 30 days), time-gated earnings (max 100 coins/hour), or variable pricing for popular items. (5) **Anti-fraud** — since coins have no real-money cost, there's a higher incentive to cheat. Implement rate limits: max 100 ad views per day, device fingerprinting to prevent bot farms, and server-side validation of ad completion events. (6) **Export restrictions** — virtual currencies cannot be converted back to real money (gambling regulations). The system must enforce one-way flow: real money can buy coins, but coins cannot be withdrawn as cash. This avoids regulatory classification as a money transmitter.

## Q5: Design the wallet's security architecture. How do you protect against account takeover (ATO) and unauthorized transfers?

**Answer:** Layered security: (1) **Authentication** — strong customer authentication (SCA) for all sensitive operations: login (2FA/TOTP), sending money (biometric approval on mobile), changing the linked bank account (email + SMS OTP). (2) **Transaction signing** — every outbound transfer request includes a digital signature from the approved device. The server verifies: `HMAC(device_secret, amount || recipient || timestamp)`. (3) **Device binding** — the wallet is bound to known devices. If a login comes from a new device, a cooling period (24 hours) is enforced for new recipient transfers. (4) **Velocity limits** — per-user: max 10 transfers per day, max $10,000 cumulative per day, max $50,000 cumulative per month. Per-beneficiary: max $5,000 per day to a new recipient (24-hour clock). (5) **Beneficiary whitelist** — new beneficiaries require a verification step: the wallet deposits a small amount (e.g., $0.12) to the beneficiary account and asks the user to confirm the amount (micro-deposit verification). This is standard in the US for ACH verification. (6) **Anomaly detection** — an ML model monitors for: login from unusual location, transfers to unusual destinations, rapid device changes, and check-cashing patterns. Anomalies trigger Transaction Hold: the transfer is queued for manual review. (7) **Session management** — all sensitive operations require a fresh authentication session. Idle timeout: 5 minutes for mobile. (8) **Encryption** — PII and financial data is encrypted at rest (AES-256) and in transit (TLS 1.3). The wallet balance column is encrypted using application-level encryption (envelope encryption with AWS KMS).

## Q6: How does the wallet handle international remittances — sending money from a user in the US to a recipient in the Philippines? Design the currency conversion and local payout rail integration.

**Answer:** Cross-border remittance: (1) **Origination** — the US user initiates a transfer of $200 to a Philippine recipient. The wallet debits $200 from the user's USD balance (plus a remittance fee, e.g., $2.99). (2) **FX conversion** — the wallet uses its FX provider (e.g., CurrencyCloud, Wise) to convert $200 USD to PHP. Rate: $1 USD = PHP 56.50 → PHP 11,300. The FX rate includes a spread (0.5-1% margin for the wallet). (3) **Local payout rail** — in the Philippines, the funds are paid out to the recipient via: GCash (mobile wallet), PayMaya, or bank transfer (PESONet or InstaPay). PESONet: batch settlement, 1-day delay. InstaPay: real-time 24/7. The wallet selects the fastest available rail that the recipient is registered for. (4) **Recipient notification** — the recipient receives an SMS or email: "You received PHP 11,300 from [sender]. Click here to claim." If the recipient is already a wallet user, the funds arrive instantly in their wallet. (5) **Regulatory compliance** — each jurisdiction has specific requirements. Philippines: BSP (Bangko Sentral ng Pilipinas) regulations require KYC verification for recipients if the amount exceeds PHP 500,000. US: FinCEN (Financial Crimes Enforcement Network) requires reporting of remittances over $1,000. The wallet logs all remittances and files regulatory reports. (6) **Cost breakdown** — to the sender: $202.99. The wallet's cost: $0.50 FX fee to CurrencyCloud + $0.10 GCash payout fee. Profit: $2.39 per remittance. (7) **Tracking** — the sender can track the transfer status: "FX converted" → "Sent to Philippines" → "Recipient credited." Tracking uses the partner API's status webhooks.

## Q7: Design the wallet notification system. How does the user get real-time push notifications for every transaction (deposit, withdrawal, transfer)?

**Answer:** The notification pipeline: (1) **Event sourcing** — every wallet transaction is an event published to Kafka: `wallet_transaction.{user_id}.{type}`. (2) **Notification service** — a consumer reads from the event stream and constructs the notification message: "You received $50.00 from John Doe." / "Your withdrawal of $200.00 to Bank Account ending in 1234 was successful." (3) **Delivery channels** — (a) Push notification (Firebase Cloud Messaging for Android, APNS for iOS) — delivered within seconds. (b) SMS — for high-value transactions (> $10,000) or security alerts (login from new device). (c) Email — weekly digest or large deposits. (d) In-app notification — real-time banner in the wallet app. (4) **User preferences** — users can configure notification thresholds: "Notify me for all transactions" vs. "Only notify me for transactions over $100". Preferences are stored in a user profile service and evaluated at notification time. (5) **Channel prioritization** — push is the default for daily transactions. SMS is used only for security-critical events (password change, withdrawal to a new account). (6) **Batch notifications** — if the user makes 20 micro-deposits in a minute (e.g., from a payment splitter), the notification system batches them: "You received 20 payments totaling $134.50." (7) **Delivery guarantees** — the notification system guarantees at-least-once delivery. If the push notification service is unavailable (FCM/APNS downtime), the notification is queued and retried. (8) **Opt-out** — users can opt out of marketing notifications but not transaction notifications (regulatory requirement for financial apps).

## Q8: How do you design the wallet's P2P (peer-to-peer) transfer system — "Send money to a friend by phone number or username"?

**Answer:** P2P transfer flow: (1) **Recipient lookup** — sender enters the recipient's phone number, email, or username. The wallet's directory service resolves this to a `recipient_user_id` using an inverted index: `phone_index (phone_hash → user_id)`. (2) **Initiation** — sender enters amount ($50) and hits "Send". (3) **Authorization** — the sender authorizes the transfer: biometric (FaceID) or PIN. The server checks: `balance >= amount + fee`. (4) **Transaction execution** — the system debits the sender's wallet and credits the recipient's wallet in a single database transaction: `UPDATE wallets SET balance = balance - 50 WHERE user_id = sender AND balance >= 50; UPDATE wallets SET balance = balance + 50 WHERE user_id = recipient; INSERT INTO wallet_transactions ...`. (5) **Recipient notification** — the recipient gets a push notification: "You received $50.00 from [Sender Name]." (6) **Scheduled transfers** — support "Send $50 to John on the 1st of every month." Implemented as a cron job that checks scheduled_transfers where `next_run_date <= now()`. (7) **Request money** — recipient can send a payment request: "John requests $25.00 from you." The sender approves or declines. (8) **Limits** — unverified users: max $500 per day P2P. Verified (KYC completed): max $10,000 per day. Limits are configurable and affect the transfer at authorization time. (9) **Dispute resolution** — if the sender claims they didn't authorize a P2P transfer, the system flags it for review. If the transfer was to a new recipient from a new device, the burden of proof is on the recipient. This is the standard "unauthorized transaction" process for P2P.

## Q9: How does the wallet handle currency conversion when sending money to a user in a different currency zone?

**Answer:** Multi-currency P2P: (1) **Scenario** — sender (USD) sends $50 to recipient in Europe who holds EUR. (2) **Conversion decision** — does the sender pay in USD (and the system converts) or does the recipient receive USD and convert themselves? The system shows both options with estimated amounts. (3) **FX rate quote** — the FX service provides a real-time quote: `GET /fx/quote?from=USD&to=EUR&amount=50`. The quote is valid for 30 seconds (the user sees "Rate expires in 15 seconds"). (4) **Execution** — when the sender confirms, the system: locks the rate, converts $50 USD → €46.22 EUR using the quoted rate, credits the recipient's EUR wallet with €46.22. The rate lock ensures the user gets the quoted amount even if market rates change during the 5-second confirmation flow. (5) **Spread** — the wallet adds a spread (e.g., 0.5%) to the mid-market rate. Mid-market: 1 USD = 0.93 EUR. With spread: 1 USD = 0.924 EUR. The spread is the wallet's revenue for providing the conversion service. (6) **Recipient choice** — the recipient can choose to receive in USD (keep the funds in a USD wallet) or convert to EUR. This choice affects the FX rate and any conversion fees. (7) **Multi-currency wallets** — the wallet system maintains per-user, per-currency balances. User A may hold both USD and EUR wallets. A P2P transfer from User A (USD) to User B (EUR) is: debit User A's USD wallet, credit User B's EUR wallet (with FX conversion). This is a single atomic operation in the ledger.

## Q10: Design the wallet's regulatory compliance system — how do you enforce Know Your Customer (KYC), Anti-Money Laundering (AML), and sanctions screening at scale?

**Answer:** Compliance is non-negotiable for a wallet operating globally: (1) **KYC tiers** — (a) Tier 0: email + phone only — max balance $100, max monthly volume $500. (b) Tier 1: Tier 0 + identity verification (government ID scan + selfie) — max balance $10,000, max monthly volume $25,000. (c) Tier 2: Tier 1 + proof of address (utility bill) — unlimited. The KYC verification uses a third-party provider (Jumio, Onfido, Stripe Identity) for document verification and liveness detection. (2) **AML monitoring** — all transactions are scored by an AML engine (e.g., Chainalysis for crypto, or a custom rules engine for fiat). Rules: (a) Structuring detection — multiple deposits under $10,000 within 24 hours (CTR threshold in the US). (b) Rapid movement — funds deposited and withdrawn to a different account within minutes. (c) High-risk jurisdictions — transactions to/from OFAC-sanctioned countries are blocked. (3) **Sanctions screening** — every transaction participant (sender, recipient, bank account holder) is screened against: OFAC SDN list, EU sanctions list, UN sanctions list, and local regulatory lists. Screening is done at signup and on every transaction using a fuzzy name-matching service (e.g., World-Check, LexisNexis). (4) **Suspicious Activity Report (SAR)** — transactions flagged by AML rules are escalated to a compliance team. If confirmed suspicious, a SAR is filed with FinCEN (in the US) or the relevant FIU. (5) **Travel Rule** — for transfers over $3,000 (FinCEN) or €1,000 (EU), the wallet must share sender/recipient information with the counterparty wallet (name, address, account number). This is implemented as a secure message exchange between compliant wallet providers (using the TRISA protocol or similar). (6) **Audit trail** — all compliance decisions (KYC approvals, AML flags, SAR filings) are logged immutably for regulatory audit. Retention: 5 years after account closure (per BSA/AML requirements).


## Q11: Design a virtual card issuance (VCN) system within the wallet. How do you generate, authorize, and settle single-use or merchant-locked virtual card numbers at scale?

**Answer:** VCN architecture: (1) **Card BIN sponsorship** — the wallet partners with a issuing bank (e.g., Stripe Issuing, Marqeta, Solarisbank) to obtain a block of BINs (Bank Identification Numbers). The wallet is the Program Manager; the partner bank holds the regulatory license. (2) **VCN generation** — on user request, the wallet calls the card processor's API to create a virtual card: `POST /cards {user_id, currency, spending_limit, merchant_locked (optional), single_use (bool)}`. The processor returns PAN (16-digit), expiry, CVV. For single-use cards, the PAN is deactivated immediately after the first authorization. For merchant-locked cards, the processor stores a `merchant_id` whitelist and rejects authorizations from any other merchant. (3) **Authorization flow** — when the VCN is used at a merchant: (a) Merchant sends ISO 8583 authorization message → Acquiring bank → Card network (Visa/MC) → Issuing processor → Wallet's authorization service. (b) The wallet checks: is the card active? Is the amount within the spending limit? Is the merchant on the allowed list (if merchant-locked)? Does the user's wallet balance cover the hold? (c) If approved, the wallet places a **hold** (pending authorization) against the user's balance: `balance_hold = balance_hold + amount; available_balance = balance - balance_hold`. This is a ledger entry separate from the confirmed balance. (d) The processor responds to the network: authorization approved or declined. (4) **Settlement** — at end of day, the processor sends a settlement file (all captured authorizations). The wallet settles: converts holds to confirmed debits, releases expired holds. (5) **3DS authentication** — for card-not-present transactions, the wallet's processor runs 3D Secure (3DS) authentication. The wallet must support the challenge flow (redirect the user to their banking app to approve). (6) **Key design decisions** — VCN data is **not stored in the wallet's DB** (PCI scope). Instead, the wallet stores a tokenized reference (`card_ref`) and the processor stores the PAN. The wallet's authorization system must be **multi-region active-active** with <100ms p99 auth latency because card authorization timeouts default to "decline" on the network, causing cardholder frustration.

## Q12: Design the P2P transfer fraud network — how do you detect and disrupt rings of fraudulent accounts that send money to each other to simulate legitimate activity or launder funds?

**Answer:** Fraud network detection requires graph analysis: (1) **Transaction graph construction** — build a directed graph where nodes are `user_id` and edges are P2P transfers with attributes: `{amount, timestamp, device_id, ip_address, location}`. The graph is stored in a graph database (Neo4j) or a specialized column store (e.g., TigerGraph) for real-time traversal. (2) **Community detection (Louvain/Connected Components)** — run daily batch algorithms to find highly interconnected subgraphs. A "fraud ring" pattern: 5–20 accounts that all send to each other in a dense cluster, with no external edges to legitimate users. (3) **Flow analysis** — track money flow through the graph. Suspicious pattern: Account A deposits $10K → sends to B → B sends to C → C sends to A (circular flow). The net position of all accounts is zero, but $30K of volume was generated — classic layering. (4) **Real-time edge scoring** — each new P2P transfer triggers a **neighborhood scan**: traverse 2 hops from sender and recipient. Score = `f(shared_devices, shared_ip, velocity_to_common_accounts, recency_of_mutual_transactions)`. If score > threshold, the transfer is held for manual review. (5) **Device fingerprint clustering** — if 10 accounts all logged in from the same device ID (emulator farm), they're likely synthetic identities. Block all transfers involving these accounts and escalate to KYC review. (6) **Temporal pattern detection** — fraud rings often transfer in lockstep: all accounts active between 2–4 AM, all transfers completed within 60 seconds of each other. Use **time-series anomaly detection** (Twitter's AnomalyDetection or Facebook's Prophet) on per-account transfer intervals. (7) **Architecture** — the graph database is updated asynchronously (sub-second delay via Kafka). The real-time scoring service is a Go/C++ service that holds the 2-hop neighborhood in a Redis graph cache for hot accounts. Cold accounts fall through to the Neo4j backend. (8) **Feedback loop** — confirmed fraud rings are used as training data for a GNN (Graph Neural Network) model that predicts the probability of a new account being part of a ring based on its initial transfer patterns.

## Q13: Design the multi-currency wallet rebalancing system. How do you automatically move funds between currency wallets (USD, EUR, GBP) to minimize FX conversion costs while maintaining sufficient liquidity in each currency?

**Answer:** Multi-currency rebalancing as an optimization problem: (1) **Data model** — each user has N currency wallets: `{user_id, currency, balance, target_pct, min_balance, max_balance}`. The user or the system sets a target allocation (e.g., 50% USD, 30% EUR, 20% GBP). (2) **Rebalancing triggers** — (a) Scheduled (daily at midnight). (b) Threshold-based: when any currency balance deviates > 10% from target. (c) Event-driven: after a large incoming transfer in a non-target currency. (3) **Optimization engine** — on each trigger, solve: `minimize(sum(FX_cost(i,j) * amount(i,j)))` subject to: `final_balance(c) ≈ target_pct(c) * total_balance`. For each pair of currencies, `FX_cost` includes the spread and any fixed transaction fee. The optimizer (linear programming or greedy heuristic) selects which currencies to convert and in what amounts. (4) **FX cost minimization** — it's cheaper to convert USD→EUR directly than USD→GBP→EUR. The optimizer builds a **cost matrix** of all pairwise conversion costs and finds the shortest path (Dijkstra-like) for each required conversion. (5) **Liquidity constraints** — each currency pool has a minimum operational balance (e.g., EUR wallet must always have at least €1,000 to cover pending payouts). The optimizer never converts below this floor. (6) **Execution** — the optimizer generates a batch of conversion orders: `{from_currency, to_currency, amount, target_rate}`. These are submitted to the FX provider (Wise, CurrencyCloud) as a batch for best execution. The provider may offer a better rate for larger batch volumes. (7) **User experience** — the user sees a single consolidated balance with a "currency breakdown" tab. Rebalancing happens silently in the background unless the user opts for manual control. (8) **Edge case** — if the total balance is too small (< $100 equivalent), rebalancing is skipped because FX costs would exceed the benefit. The threshold is configurable and should be tested via simulation against historical transaction data.

## Q14: Compare and contrast micro-deposit verification vs. instant account verification (via third-party APIs like Plaid) for linking external bank accounts. Under what conditions would you choose one over the other at principal-engineer scale?

**Answer:** (1) **Micro-deposit verification** — the wallet sends two small deposits (< $1 total) to the user's bank account. The user must check their bank statement and report the amounts back. Pros: works with any bank in the US (ACH-enabled). No third-party dependency. Full control over the verification flow. Cons: 2–3 day settlement delay (ACH). Poor UX — user must leave the app, check their bank, return. 20–30% abandonment rate at this step. Cannot verify account ownership before the deposits arrive (fraud window). (2) **Instant verification (Plaid/Finicity/Yodlee)** — the user logs into their bank via OAuth within the wallet app. The provider returns: `account_name, routing_number, account_number, balance, transactions (optional), and ownership_confirmed`. Pros: instant (<10 seconds). Rich data (balance check, transaction history for credit decisions). 90%+ completion rate. Supports account ownership verification in real-time. Cons: dependency on third-party API (Plaid outages affect the wallet). Only works with supported banks (covers ~95% of US consumer accounts, but not credit unions or smaller banks). Costs $0.50–$1.50 per verification. (3) **Principal-level decision framework** — use **instant verification as the default** for consumer-facing flows (superior UX, higher conversion). Fall back to **micro-deposit for unsupported banks or when the instant provider is unavailable**. For **commercial/b2b accounts**, micro-deposit may be preferred because: (a) B2B users are more patient; (b) Plaid's business account coverage is weaker; (c) the fraud surface is different (corporate accounts are harder to open fraudulently). For **high-risk accounts** (suspected fraud), require instant verification to get ownership proof before allowing withdrawals. For **low-value accounts** (< $100 lifetime), micro-deposit is acceptable since the fraud loss is bounded. (4) **Hybrid approach** — attempt instant verification; if it fails, offer micro-deposit with a real-time notification ("We deposited $0.12 and $0.34 to your account. Enter the amounts to verify."). This captures the 95% that can use instant and provides a graceful fallback for the remaining 5%.

## Q15: Design the recurring payment authorization model for subscription billing. How does the wallet handle authorization, capture, retry logic, dunning, and card expiry?

**Answer:** Recurring payment lifecycle: (1) **Initial authorization** — when the user subscribes, the wallet obtains a **network token** (see Q16) or a card reference. It runs an initial $0 authorization to verify the card is active, then a $1 authorization hold to confirm sufficient funds (released immediately). (2) **Recurring billing schedule** — a scheduler service (cron/Quartz) triggers on the billing date. For each active subscription due today, the system: (a) Checks if the user's wallet balance is sufficient. (b) If yes, performs an internal ledger transfer (wallet→merchant). (c) If no, attempts to **pull funds from the linked payment method** (card or bank account) via the processor. (3) **Retry logic** — if the initial payment attempt fails (insufficient funds, card declined), the system enters a **dunning management** flow. Configurable retry schedule: Day 0: initial attempt. Day 3: retry 1. Day 7: retry 2. Day 14: retry 3. Day 21: final retry. Each retry uses exponential backoff on the retry count. The user receives email/SMS notifications before each retry. (4) **Network token updater** — cards expire, are reissued, or are replaced. The wallet subscribes to the **Visa Account Updater (VAU)** / **Mastercard Automatic Billing Updater (ABU)** service. When the card network notifies that a card has been reissued, the wallet automatically updates the stored card credentials, preventing involuntary churn from card expiry. (5) **Authorization model** — recurring payments use **credential-on-file** (CoF) or **merchant-initiated transactions** (MIT) under PSD2/SCA. The initial authorization requires Strong Customer Authentication (SCA). Subsequent recurring payments are exempt from SCA (merchant-initiated). The wallet must flag the transaction as `recurring` in the ISO 8583 message to avoid unnecessary 3DS challenges. (6) **Failed payment handling** — after max retries, the subscription is marked `lapsed`. The user must re-authenticate (SCA) to reactivate. The system sends a final notification: "Your subscription has been paused. Update your payment method to continue." (7) **Partial capture** — if the user has insufficient funds for the full amount, the wallet can capture a partial amount and adjust the subscription (e.g., downgrade to a lower tier for this cycle). This is a product decision, not purely technical. (8) **Architecture** — the scheduler emits `billing_due` events to a Kafka topic. A consumer processes each event, calls the payment processor, and writes the result to a `subscription_billing_attempts` table. The dunning state machine is implemented as a lightweight workflow engine (Temporal or AWS Step Functions) for durability and observability.

## Q16: Design the digital wallet's tokenization strategy — how do you replace sensitive card PANs with network tokens (Visa Token Service, MDES) and what architectural changes are required?

**Answer:** Network tokenization (Visa Token Service — VTS, Mastercard Digital Enablement Service — MDES) replaces the 16-digit PAN with a **network token** — a digital identifier issued by the card network that is unique per device-merchant pair. Architecture: (1) **Token request** — when the user adds a card to the wallet, the wallet's mobile SDK calls the card network's token service: `POST /tokens {pan, expiry, cvv, merchant_id, device_id}`. The network returns a `token` (16-digit, Luhn-valid, but routed only to that network) and a `token_expiry`. (2) **Token storage** — the wallet stores the token (not the PAN) in its token vault. The PAN is discarded. The token is useless if stolen — it only works with this specific merchant (the wallet provider) and device pairing. (3) **Transaction flow** — on payment: (a) Wallet sends `{token, amount, merchant_id}` to the acquirer. (b) Acquirer routes to the card network. (c) The network de-tokenizes (token → PAN) and sends to the issuing bank. (d) The issuing bank sees the original PAN and authorizes. (4) **Token lifecycle** — tokens are network-managed: if the user's card is reissued (lost/stolen/expired), the card network automatically updates the token mapping. The wallet receives a webhook: `token_updated {old_token, new_token, reason}`. This eliminates the need for the wallet to store new card details. (5) **Domain restriction** — tokens are cryptographically bound to a **domain** (the wallet's merchant ID). A token stolen from the wallet cannot be used at a different merchant. This is a critical security advantage over PAN storage. (6) **Architectural changes required** — (a) **PCI scope reduction**: with tokenization, the wallet never stores, transmits, or processes PANs. The PCI DSS scope shrinks from the entire wallet infrastructure to just the token API call path. (b) **Token vault**: the wallet needs a secure token database (HSM-backed or using AWS CloudHSM) to store the token↔user mapping. (c) **Network integration**: the wallet must integrate with VTS/MDES APIs, which requires certification (Visa Ready, Mastercard Digital Enablement). (d) **Fallback**: if tokenization fails (e.g., unsupported card network), the wallet falls back to PAN storage with PCI DSS compliance. (e) **Recurring payments**: tokens support credential-on-file (CoF) and recurring payments — the token is re-usable for subsequent charges without additional authentication. (7) **Trade-offs** — tokenization reduces PCI scope and improves security, but adds dependency on the card network's token service. Token provisioning adds ~200ms to card addition latency. For high-volume wallets, this is an acceptable trade-off to eliminate PAN breach risk.

## Q17: Design a yield optimization system for idle currency holdings. How does the wallet automatically sweep idle balances into interest-bearing instruments while maintaining liquidity for daily operations?

**Answer:** Yield optimization for wallet balances is essentially a **cash management** problem: (1) **Tiered balance structure** — each currency's total pool is divided into three tiers: (a) **Operating cash** — 5% of total — held in the settlement account at the partner bank (0% interest). Immediate liquidity for same-day withdrawals and card settlements. (b) **Sweep cash** — 60% — swept automatically to money market funds (MMFs) or government bonds via an automated sweep program. These are next-day liquid (T+1 redemption). Yield: 4.5–5.5% (as of 2025). (c) **Investment cash** — 35% — invested in short-term treasury bills (1–3 month duration) or high-grade corporate bonds via the partner bank's wealth management arm. Yield: 5–6%. T+2–3 to redeem. (2) **Sweep mechanism** — at end of each business day, a batch job calculates: `sweep_amount = total_balance - operating_cash_target - projected_next_day_outflows`. The positive sweep amount is transferred to the MMF provider via the partner bank's API. Reverse sweep happens in the morning: funds are pulled back to meet the day's projected outflows. (3) **Outflow prediction** — a time-series forecasting model predicts the next day's withdrawals based on: historical patterns (day of week, seasonality), scheduled payments (subscriptions billed tomorrow), and known large withdrawals (user-scheduled). The model's 95th percentile prediction defines the operating cash target. (4) **Yield sharing** — the wallet provider keeps a spread (e.g., earns 5% on swept funds, pays user 3.5%). The 1.5% spread is the wallet's revenue for providing the service. Some wallets (e.g., Revolut, Wealthfront) pass through the full yield as a competitive differentiator. (5) **Regulatory constraints** — in the US, sweeping customer funds to MMFs requires the wallet to be a **broker-dealer** (SEC-regulated) or partner with one. In the EU, the wallet needs a **banking license** (or partner with a bank) to offer interest on e-money balances. (6) **Risk management** — the MMF must be a **government or prime money market fund** (Rule 2a-7 under the Investment Company Act) — these maintain a stable $1 NAV and are considered low-risk. The portfolio duration is capped at 60 days. The investment cash tier uses only **investment-grade bonds** (BBB rating or above). (7) **Architecture** — the sweep engine runs as a scheduled job (Temporal workflow). It reads the current balance from the ledger service, calls the forecast model API, computes the sweep amount, and executes the transfer via the bank partner's API. All sweep activity is logged for regulatory audit.

## Q18: Design the cross-border remittance compliance system. How do you handle OFAC sanctions screening, travel rule information sharing, and suspicious activity reporting at global scale?

**Answer:** Cross-border remittance compliance requires integration with multiple regulatory frameworks simultaneously: (1) **OFAC/Sanctions screening** — every transaction participant (sender, recipient, intermediary) is screened against: (a) OFAC SDN (Specially Designated Nationals) list, (b) EU consolidated sanctions list, (c) UN sanctions list, (d) HMT (UK) sanctions list, (e) local lists (e.g., DFAT for Australia). Screening is done at **account creation** (ID verification) and **per transaction**. Implementation: a fuzzy name-matching engine (World-Check One, LexisNexis Bridger) compares the participant's name and date of birth against the lists. The engine returns a **match score** (0–100). Scores > 95 are auto-blocked. Scores 75–95 are queued for manual review within 1 hour (SLAs require same-day resolution for remittances). Scores < 75 are passed. (2) **Travel Rule compliance** — under FinCEN's Travel Rule (31 CFR § 1010.100(f)) and FATF Recommendation 16, for transactions > $3,000 (US) or > €1,000 (EU), the wallet must share with the counterparty wallet: sender's name, address, account number; recipient's name, address, account number. Implementation: the wallet uses the **TRISA protocol** (Travel Rule Information Sharing Architecture) or **OpenVASP** for secure, encrypted message exchange between compliant VASPs (Virtual Asset Service Providers). The wallet has a **TRISA node** that listens for Travel Rule requests, validates them, and responds with the required PII. (3) **Suspicious Activity Report (SAR) filing** — transactions flagged by the AML engine (see Q10) are investigated by the compliance team. If confirmed suspicious, a SAR must be filed with the local Financial Intelligence Unit (FIU) within 30 calendar days (in the US: FinCEN). Implementation: the SAR filing is partially automated — the compliance system generates a draft SAR with transaction details, supporting documents, and the reason for suspicion. The compliance officer reviews and submits via the BSA E-Filing System (US) or the relevant FIU's portal. (4) **Record keeping** — all compliance data (screening results, Travel Rule messages, SAR filings) must be retained for 5 years after account closure (per BSA requirements). Data is stored in an immutable audit log (e.g., AWS S3 with Object Lock, or a blockchain-based audit trail). (5) **Scalability** — at 1M+ remittances per day, manual compliance review is impossible. The system uses a **tiered automation**: (a) Auto-block (score > 95) — fully automated, no human touch. (b) Auto-pass (score < 75, transaction amount < $10K, sender KYC tier 2) — no review. (c) Manual review (score 75–95, or any amount > $10K, or sender tier 0) — queued for compliance team. (d) Batch review — low-risk corridors (US→UK, EU→CA) have simplified review processes. (6) **Geographic routing** — for sanctioned countries or high-risk jurisdictions (FATF gray list), all transactions are blocked at the gateway, before any screening cost is incurred. The list of blocked jurisdictions is loaded from a config store and cached hot in the API gateway with a 5-minute TTL.

## Q19: Design the chargeback liability shift model. How does the wallet determine who bears the cost of a disputed transaction — the merchant, the wallet provider, or the user — and how does this affect the transaction routing decision?

**Answer:** Chargeback liability follows a complex set of card network rules (Visa Core Rules, Mastercard Chargeback Guide): (1) **Liability shift scenarios** — (a) **CNP (Card Not Present) fraud**: if the transaction was 3DS-authenticated, the liability shifts to the issuing bank (and thus to the wallet). If not 3DS-authenticated, the merchant is liable. The wallet as the issuer bears the fraud cost for 3DS-authenticated transactions. (b) **Counterfeit fraud (Chip vs. Magstripe)**: in EMV chip transactions, liability shifts to the party that doesn't support chip (usually the merchant if they use magstripe). For digital wallets, the token (VTS/MDES) is considered chip-equivalent, so liability shifts to the merchant if they don't support tokenization. (c) **Dispute types**: fraud (user claims they didn't authorize): wallet/issuer liability if 3DS-authenticated, merchant liability if not. Service not provided: merchant liability. Processing error (duplicate charge, wrong amount): merchant liability. Credit not processed: merchant liability. (2) **Transaction routing decision** — the wallet can decide (at time of payment) whether to route as: (a) **3DS-authenticated** (liability shift to issuer/wallet) — lower merchant cost (no chargeback risk for fraud) but higher wallet cost and higher friction (user must complete 3DS challenge). (b) **Non-3DS** (liability on merchant) — lower friction, but merchant bears fraud chargeback risk. For high-trust merchants (subscription, low fraud rate), the wallet may route without 3DS to reduce friction. For high-risk merchants (e-commerce, digital goods), 3DS is forced. (3) **Chargeback cost allocation** — when a chargeback occurs and the wallet is liable: (a) The card network charges the wallet a **chargeback fee** ($20–$100 per dispute, network-specific). (b) The wallet debits the merchant's reserve account if the merchant is liable, or absorbs the cost if the wallet is liable. (c) If the user is found to be committing **friendly fraud** (claiming a legitimate transaction is fraudulent), the wallet blocks the user and files a **representment** package to contest the chargeback. If the wallet wins, the chargeback is reversed. (4) **Chargeback analytics** — the wallet tracks: `chargeback_ratio = total_chargebacks / total_transactions` per merchant. If a merchant exceeds Visa's threshold (0.9% chargeback ratio or 100+ chargebacks/month), the merchant enters the Visa **Chargeback Monitoring Program** and faces fines of $25,000–$50,000. The wallet proactively terminates merchants approaching this threshold. (5) **Operational system** — a chargeback management service ingests chargeback notifications from the card network (via the processor's webhook). It determines liability, notifies the merchant/user, and manages the representment workflow (deadline: 30–45 days from chargeback). The system tracks the **chargeback lifecycle**: filed → represented → won/lost → closed.

## Q20: Design the wallet migration system — how do you migrate users and their balance history from a legacy wallet provider (e.g., Stripe Connect) to a new in-house wallet platform without disrupting user experience or losing transaction history?

**Answer:** Wallet migration is a high-risk, phased operation: (1) **Data extraction phase** — (a) Export all transaction history from the legacy platform via their API (time-bound, with pagination). For Stripe Connect: `GET /v1/balance_transactions` and `GET /v1/transfers`. (b) Export user PII (KYC data, linked accounts, saved cards). For cards, export only the tokenized reference (not the PAN) — the new wallet will need to re-collect card details or use the legacy platform's network token for continuity. (c) Export current wallet balances. This is the critical number: each user's `(currency, balance)` at the migration time. (2) **Reconciliation phase** — compare the exported balance from the legacy platform against the new platform's ledger for a sample of users (1%). The deviation should be zero. If not, fix the data transformation pipeline. This is the **highest-risk step** — any mismatch means user funds are misrepresented. (3) **Read-only migration** — phase 1: the new wallet platform is deployed in **read-only shadow mode**. All users can log into the new app and view their transaction history (sourced from the imported data), but all transactions still execute on the legacy platform. Run in this mode for 2–4 weeks, monitoring: error rates, data discrepancy reports from users, and performance. (4) **Dual-write phase** — phase 2: new transactions are written to **both** the legacy and new platforms. A dual-write proxy intercepts all wallet API calls, sends them to both platforms, and returns the legacy platform's response to the client. The new platform's response is compared asynchronously — discrepancies are logged and alerted. If the new platform has a bug, the legacy platform's result is authoritative. (5) **Cut-over** — phase 3: after 2+ weeks of zero discrepancies, the dual-write proxy is inverted: the new platform becomes authoritative, and the legacy platform receives shadow writes. If no issues arise in 1 week, the legacy platform is decommissioned. (6) **Balance freeze** — at the exact cut-over moment, all user balances are frozen for 5 minutes (scheduled during lowest traffic — 3 AM Sunday). No transactions can occur during the freeze. The final balance from the legacy platform is loaded into the new platform. After freeze, the new platform is live. (7) **Rollback plan** — if the cut-over fails, the dual-write proxy switches back to the legacy platform as authoritative within 30 seconds. This requires both platforms to have independent state — no data corruption from the failed cut-over. (8) **User communication** — users are notified 30 days, 7 days, and 24 hours before the migration. During the 5-minute freeze, the app displays: "Scheduled maintenance — your funds are safe. We'll be back online in a few minutes." Post-migration, users may need to re-verify identity (depending on KYC requirements of the new platform). (9) **Card & bank account migration** — for saved payment methods: (a) Cards → request the legacy processor to migrate the network tokens to the new processor (if both use VTS/MDES, this is a token-assignment change). If tokens cannot be migrated, users must re-add their cards. (b) Bank accounts → micro-deposit verification must be re-done unless the new platform partners with the same bank verification provider (Plaid's account identity can be carried over with user consent). (10) **Principal-level risks** — (a) **Balance drift**: during dual-write phase, user balances may diverge between platforms due to different latency in processing. The reconciliation system must detect and resolve these within seconds. (b) **Transaction drain**: high-volume users may exhaust API rate limits on the legacy platform during the extract phase. Batch extraction with adaptive throttling is required. (c) **Regulatory continuity**: the wallet must maintain an unbroken audit trail. The new platform's transaction IDs must map to the legacy platform's IDs via a cross-reference table.

## Q21: Design the wallet balance management system. How do you track real-time balances across multiple currencies, handle concurrent deposits/withdrawals, and ensure no balance goes negative?

**Answer:** Wallet balance management is a double-entry accounting system with locking. (1) **Ledger model**: (a) Each wallet has multiple currency balances: `wallets (user_id, currency, balance_cents, version)`. (b) Every transaction writes two ledger entries: debit one account, credit another (double-entry). (c) Example: deposit $100: `LedgerEntry(from: null, to: user:123, currency: USD, amount: 10000, type: DEPOSIT)`. (2) **Concurrency control — Optimistic locking**: (a) `UPDATE wallets SET balance_cents = balance_cents + ?, version = version + 1 WHERE user_id = ? AND currency = ? AND version = ?`. (b) If the version doesn't match, retry the transaction. (c) This handles concurrent deposits/withdrawals without pessimistic locks. (3) **Negative balance prevention**: (a) Check before withdrawal: `SELECT balance_cents FROM wallets WHERE user_id = ? AND currency = ? FOR UPDATE`. (b) If balance_cents < withdrawal amount, reject with INSUFFICIENT_FUNDS. (c) For atomicity: wrap the check and update in a database transaction. (4) **Reserved (pending) balances**: (a) When a user initiates a payment, the amount is "reserved" — deducted from available but not yet settled. (b) Available = balance - sum(pending_reservations). (c) `reservations (id, user_id, currency, amount, status, expires_at)`. (5) **Multi-currency balance**: (a) Each currency has its own balance row. (b) Total wallet value: sum(balance_cents × exchange_rate_to_base) across all currencies. (6) **Reconciliation**: (a) Daily job: sum all wallet balances and compare against the bank account balance. (b) Difference = in-flight transactions (not yet settled). If difference > epsilon, alert.

## Q22: Design the transaction history service. How do you store and query billions of transactions for real-time display and historical analysis?

**Answer:** Transaction history at wallet scale requires time-partitioned storage. (1) **Data model**: (a) `transactions (id, user_id, type, amount, currency, balance_before, balance_after, counterparty, description, status, created_at)`. (b) Partition by `user_id` hash or by `created_at` month. (2) **Hot vs cold storage**: (a) Hot (last 90 days): stored in a relational DB (PostgreSQL) partitioned by month. (b) Warm (90 days to 1 year): stored in ClickHouse for fast analytics queries. (c) Cold (>1 year): archived to Parquet files on S3, queryable via Athena. (c) Cached in Redis for the top 5 pages (most recent transactions). (4) **Search**: (a) Allow searching by amount range, date range, transaction type, counterparty name. (b) Use a search index (Elasticsearch) for full-text search on counterparty and description. (5) **Export**: (a) CSV export: `GET /wallet/transactions/export?from=2025-01-01&to=2025-12-31`. (b) A background job generates the CSV and emails the download link. (6) **Analytics queries**: (a) Monthly spending by category. (b) Average balance over time. (c) Top merchants by spend. These run on ClickHouse for speed.

## Q23: Design P2P (peer-to-peer) transfer between wallet users. How do you handle transfers between users of the same wallet platform, including escrow, reversal, and dispute resolution?

**Answer:** P2P transfers require atomic debits and credits across two accounts. (1) **Transfer flow**: (a) Sender initiates transfer: amount, currency, recipient identifier (email/phone/wallet ID). (b) System validates: sender has sufficient funds, recipient exists, amount within limits. (c) Atomic operation: BEGIN TRANSACTION; debit sender; credit recipient; insert transfer record; COMMIT. (d) Both users receive push notifications. (2) **Escrow**: (a) For goods/services (marketplace), the transfer is held in escrow: `LedgerEntry(from: sender, to: escrow, amount: 100, type: ESCROW_HOLD)`. (b) On delivery confirmation, release: `LedgerEntry(from: escrow, to: recipient, amount: 100, type: ESCROW_RELEASE)`. (c) If dispute, escrow is held until resolved. (3) **Reversal**: (a) If the recipient never received the funds (unlikely — the transfer is atomic), the sender can request a reversal within 24 hours. (b) Reversal: `LedgerEntry(from: recipient, to: sender, amount: 100, type: TRANSFER_REVERSAL)`. (c) Requires the recipient's consent or admin override. (4) **Dispute resolution**: (a) Recipient claims they didn't receive the service. (b) Sender claims the transfer was fraudulent. (c) Both parties submit evidence. (d) Admin reviews and decides: release to recipient or reverse to sender. (5) **Limits**: (a) Per-transfer limit: $10,000. (b) Daily cumulative limit: $50,000. (c) Monthly limit: $500,000. (d) Limits are configurable per user tier (verified users get higher limits). (6) **Anti-fraud**: (a) Monitor for unusual activity: multiple transfers to new recipients in short time, large transfers from new accounts. (b) Flag for manual review if risk score > threshold.

## Q24: Design the QR code payment system for in-store and online payments. How do you generate dynamic QR codes, handle payment confirmation, and support merchant-presented vs customer-presented flows?

**Answer:** QR code payments require a real-time payment session between customer and merchant. (1) **Merchant-presented QR (MPQR)**: (a) The merchant's POS generates a QR code encoding: payment_session_id, merchant_id, amount, currency. (b) The customer scans the QR with their wallet app. (c) The wallet app resolves the payment session: `GET /payment-sessions/{session_id}` returns merchant details and amount. (d) Customer confirms payment. (e) Wallet app calls `POST /payments {session_id, amount, currency, from_wallet}`. (f) The payment system attempts the transfer. (g) The merchant POS polls: `GET /payment-sessions/{session_id}/status`. On `COMPLETED`, the POS shows "Payment successful." (2) **Customer-presented QR (CPQR)**: (a) The customer opens their wallet app, which generates a QR code with: wallet_id, merchant_id (optional for dynamic), or just a static wallet address. (b) The merchant scans the QR (at checkout). (c) The merchant's POS sends: `POST /payments {from_wallet_id, to_merchant_id, amount, currency}`. (d) The wallet app sends a push notification: "Authorize payment of $15.50 to Starbucks?" (e) Customer approves on their phone. (3) **Dynamic vs static QR**: (a) Dynamic QR: includes a session ID that expires in 5 minutes. Amount is pre-set by the merchant. (b) Static QR: just the merchant's wallet address. Customer enters the amount manually. (4) **Offline QR**: (a) If the customer has no internet, the wallet app can store the QR content and process the payment when connectivity is restored. (b) The merchant can also process offline and sync later. (c) Risk: the payment may fail (insufficient funds). Handle via guaranteed payments (the wallet platform covers the risk). (5) **Confirmation UX**: (a) Sound/vibration on both phones when payment completes. (b) Receipt screen: transaction ID, amount, merchant name, timestamp. (c) Failure screen: reason (insufficient funds, expired QR, network error) and retry option.

## Q25: Design the wallet top-up system. How do users add funds to their wallet via bank transfer, credit card, debit card, and direct deposit?

## Q29: Design the fraud detection system specific to digital wallets. How do you detect account takeover, unauthorized transfers, money laundering (structuring), and loyalty point abuse?

**Answer:** Wallet fraud detection must catch both external attacks and internal abuse. (1) **Account takeover detection**: (a) Monitor login patterns: new device, new IP, new location, unusual time (3 AM from a foreign country). (b) Step-up authentication: if risk score > threshold, require OTP or biometric verification before allowing transfers. (c) Behavioral biometrics: typing speed, mouse movement, swipe pattern — if the current session differs from the user's baseline, flag. (2) **Unauthorized transfer detection**: (a) If a user sends money to a new recipient (never transferred to before), flag for review. (b) If transfer amount exceeds the user's 95th percentile of historical transfers, add friction: "This transfer is unusually large. Please confirm via OTP." (c) Velocity check: more than 3 transfers to different recipients in 5 minutes is suspicious. (3) **Money laundering (structuring) detection**: (a) Detect smurfing: multiple small transfers from different accounts to a single account, just below the reporting threshold. (b) Pattern: 7 transfers of $9,500 from 7 different senders to the same recipient in 24 hours. (c) Alert and file SAR (Suspicious Activity Report) with FinCEN. (4) **Loyalty point abuse**: (a) Monitor for point farming: creating multiple accounts to earn sign-up bonuses → device fingerprint clustering. (b) Points transfer between accounts in a ring pattern → flag. (c) Redeeming points for cash or gift cards shortly after earning → possible money laundering. (5) **Machine learning model**: (a) Features: transaction amount, frequency, time since account creation, number of unique recipients, average balance, device fingerprint, IP geolocation distance from home address. (b) Online learning: the model updates every hour with new labeled data (confirmed fraud → positive, legitimate transactions reviewed → negative). (6) **Manual review queue**: (a) Transactions flagged with score > 0.7 go to manual review. (b) Reviewer dashboard: transaction details, risk factors highlighted, one-click actions (approve, decline, freeze account). (c) SLA: review within 5 minutes for high-value transactions (>$10,000).

## Q30: Design the regulatory compliance (KYC/AML) system for digital wallets. How do you verify user identity, monitor transactions for suspicious activity, and file reports with financial authorities?

**Answer:** Wallet compliance is governed by AML/KYC regulations, including the Bank Secrecy Act (US), 5AMLD (EU), and FATF recommendations. (1) **KYC tiers**: (a) **Tier 0 (unverified)**: email + phone only. Can receive money, cannot send more than $500/month. (b) **Tier 1 (basic verified)**: add government ID verification + selfie. Send up to $10,000/month. (c) **Tier 2 (fully verified)**: Tier 1 + proof of address + source of funds declaration. Unlimited sends. (2) **Identity verification**: (a) Document verification: passport/driver's license → extract MRZ (Machine Readable Zone) + face photo. (b) Liveness check: user records a short video (blink, turn head). (c) Database check: verify against government databases (via Jumio/Onfido). (d) Sanctions screening: check against OFAC, UN, EU sanctions lists. (3) **AML transaction monitoring**: (a) Rules-based triggers: (i) Single transaction > $10,000 → Currency Transaction Report (CTR) filed. (ii) Multiple transactions aggregating > $10,000 in 24 hours → CTR. (iii) Transactions involving sanctioned countries → immediately freeze. (iv) Rapid movement of funds (deposit → withdrawal in < 1 hour without activity) → flag. (b) ML-based anomaly detection: model trained on historical AML cases. (4) **Reporting**: (a) CTR (Currency Transaction Report): filed for any transaction > $10,000 (physical currency or cash equivalent). Filed within 15 days. (b) SAR (Suspicious Activity Report): filed for suspicious transactions of any size. Filed within 30 days. No tipping off — the customer must NOT be notified that a SAR was filed. (c) Travel Rule: for crypto transfers > $1,000, share sender/recipient information between VASPs (Virtual Asset Service Providers). (5) **Record keeping**: (a) Retain KYC documents for 5 years after account closure. (b) Retain transaction records for 5 years. (c) All compliance data must be immediately producible to regulators upon request within 24 hours. (6) **Audit**: (a) Annual independent audit of the AML program. (b) Quarterly compliance review: sampling of SAR filings, reviewing process adherence. (c) Regulator examinations: expect on-site examinations every 12-18 months. Provide a dedicated compliance room with all requested documents.
