# Chapter 28 — Design a Stock Exchange

## Q1: Design the core order matching engine. How does it match buy and sell orders for thousands of symbols with microsecond latency?

**Answer:** The matching engine is the heart of a stock exchange. (1) **Order book** — for each symbol (e.g., AAPL), the exchange maintains two sorted data structures: (a) **Bids** (buy orders) — sorted by price descending (highest price first), then by time ascending (earliest order first — price-time priority). (b) **Asks** (sell orders) — sorted by price ascending (lowest price first), then by time ascending. (2) **Data structure** — in C++/Java, use a concurrent skip list or a custom lock-free linked list with sorted insertion. The order book for a liquid symbol may have 10K+ active orders. (3) **Matching logic** — when a new order arrives: (a) Buy order: compare its price with the best ask. If `buy_price >= best_ask.price`, match: execute at `best_ask.price` for `min(buy_qty, best_ask.qty)`. Repeat until the order is fully filled or no more matching asks. (b) Sell order: the reverse — match against the best bid. (4) **Price-time priority** — within the same price level, orders are ordered by arrival time (FIFO). This is enforced by the sorted structure: if two orders have the same price, the one with the earlier timestamp is matched first. (5) **Latency** — the hot path (order matching) is implemented in a lock-free, single-threaded event loop per symbol to avoid context switching and lock contention. The matching logic for a single order against the book is O(K) where K is the number of price levels matched, typically < 5 for liquid stocks. Total round-trip time: 10-50 µs for the matching engine alone. (6) **Memory model** — the order book is stored in shared memory (for multi-threaded access) or on the heap with NUMA-aware allocation. No disk I/O on the hot path. (7) **Order types** — market orders (execute immediately at best available price), limit orders (execute at specified price or better), icebeg orders (only show part of the order quantity), stop orders (trigger when the price reaches a specified level).

## Q2: How does the exchange handle market data dissemination? The system must publish the top-of-book (best bid/ask) and full order book depth to thousands of subscribers in real-time.

**Answer:** Market data feeds: (1) **Top-of-book (Level 1)** — the best bid price and quantity, best ask price and quantity, last traded price, and daily volume. Changes are published every time the order book's top level changes. (2) **Full depth (Level 2)** — all visible price levels on both sides (typically up to 10-20 levels). Published as incremental updates: `{symbol, side, price_level, new_quantity}` or `{clear_level, symbol, side, price_level}`. (3) **Feed protocol** — binary protocols (ITCH from Nasdaq, OUCH, or FAST protocol from FIX) are used for lowest latency. Messages are sequenced and include a sequence number for gap detection. (4) **Fan-out architecture** — the matching engine publishes order book updates to a multicast group (UDP multicast) on the trading floor LAN. Co-located clients (HFT firms) subscribe to the multicast feed for the lowest possible latency (sub-100 µs). Messages are timestamped with the exchange's atomic clock (PTP or GPS-sourced). (5) **WAN dissemination** — for remote clients, the feed is carried over a combination of: microwave links (for lowest latency between major exchanges), fiber, and satellite (for redundancy). (6) **TCP retransmission** — since UDP multicast is lossy, a TCP retransmission service allows clients to request missed messages by sequence number. (7) **Data volume** — for a liquid stock like AAPL, the order book may change 1000+ times per second. Each change is a ~50-byte message (Level 1) or ~2 KB (Level 2 full depth). Total bandwidth for 10,000 symbols: ~500 Mbps-2 Gbps. (8) **Recovery** — a new client connecting mid-day must receive a "snapshot" of the current order book (current state) plus all subsequent updates. The exchange periodically (every 1 second) publishes a snapshot message. The client subscribes to the snapshot and then starts receiving incremental updates.

## Q3: Design the exchange's risk management system. How do you prevent a "fat finger" error (a trader accidentally submitting a market order for 1 million shares instead of 1,000)?

**Answer:** Pre-trade risk checks executed before the order reaches the matching engine: (1) **Price collars** — reject any order priced more than 20% away from the last traded price (for equities). For highly volatile securities, the percentage is adjustable. (2) **Maximum order size** — per-trader, per-security, and per-symbol limits. If a trader normally trades 100-1000 shares of AAPL, an order for 1M shares is automatically rejected. (3) **Order velocity** — if the trader sends more than 10 orders per second, they are throttled (excess orders are queued or rejected). (4) **Notional limits** — per-trader max notional exposure (order value = qty × price). A single order > $10M for a retail trader triggers rejection. (5) **Duplicate detection** — if the same order (price, side, quantity, trader) arrives within 100ms of the first, it's rejected (likely from a stuck keyboard or auto-spoofer). (6) **Net exposure** — the trader's existing positions + the new order cannot exceed the trader's credit limit. If the trader has a $5M limit and already holds $4.9M in AAPL, an order for $200K more AAPL is rejected. (7) **Circuit breakers** — if a stock's price moves more than 10% in 5 minutes, trading is halted for 5 minutes (Limit Up-Limit Down rules in the US). This is enforced by the exchange-level risk system, not per-trader. (8) **Kill switch** — the CRO (Chief Risk Officer) has a physical "big red button" that disconnects the exchange's matching engines from the network, halting all trading in an emergency. All these checks must complete in <10 µs to not add significant latency.

## Q4: How does the exchange handle market volatility and crashes? Design the circuit breaker and trading halt system.

**Answer:** Market-wide and security-specific circuit breakers: (1) **Level 1 (7% drop)** — if the S&P 500 drops 7% from the previous day's close, trading halts for 15 minutes. (2) **Level 2 (13% drop)** — if the market drops 13%, another 15-minute halt. (3) **Level 3 (20% drop)** — trading halts for the remainder of the day. These are SEC-mandated market-wide circuit breakers. (4) **Single-security circuit breakers (Limit Up-Limit Down)** — if a stock's price moves outside a specified percentage band (varies by price and time-to-close), trading in that stock is paused for 5 seconds (Limit State), then resumes at the band boundary. If the stock can't find a match at the band boundary, it's paused for 5 more seconds (Straddle State), then for 10 seconds. This repeats until the market can absorb the imbalance. (5) **Implementation** — the exchange's surveillance system monitors the price feed from the matching engine. If the price crosses the circuit breaker threshold, the surveillance system generates an administrative message to halt trading in that symbol. The matching engine processes the halt command (a privileged order type) and rejects all new orders for that symbol. (6) **Resumption** — at the end of the halt period, the exchange opens an auction (call auction) to determine the opening price. All orders accumulated during the halt are matched at the single auction price. (7) **Integrity** — circuit breakers are hardcoded in the matching engine's firmware, not in software, to prevent software bugs from disabling them.

## Q5: Design the trade settlement system. After a trade is executed, how does the ownership transfer of the stock and the payment happen in the T+1 settlement cycle?

**Answer:** Trade settlement in the US moved to T+1 (trade date + 1 business day) in May 2024: (1) **Trade capture** — at the moment of execution, the matching engine generates a trade confirmation: `{trade_id, symbol, qty, price, buyer_id, seller_id, counterparty, timestamp}`. (2) **Clearing** — the trade is sent to the Central Counterparty Clearing House (CCP — NSCC in the US). The CCP interposes itself as the buyer to every seller and the seller to every buyer (novation). The CCP nets all buy and sell obligations for each member firm to a single net payment and net share delivery. (3) **Settlement instruction** — the CCP generates settlement instructions for the Depository Trust & Clearing Corporation (DTCC). For each member firm, the instruction says: "Member A delivers 10,000 shares of AAPL to Member B and receives $1,850,000." (4) **Securities settlement** — shares move from Member A's account at DTC to Member B's DTC account. This is a book-entry transfer in the DTC ledger. (5) **Cash settlement** — the cash payment is settled via the Fedwire Securities Service (same-day funds). Member A's settlement bank transfers the cash to the CCP's settlement account, and the CCP transfers it to Member B's settlement bank. (6) **Fail management** — if Member A fails to deliver the shares by the settlement deadline, the CCP: (a) borrows shares from another member on the buy-in market. (b) Charges Member A a fail fee (based on the cost of borrowing). (c) If the fail persists > 5 days, the CCP forces a buy-in (purchases the shares in the open market at the defaulting member's expense). (7) **Blockchain/DLT** — newer exchanges (ASX, SDX) use distributed ledger technology (DLT) for settlement, where ownership transfer is atomic (cash-vs-share swap on a shared ledger), eliminating the reconciliation step.

## Q6: How does the exchange provide fairness — ensuring no participant has a latency advantage over others?

**Answer:** Fairness mechanisms: (1) **Co-location** — exchange data centers offer rack space where member firms can place their servers. All co-location racks are equidistant from the matching engine's network switch (within a few meters). All participants get the same physical connection latency. (2) **Latency floor** — the exchange adds a randomized delay (jitter) of 1-10 µs to every incoming order. This prevents HFT firms from predicting order arrival timing based on physical distance. This is controversial and used by some exchanges (IEX) but not others. IEX uses a 350 µs "speed bump" on inbound orders. (3) **Batch auctions** — some exchanges (e.g., CBOE) use periodic batch auctions (every 100 ms or 500 ms) instead of continuous order matching within a batch. This ensures that within a batch, no single participant has a time advantage. This is used by IEX's "D-Limit" discretionary peg orders. (4) **Pro-rata matching vs. FIFO** — some exchanges use a pro-rata matching algorithm (orders at the same price level are matched proportionally to their size) instead of strict price-time priority. This reduces the incentive for ultrafast order cancellation and resubmission to gain queue position. (5) **Market data fairness** — the market data feed is published via multicast on a first-come-first-served basis, but all participants within the same data center receive it within the same network round-trip. The exchange also offers a "long feed" (slower) that includes the same data for participants in different data centers. (6) **Transparency** — the exchange publishes its matching algorithm, co-location pricing, and any latency-enhancing mechanisms in its rulebook for regulatory compliance.

## Q7: Design the exchange's surveillance system for detecting market manipulation (spoofing, layering, wash trading).

**Answer:** The surveillance system analyzes orders and trades in real-time (T+0) and retrospectively (T+1): (1) **Spoofing detection** — a trader places a large visible order on one side of the book to create false demand/supply, then a small order on the opposite side that they actually intend to execute. Detection: correlate order cancellations — if a trader cancels >90% of their large orders and the remaining small orders are frequently executed on the opposite side, flag for investigation. (2) **Layering detection** — a trader places multiple orders at escalating prices on one side, creating a visible "wall," then a market order on the opposite side, then cancels the wall. Detection: detect patterns of large orders placed and then cancelled within short time windows (< 2 seconds) with a market order on the opposite side in between. (3) **Wash trading** — a trader simultaneously buys and sells the same security to create fake volume. Detection: detect trades where (a) buyer and seller share a beneficial owner, (b) matched orders have the same price and quantity, (c) the trader has not changed their net position after the trade. (4) **Quote stuffing** — rapidly submitting and canceling orders to overwhelm competitors or delay their order processing. Detection: order-to-trade ratio threshold: any trader exceeding a 100:1 order-to-trade ratio is warned; exceeding 500:1 triggers a fine/trading suspension. (5) **Cross-market manipulation** — detecting manipulation in one market (e.g., a correlated stock) to benefit a position in another market (options, futures). This requires cross-market surveillance: correlate order flow in AAPL stock with options positions. (6) **Architecture** — the surveillance system ingests all order and trade events from the matching engine (Kafka topic). Real-time alerts are generated by pattern-matching rules (e.g., Esper/Coralogix complex event processing). Historical analysis uses a Splunk/Elasticsearch store of all events. (7) **Response** — confirmed manipulation results in: trading halt for the participant, a fine, and referral to the SEC/FCA for prosecution.

## Q8: How does the exchange handle the opening and closing auctions? These are high-volume events where price discovery is critical.

**Answer:** Auctions establish the opening price (pre-market) and closing price (post-market). (1) **Opening auction** — (a) Pre-market phase (7:00-9:30 AM ET): traders can enter, modify, or cancel orders. No matching occurs. Orders accumulate in the book. (b) Indicative price calculation: the exchange computes what the price would be if the auction were held right now (the price that maximizes the number of shares traded). The indicative price and imbalance (net shares from unmatched side) are published periodically (every second). (c) Call phase (9:30 AM): in the final seconds, no cancellations are allowed (only new orders). At exactly 9:30:00, the auction runs: all marketable orders are matched at the single auction price. (d) The opening price is the first trade of the day. (e) Underlying calculations: `AuctionPrice = MAX(price) such that: BuyQty(price) >= SellQty(price) AND SellQty(price) >= BuyQty(price)` — the price that maximizes matched volume. (2) **Closing auction** — similar process from 3:50-4:00 PM ET. The closing price (from the 4:00 PM auction) is the official closing price used for NAV calculations on mutual funds. (3) **Volatility interruption** — if the indicative price is outside a predefined band (+/- 5% from the last traded price), the auction is extended by 5 minutes to allow the market to absorb the new price level. (4) **Order types** — in addition to standard limit orders, auctions support Market-on-Close (MOC) and Limit-on-Close (LOC) orders that are only executable in the closing auction. (5) **Scalability** — the auction engine processes all accumulated orders (millions for popular ETFs) in a single batch computation: sort by price, compute cumulative volume, find the clearing price. This is an O(N log N) computation for N orders. For a liquid ETF like SPY, N may be 500K+ at the close. The computation must complete within 100 ms to meet the 4:00:00 PM deadline.

## Q9: Design the exchange's disaster recovery and business continuity system. What happens if the primary data center is destroyed?

**Answer:** An exchange cannot afford downtime. Business continuity: (1) **Primary site** — main matching engine in Data Center A (e.g., Secaucus, NJ — primary for Nasdaq). (2) **Secondary site** — a fully redundant matching engine in Data Center B (e.g., Ashburn, VA, 200 miles away) with synchronous log replication. (3) **Active standby** — Data Center B runs as a "warm standby": it receives order events via synchronous replication and maintains an identical state in memory. If Data Center A fails, Data Center B takes over within 1-2 seconds. (4) **Replication protocol** — the matching engine writes every order and trade to a replicated log. The log is replicated synchronously to Data Center B before acknowledging the order to the member. This ensures no order loss. (5) **Network failover** — member firms connect via dual circuits (fiber path A and fiber path B) to both data centers. If Data Center A fails, BGP routing automatically fails over to Data Center B within 30 seconds. (6) **State recovery** — after a failover, the new primary (Data Center B) loads the order book state from the replicated log and resumes processing. (7) **Testing** — every month, the exchange runs "disaster recovery drills" where trading is intentionally failed over to the secondary data center. Members are required to participate. (8) **Past lessons** — the 1999 outage of the NYSE and the 2012 BATS IPO fiasco taught the industry that exchanges must have hot-warm pairs. Nasdaq's current architecture uses geographically dispersed matching engines with fully synchronized state.

## Q10: Design how the exchange charges fees (maker-taker model). A "maker" adds liquidity by placing a limit order, a "taker" removes liquidity by placing a market order. How does the fee calculation work at the end of each trading day?

**Answer:** The maker-taker fee model: (1) **Fee schedule** — typical US equities fee: maker rebate: $0.0025/share (the exchange pays the liquidity provider), taker fee: $0.0030/share (the exchange charges the liquidity remover). Net exchange revenue: $0.0005/share on a matched trade. (2) **Fee calculation** — at the end of each trading day, the clearing system calculates: for each member, `net_fee = sum(taker_fees) - sum(maker_rebates)`. (3) **Volume tiers** — members are charged based on their monthly trading volume: Tier 1 (0-10M shares): taker $0.0035, maker rebate $0.0020. Tier 2 (10M-100M shares): taker $0.0030, maker rebate $0.0025. Tier 3 (100M+ shares): taker $0.0025, maker rebate $0.0030. (4) **Inverted fee model** — some exchanges (e.g., EDGX) use an inverted model: maker fee ($0.0005) and taker rebate ($0.0005). This incentivizes taking liquidity (market orders) and is used for less liquid securities. (5) **Billing system** — a nightly batch job processes the trade log, computes fees per member per security, and generates invoices. The billing system integrates with the clearing house to deduct fees from the member's settlement amount. (6) **Fee cap** — some exchanges cap the total fee per trade (e.g., max $5 per trade for retail-sized orders). (7) **Reporting** — each member receives a daily trade summary with individual trade-level fee calculations: `{trade_id, symbol, qty, price, side (buy/sell), liquidity_flag (maker/taker), fee_amount}`. (8) **Disputes** — members have 3 business days to dispute fees. The exchange's fee verification process: the member's trade log must match the exchange's trade log. Discrepancies are rare (< 0.001% of trades) and are resolved by comparing both logs side-by-side.


## Q11: Design the market data feed handler optimization. How do you parse and process 2+ Gbps of binary market data (ITCH/FAST protocol) with sub-microsecond jitter, and handle packet loss recovery?

**Answer:** Market data feed handler architecture: (1) **Kernel bypass** — the feed handler runs on a dedicated NIC with kernel bypass (DPDK, Solarflare EFVI, or AWS EFA). The NIC's ring buffer delivers raw Ethernet frames directly to userspace with zero-copy. No kernel context switches on the hot path. (2) **Protocol parsing** — binary ITCH 5.0 messages are fixed-length or short variable-length records. The parser is a hand-written state machine in C++ (no virtual functions, no dynamic dispatch) that decodes messages in a tight loop: `switch(message_type) { case 'A': parse_add_order(...); break; ... }`. Each message decode takes ~20–50ns. (3) **Message sequencing** — each message carries a sequence number (monotonically increasing). The handler maintains a `next_expected_seq`. If a gap is detected (e.g., received seq 1001, expected 1000), the handler immediately requests a retransmission from the exchange's TCP retransmission service. While waiting, it buffers the out-of-order messages. This introduces latency jitter (50–200μs) for the gap-filled window. (4) **Reassembly** — the handler maintains the full order book in shared memory (see Q2). On receiving an ITCH Add Order message, it inserts into the book. On an Execute message, it reduces the quantity or removes the order. The book state is updated lock-free using RCU (Read-Copy-Update) pointers — readers can traverse the book without blocking on writes. (5) **Latency optimization** — (a) **CPU pinning**: the handler thread is pinned to a dedicated physical core, isolated from the OS scheduler (isolcpus). (b) **NUMA awareness**: the handler's memory is allocated on the same NUMA node as the NIC. (c) **No heap allocations on hot path**: all book nodes are pre-allocated from a memory pool (tcmalloc or jemalloc). (d) **Hot/cold splitting**: frequently updated fields (price, qty) are in one cache line; rarely used fields (client ID, timestamp) are in another. This reduces cache misses. (6) **Recovery on loss** — if the retransmission request is not fulfilled within 1ms, the handler drops the in-flight book state and requests a full **snapshot** of the order book from the exchange. This causes a ~1ms "blind window" during which trading decisions must rely on stale data. (7) **Monitoring** — key metrics: `messages_parsed_per_sec`, `parse_latency_p50/p99/p99.9`, `gap_count`, `retransmission_latency`. Alerts fire if gap count exceeds 10/min (indicates network degradation).

## Q12: How does the exchange ensure correct order book reconstruction from decimal-encoded price fields in binary market data feeds? What edge cases arise with decimal precision and rounding in matching?

**Answer:** Decimal encoding in exchange feeds: (1) **Representation** — prices are typically encoded as a 4-byte integer with an implicit decimal shift (e.g., Nasdaq ITCH: `price = integer_value / 10000` for most equities, giving 4 decimal places). Some exchanges use a separate `scale` field: `price = integer_value × 10^(-scale)`. (2) **Reconstruction** — the feed handler converts the encoded integer to a fixed-point representation (e.g., `int64_t price_cents = integer_value * (100 / 10000)`). Avoid floating-point (`double`) for prices — floating-point rounding errors cause order book mismatches when reconstructing the book from the feed. (3) **Edge cases** — (a) **Zero-price orders**: some dark pools or auction-only orders have a price of 0 (market order equivalent). These must be handled as matching at the best opposite price, not literal zero. (b) **Decimal overflow**: decimal 5.12345 → integer 512345 at scale 100000. If the receiving system expects only 2 decimal places (cents), truncation to 512 (=$5.12) loses $0.00345 per share — on 1M shares that's $3,450 of unexplained difference. The feed handler must use the **exact scale** from the protocol spec. (c) **Negative prices**: rare but possible in certain derivatives (e.g., oil futures briefly went negative in 2020). The encoding must support signed integers. (d) **Price ticks**: each symbol has a minimum price variation (e.g., $0.0001 for most US equities, $0.01 for options). Orders with price increments smaller than the tick are rejected by the exchange — the feed handler should validate tick compliance when adding orders locally. (4) **Cross-exchange reconciliation** — when the same symbol trades on multiple exchanges (Reg NMS requires best execution across all), price representations may differ (Nasdaq uses 4 decimal places, IEX uses 4, NYSE uses 2 for some securities). The smart-order router must normalize all prices to a common representation (e.g., microcents) before comparing. A mismatch of 1 microcent can cause a trade-through (trading at a worse price than available on another exchange). (5) **Testing strategy** — replay historical trade data and compare the reconstructed order book's match events against the exchange's official trade log. Any discrepancy > 0 in price or quantity indicates a precision bug.

## Q13: How do you design colocation latency arbitrage prevention? Some HFT firms may try to gain an unfair advantage by positioning their servers closer to the matching engine or by using microwave links that bypass the exchange's latency floor.

**Answer:** Latency arbitrage prevention is a regulatory and architecture challenge: (1) **Physical colocation equality** — the exchange's data center has a standard colocation cabinet layout. All member cabinets are within a **defined radius** (e.g., 10 meters) of the matching engine's network switch. The exchange publishes a **latency map** showing the base round-trip latency from each cabinet position to the matching engine — these should be identical within measurement noise (±0.1μs). (2) **Cable length equalization** — the network cables from each cabinet to the matching engine switch are **equal-length** (same number of meters of fiber). Some exchanges bury extra cable in a trench to match lengths. The variance is < 1 meter ≈ 5ns of propagation delay. (3) **Speed bump / latency floor** — to neutralize the advantage of third-party microwave links (which bypass the exchange's fiber), the exchange enforces a **minimum inbound latency**. IEX uses a 350μs "speed bump" — a coil of fiber that adds 350μs of propagation delay to every order. The speed bump is placed at the entrance to the matching engine, after the colocation racks, so even microwave-fed orders must wait the same 350μs. (4) **Disclosure and audit** — the exchange's latency floor mechanism is disclosed in its rulebook (SEC Rule 611 / Reg NMS). An independent auditor measures the actual latency from each colocation position every quarter and publishes the results. If any firm is found to have a measurable advantage, the exchange must remediate. (5) **Detection of arbitrage strategies** — the surveillance system monitors for **latency arbitrage patterns**: (a) A firm receives a trade on Exchange A (faster), then immediately sends an order on Exchange B (slower) that exploits the stale price on B before Exchange B's price update arrives. This is a form of "cross-exchange latency arbitrage." Detection: correlate trade timestamps on Exchange A with order arrivals on Exchange B — if the time delta is less than the known inter-exchange latency, flag for review. (6) **Alternative model: batch auctions** — some exchanges (e.g., IEX's D-Limit, CBOE) replace continuous matching with periodic batch auctions (every 100–500μs). Within a batch, all orders are matched at the same clearing price, eliminating the value of microsecond speed advantages. This is the most effective structural prevention but comes at the cost of increased latency for all participants.

## Q14: Design the FIX protocol session management system for an exchange that handles 50,000+ concurrent FIX sessions from member firms. How do you handle session authentication, sequence number recovery, and heartbeats at scale?

**Answer:** FIX session management at exchange scale: (1) **Connection architecture** — a **FIX gateway** cluster (Nginx + custom FIX module, or a purpose-built FIX engine like OnixS, Citi FIX) terminates all member sessions. Each session is a persistent TCP connection (one per member per trading session). 50,000 concurrent sessions require ~50,000 open file descriptors — ensure the OS `fs.file-max` and `ulimit -n` are configured appropriately (Linux default is often 1024 → must be raised to 100K+). (2) **Authentication** — each member is assigned a `SenderCompID` / `TargetCompID` pair. On login (`35=A`), the gateway validates: (a) The SenderCompID is a registered member. (b) The password (96=Password) matches the stored hash (bcrypt or argon2). (c) The IP address is in the member's allowed range (whitelist per member). (d) The SSL/TLS certificate (if using FIXS over TLS) chains to the exchange's CA. (3) **Sequence number recovery** — FIX sessions track `MsgSeqNum` (34=) inbound and outbound. On login, the member sends `34=NextExpectedSeqNum`. The gateway checks: (a) If `34 = stored_outbound_seq + 1` → normal resume. (b) If `34 > stored_outbound_seq + 1` → gap detected. The gateway sends a **ResendRequest** (35=2) for the missing range. The member must retransmit those messages. (c) If `34 < stored_outbound_seq + 1` → possible duplicate session. The gateway rejects with `35=5 (Reject)` and `58=MsgSeqNum too low`. The member must reset the sequence (35=4 Sequence Reset) before proceeding. (4) **Heartbeat management** — every FIX session exchanges heartbeats (35=0) at a configurable interval (default 30 seconds). If no message is received within `HeartBtInt × 2`, the gateway sends a **TestRequest** (35=1). If no response within another `HeartBtInt`, the session is disconnected. The gateway maintains a **heartbeat timer wheel** (hierarchical timing wheel) to track 50,000 sessions' heartbeat expectations without O(n) scan. (5) **Session state machine** — FIX session states: NotConnected → Connected (TCP) → LogonReceived → LoggedOn (session established) → ResendRequested (gap fill in progress) → LoggedOn. Each state transition is validated. Invalid transitions (e.g., a new `35=D` order before `35=A` logon) trigger a disconnect. (6) **Throughput** — a single FIX gateway node can handle ~5,000–10,000 sessions (CPU-bound on SSL termination and log compaction). Scale out behind a load balancer that routes by `SenderCompID` hash to ensure session affinity (a member's session must stay on one gateway node for sequence number continuity). Use **consistent hashing** on `SenderCompID` to minimize connection migration during scale events. (7) **Resilience** — if a gateway node fails, active sessions on it are disconnected. Members should have a **backup connection** to a secondary gateway IP. The backup connection picks up from the last sequenced message (using FIX's sequence number recovery). The exchange's gateway cluster also runs in active-active across two data centers — members connect to both and use one as primary.

## Q15: Design the best execution obligation compliance system. How does the exchange (or a broker using the exchange) ensure that every trade meets the broker's best execution obligation under SEC Rule 606 and MiFID II RTS 28?

**Answer:** Best execution compliance requires capturing, storing, and reporting every decision factor: (1) **Order routing decision** — when a broker receives a customer order, it must evaluate multiple execution venues (exchanges, dark pools, ATSs, market makers) for best execution. Factors: (a) **Price**: the NBBO (National Best Bid and Offer) price at the time of decision. (b) **Cost**: explicit fees (maker-taker, routing fees) and implicit costs (spread, market impact). (c) **Speed**: expected fill latency on each venue. (d) **Likelihood of execution**: probability that the order will be filled based on historical fill rates. (e) **Size**: ability to execute the full order size at the best price. (2) **Smart Order Router (SOR)** — the broker's SOR receives the order and runs a **venue selection algorithm** that scores each venue: `score(v) = w1 * price(v) + w2 * cost(v) + w3 * speed(v) + w4 * fill_probability(v)`. The SOR slices the order across the top-N venues, routing the largest slice to the highest-scoring venue. (3) **Record keeping (SEC Rule 606)** — the broker must capture for each order: (a) The identity of the venue where the order was routed. (b) The NBBO at the time of routing. (c) The execution price and quality (fill rate, speed, any price improvement). (d) Any material relationship between the broker and the venue (e.g., payment for order flow). (4) **Reporting (MiFID II RTS 28)** — the broker publishes an annual "best execution report" showing: (a) Top 5 execution venues by asset class. (b) The quality of execution achieved on each venue (price improvement, fill rate, latency). (c) Any conflicts of interest. (5) **Execution quality analysis (post-trade)** — a batch job compares the actual execution price against the **NBBO at the time of execution**. If the execution was at a worse price than NBBO (a "trade-through"), it's an auto-flagged compliance exception. The broker must justify why a trade-through occurred (e.g., the NBBO quote was locked or crossed, or the order was a large block with market impact). (6) **System architecture** — the SOR is a low-latency C++ service co-located in the exchange's data center. It maintains a real-time feed of venue status (latency spread, fill probability) updated every 100ms. The compliance recording service is a Kafka-based pipeline that captures every order routing decision as a protobuf message and lands it in an immutable store (Apache Parquet on S3). (7) **Audit** — the compliance team runs queries against the SOR logs to answer: "For this order, was the customer's execution at least as good as what was available on the primary exchange?" If the answer is "no" for >0.01% of orders, the SOR algorithm is re-tuned.

## Q16: Design the short sale locate and delivery system. How does the exchange ensure that short sellers have located borrowable shares before executing a short sale (SEC Rule 203) and that delivery of borrowed shares settles correctly?

**Answer:** Short sale compliance under SEC Rule 203: (1) **Locate requirement** — before executing a short sale, the broker must have a "reasonable basis" that borrowable shares exist. This is the **locate**: the broker queries a securities lending database (or an internal pool) and receives a locate confirmation: `{symbol, qty_located, lending_firm, expiry}`. The locate is valid only for the day of execution. (2) **Easy-to-borrow (ETB) list** — for liquid stocks (e.g., AAPL, MSFT), the broker maintains an automatic ETB list — shares are pre-located and the locate check is bypassed. ETB lists are updated daily based on lending supply. (3) **Hard-to-borrow (HTB) list** — for HTB securities, the broker must perform a full locate before each short sale. The locate may come from: (a) The broker's own inventory (prime brokerage desk). (b) A third-party securities lender (e.g., EquiLend, Aequitas). (c) Another broker-dealer (rare). (4) **Execution** — once the locate is confirmed, the short sale order is flagged with a `ShortSaleIndicator = S` (as defined by FINRA) and sent to the exchange. The exchange checks the flag but does not validate the locate (it is the broker's responsibility). (5) **Delivery (settlement)** — on T+1 settlement, the short seller must deliver the borrowed shares to the buyer. The borrowed shares are typically obtained via: (a) **Prime broker lending pool**: the prime broker has a pool of client margin shares that can be lent. (b) **Stock loan market**: the broker borrows from another broker's inventory via the securities lending market (EquiLend). (6) **Fail to deliver** — if the short seller fails to deliver shares by settlement deadline (T+1), a **fail** is registered at the clearing house (NSCC/DTCC). Consequences: (a) The failing broker pays a **fail charge** (based on the cost of borrowing the shares in the open market, currently SOFR + 100–300bps). (b) After 5 consecutive days of fail, the buying broker can force a **buy-in**: the NSCC purchases the shares in the open market at the defaulting broker's expense. (c) The short seller's broker may close the short position by force (buy to cover) if the fail persists. (7) **Threshold securities** — if a security has 10,000+ shares of fails for 5+ consecutive settlement days, it is a **threshold security** (per SEC Regulation SHO). For threshold securities, any new short sale requires a **pre-borrow** (the shares must be in hand before the short sale, not just located). (8) **Architecture** — the locate system is a real-time service that maintains a cache of the lending pool (updated via FIX from the securities lending market). Locate requests are checked against the cache and confirmed within 1ms. The fail tracking system is a nightly batch that reconciles the clearing house's fail report against the broker's short positions.

## Q17: Design the market maker obligation algorithms. How does a designated market maker (DMM) or registered market maker maintain continuous two-sided quotes within the exchange's required spread and size parameters?

**Answer:** Market maker quoting algorithms: (1) **Obligations** — a registered market maker on an exchange must provide continuous two-sided quotes (bid and ask) for assigned securities during regular trading hours. Requirements (Nasdaq): (a) **Minimum quote size**: at least 100 shares per side. (b) **Maximum spread**: for stocks ≥ $1.00, the spread cannot exceed 25% of the NBBO spread (for registered market makers) or 8% for certain ETFs. (c) **Quote presence**: at least 90% of the trading day (with allowed exceptions for risk management). (2) **Quote pricing algorithm** — the market maker computes: `bid = fair_value - spread/2 - inventory_skew`, `ask = fair_value + spread/2 + inventory_skew`. `fair_value` = NBBO midpoint or a proprietary pricing model (e.g., based on correlated securities, options implied volatility). `spread` = f(volatility, volume, time_to_close). `inventory_skew` = g(current_inventory, target_inventory). (3) **Spread calculation** — `spread = base_spread × (1 + k1 × volatility_ratio + k2 × (1 - volume_ratio) + k3 × time_factor)`. During high volatility (VIX > 30), the spread multiples by 2–3×. During low volume (afternoon lull), spread widens. In the final 30 minutes of trading, spread widens due to increased risk of overnight gap. (4) **Inventory management** — the market maker tracks its current position in each security. If long 10,000 shares of AAPL (net positive inventory), the algorithm **skews quotes lower**: lower bid (to avoid buying more) and lower ask (to encourage selling). The skew is proportional to inventory: `inventory_skew = inventory × dollar_risk × skew_factor`. Target inventory is 0 (delta-neutral). (5) **Quote refresh** — the market maker updates its quotes every 10–100ms (or faster for HFT market makers). Each update is a cancel-replace cycle: cancel existing quotes, send new quotes. This generates significant order traffic — a market maker for 800 symbols may send 500+ messages per second. (6) **Risk controls** — (a) **Max position limit**: if long > 50,000 shares of AAPL, stop quoting the bid (go to one-sided). (b) **Max loss limit**: if P&L on the day drops below –$50,000, stop all quoting (circuit breaker for the market maker). (c) **Fat-finger protection**: if the ask is accidentally set to $0.01 (vs. $150 fair value), the system detects a price anomaly > 50% from fair value and rejects the quote. (7) **Regulatory reporting** — the market maker submits daily quote and trade data to FINRA/Nasdaq for **market maker performance monitoring**. The exchange evaluates: did the firm meet the 90% quoting obligation? Was the spread within the allowed maximum? Underperformance results in loss of market maker status (and associated rebates).

## Q18: Design the regulatory reporting system (CAT and OATS). How does the exchange aggregate and report order and trade data for the consolidated audit trail (CAT) and order audit trail system (OATS)?

**Answer:** Regulatory reporting for US equities: (1) **CAT (Consolidated Audit Trail)** — the SEC-mandated CAT system captures every order, cancellation, modification, and trade for all NMS securities from the moment of receipt to execution or cancellation. The reporting is per **order event**: (a) New order: `{order_id, customer_id, symbol, side, quantity, price, order_type, receipt_timestamp, receiving_firm, sending_firm}`. (b) Route: `{order_id, route_timestamp, destination_venue, route_reason}`. (c) Cancel: `{order_id, cancel_timestamp, remaining_quantity}`. (d) Execution: `{order_id, execution_timestamp, execution_price, executed_quantity, contra_firm}`. (2) **CAT reporting deadline** — by 8:00 AM ET the next business day (T+1), all CAT-reportable events must be submitted to the CAT processor (FINRA). Late reporting results in fines ($5,000–$50,000 per late day for systemic failures). (3) **OATS (Order Audit Trail System)** — the FINRA OATS system captures similar data for over-the-counter (OTC) equity trades. OATS is being phased out (replaced by CAT) but is still in effect for certain securities and member firms. (4) **Data ingestion architecture** — the exchange's CAT reporting pipeline: (a) A **CAT event collector** service subscribes to the matching engine's event stream (Kafka). For each order event, it enriches the event with the required CAT fields: `customer_id` (from the member's customer-ID mapping file — the member sends a separate linkage file to CAT mapping account numbers to customer IDs). (b) The enriched events are written to a **CAT event log** (Apache Avro on HDFS/S3). (c) A **CAT batch submitter** runs at 7:00 AM each day: it reads the previous day's events, generates the CAT XML/JSON payload (per the CAT technical specifications), validates it (schema check, mandatory field check), and submits it to the CAT processor via SFTP or HTTPS. (5) **Customer ID linkage** — each broker firm must submit a **customer-ID linkage file** to the CAT processor that maps each customer account number to a unique **CAT Customer-ID** (a hash-based identifier). The exchange does NOT submit customer IDs directly — it receives them in the linkage file from the member firm and includes them in the order event. (6) **Timeline compliance** — the CAT reporting system tracks submission deadlines per event date. A dashboard shows: "Oct 15, 2025: 99.8% submitted, 0.2% pending (3,456 late events — alert fired at 7:55 AM)." Late events are auto-escalated to the compliance team. (7) **System design considerations** — CAT generates ~100B events per year across all exchanges. Each event is ~500 bytes → ~50 TB/year. Storage is tiered: hot (last 30 days on SSD, for regulatory queries), warm (31–365 days on HDD, for compliance review), cold (1–5 years on Glacier, for retention). The system must support ad-hoc queries from the SEC (who subpoenas specific order data during investigations). A search over 5 years of data must return results within 24 hours.

## Q19: Design the exchange's disaster recovery testing methodology. How do you validate that the secondary data center can take over without loss of state, and how do you test without disrupting live trading?

**Answer:** DR testing for a stock exchange is high-risk because a failed test can delay market open or cause data loss. Methodology: (1) **Testing tiers** — (a) Tier 1 (Component): test individual components (matching engine, market data feed, FIX gateway) failover in isolation. Run weekly. (b) Tier 2 (Integrated): test the full trading pipeline — FIX gateway → matching engine → market data dissemination → clearing. Run monthly on a weekend (no live market). (c) Tier 3 (Live failover): simulate a real failure by switching live traffic from DC-A to DC-B during a low-volume period (e.g., 2:00–3:00 AM ET, when only overnight futures trade). Run quarterly. (2) **State verification** — the critical question: after failover, does the secondary DC have exactly the same order book state as the primary at the moment of failure? Verification: (a) **Replicated log checksum**: at the end of each trading day, compute a checksum of the primary's transaction log and compare it with the secondary's replicated log. Any mismatch → DR bug. (b) **Deterministic replay**: the secondary records every input (order, cancel, modify) with its timestamp. After failover, the secondary replays these inputs against a "reference matching engine" (a software simulator of the primary's matching engine). The output (trade log) must match the primary's trade log exactly. (c) **Mirror trade comparison**: during Tier 2 tests, the secondary's matching engine processes the same input stream as the primary, but its output is discarded (not sent to the market). After the test, compare the secondary's internal trade log with the primary's — any difference means the secondary has a bug. (3) **Network cutover testing** — (a) **BGP convergence**: measure how long it takes for the BGP route withdrawal/advertisement to propagate to all member firms' routers. Target: < 30 seconds. Test by announcing a test IP prefix from DC-B and measuring ping reachability from member firms' test points. (b) **Multicast failover**: for market data feeds (UDP multicast), the secondary must start publishing the feed on the same multicast group with zero gap. Test by having a monitoring client subscribe to the feed during failover and measuring: `number_of_messages_lost / total_messages`. (4) **Rollback procedure** — every DR test has a **rollback script**: "If the test fails, switch traffic back to the primary within 60 seconds." The rollback must be tested at least once per year. The rollback script must not assume the secondary has clean state (it may have corrupt state from a failed test). (5) **GMP (Good Market Practice)** — the exchange follows the **FIA DR test guidelines**: all DR tests are scheduled 2 weeks in advance, announced to members via circular, and run outside regular trading hours. Members are required to participate in the live failover test once per quarter (validate their connections to DC-B). (6) **Post-test report** — after each test, the exchange publishes: test date, duration, any data loss (should be zero), failover time (should be < 2 seconds), rollback time (if applicable), and corrective actions for any issues.

## Q20: Design the dark pool matching logic. How does a dark pool (alternative trading system) match orders without displaying quotes to the public market, and how does it interact with the displayed exchange to avoid trade-throughs?

**Answer:** Dark pool matching operates differently from lit exchanges: (1) **Order types** — dark pools accept **non-displayed orders** (also called "reserve orders" or "hidden orders"). The order is not visible in the public order book. Common dark pool order types: (a) **Midpoint peg**: price is pegged to the NBBO midpoint (e.g., if NBBO bid=$100.00, ask=$100.05, midpoint=$100.025). This ensures the dark pool always offers price improvement (no worse than the NBBO). (b) **Primary peg**: price = best bid for a buy, best ask for a sell (no price improvement). (c) **Market peg**: price = current market price (can drift). (d) **Discretionary peg**: the user specifies a limit, but allows the price to improve toward the midpoint at the pool's discretion. (2) **Matching logic** — when a new order arrives: (a) Check if it crosses any existing non-displayed order on the opposite side at the pegged price. (b) If yes, execute at the best available price (midpoint or better). (c) If no, the order is added to the dark pool's order book (non-displayed). (3) **Trade-through avoidance (Reg NMS Rule 611)** — a dark pool cannot execute a trade at a price worse than the displayed NBBO. If a buy order arrives at $100.00 and the NBBO ask is $100.00, the trade is valid. If the NBBO ask is $100.05, the trade cannot execute at $100.00 (it would be a trade-through). The dark pool must either: (a) Route the order to the exchange showing the best displayed price (where it will execute at $100.05). (b) Wait until the NBBO ask drops to $100.00. (4) **Block trading** — dark pools specialize in large block trades (10,000+ shares). A block order is matched only when the contra-side is at least the block size. The matching algorithm searches for a block-sized contra order before considering smaller orders. This prevents "information leakage" from a large order being filled by many small orders. (5) **Minimum execution quantity (MEQ)** — the user can specify: "Don't execute my order unless at least 5,000 shares can be matched." The dark pool respects this — partial fills below MEQ are rejected. (6) **Interaction with lit exchanges** — many dark pools are operated by exchange operators (e.g., Nasdaq BX, NYSE American) and share infrastructure. A "sweep" order type will: check the dark pool first (for price improvement), then route remaining to the lit exchange. This is the **dark-first** routing strategy required by some institutional clients. (7) **Anti-gaming** — dark pools are vulnerable to "pinging": an HFT sends small orders to detect if a large block is sitting in the dark pool. Mitigations: (a) **Randomization**: the dark pool adds jitter (1–10ms) to match confirmations, making it harder for pings to identify block orders. (b) **Minimum size**: ignore orders below a threshold (e.g., 100 shares) for securities with a large average trade size. (c) **Toxic flow flagging**: if a counterparty's fill ratio is consistently below 10% (they cancel most orders), they are flagged as potentially toxic and their orders are deprioritized. (8) **Regulatory reporting** — dark pools report trades to FINRA's **Alternative Display Facility (ADF)** within 10 seconds of execution. They also file quarterly Form ATS-N disclosure with the SEC, detailing their matching logic, fee model, and any conflicts of interest.

## Q21: Design the order types supported by the exchange. How do you implement market orders, limit orders, stop-loss orders, and iceberg orders in the matching engine?

**Answer:** Each order type has specific matching semantics. (1) **Market order**: (a) Buys at the best available ask price (for buy) or best bid (for sell). (b) Consumes all liquidity at the top of the book, then moves to the next price level until fully filled. (c) Risk: the fill price can be worse than expected if the market moves between submission and execution. (2) **Limit order**: (a) Buy/sell at a specified price or better. (b) If the limit price crosses the book, it executes immediately. If not, it rests on the book as a maker order. (c) GTC (Good-Til-Cancelled): stays on the book until filled or cancelled. Day order: expires at market close. (3) **Stop-loss order**: (a) Converts to a market order when the last trade price crosses the stop price. (b) Implementation: the matching engine checks stop orders on every trade event. If `last_trade_price >= stop_price` for a sell stop, activate the order. (c) Stop-limit: converts to a limit order (not market) when triggered. (4) **Iceberg order**: (a) A large limit order with a visible (peak) quantity and an undisclosed quantity. (b) When the visible quantity is fully filled, a new visible quantity is revealed from the hidden reserve. (c) Implementation: the matching engine stores the total quantity and only shows `peak_size` in the order book. On partial fill, it decrements the visible quantity. When visible quantity hits 0, refresh from reserve. (d) This prevents market impact — other traders don't see the full size of the order. (5) **Order validation**: (a) All orders are validated: price > 0, quantity > 0, not exceeding position limits. (b) Market orders require sufficient liquidity on the opposite side (or reject with INSUFFICIENT_LIQUIDITY). (c) Stop orders are validated but not pre-funded — the conversion to market order is what triggers the actual trade.

## Q22: Design the order book data structure. How do you store buy and sell orders sorted by price-time priority, support 10M+ order updates per second, and provide real-time snapshots to market data feeds?

**Answer:** The order book is the core data structure of any exchange — every microsecond matters. (1) **Price-time priority**: (a) Orders are sorted: highest bid first (buyers willing to pay more), lowest ask first (sellers willing to accept less). (b) Within the same price level, orders are sorted by arrival time (first-in-first-out). (2) **Data structure — Price levels**: (a) Use a sorted map (e.g., C++ `std::map` or Java `TreeMap` where key = price, value = a FIFO queue of orders). (b) For each price level, maintain a doubly-linked list of orders in arrival order. (c) This gives O(log P) for adding/removing a price level (P = unique price levels) and O(1) for adding/removing an order at a price level. (3) **Performance optimizations**: (a) Pre-allocate order objects in a memory pool to avoid GC pauses. (b) Use lock-free data structures (Disruptor pattern) for the event bus. (c) Batch order processing: process up to 64 orders per batch to amortize synchronization costs. (4) **Snapshot for market data**: (a) Every time the top of the book changes, publish a market data increment: price, size, side. (b) Periodically (every 1 second), publish a full order book snapshot for new subscribers. (c) Snapshot format: array of (price, size, order_count) per side, top 10 levels. (5) **Recovery**: (a) On restart, the matching engine replays the transaction log (all order events since last snapshot) to rebuild the order book. (b) Snapshots are taken every 100K events to bound recovery time. (6) **Memory management**: (a) A single order is ~64 bytes (price, quantity, timestamp, order_id, next_ptr, prev_ptr). For 10M orders, ~640MB — acceptable for an in-memory data structure. (b) Use off-heap memory (Java DirectByteBuffer or C++ mmap) to avoid GC overhead.

## Q23: Design the matching engine logic. Given incoming buy and sell orders, how does the matching engine determine trade price and quantity, handle partial fills, and generate trade confirmations?

**Answer:** The matching engine is the heart of the exchange — it must produce deterministic, fair trade outcomes. (1) **Matching logic for a buy order**: (a) Check the lowest ask (best offer). If buy price >= ask price: (i) Trade price = ask price (aggressive order pays the resting order's price — "price priority"). (ii) Trade quantity = min(buy_qty, ask_qty). (iii) Reduce ask qty by trade qty. If ask qty == 0, remove the ask order. (iv) Reduce buy qty by trade qty. If buy qty == 0, done. Otherwise, move to the next ask level. (2) **Matching logic for a sell order**: (a) Check the highest bid (best bid). If sell price <= bid price, trade at bid price. (3) **Price-time priority in action**: (a) If two buy orders at price $100 arrive at 10:00:01.000 and 10:00:02.000, and a sell order at $100 arrives at 10:00:03.000, the sell matches with the 10:00:01 buyer first. (b) If a buy order at $101 arrives at 10:00:04.000, it has price priority over the $100 buyers — it matches with the lowest ask immediately. (4) **Partial fills**: (a) If a buy order for 1000 shares only matches 300 shares, the remaining 700 shares rest on the book (for limit orders). (b) For market orders with partial fill: the unfilled portion is cancelled (not rested). (5) **Trade confirmation**: (a) On each match, the engine generates: `Trade(trade_id, buy_order_id, sell_order_id, price, quantity, timestamp)`. (b) Trade ID is monotonically increasing and globally unique across the exchange. (c) Both parties receive a trade confirmation: you bought/sold X shares at $Y. (6) **Determinism**: (a) For a given set of input orders in a given sequence, the matching engine must produce exactly the same trades every time. (b) This is critical for auditing and replay. (c) No randomness, no time-of-day dependencies — only the order of arrival and order book state.

## Q24: Design the market data feed system. How do you publish real-time order book snapshots, trade ticks, and quote updates to thousands of subscribers with minimal latency?

**Answer:** Market data feeds must be low-latency and high-throughput. (1) **Feed tiers**: (a) **Level 1**: top-of-book (best bid/ask, last trade price, volume). Low bandwidth, suitable for retail apps. (b) **Level 2**: full order book (top 10+ price levels on bid/ask). Higher bandwidth, for professional traders. (c) **Level 3**: full order book with order IDs (for co-located algorithmic traders). (2) **Publishing architecture**: (a) Matching engine publishes events (OrderAdded, OrderExecuted, OrderCancelled) to a UDP multicast group. (b) Each event is ~50 bytes. At 1M events/second, bandwidth ~50 MB/s — manageable with 10GbE. (c) Subscribers join the multicast group and receive events directly — no server-side fan-out overhead. (3) **Snapshot vs incremental**: (a) Incremental updates: each event carries the delta (change to the book). Subscribers maintain a local copy of the order book and apply deltas. (b) Full snapshots: every 1 second, publish a full L2 snapshot (top 10 levels). New subscribers use this to initialize their local book. (4) **Sequence numbers and gap detection**: (a) Each event has a monotonically increasing sequence number per symbol. (b) Subscribers detect gaps: if sequence jumps from 1000 to 1005, the subscriber missed 4 events. (c) Gap recovery: request missed events from a recovery server (HTTP or TCP), then resume processing. (5) **Latency**: (a) Matching engine → network = <1 microsecond. (b) Network propagation = ~1 microsecond per km (fiber optic). (c) Co-location: traders place servers in the same data center as the exchange to minimize latency. (6) **Protocols**: (a) Binary protocol (FAST or SBE) for low-latency. (b) Normalized JSON over WebSocket for retail subscribers (slower but more accessible). (7) **Regulatory requirements**: (a) Market data must be archived for 7+ years. (b) Each data feed must be timestamped with the exchange's atomic clock (NTP or PTP). (c) Regulators (SEC, FCA) can request a replay of market data for any past period.

## Q25: Design the pre-trade risk checks. Before an order enters the matching engine, what validations must pass to prevent erroneous or malicious trades?

**Answer:** Pre-trade risk checks prevent fat-finger errors, credit breaches, and market manipulation. (1) **Price validity**: (a) Order price must be within a percentage band of the last trade price (e.g., +-20% for stocks, +-5% for ETFs). (b) If no last trade price, use the previous close. (c) Market orders cannot exceed a maximum notional value ($10M per order). (2) **Position limits**: (a) Check the trader's current position + order quantity <= maximum position limit. (b) For leveraged products, check margin adequacy: `(position_value + order_notional) x margin_rate <= available_collateral`. (c) For short sales: verify the trader has a locate (borrowed shares available). (3) **Order rate limiting**: (a) Max 100 orders per second per trader (fair usage). (b) Max 10,000 orders per day. (c) If exceeded, reject with RATE_LIMITED. (4) **Duplicate detection**: (a) Each order has a `client_order_id` (provided by the trader). (b) If two orders with the same `client_order_id` arrive within 5 seconds, reject the second (potential duplicate from retry). (5) **Circuit breakers**: (a) If a single trader's orders exceed 10% of the traded volume in a symbol in the last 5 minutes, reject additional orders. (b) If the trader's cancellation rate exceeds 90% (large number of orders cancelled before execution), flag as suspicious (quote stuffing). (6) **Settlement checks**: (a) Buyer must have sufficient cash (if trading on a cash basis) or sufficient margin. (b) Seller must have sufficient inventory. (c) Real-time check: debit the buying power / inventory at order entry, credit on cancellation. (7) **Latency budget**: all pre-trade checks must complete in <10 microseconds. Implemented in C++ or Rust, running on the same server as the matching engine. (8) **Override**: (a) Exchange operations can override risk checks for market makers (who need tighter spreads). (b) All overrides are logged and reported to compliance.

## Q26: Design the circuit breaker and market halt system. When should trading be paused, how do you coordinate across multiple exchanges for the same stock, and how do you resume trading?

**Answer:** Circuit breakers prevent panic selling and flash crashes. (1) **Price band circuit breaker**: (a) If a stock's price moves more than X% in 5 minutes (e.g., 10% for S&P 500 stocks), pause trading for 5 minutes. (b) Implemented in the matching engine: on each trade, compare the trade price to the reference price (last close or 5-minute VWAP). If the deviation exceeds the threshold, trigger a halt. (2) **Market-wide circuit breakers**: (a) Level 1: S&P 500 drops 7% -> market halts for 15 minutes. (b) Level 2: drops 13% -> halt for 15 minutes. (c) Level 3: drops 20% -> trading halted for the day. (3) **Limit up/Limit down (LULD)**: (a) Each stock has a price band based on its average price over the last 5 minutes. (b) If the price tries to trade outside the band, the stock enters a 15-second limit state -- orders can only be entered within the band. (c) If the limit state doesn't resolve in 15 seconds, trading pauses for 5 minutes. (4) **Cross-exchange coordination**: (a) If one exchange halts a stock, other exchanges should also halt (if they list the same stock). (b) The SIP (Securities Information Processor) disseminates halt status. (c) The exchange subscribes to the SIP's halt/resume feed and acts accordingly. (5) **Resumption**: (a) After the halt period, the exchange conducts an auction (call auction) to establish a fair opening price. (b) The auction collects interest (bids and offers) for 5 minutes, then matches at a single clearing price. (c) Continuous trading resumes after the auction completes. (6) **Volatility interruption**: (a) For products with daily price limits (e.g., commodity futures), if the price hits the limit, trading continues but only within the limit price. (b) If the limit is hit and no new orders arrive within 30 seconds, trading pauses for 2 minutes to allow the market to digest.

## Q27: Design the market making program for the exchange. How do you incentivize market makers to provide tight spreads and continuous two-sided quotes?

**Answer:** Market makers (MMs) are essential for liquidity -- the exchange incentivizes them via rebates and obligations. (1) **MM obligations**: (a) An MM for a symbol must continuously provide two-sided quotes (bid and ask) within a maximum spread (e.g., $0.05 for a $100 stock). (b) Minimum quote size: 100 shares on each side. (c) Uptime: MM must be quoting at least 95% of the trading day. (2) **Rebate structure**: (a) Maker rebate: if the MM's limit order is passive (rests on the book and is executed by an incoming aggressive order), the MM receives a rebate (e.g., $0.002 per share). (b) Taker fee: aggressive orders pay a fee (e.g., $0.003 per share). (c) Net: the MM earns the rebate, the taker pays the fee. (3) **Performance monitoring**: (a) Track per-MM: time-weighted spread, quoted depth, uptime percentage, fill ratio. (b) If an MM fails to meet obligations, reduce their rebate tier or revoke MM status. (4) **Risk controls for MMs**: (a) MMs get higher rate limits (10,000 orders/second) but face stricter position limits. (b) MM orders cannot be cancelled if they would leave an unhedged position above the limit. (c) MM orders are subject to auto-cancellation if the MM's connection drops (to prevent stale quotes). (5) **Market making in volatile markets**: (a) MMs can widen spreads during high volatility (allowed -- the market needs realistic quotes, not artificial tightness). (b) During circuit breakers, MM obligations are paused resume when trading resumes. (6) **Auction participation**: MMs are expected to submit balancing orders during opening/closing auctions to reduce the auction imbalance. If the auction would clear with a large price gap, MM participation narrows it.

## Q28: Design the settlement (clearing) system. How do you handle trade confirmation, netting, and delivery-versus-payment (DvP) to settle trades by T+1?

**Answer:** Trade settlement transfers ownership and funds between buyer and seller. (1) **Trade confirmation**: (a) After a trade is executed, both parties receive a trade confirmation with trade ID, symbol, quantity, price, gross amount, fees, net amount. (b) Confirmations are compared through the CCP (Central Counterparty) -- if they match, the trade proceeds to clearing. (2) **Clearing -- novation**: (a) The CCP interposes itself between buyer and seller. (b) The buyer's counterparty becomes the CCP; the seller's counterparty becomes the CCP. (c) This eliminates counterparty risk: if the buyer defaults, the CCP still pays the seller. (3) **Netting**: (a) Instead of settling each trade individually, net all positions between the CCP and each clearing member. (b) Example: Broker A bought 1000 shares of AAPL and sold 400 shares of AAPL on the same day -> net obligation: deliver 600 shares. (c) Netting reduces settlement volume by 90%+. (4) **DvP (Delivery versus Payment)**: (a) Securities and cash are exchanged simultaneously -- the transfer of securities only happens if the corresponding cash transfer also happens. (b) Implemented via the CSD (Central Securities Depository, e.g., DTCC in the US). (c) DVP Model 2: securities transfer at end of day, cash transfer at end of day. (d) DVP Model 3: both transfer simultaneously (real-time gross settlement). (5) **T+1 settlement**: (a) As of May 2024, US equities settle T+1 (trade day + 1 business day). (b) The matching engine time-stamps each trade with trade date. (c) The clearing system processes all trades by end of trade date, nets positions, and submits to DTCC for overnight settlement. (d) By the next morning, shares are in the buyer's account and funds are in the seller's account. (6) **Failed trades**: (a) Seller fails to deliver shares (shortage): the CCP borrows shares from a lending pool and delivers to the buyer. (b) Buyer fails to pay: the CCP uses the buyer's margin/collateral to complete the settlement. (c) Penalty: failed trades incur a penalty fee (increasing per day). (d) Persistent failures: the clearing member is fined and may lose clearing privileges.

## Q29: Design the market surveillance system. How do you detect insider trading, market manipulation (spoofing, layering, wash trading), and reporting to regulators?

**Answer:** Market surveillance uses pattern detection and data analysis to catch illegal trading. (1) **Spoofing detection**: (a) Spoofing: placing a large order on one side of the book to create a false impression of demand, then cancelling it. (b) Pattern: submit a large limit order (e.g., 10,000 shares bid), then a small order on the opposite side to execute at an improved price, then cancel the large order. (c) Detection: compute the order-to-trade ratio (OTR). If a trader has OTR > 90% (most orders are cancelled), flag for investigation. (2) **Layering**: (a) Similar to spoofing, but with multiple orders at different price levels to create depth. (b) Detection: identify orders that are repeatedly placed and cancelled, with small executions on the opposite side between them. (3) **Wash trading**: (a) A trader buys and sells the same security to create artificial volume. (b) Detection: identify trades where buyer and seller have the same beneficial owner, same IP address, or same device fingerprint. (c) Also detect: matching order patterns -- a sell order at $100.01 arrives milliseconds after a buy order at $100.01 from the same trader group. (4) **Insider trading**: (a) Detection: trade immediately before a material non-public event (earnings, M&A announcement). (b) Cross-reference trader profiles with: corporate insiders, connected persons, unusual communication patterns. (c) The system doesn't directly detect insider trading -- it flags unusual patterns and refers them to the SEC/FCA for investigation. (5) **Real-time alerts**: (a) The surveillance system processes all order events and trade events in real-time via a stream processor (Kafka + Flink). (b) Alerts are generated when a pattern matches within 30 seconds. (c) Alert severity: LOW (investigate within 24h), MEDIUM (within 1h), HIGH (immediately freeze positions). (6) **Reporting**: (a) The system produces a daily surveillance report for the compliance department. (b) Suspicious activity reports (SARs) are filed with the regulator. (c) All surveillance data is retained for 7 years per SEC rule 17a-4.

## Q30: Design for HFT (High-Frequency Trading) considerations. How does the exchange infrastructure support low-latency trading, co-location, and fair access for all market participants?

**Answer:** HFT infrastructure is about minimizing every microsecond while ensuring fairness. (1) **Co-location**: (a) The exchange provides racks in the same data center for HFT firms. (b) Co-located servers connect to the matching engine via a dedicated, ultra-low-latency network (100GbE with RDMA). (c) Distance from co-lo rack to matching engine: <50 meters (fiber), ~0.25 microseconds latency. (2) **Network topology**: (a) All co-located participants connect through a single multicast network. (b) The matching engine publishes market data events to all subscribers simultaneously. (c) Order entry is via a separate unicast path -- each participant has a dedicated TCP or RDMA connection. (3) **Fairness -- FIFO in the network**: (a) All incoming orders are timestamped by a hardware timestamp (PTP/IEEE 1588) at the network switch before reaching the matching engine. (b) The exchange operates a FIFO gate: orders arriving at the switch are forwarded to the matching engine in strict arrival order, regardless of which HFT firm sent them. (c) This prevents near-equal latency participants from jumping ahead. (4) **Hardware acceleration**: (a) The matching engine can be implemented on FPGA or ASIC for sub-microsecond matching. (b) FPGA-based matching engine: order entry to trade confirmation in <100 nanoseconds. (c) For most exchanges, software (C++ on tuned Linux) achieves <1 microsecond -- sufficient for most use cases. (5) **Latency measurement**: (a) Each order's latency is measured: from network switch -> matching engine -> response. (b) The exchange publishes a latency report: avg, p50, p99, p99.9 for each service. (c) SLA: 99% of orders processed within 10 microseconds. (6) **Anti-competitive practices**: (a) The exchange must NOT give preferential latency to any participant. (b) All co-location racks are identical -- same server specs, same network connectivity, same distance (within 1 meter). (c) The exchange's own trading desk (if any) is not co-located -- must be at least 10 meters from the matching engine. (7) **Transparency**: (a) The exchange publishes its matching engine logic, fee schedule, and latency statistics. (b) Any changes to the matching engine are announced 30 days in advance and tested on a simulated environment. (c) Regulators audit the exchange's fairness annually.
