# Chapter 17 — Design Nearby Friends

## Q1: A feature like Snapchat's Snap Map or Find My shows a user's friends within a certain radius on a map. How do you balance location update frequency against battery life and bandwidth?

**Answer:** Use adaptive location reporting based on user state. When the app is in the foreground and the user is actively viewing the map, poll GPS at high frequency (every 5-10 seconds). When in the background or the app is closed, use geofence-based updates — a significant location change (>500 meters) or periodic updates (every 15-30 minutes). iOS and Android provide "significant location change" APIs that are battery-efficient (wake the device only on cell tower handoffs). Further optimization: if the user hasn't moved in 10 minutes, switch to a "stationary" state and reduce polling to once every 5 minutes.

## Q2: Design the server-side architecture for a "Nearby Friends" feature supporting 100 million daily active users. How do you match friends within a geospatial radius without a full table scan?

**Answer:** Shard by friend group or region. Two main approaches: (1) **User-centric sharding**: each user's friends list maps to a small set of user IDs. Instead of geo-indexing all users globally, only index the locations of a user's friends. When user A opens the map, fetch the last known locations of all of A's friends (a bounded set, typically < 1000) from a Redis cache keyed by `user_id`. Then filter by distance in the application layer. This avoids a global geo-index entirely. (2) **Geo-centric sharding**: for the open-world approach (Snap Map where you can see public stories near you), use a geohash-based index in Redis. For each user, `GEOADD friends:<user_id> <lng> <lat> <friend_id>`. When querying, `GEORADIUS friends:<user_id> <lng> <lat> <radius>`. The `friends:<user_id>` sorted set is stored on a Redis node determined by consistent hashing on `user_id`.

## Q3: How do you handle privacy in a Nearby Friends system? Users should be able to hide from specific people or enable "ghost mode" at any time.

**Answer:** Implement a permission filter layer. When a user updates their location, it's written to a location cache with `{user_id, lat, lng, timestamp, visibility}`. When friend B requests to see friend A's location: (1) Check if A has B blocked (block list lookup, O(1)). (2) Check if A has "ghost mode" enabled (boolean flag on A's profile). (3) Check if A is sharing with B specifically or with a custom list. This filtering is done in the read path, not the write path, because a user's privacy settings can change between location updates. The permission check must be fast (<5 ms per friend) — cache the user's privacy settings in Redis. For batch requests (fetching locations of 50 friends), issue 50 parallel Redis `MGET` calls with a 50-ms deadline.

## Q4: A user has 5,000 friends. Fetching all their locations on map open would be expensive. How does Snap Map handle this at scale?

**Answer:** Snap Map uses a combination of techniques: (1) The client only displays friends visible within the current map viewport. The server only returns friends whose last known location falls within the viewport bounds + some margin. (2) The client sends the current viewport bounds (SW/NE lat/lng) along with the request. The server performs a bounding-box filter on the cached friend locations. (3) Location data is stored in a spatial-temporal key-value store keyed by `user_id` with a TTL. The server iterates through the user's friend list (pulled from a social graph service) and checks each friend's last location against the viewport. (4) This is an O(N_friends) operation, but for 5,000 friends, a bounding-box check is a few float comparisons — it completes in <50 ms if the data is in Redis. (5) Paginate — return the closest 100 friends first, then lazy-load others as the user pans.

## Q5: Design the WebSocket-based real-time push mechanism. When friend B moves, how does friend A's map update within seconds?

**Answer:** When user A opens the map, the mobile client establishes a persistent WebSocket connection to a notification server. The server subscribes A to location updates from A's friends (essentially a pub/sub channel per friend group). When B's location changes: (1) B's client sends the new location to the API. (2) The API writes to the location cache. (3) An event is published to a Redis pub/sub or Kafka topic for B's user ID. (4) A fanout service reads B's friend list from the social graph (cached) and routes the location event to the notification servers that hold WebSocket connections for each of B's friends. (5) Those notification servers push the location update to A's client. This avoids A's client polling. To scale fan-out: use a Redis Cluster fronting the notification servers. Each notification server maintains an in-memory map of `user_id → WebSocket connection`. When a friend moves, the notification router uses a consistent-hash ring to find which notification server holds that friend's WebSocket.

## Q6: How do you handle the "last seen" timestamp for location data? What's the staleness threshold before showing "location unavailable"?

**Answer:** Location data has an associated timestamp and a TTL in the cache (e.g., 2 hours). When the client requests friend locations, the server returns `{lat, lng, timestamp}` for each friend. The client decides how to display it based on staleness: <1 minute → bright dot; 1-15 minutes → faded dot; >15 minutes → show "X minutes ago" label; >2 hours → "location unavailable". The server-side TTL ensures stale data doesn't accumulate. For the "live" indicator (green dot), the server checks if the timestamp is within the last 2 minutes. Configurable thresholds are useful — during a hurricane or festival, you might extend the freshness threshold to reduce battery drain from frequent polling.

## Q7: How would you test chaotic scenarios — what happens when 100,000 people are in the same concert venue and all refreshing their map simultaneously?

**Answer:** This creates a thundering herd on the location read path. Mitigations: (1) Cache the location data aggressively — since many users are querying for the same set of nearby friends. Use a local cache (e.g., in-process LRU) on each API server with a 10-second TTL. (2) Rate-limit the client — the mobile app should back off if the server returns 429 responses, using exponential backoff. (3) Use a "viewports" cache — if two clients query the same geohash area within seconds, serve the same cached response. (4) Degrade gracefully — if the map service is overloaded, show cached locations from the last successful poll with a "last updated X ago" label rather than showing nothing.

## Q8: How do you design the location sharing invitation flow — "Share my location for 1 hour" — with temporary, revokable access?

**Answer:** Temporary sharing uses expiring tokens with access control entries (ACE). When user A shares their location with user B for 1 hour: (1) Create an ACE `{grantor: A, grantee: B, permission: READ_LOCATION, expires_at: now + 1h}` in a fast KV store keyed by `(grantee_id, grantor_id)`. (2) Set a TTL on the ACE equal to the sharing duration. (3) On each location read, the server checks the ACE — if expired, deny access. (4) Revocation is immediate — delete the ACE from the KV store before its TTL expires. (5) For recurring sharing (every day from 9-5), use a cron-style eval that re-creates the ACE on schedule. The ACE check must be in the critical path (read path) and uses a cache with sub-millisecond latency. Write the ACE to both the primary DB (for durability) and Redis (for fast reads).

## Q9: How does Nearby Friends handle cross-platform and cross-region latency when users are distributed globally?

**Answer:** Use a multi-region deployment with active-active geo-replicated services. Location writes go to the nearest regional endpoint (DNS-based routing). Location data is replicated across regions asynchronously (e.g., via Kafka MirrorMaker or Cassandra multi-region replication). When user A in Europe looks for friend B in Asia: (1) A's request hits the EU region. (2) The EU region has asynchronously replicated B's location (typically <5 second lag). (3) The response shows B's location from the replicated data. Cross-region read latency is negligible because the data is cached in the local region. For the "live" indicator (green dot), the freshness is defined relative to the original timestamp, not the region clock. Cross-region latency of 100-200 ms for the write path is acceptable for a feature that doesn't require sub-second consistency.

## Q10: A friend is traveling on a highway at 120 km/h. How does the system handle the burst of location updates without overwhelming the pipeline?

**Answer:** Windowing and deduplication. At 120 km/h (33 m/s), a location update every second would flood the system. Strategies: (1) Client-side throttling — the mobile app enforces a minimum interval between uploads (e.g., 5 seconds) regardless of GPS changes. (2) Server-side dedup — if the new location is within 50 meters of the last reported location (at highway speed, 5 seconds × 33 m/s = 165 meters), the update is still significant. The server can batch updates: receive every update but only publish to subscribers every 10 seconds. (3) Adaptive resolution — at high speed, location accuracy requirements are lower (you don't need centimeter precision for a map dot). The server can quantize the coordinates to fewer decimal places (e.g., 3 decimal places ≈ 110 meter precision vs. 5 decimal places ≈ 1 meter). This increases the effective cache hit rate and reduces the cost of geo-distance computations.

## Q11: How does background location tracking differ architecturally between iOS and Android, and how do you design a unified abstraction layer that handles both platform constraints?

**Answer:** iOS and Android have fundamentally different background execution models. iOS uses a "lease-based" model — the system grants the app background time in short, non-deterministic batches via `beginBackgroundTask` (typically 30 seconds to a few minutes). For continuous location, iOS provides three APIs: `significant-change location service` (cell-tower handoff events, 500m+ granularity), `region monitoring` (geofence crossing), and `CLMonitor` (iOS 17+ with beacon regions). Crucially, iOS can kill the app at any time if it's been in the background too long. Android uses a "service-based" model — a `ForegroundService` with a persistent notification keeps the process alive indefinitely, and `FusedLocationProviderClient` provides batched locations from GPS, Wi-Fi, and cellular radios. A unified abstraction layer must: (1) Expose a `LocationProvider` protocol with platform-specific implementations — iOS uses `CLLocationManager` with background modes capability, Android uses `FusedLocationProviderClient` with `PendingIntent` for battery-efficient callbacks. (2) Implement a "heartbeat escalation" — if the app detects it hasn't received a location update from the OS in more than 2× the expected interval, it requests a higher-accuracy mode or wakes the device via silent push notification. (3) Track the app lifecycle state — foreground (high frequency), background-suspended (iOS only — defer to significant change), background-active (Android service — use Wi-Fi/cellular fused location), killed (rely on push-to-wake). The critical architectural insight: never fight the OS. On iOS, accept indeterminate background updates and design the server to tolerate stale-while-revalidate. On Android, use `interval` and `fastestInterval` parameters to let the OS batch sensor reads efficiently. A common mistake is trying to poll at fixed intervals on iOS — it doesn't work because the OS can suspend the app mid-poll. The correct pattern is event-driven: the OS delivers location updates as they become available, and the app processes them opportunistically.

## Q12: Design a geofence-based wake-up strategy that balances battery life with freshness for a "friend nearby" alert system. How do you handle the case where a user crosses into a geofence while the app is killed?

**Answer:** The system uses a hierarchy of geofence rings: (1) **Coarse satellite geofences (10 km radius)** — iOS `CLRegion` / Android `GeofencingClient` with a wide radius. These are registered with the OS and wake the app on boundary crossing with zero app-side battery drain. Limited to ~100 simultaneous monitored regions per app on both platforms. (2) **Fine active geofences (100 m radius)** — when the coarse geofence fires and the app wakes, the app registers a smaller, precise geofence around the friend's actual location. These expire after a TTL (e.g., 10 minutes) and are re-registered each time the coarse geofence fires. (3) **The killed-app case** — when the app is killed, iOS removes all monitored regions. Android may keep regions registered via `PendingIntent`. To handle this: use a silent push notification (iOS `content-available: 1` / Android `onMessageReceived` in a `BroadcastReceiver`) to re-register geofences. The push triggers the OS to launch the app briefly, allowing it to re-register regions. If push delivery is delayed (common on iOS with Doze mode), the system falls back to the server polling the device's last known location on the next foreground launch and backfilling missed proximity events. The key design decision: geofences are ephemeral and should be re-registered aggressively — every time the app receives any location update, it re-evaluates which geofences are relevant and re-registers them with the OS. This is cheap because OS-level region monitoring uses dedicated hardware (cellular/baseband co-processor) that draws microamps, not the main CPU.

## Q13: You need to detect and protect "privacy zones" — a user's home, work, or hospital location — that should never be shared with friends, even accidentally. How do you design this without leaking the very location you're trying to protect?

**Answer:** This is a zero-trust privacy problem — the server should never see the raw privacy zone coordinates. Solution: (1) **On-device zone detection** — the user defines privacy zones by dropping a pin on their own device. The coordinates never leave the device. The device stores them in the iOS Keychain / Android EncryptedSharedPreferences with biometric protection. (2) **On-device blurring pipeline** — before any location is sent to the server, the client checks if the current location falls within any privacy zone (distance from center + radius). If yes, the client either: (a) **Snaps to zone perimeter** — adjusts the reported location to the nearest point on the zone boundary outward, so the friend sees "nearby but not inside," or (b) **Suppresses the update entirely** — sends a "still active" heartbeat without coordinates, and the server shows "currently unavailable" for that friend. (3) **Avoiding inference attacks** — if the user always suppresses location at 9:00 AM Mon-Fri at a certain coordinate cluster, an attacker could infer "work." Mitigation: add random jitter to the suppression — the device can randomly share a slightly-off location (within 200m of the actual location but still outside the zone) with some probability, making suppression patterns non-deterministic. (4) **Temporal zones** — a user might want to suppress location during certain hours (e.g., 10 PM - 7 AM at home). The device evaluates time-of-day + location against privacy rules before sending. The server protocol must support this: the client sends `{latitude, longitude, accuracy, timestamp, visibility_hint}` where `visibility_hint` is an enum — `VISIBLE`, `ZONED_OUT` (location is fuzzed), `HIDDEN` (no coordinates). The server never knows which zone was triggered or where the zone is located. This requires trust in the client — to harden it, use attestation (iOS DCAppAttestService / Android SafetyNet) to verify the client binary hasn't been tampered with to bypass zone checks.

## Q14: Design a friend clustering strategy for batch location updates when a user has 200+ friends in the same city. How do you group and prioritize which friends to update first when the user opens the map?

**Answer:** The system uses spatial friend clustering with priority tiers: (1) **Tier 1 — Viewport-visible (0-30 friends)** — friends whose last known location falls within the current map viewport bounds. These are fetched immediately and displayed. (2) **Tier 2 — Nearby but off-screen (30-200 friends)** — friends within a 10 km radius of the user's current location but outside the viewport. These are fetched in the background and rendered as clustered markers (e.g., "12 friends nearby" with a count badge). (3) **Tier 3 — Distant friends (200+)** — friends further than 10 km. These are lazy-loaded only if the user pans the map to their region. Server-side clustering algorithm: divide the city into a grid of 500m × 500m cells. For each cell, maintain a count of friends. The server returns the grid cells (with friend counts) plus the actual locations of friends in cells within viewport bounds. This reduces the payload from 200 individual locations to ~30 detailed locations + 20 grid cells with counts. The client renders cell counts as heatmap overlays. For the real-time update stream, events are also batched at the cell level — instead of "friend A moved to (x1,y1), friend B moved to (x2,y2)" the server pushes "cell (5,12) now has 8 friends." The client only expands cell updates when the user taps or zooms into that cell. This is exactly how Snap Map's "ActionMoji clustering" works — at low zoom, you see clusters of friends; at high zoom, individual Bitmojis appear.

## Q15: Design an ephemeral sharing system where user A shares their live location with user B for exactly 90 minutes, with the ability to extend mid-session. How do you handle the token lifecycle, revocation, and clock skew across devices?

**Answer:** The system uses time-bound access tokens backed by a capability-based authorization model: (1) **Token issuance** — When A starts sharing with B, the server creates a capability token: `{issuer: A, subscriber: B, resource: "location:A", iat: now, exp: now + 90min, scope: "read", jti: <UUID>, signature: HMAC(secret, payload)}`. The token is returned to A's client and pushed to B's client via WebSocket. (2) **Token propagation** — B's client stores the token and includes it in the `Authorization` header of every location fetch request for A. The server validates the token on every request — signature, expiration, and revocation status. (3) **Clock skew handling** — Use a 5-minute grace window on token validation. If the server's clock is ahead of the issuer's, tokens expire 5 minutes later than their `exp` claim. If the server detects a clock skew >30 seconds between the token issuer and itself, it logs a warning and uses the issuer's timestamp (embedded in `iat`), not wall clock. (4) **Extension** — When A extends sharing, the server issues a new token with a later `exp` and invalidates the old one via a revocation list. The revocation is a Redis set keyed by `jti` with a TTL equal to the original token's expiry, preventing replay of stale tokens. (5) **Revocation** — immediate revocation works by adding the token's `jti` to a blocklist (Redis `BLOCKED_TOKENS` sorted set). Blocked tokens are checked before the HMAC signature. The blocklist entry has a TTL of 90 minutes (maximum possible lifetime), after which it's auto-removed. (6) **Renewal race** — if A extends sharing while B's client is using an old token, B gets a 401 on the next location fetch, triggering B to request a fresh token from A's shared-with list endpoint. This is a standard OAuth2-style refresh flow: the token is short-lived; the relationship (A sharing with B) is long-lived and session-based. The token is just a cached capability — if it expires, B fetches a new one as long as the relationship is still active.

## Q16: You're integrating a third-party mapping SDK (Mapbox or Google Maps) for rendering friend locations. How do you abstract the mapping layer so you can switch providers without rewriting the Nearby Friends feature, and what are the critical API differences you must account for?

**Answer:** The abstraction is a `MapProvider` protocol with the following contract: (1) **Annotation management** — `addAnnotation(location: CLLocationCoordinate2D, metadata: FriendAnnotation)` and `removeAnnotation(identifier: String)`. Google Maps uses `GMSMarker`, Mapbox uses `PointAnnotation`. The abstraction must handle the fact that Mapbox's `PointAnnotation` is a struct (value type), while `GMSMarker` is a reference type with KVO-observable properties. (2) **Cluster support** — `setClusterIdentifier(_:maxZoom:)` — Google Maps provides `GMUClusterManager` (external library), Mapbox has native `SymbolLayer` clustering with `cluster` and `cluster_count` properties. The abstraction must normalize cluster tap behavior: Google Maps default clusters auto-expand when tapped; Mapbox requires explicit `CLLocationCoordinate2D` calculation for the cluster's center. (3) **Viewport management** — `animateToBounds(southWest:northEast:)` — Google Maps uses `GMSCoordinateBounds` with padding; Mapbox uses `setVisibleCoordinateBounds` with `UIEdgeInsets`. (4) **Custom annotation views** — the friend dot is not a system pin. It needs to show a profile picture circle with a "live" pulsing ring. Google Maps allows custom `UIView` as `iconView` on markers but limits it to one view per marker. Mapbox uses `iconImage` with a sprite sheet — you must pre-render the profile image onto a canvas and load it as a sprite. This is a major architectural difference: Mapbox's rendering is GPU-based and requires all icons to be in a sprite atlas; Google Maps' rendering is CPU-based for custom views. (5) **Fee considerations** — Google Maps charges per tile request (beyond the $200 monthly credit, ~$7 per 1000 requests). Mapbox charges per tile request for the Maps SDK (~$5 per 1000 requests for the base layer plus $5 per 1000 requests for the friends layer). If the Nearby Friends feature updates every 5 seconds, a single user generates 17,280 tile requests per day. At 1M DAU, that's 17B daily tile requests. The cost structure alone often dictates the provider choice — most production Nearby Friends systems use a self-hosted tile server (OpenMapTiles + Planetiler) for the base map and only use the third-party SDK for the friend overlay, dramatically reducing SDK-dependent tile loads.

## Q17: Design the system for a large group sharing scenario — 50+ friends are at the same event (concert, festival). How does the system handle the thundering herd of location updates and the dense clustering on the map?

**Answer:** This creates three simultaneous scaling challenges: high write volume (50+ devices writing rapidly in the same geohash), high read volume (50+ users all querying "friends nearby" at the same time), and visual clustering (50+ markers overlapping on the same screen area). Solutions: (1) **Write coalescing** — when 50 devices are in the same 100m² area, their GPS coordinates will differ by only a few meters. Instead of 50 independent writes, elect a "group anchor" device (e.g., the device with the lowest latency to the server) that reports the group's approximate centroid. All other devices report only deltas: "I am 3m southwest of the anchor." The server publishes the anchor location + deltas. This reduces writes from 50 to ~5 per interval. (2) **Read subscription batching** — instead of each client polling individually, the server broadcasts updates to all subscribed clients in the same geohash cell via a shared multicast channel (e.g., a Redis pub/sub channel per cell). The first client to subscribe to the cell triggers the broadcast; subsequent clients receive the same broadcast without additional load on the location service. (3) **Visual declustering** — on the client, use a force-directed layout to spread overlapping markers. Calculate the bounding box of all markers; if any two markers overlap at the current zoom level, offset them radially from the centroid based on a spiral algorithm. The offset is cosmetic — the actual location is still the centroid, but the visual position is spread so the user can tap each friend. (4) **Priority fan-out** — within the group, the user's closest friends (by interaction frequency) get individual dots. Others get a numbered cluster bubble. This is data-driven: the server ranks friends by a combination of physical proximity and social graph weight (Snap calls this "Best Friends" order).

## Q18: How do you handle cross-platform location sharing between iOS and Android devices when the two platforms have fundamentally different background location capabilities and battery management policies?

**Answer:** The core challenge is that Android's background location is more reliable than iOS's, but iOS devices tend to have stricter battery management (especially with Low Power Mode). The system must normalize the experience across platforms without being constrained by the weaker platform. Approach: (1) **Platform-aware staleness** — the server annotates each location update with the platform type. When serving a batch of friend locations to client A, the server includes `{friend_id, lat, lng, timestamp, platform, reliability}`. The client renders different staleness thresholds per platform: iOS updates older than 5 minutes are considered "fuzzy" (show with reduced opacity); Android updates older than 15 minutes are fuzzy. This sets user expectations correctly — you see an Android friend's dot as "live" for longer. (2) **Compensatory reporting** — iOS devices, when in the foreground even briefly (e.g., user checks another app and the Nearby Friends app gets a few seconds of background time), should report a burst of cached locations. The iOS app maintains a local SQLite table of "missed locations" with timestamps. When the app wakes briefly, it sends up to the last 10 missed locations in a batched payload: `{friend_id, updates: [{lat, lng, timestamp}, ...]}`. The server interpolates the path between these points so Android friends see a smooth path, not teleportation. (3) **Cross-platform push reliability** — iOS push notifications (APNs) have no delivery guarantee and can be delayed by the OS's power management. Android FCM is more reliable. To compensate, the server sends redundant updates via WebSocket (if the client is in foreground) and push (if background). iOS clients that haven't received a push location update within 10 minutes should, on next foreground launch, request a "catch-up" delta from the server covering the missed period. (4) **Battery policy adaptation** — on iOS Low Power Mode, the system should use only significant-change location (cell tower handoffs, not GPS). On Android Doze mode, the system should use `setWhitelisting` judiciously — only whitelist the app for location access if the user has explicitly marked the app as "not optimized" for battery. The key principle: never promise equal behavior across platforms. Design the experience tiers explicitly — "Powered by iOS" (updates when app is open or briefly backgrounded) vs "Powered by Android" (continuous background updates with a persistent notification). Users understand this trade-off.

## Q19: A user in the EU requests deletion of all their location history under GDPR Article 17 ("Right to Erasure"). Your system stores location data in multiple caches, logs, and backup archives. How do you implement a verifiable deletion pipeline that covers all storage layers, and how do you handle the 30-day compliance window?

**Answer:** GDPR right to erasure for location data is hard because location is stored in multiple derived forms. The pipeline: (1) **Logical deletion (instant, < 1 hour)** — Immediately mark the user's record in the primary user database as `GDPR_DELETED: true`. This stops all location collection for this user. Delete the user's location from: the Redis location cache (`DEL user:<uid>:location`), the friend subscription registry (`DEL friend:<uid>:subscriptions`), and the geofence region store. At this point, the user is invisible to friends and no new data is collected. (2) **Hard deletion of primary data (within 30 days)** — Delete all rows from the location history database (time-series store like TimescaleDB or Cassandra) where `user_id = uid`. Use a TTL-aware batch delete — issue `DELETE FROM locations WHERE user_id = uid AND timestamp < NOW() - 30d` first (for data that could still be needed for pending disputes), then delete the rest. For Cassandra, use `DELETE FROM locations WHERE user_id = uid` (which marks tombstones). (3) **Log scrubbing** — Application logs (error logs, access logs) may contain location data. Use a batch processing job that reads logs from the past 90 days, redacts the user's user_id pattern and any coordinate-looking patterns associated with that user_id, and rewrites the log files. This is the hardest part because logs are append-only in most systems. The pragmatic approach: for logs older than 30 days, truncate the log file at the partition boundary (e.g., rotate the log file and delete the old one). For current logs, run a streaming scrubber that replaces the user ID with a salted hash before writing. (4) **Backup archival** — Backups from S3/GCS that contain the deleted data must be overwritten at the next backup cycle. This means for up to 30 days (the typical backup retention cycle), the deleted user's data exists in old backups. The GDPR recognizes this — the "30-day window" allows for standard backup cycles. Document in the deletion report: "User data deleted from primary and cache stores as of [date]. Data in encrypted backups will be overwritten by next backup rotation on [date+30d]." (5) **Verification report** — generate a signed deletion report containing hashes of the deletion operations, timestamps, and an attestation log. This report is kept for the regulatory audit trail (but must not contain the deleted data itself — it contains only operation metadata). For a high-scale system, implement a "deletion pipeline" as a Kafka consumer that reads from a `gdpr_deletion_requests` topic and fans out deletion jobs to all storage services, with a completion callback that aggregates into the verification report.

## Q20: Design a fault-tolerance strategy for the location service. What happens when the location cache (Redis) is completely down, or a region's API servers are unreachable? How do you ensure the user still sees some map state rather than a blank screen?

**Answer:** The system must degrade gracefully into a "last known good state" display — never a blank map. Layered fault tolerance: (1) **Local fallback cache** — each mobile client maintains a local SQLite database of the last known locations for the user's top 50 friends (by interaction frequency). When the server is unreachable or returns 5xx, the client displays this local cache with a "Last updated X minutes ago" banner. The local cache is refreshed every time a successful server response is received. (2) **Read-through cache hierarchy** — On the write path: API server → Redis (primary cache) → Cassandra (long-term store). On the read path: API server checks L1 (in-process LRU cache on the API server, 5-second TTL), then L2 (Redis cluster), then L3 (Cassandra — returns the last known location for each friend, even if stale). If Redis is down, the read path skips L2 and reads from Cassandra directly. Cassandra is highly available (no single point of failure), so the system operates at degraded performance (higher latency, 50-100ms instead of 5ms) but still returns data. (3) **Circuit breaker per storage dependency** — use Hystrix-style circuit breakers for Redis and Cassandra calls. If Redis latency exceeds 100ms for 50% of requests in a 10-second window, the circuit opens and all reads go directly to Cassandra for 30 seconds. The circuit then half-opens to test a single request before closing. This prevents cascading failures where a slow Redis cluster causes API server thread pool exhaustion. (4) **Region failover** — if an entire AWS region goes down (e.g., us-east-1), traffic is routed to us-west-2 via Route53 latency-based DNS. The failover region has its own Redis cluster (asynchronously replicated from the primary region using Redis Enterprise CRDTs or Active-Active). The failover is not transparent — there is a 30-60 second DNS propagation window. During that window, clients receive HTTP 503 from the primary region's load balancer. The client handles this with a retry to the secondary endpoint (pre-configured in the app's config). (5) **Chaos engineering validation** — regularly run GameDay scenarios: kill the Redis cluster, kill one AZ's API servers, inject latency on the Cassandra read path. Verify that the client displays local cached data with appropriate staleness indicators and that the system self-recovers within 30 seconds of dependency restoration. The SLA: P99 of users see a non-blank map within 2 seconds of app launch, even under a single-region or single-cache failure.

## Q21: How does background location tracking differ architecturally between iOS and Android, and how do you design a unified abstraction layer that handles both platform constraints at scale?

**Answer:** iOS uses a lease-based model — `beginBackgroundTask` grants intermittent execution batches. Android uses a service-based model — `ForegroundService` keeps the process alive. Unified abstraction: (1) **LocationProvider protocol**: iOS impl uses `CLLocationManager` with background modes; Android impl uses `FusedLocationProviderClient` with PendingIntent. (2) **Heartbeat escalation**: if no update received in 2x expected interval, request higher accuracy or wake device via silent push. (3) **Lifecycle state tracking**: foreground (high freq), background-suspended iOS (significant-change only), background-active Android (Wi-Fi/cellular fused), killed (rely on push-to-wake). (4) **Never fight the OS**: on iOS, accept indeterminate updates and design server for stale-while-revalidate. On Android, use interval/fastestInterval to let OS batch sensor reads. The server annotates each location with platform and reliability, so clients render different staleness thresholds per platform.

## Q22: Design a geofence-based wake-up strategy. How do you handle a user crossing into a geofence while the app is killed, and how do you balance battery life with freshness?

## Q23: You need to detect and protect "privacy zones" (home, work) that should never be shared, even accidentally. How do you design this without leaking the very location you're protecting?

**Answer:** Zero-trust on-device design: (1) **On-device zone detection**: privacy zone coordinates never leave the device. Stored in iOS Keychain / Android EncryptedSharedPreferences. (2) **On-device blurring pipeline**: before sending location, client checks if current location falls within any zone. If yes: either snap to zone perimeter (report nearest point on boundary outward) or suppress entirely (send "still active" heartbeat without coordinates). (3) **Inference attack mitigation**: if user always suppresses at 9 AM Mon-Fri, attacker could infer "work." Add random jitter — device randomly shares slightly-off location (within 200m but outside zone) with some probability. (4) **Temporal zones**: suppress during certain hours (10 PM - 7 AM at home). The server never knows which zone or where it is. Hardening: use attestation (iOS DCAppAttestService / Android SafetyNet) to verify the client binary hasn't been tampered with to bypass zone checks.

## Q24: Design a friend clustering strategy for batch location updates when a user has 200+ friends in the same city. How do you group and prioritize which friends to update first when the user opens the map?

## Q25: Design an ephemeral sharing system where user A shares location with user B for exactly 90 minutes, with mid-session extension. Handle token lifecycle, revocation, and clock skew.

**Answer:** Capability-based tokens: (1) **Token issuance**: server creates JWT: `{issuer: A, subscriber: B, resource: "location:A", iat, exp: now+90m, scope: "read", jti: UUID, signature: HMAC}`. Token pushed to B via WebSocket. (2) **Clock skew handling**: 5-minute grace window on expiry. If server clock is ahead of issuer, tokens expire 5 minutes later. Log if skew > 30s. (3) **Extension**: issuer creates new token with later exp, invalidates old via revocation list (Redis set keyed by jti, TTL = original token expiry). (4) **Revocation**: add token's jti to BLOCKED_TOKENS sorted set, checked before signature validation. (5) **Renewal race**: if B uses old token during extension, B gets 401, fetches fresh token from shared-with list endpoint — standard OAuth2 refresh flow.

## Q27: Design for large group sharing — 50+ friends at the same concert venue. How do you handle the thundering herd of location updates and dense clustering?

**Answer:** Three simultaneous challenges — write volume, read volume, visual clustering: (1) **Write coalescing**: when 50 devices are in the same 100m² area, elect a "group anchor" device reporting the centroid. All others report deltas ("3m southwest of anchor"). Reduces writes from 50 to ~5 per interval. (2) **Read subscription batching**: instead of individual polling, server broadcasts to all subscribed clients in the same geohash cell via shared multicast (Redis pub/sub per cell). First subscriber triggers broadcast; all subsequent receive without additional load. (3) **Visual declustering**: force-directed layout to spread overlapping markers. Offset radially from centroid using spiral algorithm when markers overlap at current zoom.

## Q29: GDPR right to erasure for location data — your system stores location in caches, logs, and backups. Design a verifiable deletion pipeline covering all layers.

**Answer:** (1) **Logical deletion (instant, <1 hour)**: mark user as GDPR_DELETED, stop collection. Delete from Redis location cache, friend subscription registry, geofence region store. ## Q30: Design a fault-tolerance strategy for the location service. What happens when Redis is completely down or a region's API servers are unreachable? Ensure the user sees map state rather than a blank screen.

**Answer:** Layered degradation: (1) **Local fallback cache**: mobile client maintains SQLite of last known locations for top 50 friends. On 5xx or timeout, display local cache with "Last updated X minutes ago" banner. (2) **Read-through cache hierarchy**: L1 = in-process LRU (5s TTL), L2 = Redis, L3 = Cassandra. If Redis down, skip to Cassandra directly (50-100ms instead of 5ms). (3) **Circuit breaker**: if Redis latency > 100ms for 50% of requests in 10s window, open circuit and read from Cassandra for 30 seconds. (4) **Region failover**: Route53 switches to secondary region with its own Redis cluster (async replicated). 30-60s DNS propagation — clients retry with pre-configured secondary endpoint. (5) **Chaos engineering**: regular GameDay scenarios killing Redis, AZs, injection latency. Verify client shows non-blank map within 2 seconds P99 under any single-region or single-cache failure.
