# Chapter 16 — Design a Proximity Service (Yelp / Nearby Places)

## Q1: Compare Geohash, Google S2, and QuadTree for geospatial indexing at Yelp scale (100M+ places). What are the concrete trade-offs in query latency, write throughput, and cell neighbor complexity?

**Answer:** Geohash is a z-order curve with base32 encoding. Query: compute geohash prefix, scan that prefix's range in a KV store. Write: trivial single KV put. Neighbor lookup: must compute all 8 adjacent geohash cells manually — tedious but well-understood. Precision is fixed per prefix length. S2 uses a Hilbert curve on a cube-projected sphere, producing 64-bit cell IDs. Query: compute the covering (set of cells at a chosen level that covers the search circle). Write: single put. Neighbor lookup: adjacent cells have consecutive IDs due to Hilbert curve spatial locality, so range scans are efficient. S2 handles the spherical geometry correctly (geohash has distortion near poles). QuadTree: recursively subdivide space into four quadrants. Query: traverse the quadtree to find points in the target cell. Write: insert point, potentially split cells that exceed a capacity threshold. Neighbor query: walk up to parent then down to adjacent leaf. Quadtrees are dynamic (density-adaptive) but require more complex balancing and are harder to shard across servers. For 100M+ places, S2 is preferred by Google (it's used in Google Maps) because of spherical correctness, 64-bit efficiency, and excellent library support.

## Q2: A user searches for "coffee shops near me" at 14:30 on a Tuesday. How does the system combine location filtering, text relevance, and real-time business hours to return results in under 200 ms?

**Answer:** The system uses Elasticsearch with a geo-distance query filter: `{"bool": {"must": {"match": {"name": "coffee shop"}}, "filter": {"geo_distance": {"distance": "5km", "location": {"lat": X, "lon": Y}}}}}`. Business hours filtering is applied as a post-filter because it's not geo-indexed — each document has a `hours` field (e.g., `{"tuesday": [{"open": "07:00", "close": "22:00"}]}`). Elasticsearch returns matched docs, then the service layer filters by current time against business hours. To stay under 200 ms: (1) Elasticsearch geo queries with a bounded radius use the geohash cell prefix filter internally, making them fast. (2) The post-filter in the application layer is O(N) on the result set (typically < 100). (3) A CDN or Redis cache layers popular (lat/lng, query, time_of_day) combos. (4) Business hours can be precomputed into an "is_open" boolean field updated hourly via a batch job, moving the filter into the Elasticsearch query itself for faster performance.

## Q3: How would you shard a 100-billion-row places database across multiple database nodes while supporting efficient geo-range queries?

**Answer:** Shard by geohash prefix (e.g., 3-character prefix = ~150 km cells). Each shard owns one or more geohash prefixes. A query with a (lat, lng, radius) converts to a set of covering geohash prefixes at the appropriate level. The query is fanned out to the shards that own those prefixes, then results are merged in the application layer. This is how DynamoDB's geo library works. Challenges: (1) hot shards — Manhattan has far more places than rural Wyoming. Mitigate by splitting hot shards into smaller prefixes (longer geohash). (2) Cross-shard queries for the covering cells — the fan-out adds latency. Mitigate with parallel async queries. (3) Adding/removing shards requires rebalancing — use consistent hashing on the geohash prefix to minimize data movement.

## Q4: Design the proximity service's write path for a new restaurant opening. How does the system handle the update propagation across search indexes, caches, and secondary replicas within seconds?

**Answer:** The write path: (1) Restaurant owner submits details via API → the write service validates and writes to the primary metadata DB (PostgreSQL). (2) An event is published to a CDC stream (Debezium → Kafka). (3) Downstream consumers: (a) the search indexer consumes the event and upserts the document into Elasticsearch with the geo-point field; (b) the cache invalidator sends a cache invalidation message to Redis for any cached queries that might include this area (broadcast to nearby geohash prefixes); (c) the review indexer updates the review aggregation. (4) The API returns success to the client immediately after step 1 — the remaining propagation is async. Total end-to-end visibility: 1-3 seconds. For stronger consistency within seconds, Elasticsearch's refresh interval can be lowered to 1 second (at the cost of write throughput).

## Q5: A rideshare app needs to find all available drivers within 1 km. How do you design this for 10 million active drivers updating their location every 3 seconds? Compare pub/sub versus polling.

**Answer:** This is fundamentally different from static POI search because driver locations change every 3 seconds. Recommended approach: use a geospatial pub/sub system. Each rider subscribes to a geohash cell (e.g., precision level 7 ≈ 150 m). Drivers publish their location to the single cell they occupy. When a rider requests nearby drivers, the system subscribes to the rider's cell plus 8 neighbors. In Redis, use `GEOADD` to update driver location and `GEORADIUS` to query. Redis can handle ~100K GEOADD/s per node. For 10M drivers: shard by region across multiple Redis clusters. Alternative: use Apache Kafka with key by geohash cell — each consumer group receives all location updates for a specific area. Polling (e.g., driver writes to DB, rider queries DB) creates too much read/write amplification. The pub/sub approach reduces query latency because rider receives pushed updates rather than polling.

## Q6: How does the proximity service handle "edge cases" like a user standing exactly on a geohash boundary, or searching near the International Date Line or the North Pole?

**Answer:** For geohash boundaries: the query always expands the search to include all adjacent (8) neighbor cells. This 9-cell covering ensures that points near the boundary are included. For the International Date Line (±180° longitude): this is a discontinuity in most coordinate systems. Google S2 handles this natively with a Hilbert curve on a cube projection that wraps correctly. Geohash also handles it since longitude 180° wraps to -180° in the encoding, but you must compute neighbors that cross the date line correctly. For the North Pole (latitude 90°): geohash cells converge at the poles, making higher-precision geohash prefixes very small near the poles. S2 avoids polar distortion by projecting the sphere onto the faces of a cube. If your service doesn't operate near the poles (e.g., Uber, Yelp), geohash is fine. If global with polar regions (e.g., marine tracking), prefer S2.

## Q7: Design the "popular times" feature — showing how busy a place is at different hours. What's the data pipeline and storage format?

**Answer:** The pipeline: (1) Each check-in, search click, or explicit "I'm here" action produces a raw event with `{place_id, timestamp}`. (2) A stream processor (Flink/Dataflow) aggregates events into 15-minute buckets per place per day-of-week, running hourly. (3) The aggregated data `{place_id, day_of_week, hour_slot, visit_count}` is stored in a time-series friendly store (Bigtable or Cassandra) keyed by `(place_id, day_of_week)`. (4) The read path: when a user views a place page, the service fetches the 24-hour histogram for the current day-of-week and normalizes it to a percentage of peak (0-100%). Storage format: a compressed protobuf or array of 96 uint16 values (one per 15-minute slot). For "live busyness", compare the current slot's real-time count against the historical baseline.

## Q8: How would you design a recommendation system that biases search results toward places the user has previously visited or rated highly?

**Answer:** This requires a personalization layer that merges with the geo-text search. Approach: (1) Maintain a user embedding or preference vector (favorite cuisines, price range, rating threshold) in a user service. (2) When a search query arrives, the search service enriches the Elasticsearch query with a boosting function: `{"function_score": {"functions": [{"filter": {"term": {"place_id": <visited_place_id>}}, "weight": 1.5}]}}`. (3) For collaborative filtering, store a user-place interaction matrix in a graph DB or sparse matrix store. Use approximate nearest neighbor (ANN) search on place embeddings to find similar places the user liked. (4) Merge results: 70% weight on geo-text relevance, 30% on personalization score. The personalization model is trained offline (nightly batch), and the embeddings are loaded into a sidecar cache on each search node for low-latency lookup.

## Q9: The proximity service receives a sudden spike in traffic (e.g., New Year's Eve). How does the system scale to handle 100x normal load?

**Answer:** Multi-layered approach: (1) Read path: most queries hit the CDN edge cache for popular area searches. Geo queries that cannot be cached (the user is in a unique location) still require origin hits. (2) Elasticsearch cluster auto-scales by adding search replicas (horizontal scaling for read throughput). (3) The API gateway applies load shedding: non-critical queries (reviews, photos) are degraded or queued; critical queries (nearby results) get priority. (4) The metadata DB uses read replicas to absorb the increased query load. (5) Caching: precompute and cache queries for known high-density event areas (e.g., Times Square on NYE) in a local Redis cluster. (6) If the system is still overwhelmed, return cached results with a "last updated X minutes ago" notice rather than failing.

## Q10: How would you design a "real-time" leaderboard that ranks places by check-in velocity (most check-ins in the last hour)?

**Answer:** Use a sliding window counter per place in Redis: `ZADD velocity:current_hour <timestamp> <place_id>` with an expiry. Maintain a sorted set `ZINCRBY velocity:leaderboard 1 <place_id>` on each check-in event. Every minute, a cron job expires old entries (outside the 1-hour window) by comparing against a cursor of the oldest allowed event. The leaderboard is simply `ZREVRANGE velocity:leaderboard 0 99`. For high throughput (millions of check-ins/min): windowed aggregation in a stream processor (Flink), writing per-minute counts to Redis: `ZINCRBY velocity:leaderboard <count> <place_id>`. The Flink job handles windowed deduplication, and Redis handles the real-time ranking query. Use a separate Redis instance for the leaderboard to avoid impacting the geo-indexing Redis.

## Q11: Compare Apache GeoMesa and GeoWave for real-time location indexing at Yelp scale (100M+ POIs, 10K updates/sec). What are the concrete trade-offs in query latency, write throughput, and ecosystem integration?

**Answer:** Apache GeoMesa (built on Accumulo/HBase/Redis) and GeoWave (built on Accumulo/Bigtable) both provide distributed geospatial indexing using space-filling curves (Z-order and Hilbert respectively) on top of key-value stores. GeoMesa uses geohash/Z-order and supports multiple temporal dimensions (for tracking moving objects). GeoWave uses Hilbert curve with more sophisticated partition splitting algorithms. At Yelp scale: **GeoMesa** wins on query expressiveness � it supports CQL (Contextual Query Language) filters, WKT geometry predicates (ST_Contains, ST_Intersects), and attribute+spatial combined filters natively. This is critical for queries like "find restaurants open now within 5km with rating > 4.0." GeoWave wins on write throughput with BigTable due to its automatic tablet splitting and pre-split strategy. **Trade-off decisions**: (1) If your query pattern is mostly geo-bounding-box + simple attribute filters, GeoWave's Hilbert curve offers ~20% better query latency due to better spatial locality. (2) If you need complex temporal-spatial queries (e.g., "find all food trucks that were within 1km of this location in the last hour"), GeoMesa's spatio-temporal index (XZ2/XZ3) is purpose-built for this. (3) Write throughput: GeoMesa on Accumulo tops out at ~50K writes/node, GeoWave on BigTable at ~200K writes/node due to BigTable's separation of compute from storage. (4) Operational complexity: GeoMesa requires managing Accumulo (or HBase) which is significantly more complex than BigTable. Recommendation: start with GeoMesa if query expressiveness is the priority (likely for a Yelp-like service), and if write throughput becomes a bottleneck, migrate the indexing layer to GeoWave while keeping the query layer abstracted behind a common interface.

## Q12: Design a spatial join query: "Find all restaurants that are open now within 5km of the user, have an average rating > 4.0, and are within 200m of a parking garage." How do you execute this efficiently without performing O(N*M) cross-joins?

**Answer:** This is a multi-stage spatial join optimization problem. Approach: (1) **Stage 1 � User location filter**: compute S2/geohash covering cells for a 5km radius around the user. Query Elasticsearch/S2 index for all restaurants in those cells. Let's say this returns 1000 restaurants. (2) **Stage 2 � Open now + rating filter**: apply the business-hours filter and rating >= 4.0 predicate. This is a post-filter in the application layer or built into the Elasticsearch query as a script filter. Down to ~150 restaurants. (3) **Stage 3 � Spatial join with parking garages**: for each of the 150 restaurants, find all parking garages within 200m. This is the expensive part � naive O(150 * M) where M = number of garages in the city. Optimization: compute a second covering for each restaurant's 200m radius. The combined geo shapes for all 150 restaurants form a union polygon. Query the parking garage index for all garages within the union bounding box. This is a single query. Then for each restaurant, check if the garage is within 200m. This converts the cross-join to a grouped join: O(Restaurants + Garages) for the query, O(Restaurants * avg_garages_per_200m) for the post-filter. (4) **Stage 4 � Precomputation for hot spots**: for popular areas (downtown SF, Manhattan), precompute the restaurant-garage pairs in a daily batch and store as a proximity list in Redis: ZADD restaurants:near_garage:sf_downtown {garage_id} {restaurant_id} with score = distance. Then at query time, the spatial join is a pre-computed set intersection. (5) **Execution plan optimization**: use a cost-based optimizer. If the user's area has 5 garages and 1000 restaurants, the join order should be: filter garages first, then find restaurants near them. User-defined hinting can specify which table is the "build" side of the join.

## Q13: How would you use GPU acceleration to speed up distance computations for a proximity service? Describe the architecture: what computations move to the GPU, how do you batch them, and what's the latency/cost trade-off compared to CPU-based geospatial indexing?

**Answer:** GPU acceleration is beneficial when you need to compute distance for millions of point pairs simultaneously � common in "nearby search" across all users in a city. Architecture: (1) **CPU-optimized stage**: use traditional geohash/S2 indexing to narrow candidates � this runs on CPU since it's I/O-bound (index lookups). (2) **GPU stage**: once candidate lists are retrieved, the distance computation for each user vs. candidates is a pure matrix operation: for each user location (lat, lng), compute Haversine distance to all candidate POIs. This is embarrassingly parallel. Batching: accumulate user queries over a 50ms window (or 1000 queries, whichever comes first). Pack into a batch matrix: shape [U x P] where U = batched users (e.g., 1000), P = max candidates per user (e.g., 500). The GPU kernel computes all U*P distances in a single launch. A modern GPU (A100) can compute 500K Haversine distances in ~0.02 ms, versus ~10ms on CPU (500K operations at 50ns each). That's a 500x speedup. (3) **Mixed precision**: Haversine distance does not require double precision. Use float16 (half) for lat/lng and float32 for distances. This doubles memory bandwidth and throughput. The error in float16 haversine at 5km is < 1 cm � well within tolerance. (4) **Bandwidth bottleneck**: the limiting factor is PCIe transfer time for sending query matrices to the GPU, not the computation itself. Mitigation: keep the GPU-resident POI index in GPU memory (using GPU memory-mapped files or on-device geohash index). This avoids transferring POI data each time. For a city with 100K POIs, each POI takes 16 bytes (lat, lng, id) = 1.6 MB � easily fits in GPU memory. (5) **Cost trade-off**: GPU instances cost ~3x more than CPU instances (e.g., p3.2xlarge vs c5.2xlarge). However, you need fewer instances: 1 GPU instance can handle the compute load of ~20 CPU instances for this workload. At scale, GPU is 3-5x cheaper for the geo-distance compute workload. (6) **When NOT to use GPU**: if your query pattern is index-dominated (most candidates filtered by hash prefix) and few actual distance computations are needed, GPU overhead dominates. Only deploy GPU acceleration when the post-index filter percentage is high (>20% of candidates survive the geohash filter).

## Q14: Design a mixed-precision storage scheme for geospatial coordinates. What precision is needed for lat/lng at different scales (city-level, street-level, building-level)? How do you encode coordinates to minimize storage in the index while supporting all query types?

**Answer:** Mixed precision leverages the observation that not all applications need the same coordinate precision: (1) **Precision requirements**: 1 degree latitude ~ 111 km. Float32 gives 7 decimal digits (1.2m precision). Float64 gives 15 decimal digits (sub-mm). For proximity search with 100m resolution, float32 is sufficient. For navigation/3D mapping indoors, float64 is needed. (2) **Encoding scheme**: use a variable-length encoding similar to geohash but as a binary representation. A 64-bit geospatial encoding: 1 bit sign, 8 bits exponent (shared), 10 bits latitude increment, 10 bits longitude increment at base. Then additional 17 bits for sub-increment resolution. This gives 44-bit total for a 1-meter resolution � 31% smaller than float64 (64-bit). (3) **Hierarchical precision**: store coordinate pairs at multiple resolutions in the same compound index. Use a 4-byte "coarse" part (geohash prefix, ~150m resolution) for initial filtering, then a 4-byte "fine" part (exact position) for precise distance. The index is keyed by the coarse part, and the fine part is payload. This matches Spanner's storage model: interleave the coarse key with fine data. (4) **Storage savings**: at 100B POIs, switching from float64 (16 bytes for lat+lng) to mixed precision (8 bytes total) saves 800 GB + index space. The trade-off is that queries requiring sub-meter precision must do additional computation to reconstruct exact float64 from the compressed encoding. (5) **Query compatibility**: the mixed-precision index must support all query types: bounding box, radius, polygon contains. The coarse level filters bounding boxes efficiently. For polygon contains that requires precise point-in-polygon, decompress the fine part. (6) **Database-specific**: in PostgreSQL, use the geography type with a GIST index + ST_Simplify for mixed precision. In BigQuery, use GEOGRAPHY with a spatial index. In custom S2/H3 indices, store the 64-bit cell ID directly � this IS a mixed-precision encoding (cell level defines precision).

## Q15: Design the offline maps download and synchronization feature for a proximity service mobile app. Users download city maps for offline use, but POI data changes frequently (hours, ratings, status). How do you delta-sync 500 MB city maps over cellular data without destroying the user's data plan?

**Answer:** The key insight: most of the map payload (tiles, streets, buildings) is static. Only POI attributes (hours, ratings, open/closed status) change frequently. Design: (1) **Layered storage**: separate the downloaded data into two layers � (a) **Base layer**: vector map tiles (streets, buildings, terrain) � static, downloaded once. (b) **Dynamic layer**: POI metadata (name, hours, rating, is_open) � updated via differential sync. (2) **Differential POI sync**: maintain a version vector per city: {city_id, poi_version, tile_version}. On sync, the client sends its current poi_version (a timestamp or hash of all POI data). The server returns only POIs that changed since that version. Use protobuf encoding with delta compression: each POI update is {poi_id, changed_fields_bitmask, field_values}. Unchanged fields are not sent. (3) **Geo-indexed diff**: the server's diff is organized by geohash cell to minimize transfer. If the user is in cell "9q8yy," only diffs for that cell and its 8 neighbors are sent (assuming the user needs visibility of ~5km radius). This reduces full-city sync to neighborhood-scale sync. (4) **Tile versioning with vector tiles**: use Mapbox Vector Tiles (MVT) with a version per tile. Only tiles with version changes are re-downloaded. The tile diff is byte-level: the client and server use a shared tile cache, and the server sends only the gzipped diff (using bsdiff for binary diffs on compressed tiles). Typical tile diff is 1-2 KB vs. full tile at 50-200 KB. (5) **Sync triggers and throttling**: sync is triggered by: (a) app foreground event (check for changes, lightweight), (b) periodic background refresh (every 6 hours for full, every 30 min for favorite POIs), (c) push notification for critical changes (restaurant closes permanently). Throttle to 5 MB per sync over cellular. Queue larger syncs for WiFi. (6) **Data plan preservation**: mobile carriers charge ~/GB overage. If a city sync is 500 MB full, the baseline download is 500 MB once (hopefully on WiFi). Daily diffs average 2-5 MB per city (for 10K POIs with 5% daily churn). Over 30 days: 150 MB � acceptable. With neighborhood-scoped diff, a user who only views 1km radius uses ~200 KB/day. (7) **Consistency model**: offline data is eventually consistent. The UI shows "Updated X hours ago" for POI data. For critical real-time data (is_open), the app attempts a lightweight fetch over the network if available, falling back to the cached data with a warning.

## Q16: A proximity service combines crowdsourced data (user check-ins, reported hours) with official data (business owner claims, government registries). How do you resolve conflicts when the two sources disagree (e.g., 100 users say a restaurant is "open" but the owner officially says "permanently closed")? Design the trust scoring and conflict resolution pipeline.

**Answer:** This is a data fusion problem with asymmetric trust: (1) **Trust tiers**: assign each data source a trust score: Official owner claim = 10, Government registry = 9, Verified third-party (Yelp contractor) = 7, Crowdsourced with verification (photo receipt) = 5, Unverified crowdsourced = 2. (2) **Conflict detection**: run a reconciliation batch job hourly. For each POI field (is_open, hours, phone, address), compare the official value against the crowdsourced aggregate. If >5% of crowdsourced votes contradict the official value, it's flagged as a conflict. (3) **Quarantine with overrides**: when 100 users say the restaurant is open but the owner claims closed, the system enters a "quarantine with override" state: (a) the official value is shown as primary, (b) the crowdsourced value is shown as a secondary note ("100 users report this is open � last checked X hours ago"), (c) the system sends a notification to the owner asking them to verify. (4) **Automated resolution heuristics**: if the crowdsourced signal exceeds a confidence threshold (e.g., 200 unique check-ins over 7 days, with photo evidence), the system overrides the official value and flags the POI for priority owner re-verification. The threshold is dynamic: a high-traffic restaurant needs more corroboration than a low-traffic one. (5) **Cheating detection**: malicious crowdsourcing (false check-ins from a competitor) is detected via patterns: (a) single user checking in to non-food POIs at impossible rates (>50 check-ins/hour), (b) all check-ins from the same IP cluster. These users are shadow-banned (their data is accepted but weighted at 0). (6) **Storage and query**: official data lives in the primary metadata DB (Spanner). Crowdsourced signals are stored in a time-series store with hourly aggregation per POI. The reconciliation pipeline is a Dataflow job consuming both streams and emitting resolved POI states to Elasticsearch. (7) **User experience**: for most queries, the resolved state is served. For explicitly queried POIs with active conflicts, the UI shows a conflict indicator and allows the user to see both data sources.

## Q17: Design 3D geospatial indexing for indoor malls with multiple floor levels. A user searches for "shoe stores within 100m, floor 2" � how do you index and query 3D volumes (x, y, floor)?

**Answer:** 3D geospatial requires extending 2D indexing with a vertical dimension: (1) **Coordinate model**: represent each POI as (lat, lng, floor, altitude_m). The floor is discretized (not continuous), so we can optimize for discrete floors rather than full 3D continuous space. (2) **Indexing strategy A � 2.5D with floor prefix**: extend the geohash to include floor: key = geohash(lat, lng) + ":" + floor. This creates a separate 2D index per floor. Query: for a user on floor 2, search the "floor=2" geohash cells within 100m. Simpler and leverages existing 2D geo indexing. At query time, also check floor 1 and 3 with a 50m vertical buffer if the mall has open atriums (escalators/stairs connect floors). (3) **Indexing strategy B � True 3D S2**: Google S2 can be extended to S2 (3D) by projecting onto a cube and adding a Z-axis. Each cell ID becomes 128-bit: 64 bits for the 2D S2 cell + 64 bits for vertical range. PostGIS 3D supports ST_3DDWithin and ST_3DIntersects. (4) **Query pattern**: "shoe stores on floor 2, within 100m": S2 covers the 100m radius on lat/lng, then filter by floor_id = 2. For "within 100m including vertical (staircase accessible)": compute geohash covering, filter floor = {1, 2, 3}, then post-filter by 3D Haversine (including altitude). (5) **Indoor routing (adjacency)**: floors are connected by staircases and elevators stored in a ertical_connections (mall_id, floor_from, floor_to, connection_type, location) table. The routing algorithm treats each floor as a graph layer, with vertical connections as edges between layers. Dijkstra operates on the 3D graph. (6) **Precision vs. performance trade-off**: for most indoor use cases, 2.5D with floor prefix is sufficient and 10x faster than true 3D indexing. True 3D is needed for drone delivery routing or multi-story autonomous vehicle parking. Recommendation: use 2.5D for general indoor POI search, with a fallback to 3D PostGIS for specific routing queries.

## Q18: Design a geofencing-as-a-service platform. Users define custom geofences (polygons on a map) and get notified when they enter/exit. How do you support 10M geofences with <1 second notification latency, each evaluated against 100K device location updates per second?

**Answer:** This is a classic "reverse geofencing" problem � not "find POIs near point" but "find which geofences contain this point." Scale: 10M geofences, 100K devices updating location every 3 seconds = 33K location events/sec. Architecture: (1) **Geofence indexing**: store geofences in a spatial index (S2 or Geohash) keyed by covering cell. Each geofence's covering is precomputed: a large polygon covers many S2 cells; a small polygon covers few. Index: cell_id -> list of geofence_ids. On insert/update of a geofence, compute its covering and index it. (2) **Location evaluation path**: on each device location update (lat, lng, timestamp, device_id): (a) compute the S2 cell ID of the location (level 13, ~1km cells). (b) Query the geofence index for all geofences covering that cell. (c) For each candidate geofence, perform point-in-polygon (PIP) check (ray casting algorithm, optimized with bounding box pre-check). (d) Compare against the device's previous location's containment set � if containment changed (was OUT -> now IN), trigger enter event; if IN -> OUT, trigger exit event. (3) **Performance optimization**: (a) cache the "current containment set" per device in Redis (device_id -> Set<geofence_id>). (b) Pre-filter geofences using their bounding box before PIP � eliminates 90% of PIP checks. (c) Use SIMD-accelerated PIP (cross-platform: GEOS or Boost Geometry with -march=native). (d) Batch location updates for a single device: if a device sends updates every 3 seconds, and geofence evaluation takes 2ms, you can process 1500 locations/sec per CPU core. (4) **Scale-out architecture**: shard geofence evaluation by device_id (consistent hashing). Each evaluation node holds a replica of the entire geofence index in memory (10M geofences * 200 bytes each = 2 GB � fits in memory on one node, but replicate to 3 nodes for HA). (5) **Notification delivery**: enter/exit events are published to a Kafka topic partitioned by device_id. Consumers push notifications via APNS/FCM. For latency critical use cases (e.g., "customer enters store" triggers a welcome coupon), the push path must be under 500ms. This is achievable with end-to-end streaming. (6) **Geofence CRUD**: when a geofence is created/updated, its covering must be updated. Use a broadcast mechanism: geofence change -> Pub/Sub -> all evaluation nodes receive the update. During the propagation window (seconds), a device's location might be evaluated against old coverage � acceptable, eventual consistency is fine for geofencing.

## Q19: Design a temporal geo-query system that supports queries like "find me all coffee shops that were popular at this time yesterday," "show me bars that get busy at 11 PM on Fridays," and "what restaurants near this location were most searched for in the last hour?" How do you index both space and time simultaneously?

**Answer:** This requires a spatio-temporal index: (1) **Data model**: each POI has a time-series of "activity" metrics: check_ins, search_clicks, direction_requests, all bucketed into 15-minute windows per day-of-week. Store as (geohash_prefix, day_of_week, time_slot) -> sorted list of (poi_id, activity_score). (2) **Index structure**: use a composite key. On Bigtable/Cassandra: row_key = geohash(5) + ":" + day_of_week + ":" + time_slot. Column qualifier = poi_id, value = activity_score. This allows range queries on time slots. For query "popular at 11 PM on Friday": row_key = geohash(5) + ":5:44" (where 5 = Friday, 44 = 15-min slot for 11 PM). Scan the row for top-K POIs by activity_score. (3) **Real-time query: "most searched in the last hour"**: maintain a sliding window counter in Redis: ZINCRBY geo:popular:9q8yy <poi_id> with a TTL of 60 minutes. Every check-in/search increments the score. ZREVRANGEBYSCORE returns top POIs in the cell. The cell-level sorted set can be merged across neighbor cells at query time. (4) **Historical pattern query: "popular at this time yesterday"**: the time-series index (step 2) stores yesterday's data. Query: SELECT TOP 20 poi_id FROM spatio_temporal WHERE geohash_prefix = '9q8yy' AND day_of_week = WEEKDAY(NOW()) AND time_slot = SLOT(NOW() - 24h) ORDER BY activity_score DESC. This is a single range scan. ## Q20: Design a privacy-preserving proximity query system. A user wants to know "how many people are within 100m of me" without revealing their exact location to the server. How do you support both count queries and "find nearby friends" with differential privacy guarantees?

**Answer:** Privacy-preserving proximity requires shifting trust from the server to the client or using cryptographic protocols: (1) **Private proximity count (P4)**: uses a distributed point function protocol. The user's device generates a secret-shared representation of its location. Other devices in the area evaluate their location against the secret share without learning the user's location. The server sums partial results to compute the count. This is the approach used by Apple's "Find My" network. Latency: ~2 seconds for a 1km radius with 1000 devices. (2) **Differential privacy for aggregate queries**: the server publishes aggregate "density maps" with Laplace noise added. For "how many people within 100m": (a) the server computes the count from location data it already has, (b) adds Laplace noise calibrated to epsilon = 1.0, (c) returns noisy count. With epsilon = 1.0 and a population of 50 within 100m, the noise standard deviation is ~7 (14% error). For analytics (average density), this is acceptable. For safety ("is there anyone nearby?"), the error might be too high � in which case use a lower epsilon per-user per-day budget. (3) **Private set intersection for "find nearby friends"**: each device computes a set of geohash cells it occupies at different precisions. The server computes the intersection of the user's cells and their friends' cells using PSI (Private Set Intersection) based on ECDH or oblivious PRFs. Only the intersection (shared cells) is revealed � not individual exact locations. Latency: ~500ms for 10 friends, each with 20 cells. (4) **Geo-indistinguishability**: instead of sending exact (lat, lng) to the server, the device adds planar Laplace noise with parameter epsilon_geo. For epsilon = 0.5 and a desired privacy radius of 100m, the noise scale is 100/epsilon = 200m. So a user at (40.7128, -74.0060) reports (40.715, -74.003) � roughly 200m offset. This makes exact location recovery difficult while still allowing proximity queries at the 500m granularity. (5) **k-anonymity for location uploads**: the server only accepts location updates if at least k devices are in the same geohash cell. For k=20, if fewer than 20 devices are in a cell, the server clusters adjacent cells until the threshold is met. This prevents identifying a single device's location pattern. (6) **Trade-off analysis**: strong privacy (P4, PSI) adds 500ms-2s of latency and requires client-side computation. Weak privacy (geo-indistinguishability, k-anonymity) has negligible latency but leaks approximate location. Recommendation: use differential privacy + geo-indistinguishability for "nearby place" search (low sensitivity), and P4/PSI for "find my friends" (high sensitivity). The user selects their privacy level in the app settings.

## Q21: How would you design the "nearest POI" query (e.g., "closest gas station to my current location") at Yelp scale? What index structures and query optimizations are needed?

**Answer:** The nearest-POI query is fundamentally different from a range query (POIs within radius). It requires spatial sorting. Approaches: (1) **Incremental nearest neighbor (INN) via R-tree**: traverse cells in order of distance from the query point. An R-tree built on POI locations supports incremental nearest-neighbor search using a priority queue of nodes ordered by mindist (minimum possible distance). This is O(log N) per result. Best for offline/recommendation systems. (2) **Geohash approximation**: (a) Start at geohash level corresponding to ~100m precision (level 7). Query the cell. (b) If no POIs found, expand to the 8 neighbors (level 7). (c) Repeat expanding outward by one ring until a POI is found. (d) Among found POIs, compute Haversine distance and return the minimum. Number of cells scanned in worst case: O(k²) where k = rings expanded. (3) **S2 with Hilbert order**: because S2 cell IDs preserve spatial locality, a query can scan cells in ID order outward from the query cell. This is more efficient than geohash because neighbor cells have consecutive IDs — one range scan suffices per ring instead of 8 separate scans. (4) **Latency optimization for production**: precompute and store the nearest-N POIs for each geohash cell in an offline batch job. For cell `9q8yy`, store `[poi_id_1, poi_id_2, ..., poi_id_20]` sorted by distance from cell center. At query time: (a) identify the user's geohash cell, (b) return the precomputed nearest POIs from that cell. (c) For cells with <N POIs, the batch job merges neighbor cells. Query latency: ~2ms (single KV lookup). (5) **Real-time ranking adjustment**: the nearest POI result must account for real-time factors (open/closed, popularity). After retrieving precomputed nearest candidates, filter by open status and re-rank by a composite score. This avoids scanning large indexes while still incorporating real-time signals.

## Q22: How do you shard geospatial data across 100+ database nodes while maintaining efficient range queries? Compare grid-based, hash-based, and tree-based sharding strategies.

**Answer:** Sharding geo data for proximity queries is challenging because range queries cross shard boundaries. (1) **Grid-based sharding**: divide the Earth into fixed grid cells (e.g., 1°×1° lat/lng). Each server owns a set of grid cells. Query: determine which grid cells overlap the search radius, fan-out queries to those servers, merge results. Pros: simple, predictable shard counts. Cons: hot spots (Manhattan cell has 1000x more POIs than ocean cell); skew requires manual rebalancing. (2) **Geohash-prefix sharding**: each POI's geohash prefix (first 4 chars, ~40km precision) determines its shard: `shard_id = hash(geohash_prefix) % N`. Query: compute the geohash of the search center, generate neighbor prefixes (9 cells), hash each to find target shards, fan-out. Pros: even distribution via hash. Cons: 9 fan-out queries per request; hash means adjacent cells land on random shards (no locality). (3) **S2 cell ID range sharding**: assign contiguous ranges of S2 cell IDs to each shard. Because S2 cell IDs preserve spatial locality (Hilbert curve), a contiguous range corresponds to a geographic region. Query: identify the S2 cell ID range covering the search radius and query the shard(s) owning that range. Pros: minimal fan-out (usually 1-2 shards). Cons: range sizes must be dynamic — high-density areas need smaller ranges; requires a shard-to-range mapping service. (4) **Dynamic sharding (Google S2 approach)**: use an S2 cell covering at a fixed level (e.g., level 12 ≈ 5km²). Monitor each cell's POI count and QPS. When a cell exceeds a threshold (e.g., 10K POIs), split it into 4 children cells at the next S2 level. Each leaf cell maps to a tablet. The tablet-to-server mapping is managed by a distributed metadata service. (5) **Production recommendation**: use S2-based range sharding for primary data (allows single-shard lookups for local queries) with a hash-based secondary index for random-access queries (e.g., "get this specific POI by ID"). The metadata server tracks shard-to-server mappings and supports live migration.

## Q30: Design the caching strategy for geo-query results. How do you cache "top 20 restaurants near Times Square" across millions of unique queries while keeping cache hit rates high and staleness low?

**Answer:** Caching geo-query results is challenging because the query space is continuous — no two queries have exactly the same center. (1) **Cell-level caching**: instead of caching per-query results, cache results per geohash/S2 cell. For cell `9q8yy` (level 6, ~1km²), precompute and cache the top-50 POIs sorted by a composite score. Any query whose center falls within `9q8yy` can use this cache. Additionally, neighbor cells' cached results are merged to handle queries near cell boundaries. (2) **Cache granularity**: use multiple S2 levels: (a) Level 10 (~60km²) — cache for city-level queries. TTL: 1 hour. (b) Level 13 (~7.5km²) — neighborhood-level. TTL: 15 minutes. (c) Level 16 (~1km²) — street-level. TTL: 5 minutes. Higher zoom (smaller cells) has shorter TTL because results change more frequently at street level (a new restaurant opening). (3) **Cache key design**: `geo:pois:{s2_cell_id}:{category_filter}:{sort_order}:{limit}`. Example: `geo:pois:8c9e23b:restaurant:rating:20`. This allows category-specific caching. (4) **Staleness vs. hit rate**: (a) For freshness, the cache is invalidated when a POI in that cell is updated (new POI added, rating changed, hours changed). (b) Use a change-data-capture pipeline: when a POI is updated, publish event `poi_updated(poi_id, cell_ids)`. A cache invalidator subscribes and deletes cached entries for the affected cells. (c) Trade-off: fine-grained invalidation (per POI update) increases write load. Alternative: use TTL-only with 5-minute granularity and accept up to 5 minutes of staleness. For Yelp-scale, TTL-only is simpler and acceptable — most POI metadata doesn't change minute-to-minute. (5) **Cache hierarchy**: (a) L1: in-memory (Redis/Memcached) on the API server. 100MB per server stores ~50K cells. Hit rate: 40% (hot cells repeated across queries). (b) L2: distributed cache (Redis cluster). 100GB stores all cells in active cities. Hit rate: 85%. (c) L3: CDN edge cache for map tiles and static POI data (hours, photos). Hit rate: 95% for tiles. (6) **Cache-aside with refresh-ahead**: when a cache entry is near expiry (TTL < 20% remaining), a background goroutine refreshes the entry before it expires. This prevents thundering herd on cache miss when a popular cell's entry expires. (7) **Query collapsing**: for identical concurrent queries (rare but possible for popular intersections), use a single-flight pattern: the first request fetches the result; subsequent identical requests wait for the first to complete and share the result.
