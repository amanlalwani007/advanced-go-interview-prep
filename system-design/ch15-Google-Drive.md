# Chapter 15 — Design Google Drive

## Q1: Explain how Google Drive handles delta sync for efficient revision storage. How does it decide which blocks have changed without transferring the entire file?

**Answer:** Google Drive uses a chunk-based delta sync algorithm similar to rsync. The file is split into fixed-size blocks (e.g., 4 MB). On each sync, the client computes hash checksums (SHA-256) for all blocks and sends them to the server. The server compares these hashes against the last committed revision's block index. Only blocks with mismatched hashes are uploaded. This avoids re-uploading the entire file. For large files with small changes (e.g., a 1 GB presentation with one slide changed), only the modified ~4 MB blocks are transferred. The server then assembles the new revision as a manifest pointing to the same (unmodified) block references plus the newly uploaded blocks.

## Q2: Design the architecture for handling 10,000+ simultaneous file uploads per second. Where are the bottlenecks, and how do you shard the metadata store?

**Answer:** The upload path bottlenecks are: (1) network bandwidth at the load balancer, (2) chunk reassembly CPU at the upload service, (3) metadata database write throughput. Architecture: clients upload chunks directly to a chunk/blob storage layer (e.g., S3 or GCS) via presigned URLs, bypassing application servers. A separate async service consumes upload-completion events from a queue and updates the metadata store. For metadata sharding, partition by `user_id` (or a hash of `user_id`) across multiple PostgreSQL instances. This keeps all of a user's files co-located for transactional consistency on rename/move operations within their drive. Use read replicas for listing directory contents.

## Q3: How does Google Drive handle conflict resolution when a file is modified offline on two devices? Walk through the detection and resolution flow.

**Answer:** When a device comes online, it sends its local revision vector (a list of {file_id, revision_id, timestamp} pairs) to the server. The server compares this against the canonical revision log. If the latest canonical ancestor of both edits diverges (causal conflict), the server detects a conflict. Resolution: the server accepts both versions, marks the file as conflicted, and both copies are stored as separate branches. The user is notified via the UI to manually merge or choose which version to keep. The "winning" version becomes the new canonical head; the losing version is preserved as a conflicted copy. This is a multi-value (non-last-writer-wins) strategy — last-writer-wins would risk data loss.

## Q4: Explain the design of the file indexing and search feature in Google Drive. How does it index full-text content of documents without overloading the system?

**Answer:** Files are indexed asynchronously. After a file is uploaded, an event is published to a queue consumed by the indexing service. For text-based files (docs, PDFs, code), the indexing service extracts text content (using OCR for images, PDF parsers, etc.) and builds an inverted index stored in a search engine like Elasticsearch. To avoid overloading: (1) the queue acts as a buffer — indexing can lag behind uploads during peak traffic; (2) file types that are not text-searchable (binaries, videos) are indexed by metadata only (filename, type, size); (3) indexing priority is given to frequently accessed files. The search query path queries Elasticsearch which returns matching file IDs, then joins with the metadata DB for access control filtering.

## Q5: Google Drive throttles API requests. Design a rate-limiting strategy that differentiates between interactive user requests and sync client polling.

**Answers:** Use a dual-rate-limiter pattern. Interactive requests (UI operations: rename, delete, move) get a higher-priority token bucket with lower capacity but fast refill — they need low latency. Sync client polling requests (checking for changes) get a separate bucket with higher burst capacity but stricter long-term average because a single sync client can generate far more requests than an interactive user. Distinguish them via API path prefixes or User-Agent headers. Additionally, use a leaky bucket per `user_id` with a global cap per IP to prevent a single misconfigured sync client from starving other users.

## Q6: A user has 50,000 files in a single folder. How does Google Drive avoid fetching the entire file list on every sync?

**Answer:** The desktop sync client maintains a local SQLite database that mirrors file metadata. Instead of fetching the full directory tree, the client polls a `/changes` endpoint that returns only the incremental change log (event ID, file ID, operation type, timestamp) since the last known cursor. The cursor is an opaque token that represents the client's sync state. The server maintains a monotonically increasing change ID per user; any mutation (file create, update, delete, share) appends an event. The client fetches batches of changes, applies them to the local SQLite DB, and advances the cursor. This way, listing a folder with 50,000 files requires only O(changes since last sync) work, not O(50,000).

## Q7: How does Google Drive ensure data durability against data center failures, and what is the trade-off in cost vs. latency?

**Answer:** Google Drive stores each chunk in multiple geographically distributed data centers (typically 3). Writes are synchronously replicated within the local data center for fast acknowledgment, then asynchronously replicated across regions. For read-after-write consistency within a region, the client reads from the local replica. Cross-region replication is eventual. The trade-off: synchronous cross-region replication would add 50-200 ms of latency per write and triple the write cost, but would guarantee zero data loss. Google chooses async cross-region replication because it keeps write latency low and accepts the minute-scale replication window (during which a full data center loss could lose recent writes). For most file storage use cases, this is acceptable.

## Q8: Design the sharing permission model. How does Google Drive evaluate access for a deeply nested file shared with a group?

**Answer:** Permissions are ACL-based. Each file and folder has an ACL (list of {principal, role}). Roles: Owner, Editor, Commenter, Viewer. Inheritance: a file inherits permissions from its parent folder. When checking access for a deeply nested file: (1) the system checks the file's own ACL; if a direct entry matches the user (or any group the user belongs to), it returns that role. (2) If no direct entry, walk up the ancestor chain checking each folder's ACL. (3) Cache the resolved effective permission at the file level with a TTL to avoid O(depth) walks on every access. Group membership is resolved via a group service (e.g., LDAP). ACLs are stored in the metadata DB with an index on `(file_id, principal_id)` for O(1) direct lookup.

## Q9: How does Google Drive detect and handle large-scale abuse, such as a script creating millions of files or sharing spam?

**Answer:** Abuse detection operates at multiple layers: (1) Per-user rate limits on file creation, sharing invites, and API calls. (2) Anomaly detection — a user creating 10,000 files per minute triggers a review flag. (3) Heuristic analysis — files shared to too many users in a short window are quarantined pending abuse review. (4) Machine learning classifiers scan file content for malware, phishing, or policy violations. Quarantined files are invisible to recipients, and the uploader is notified with an appeal process. This is implemented as a stream-processing pipeline (e.g., Apache Beam/Dataflow) consuming file creation and share events, running rules and ML models in real-time, and publishing block/allow decisions to a fast key-value store checked on every file access.

## Q10: Explain the offline availability feature. How does Google Drive decide which files to cache locally, and how does it handle cache invalidation?

**Answer:** Users can mark specific files or folders as "Available offline". The desktop/mobile client downloads the latest revision of marked files to local storage. For folders, it recursively downloads the subtree. The cache is indexed by file ID and revision ID in the local SQLite DB. Cache invalidation: on the next sync poll, if a file's server revision ID differs from the locally cached one, the client downloads the new chunks and updates the local index. For space management, the client shows the offline cache size and allows the user to remove files. On mobile, the client may limit offline caching to a configurable maximum (e.g., 20 GB) and evict least-recently-used files when space runs low.

## Q11: Google Drive uses differential sync. Compare differential (rsync-style) sync against Operational Transform (OT) and CRDTs for a collaborative file storage system. When would you choose one over the other, and what are the concrete failure modes of each at Google Drive scale?

**Answer:** Differential sync (rsync-style): file is split into fixed blocks, only changed blocks are transferred. Failure mode � if concurrent edits from two devices modify the same block range, the last upload wins at block granularity, causing partial data loss. It has no merge semantics; conflicts produce "conflicted copies." Operational Transform (OT): each edit is an operation (insert "a" at position 5) transformed against concurrent ops to converge. Used by Google Docs for real-time collaboration. Failure mode � OT requires a central server to perform transformation, creating a single point of serialization. Transformation logic is O(n�) complex for multi-cursor scenarios, and edge cases (e.g., concurrent formatting + deletion at same range) produce subtle bugs that have caused Google Docs corruption incidents. CRDTs: each peer applies commutative operations and converges without a central server. Failure mode � metadata overhead for large files is severe; every character needs a unique ID, so a 10 MB document carries 100s of MBs of CRDT metadata. For Google Drive's use case (large binary files, office documents, images), differential sync is correct � it optimizes for the common case (single editor, large files). OT/CRDTs are only appropriate for the real-time collaborative editing layer (Google Docs/Sheets), not the storage layer. Drive's architecture cleanly separates the two: Drive handles blob storage with differential sync; Docs handles character-level edits with OT.

## Q12: Design the file type detection and processing pipeline for Google Drive. How do you handle a user uploading a malware.pdf.exe that bypasses extension-based detection? Cover magic bytes, deep content inspection, and the trade-off between scan latency and user experience.

**Answer:** The pipeline has multiple layers: (1) **Extension check** � fast pass, rejects .exe, .bat, .scr etc. immediately. (2) **Magic byte identification** � read the first 512 bytes and match against a fingerprint database (libmagic / file command). Detects renamed files: malware.pdf.exe with EXE magic bytes MZ is flagged regardless of extension. (3) **MIME sniffing** � Chrome's MIME sniffing algorithm (https://mimesniff.spec.whatwg.org/) as a secondary check. (4) **Deep content inspection** � for text files and PDFs, extract text and run ML-based malware classifiers (check for phishing URLs embedded in PDFs, macros in Office files). For binaries, send to a sandbox (Google's Cujo sandbox) that executes the file in an isolated VM and observes behavior. (5) **Hashing** � compute SHA-256 and check against known malware hash databases (VirusTotal integration). (6) **Scan latency trade-off** � users expect uploads to complete in <2 seconds for small files. A full sandbox scan takes 30-120 seconds. Solution: return the upload success immediately (file appears in Drive), but mark it as "Scanning" in the UI. If the scan finds malware after the fact, the file is quarantined and the user is notified. This async scanning means malware can be shared in the ~2-minute window before detection � an accepted risk. For Google Workspace Enterprise, admins can enforce "block until scan completes" at the cost of upload latency. The pipeline is implemented as a Pub/Sub-triggered Cloud Function/Dataflow pipeline, consuming upload events, processing through the detection stages, and writing results to a BigQuery table for audit.

## Q13: When a user edits a Google Doc and simultaneously renames the containing folder in Drive, a collaboration conflict can occur between the Docs real-time layer and the Drive metadata layer. How do you detect and resolve this cross-system inconsistency without corrupting data?

**Answer:** This is a cross-service consistency problem between the Docs collaboration service and the Drive metadata service. The Doc's content is stored in the Docs service keyed by document_id; the Drive metadata service stores resource_id -> document_id mapping along with the parent folder reference. If a folder is renamed in Drive while the Doc is being edited, two things happen concurrently: Docs maintains an in-memory session referencing the old path, and Drive commits the rename. To detect: Drive uses an MVCC approach � every resource has a change_id. The Docs service periodically checks with Drive that its cached path is still valid. If the change_id for the parent folder or the doc itself has advanced, Docs knows the path is stale. To resolve: (1) The Docs session is notified via a push event from Drive's change notification system. (2) The Docs UI updates the title bar to show the new path. (3) The content being edited is unaffected � the rename touches metadata, not content. The dangerous case is concurrent delete + edit: if the parent folder is trashed while the doc is being edited. When the user saves, the Docs service attempts to update Drive metadata, but Drive rejects because the parent is in trash. Resolution: Drive returns an error, and the client shows "This document's parent folder has been deleted. Save a copy to a new location." The solution is a two-phase commit with compensation: Drive performs a logical check (is the parent valid?), commits the metadata change, and if it fails, the Docs content is preserved in a detached state with a clear UI notification.

## Q14: Design an automated storage tiering system for Google Drive that moves files between hot (SSD), warm (HDD), and cold (archive) storage based on access patterns. How do you prevent thrashing when a file is accessed once after months of cold storage?

**Answer:** The tiering system uses a gradient approach: (1) **Tiers**: Hot � SSD with 3x replication within region, <5 ms read latency, .10/GB/month. Warm � HDD with erasure coding (12+3), ~50 ms read latency, .02/GB/month. Cold � tape/optical archive, minutes to hours recall latency, .001/GB/month. (2) **Promotion/demotion policy**: track last_access_time, access_frequency_90d, file_size. Hot -> Warm: if no access in 30 days. Warm -> Cold: if no access in 180 days. Cold -> Warm: on first access, recall to warm (not hot). Warm -> Hot: if accessed 3+ times in 7 days after recall. (3) **Thrashing prevention**: introduce a cooldown timer � after a file is promoted from cold, it cannot be demoted for at least 7 days regardless of access pattern. Use an LRU filter with a 24-hour window: a single access doesn't trigger promotion; only sustained pattern changes do. (4) **Manifest indirection**: files are not physically moved on every tier change. The metadata store maintains a storage_tier and chunk_location pointer. Demotion is lazy � the chunks stay in hot storage until the tiering batch job compacts them. This batch process runs daily during low-traffic windows. (5) **User bypass**: power users can pin files to hot storage via a Drive API parameter. This is used for frequently accessed shared drives in enterprises. (6) **Cost analysis**: the tiering system must save more in storage costs than it spends on read/write operations. At Google Drive's exabyte scale, moving 1% of cold data saves millions annually. The system logs tier transitions to BigQuery for cost accounting per organizational unit.

## Q15: A folder in Google Shared Drive has 100,000 users with direct or group-inherited permissions. Design the permission evaluation path that can answer "Can user U access file F?" in under 50 ms P99. Address ACL breadth, group nesting depth, and cache invalidation storms when a permission changes.

**Answer:** This is the "broad-shared-directory" problem. Naive evaluation � walk the ACL of the folder and all parent folders, resolve group membership for U against each ACL entry � takes seconds. Optimized design: (1) **Flat permission cache**: for each file, precompute and cache the effective permission set as a Bloom filter or bitmap of (principal_hash, role) pairs. Keyed by (file_id), stored in Redis with 60-second TTL. (2) **Group flattening**: when a group is added to an ACL, eagerly resolve the group's transitive membership and write individual entries: ACL (file_id, user_id, role). This adds write amplification but makes reads O(1). Flattened ACLs are stored in Spanner with an interleaved table: parent: file_id, user_id -> role. Query: SELECT role FROM flattened_acl WHERE file_id = ? AND user_id = ?. Spanner's interleaved index makes this a single row lookup. (3) **Incremental re-flattening on group change**: when group membership changes (user added/removed from group), a background worker recomputes affected folder ACLs. This is batched with a 60-second debounce to avoid storms. (4) **Permission change cache invalidation**: when a permission is added/removed, publish an invalidation message: {file_id, version}. Each cache node subscribes and evicts the key. But for 100K-user folders, a single permission change could invalidate 100K cache entries. Mitigation: use versioned ACLs � the cache key is (file_id, acl_version) not (file_id). When a permission changes, increment acl_version on the file metadata. Reads check the version from metadata (cached separately with longer TTL) and only hit the ACL cache if the version matches. This avoids bulk invalidation � old versions naturally expire. (5) **Hot-path shortcut**: for the most common 1000 files (measured by access frequency), pre-warm the flattened ACL in a local LRU cache on each serving node.

## Q16: Google Drive's trash system retains deleted files for 30 days (consumer) or custom retention (enterprise). Design the enforcement pipeline that ensures trashed files are physically deleted after the retention period while supporting legal holds and eDiscovery holds that override deletion. How do you handle the "delete from trash" vs. "purge after retention" semantics at exabyte scale?

**Answer:** The system has three phases � logical delete, retention enforcement, and physical purge: (1) **Logical delete**: when a user moves to trash, the file's lifecycle_state changes to TRASHED, and trashed_at is set. The file is still fully accessible (its chunks are not deleted). The metadata is moved to a "Trash" collection partitioned by user � this is a Spanner mutation, O(1). (2) **Retention enforcement**: a daily batch job scans the lifecycle_state = TRASHED table, filtering for trashed_at < NOW() - retention_days. But before deleting, it checks against the legal hold index � if resource_id appears in legal_holds (case_id, resource_id, hold_expiry), the file is excluded from deletion. The legal hold check uses a Bloom filter indexed by resource_id (10% false positive rate is acceptable � better to retain one extra file than delete one under hold). (3) **Physical purge**: after passing retention check, the file's chunk references are collected into a purge batch. The batch marks chunks for GC in the blob store. Chunks shared by multiple files (via deduplication) are only deleted when the reference count drops to zero. This is handled by a reference-counting service that decrements on each file deletion and triggers physical deletion when count = 0. (4) **Delete from trash**: when a user explicitly deletes from trash, the file bypasses the retention wait � it's immediately queued for purge. But legal holds still override. (5) **eDiscovery/legal hold override**: holds are stored in a separate CRDT-replicated global store. Before any purge, the system checks the hold index. Holds can be infinite or time-bound. The enforcement is checked at two points: (a) during the batch job, (b) at the chunk GC layer as a safety net. (6) **Scale handling**: at exabyte scale, the batch job is a MapReduce (Dataflow) running on the Spanner export, partitioned by user hash. The physical purge is throttled to avoid saturating the blob store's delete API � typically 10K chunk deletions per second per blob store shard.

## Q17: Google Drive File Stream (now Google Drive for Desktop) presents Drive files as a mounted volume without downloading everything. Design the caching vs. streaming trade-off: what granularity do you fetch (whole file, chunk, or stride-predicted chunks)? How do you handle the case where a user launches a 10 GB Photoshop file and immediately seeks to the last 5%?

**Answer:** The core design decision is fetch granularity. Options: (1) **Whole-file fetch** � simple but wastes bandwidth if the user only opens the file and views it once. (2) **Fixed-chunk fetch** � fetch 4 MB chunks on demand, similar to a virtual file system. (3) **Stride-predicted pre-fetch** � observe access patterns (Photoshop often reads header -> index -> specific layers) and pre-fetch chunks based on learned access stride. Google Drive File Stream uses a hybrid: on-demand chunk fetch with aggressive local caching and read-ahead. For the 10 GB PSD case: the file's chunk manifest is fetched first (a small metadata payload listing all chunk hashes). When Photoshop requests a read at byte offset 9.5 GB, File Stream computes which chunk contains that offset, checks the local cache (NVMe SSD cache, typically 20-40 GB), and if missing, fetches only that chunk (4 MB) from the cloud. It also prefetches adjacent chunks opportunistically � if the user accessed offset 9.5 GB, it's likely they'll continue reading forward. The cache eviction policy is ARC (Adaptive Replacement Cache): it balances recency and frequency. Files accessed once and never again are evicted first. Frequently accessed files have their full set of chunks cached locally. The **caching vs. streaming tension**: caching all chunks locally provides the best latency but consumes disk space and initial sync time. Purely streaming each request introduces network latency on every chunk miss. The middle ground: the system maintains a **pin count** per file � if a file is opened by an application, the chunks currently in flight are pinned in cache. When the application closes the file, the pin is released and chunks are eligible for eviction. For the seek-to-last-5% case: the seek triggers a new chunk request. If the file has been opened before, the local cache may still have that chunk (if not evicted). If not, it's a single 4 MB fetch � at 50 MB/s cloud read speed, that's ~80 ms latency to the user. Acceptable for random access in a large file.

## Q18: Design the enterprise audit logging system for Google Drive. You must log every file access (view, edit, download, print, share) for 10M+ users and support queries like "show me all files downloaded by user X in the last 90 days" in under 5 seconds. How do you handle the write volume (10M events/min) and the read latency requirements?

**Answer:** The audit logging pipeline has two paths � real-time alerting and retrospective querying: (1) **Write path**: every Drive action generates a Cloud Audit Log event with {actor, action, resource_id, timestamp, ip_address, user_agent, org_unit}. These are pushed to a Pub/Sub topic partitioned by org_unit for locality. A Dataflow streaming pipeline consumes events and writes them to two sinks: (a) **BigQuery** � partitioned by _PARTITIONDATE (daily) and clustered by (actor, resource_id). Daily partition pruning gives us 90% query cost reduction. (b) **Elasticsearch** � for real-time dashboards and alerting with sub-second latency. (2) **Read path for "show me files downloaded by user X in 90 days"**: query BigQuery: SELECT resource_id, timestamp, action FROM audit_logs WHERE actor = 'X' AND action = 'download' AND _PARTITIONDATE >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY) ORDER BY timestamp DESC. Partition pruning + clustering on actor makes this a scan of ~90 partitions x ~1000 rows per partition = 90K rows, taking 2-3 seconds. (3) **Volume handling**: 10M events/min = 166K writes/sec. BigQuery's streaming insert supports 100K rows/sec per table by default. Mitigation: shard by writing to multiple tables (one per org unit) and use a union query or application-level fan-out. Alternatively, use a time-series columnar store like ClickHouse which handles 1M inserts/sec. (4) **Retention**: raw logs retained 90 days (Enterprise), 180 days (Enterprise Plus), aggregated summaries retained 7 years. Aggregation: hourly rollups (actor, action, count, distinct_resources) stored in a separate BigQuery table for long-term trend analysis. (5) **Alerting path**: real-time rules run on the Dataflow stream (e.g., "user downloads 1000 files in 5 minutes" triggers an anomaly alert to the admin). (6) **Compliance export**: admins can export logs to an external SIEM (Splunk, Sumo Logic) via a nightly export job or a direct Pub/Sub sink. The export is in CEF (Common Event Format) for SIEM compatibility.

## Q19: Google Drive allows third-party apps (e.g., DocuSign, Slack) to access files via OAuth 2.0 scopes. Design the permission scoping system that lets a user grant "read access only to files that Slack has been specifically opened from" while preventing the app from enumerating the entire Drive. How do you handle scope creep and token rotation?

**Answer:** This is the per-window or per-file consent model, analogous to "Select a file to upload" dialog vs. "Full Drive access." (1) **Scope levels**: Google Drive API defines granular scopes: https://www.googleapis.com/auth/drive.file � access only to files created or opened by the app (narrowest). https://www.googleapis.com/auth/drive.readonly � read access to all files. https://www.googleapis.com/auth/drive.metadata.readonly � metadata only. The key mechanism is scoped tokens with resource binding. (2) **Resource-bound access tokens**: when a user authorizes Slack to access a specific file (via Drive Picker), the OAuth token is issued with a resource_id constraint embedded in the token. The token's introspection response includes {scope: "drive.file", resource_ids: ["file:abc123"]}. The Drive API checks at request time: if the token has resource-bound scopes, the requested file_id must be in the allowed list. This prevents the app from listing the user's /drive/files since that endpoint would return files not in the list. (3) **Scope creep prevention**: each scope upgrade (e.g., from drive.file to drive.readonly) requires re-consent. Google's OAuth consent screen shows a diff of new scopes. The system monitors scope expansion requests � if an app requests scope upgrade within 24 hours of initial consent, it's flagged for manual review. (4) **Token rotation and expiry**: access tokens expire in 1 hour. Refresh tokens can be revoked by the user at any time from the "Third-party apps" settings page. Google also enforces token binding � the refresh token is bound to the device/channel that performed the initial auth. If a refresh attempt comes from a different IP/user-agent, the token is rejected and the user must re-authenticate. (5) **Audit of app activity**: all API calls made on behalf of third-party apps are logged with app_id in the audit trail. If an app with drive.file scope suddenly makes 10K API calls per hour, anomaly detection flags it for possible abuse. (6) **App verification**: before requesting sensitive scopes (full Drive access), the app must go through Google's OAuth verification process, including a security assessment and penetration test. Unverified apps get a warning screen to users.

## Q20: Google operates Drive data centers globally. Design the data residency compliance system that ensures files owned by an EU-based enterprise customer are never stored outside the EU (including replicas, backups, and CDN caches). How do you handle cross-region collaboration where an EU user shares a file with a US user?

**Answer:** Data residency enforcement must cover the full data lifecycle � primary storage, replicas, backups, caches, and CDN: (1) **Data classification at upload**: when a file is uploaded, the metadata is tagged with a data_residency label derived from the user's domain (for Google Workspace, the admin sets a data region policy per organizational unit). The tag is stored in the file metadata and propagated to all replicas. (2) **Storage placement**: the upload service reads the user's residency policy and generates presigned URLs pointing only to blob storage buckets in the allowed region (e.g., eu-multiregion for EU data). The policy is enforced at the bucket level via IAM � the uploader cannot write to a bucket in the US if the policy requires EU. (3) **Replica placement**: synchronous replicas must be within the same region (e.g., EU-WEST-1 and EU-WEST-2). Cross-region async replicas are prohibited for restricted data. The blob storage layer's replication configuration is per-bucket, and the residency tag controls which bucket is selected. (4) **Backups**: backup copies must also reside in the region. Cold storage archives (e.g., Google Cloud Storage Archive class in EU-WEST-1) are used. The backup pipeline explicitly checks the residency tag before selecting the backup destination. (5) **CDN and edge caching**: CDN caches are a problem � a cache node in Virginia (US) could serve a chunk to a US user even if the data is EU-restricted. Solution: use signed URLs with a region constraint � the authorization token includes the user's region, and the CDN edge validates the token: if a US-based user requests an EU-restricted file, the CDN must fetch from the EU origin (not serve from a US edge cache). This adds ~80 ms latency for US users accessing EU data. (6) **Cross-region sharing**: when an EU user shares a file with a US user, the US user can read it, but the data stays in EU. The sharing system does not replicate the data to the US. The US user reads from the EU storage � the latency is acceptable (100-200 ms for static file download). The tricky case is Google Docs collaboration � if the EU user shares a Doc with the US user and they collab in real-time, the OT operations flow through the closest frontend server, but the authoritative document state stays in the EU region. The OT server in the US proxies operations to the EU leader. This adds 30-50 ms of latency per keystroke, which is noticeable but acceptable for enterprise compliance. (7) **Audit and enforcement**: a compliance scanner runs daily, checking that 100% of chunks for region-restricted files are located in the correct geography. Violations trigger an immediate investigation and manual remediation. The scanner queries the chunk location index (a Spanner table mapping chunk_id -> bucket_region) and cross-references with the file metadata's residency tag.

## Q21: Design a differential sync protocol for Google Drive that must handle concurrent edits from two online devices on the same file. Compare it against OT-based (Google Docs) and CRDT-based (collaborative editing) approaches. Under what real-world failure modes does each approach break at Drive scale?

**Answer:** Differential sync operates at the block level — file is split into fixed-size blocks (e.g., 4 MB). Each device computes SHA-256 hashes and sends them to the server. The server compares against the canonical block index and accepts only changed blocks. Failure modes: (1) **Concurrent block collision** — two devices edit the same 4 MB block concurrently; the last block upload wins, silently overwriting the other's changes. Drive stores the losing block in a "conflicted copy," but the user must merge manually — data loss is possible if the user ignores the conflict. (2) **Non-deterministic block boundaries** — if the file format changes (e.g., a PDF saved by two different PDF writers), the block boundaries may shift, causing the entire file to appear changed even when only metadata differs. This forces a full re-upload. (3) **Atomicity gap** — partial block upload failure leaves the file in an inconsistent state; Drive uses a manifest-based approach where the revision manifest is only committed after all new blocks are confirmed, preventing partial updates.

**Operational Transform (OT)**: used by Google Docs for character-level collaboration. Each operation (insert "hello" at position 5) is transformed against concurrent ops via a central server. Failure modes at Drive scale: (1) **OT only works for structured documents** — it cannot handle binary files (images, videos, executables) because there is no "character" granularity. (2) **Central server bottleneck** — OT requires a single serialization point for transformation, limiting throughput to ~10K ops/sec per document. For a shared Drive folder with 100 concurrent editors on different files, OT would require separate sessions per file, exploding state management. (3) **O(n²) complexity** — transforming N concurrent operations requires O(N²) comparisons. At N=1000, transformation latency spikes to seconds, causing perceived UI freezes. Google has experienced OT-related document corruption incidents when concurrent formatting and deletion overlap.

**CRDTs**: Each peer applies commutative operations without a central server. Failure modes at Drive scale: (1) **Metadata explosion** — a 10 MB text document using CRDTs (e.g., Automerge, Yjs) carries 100s of MB of metadata (unique IDs per character). For a 10 GB binary file, CRDT metadata would be terabytes. (2) **Garbage collection** — CRDT tombstones (deleted characters) must be cleaned up eventually. At Drive scale with 10B+ revisions, tombstone GC becomes a major operational burden. (3) **Conflict resolution ambiguity** — CRDTs guarantee convergence but not sensible merge results. When two users reorder paragraphs concurrently, the CRDT may produce a nonsensical ordering that neither user intended.

**Design decision**: Drive uses differential sync for blob storage (its primary use case) and delegates to OT/CRDT for collaborative editing (Docs, Sheets). The separation is clean: Drive stores opaque blobs with a manifest; Docs handles character-level ops. Any attempt to merge the two layers (e.g., using CRDTs for binary files directly) would fail due to metadata and performance constraints.

## Q22: Google Drive's storage tier automation must decide when to move a file from hot (SSD) to warm (HDD) to cold (archive). Design the machine learning model that predicts the optimal tier for each file at each point in time. What features, model architecture, and business constraints do you use, and how do you prevent thrashing (file getting promoted/demoted repeatedly)?

**Answer:** The tiering model is a **reinforcement learning or time-series classifier** that predicts the probability the file will be accessed in the next N days. Architecture: (1) **Feature engineering** — per-file features: `days_since_last_access`, `access_frequency_7d`, `access_frequency_30d`, `access_frequency_90d`, `file_size`, `file_type`, `owner_department`, `is_shared_drive`, `num_shares`, `num_editors_7d`, `num_downloads_7d`, `is_pinned_by_user`, `is_in_trash`, `holiday_indicator` (files accessed seasonally, e.g., tax documents in April). Aggregate features: `avg_access_frequency_similar_files` (files of same type in same org). (2) **Model architecture** — use a gradient-boosted decision tree (LightGBM) with a survival-analysis-style output: `P(t >= T)` where T is the time to next access. A separate classifier predicts `P(access_in_next_30_days)`. The model is retrained weekly on the full access log from BigQuery. (3) **Business constraints** — the tiering decision is not purely ML-driven. Hard rules override: if the file is on legal hold, it stays in hot storage. If the file is accessed by a VIP user (CEO, legal team), it is pinned to hot. If the file is part of an active project folder (any file in the folder accessed in the last 7 days), all files in that folder are promoted to warm at minimum. (4) **Thrashing prevention** — (a) **Cooldown period**: after a promotion, the file cannot be demoted for 7 days regardless of model output. (b) **Hysteresis threshold**: the promotion threshold (P > 0.7) is higher than the demotion threshold (P < 0.3). If P = 0.5, the file stays in its current tier. (c) **Look-ahead filter**: before demoting, check if the file is likely to be accessed in the next 7 days based on the upcoming calendar (e.g., a report scheduled for next Monday). (d) **Batch evaluation**: tiering decisions are made daily in a batch job, not per-access. This avoids the model overreacting to a single access event. (5) **Cost-aware optimization** — the model's loss function includes a cost term: `loss = cross_entropy + lambda * (storage_cost(saved) + retrieval_cost(access))`. If promoting a file costs $0.01/month extra but prevents a $0.50 retrieval penalty on cold access, the model learns to promote aggressively for files with high access probability.

## Q23: A Google Shared Drive folder has 100,000 members with complex group inheritance (group A contains groups B and C, each with different roles). Design the permission evaluation path that can answer "Can user U access file F?" in under 50 ms P99. Address the challenge of nested group resolution, ACL breadth, and cache invalidation when membership changes.

**Answer:** The critical insight is that **eager flattening** (precomputing `(file_id, user_id) -> role`) is the only way to meet 50 ms P99 for a 100K-member folder. Design: (1) **Flattened ACL table** — store in Spanner: `acls (file_id, user_id, role, version)` with an interleaved index. For a folder with 100K members, this table has 100K rows. A query `SELECT role FROM acls WHERE file_id = ? AND user_id = ?` is a single point lookup — sub-millisecond. (2) **Group flattening** — when a group is added to an ACL, a background worker resolves the group's transitive membership and writes individual rows. Implementation: `resolve_group(group_id)` returns all nested user IDs via a recursive CTE in the group membership DB (PostgreSQL with `WITH RECURSIVE`). The flattened rows are written to Spanner in a batch transaction. (3) **Incremental re-flattening** — when group membership changes (user added/removed), the group membership service publishes a `membership_changed:{group_id}` event to Pub/Sub. A worker reads affected ACLs (all file_ids where this group appears in the ACL) and re-flattens. Debounce: if 10 membership changes happen within 60 seconds for the same group, batch them into one re-flattening cycle. (4) **Caching** — the flattened ACL is cached in Redis with key `perm:{file_id}:{user_id}` and 60-second TTL. On cache miss, query Spanner. On permission change, publish an invalidation event to a Redis pub/sub channel; each API server subscribes and evicts the relevant keys. For 100K-user folders, broad invalidation (when the folder's ACL changes) could invalidate 100K keys. Mitigation: use versioned cache keys — `perm:{file_id}:{user_id}:{acl_version}`. The acl_version is stored on the file metadata (cached separately). Increment the acl_version on ACL change instead of bulk-invalidating individual keys. Old keys naturally expire via TTL. (5) **Group nesting depth** — limit to 10 levels of group nesting (A contains B contains C ...). Beyond that, deny the ACL change with a "too deep" error. This prevents computationally explosive group resolution. (6) **Consistency** — the flattened ACL is eventually consistent with group membership (up to 60-second lag). For strong consistency on sensitive files (legal, HR), the system falls back to the on-the-fly evaluation path (walk ancestor chain, resolve groups per ACL entry), which may take 200-500ms but is always correct.

## Q24: Design the legal hold enforcement system for Google Drive. A legal hold must preserve all versions of a file indefinitely, even if the user deletes it from trash or the retention policy would purge it. How do you implement holds across revisions, prevent deletion at every layer (UI, API, GCS lifecycle, backup restore), and handle the intersection of multiple holds on the same file?

Each hold targets a resource (file, folder, or org unit). Folder holds apply recursively to all children. (2) **Hold intersection** — a file can be under multiple holds (eDiscovery hold + regulatory hold + litigation hold). The effective hold is the UNION of all holds: the file is preserved until ALL holds are released. Implementation: a `holds_per_resource` materialized view: `resource_id -> Set<hold_id>`. (e) **Backup restore** — if a system administrator restores from a backup that predates the hold, the restored files must also be under hold. (c) For files in trash that were held, the release may immediately schedule them for physical deletion (if they were past the normal retention). (d) For GCS lifecycle tags, the tag is removed in a background batch. (6) **Audit trail** — every hold action (create, modify, release) is logged in the audit system with `{who, what, when, why, case_id}`. The log is immutable and stored in a separate BigQuery table with 7-year retention.

## Q25: Google Drive File Stream (desktop) presents cloud files as a local mounted volume. Design the read-ahead and caching strategy for a user who opens a 10 GB Photoshop file and seeks to the last 5%. How do you decide which chunks to pre-fetch locally, and how do you balance network bandwidth, local SSD wear, and user-perceived latency?

**Answer:** The core challenge is predicting access patterns without application-specific knowledge. Google Drive File Stream uses a **stride-based prefetcher** with adaptive window sizing: (1) **Chunk granularity** — files are divided into 4 MB chunks. The chunk manifest (list of all chunk hashes) is fetched eagerly when the file is opened — this is a small metadata request (< 100 KB for a 10 GB file). (2) **On-demand fetch** — when Photoshop requests a read at byte offset 9.5 GB, File Stream computes `chunk_index = floor(9.5 GB / 4 MB) = 2432`. It checks the local cache (NVMe SSD, up to 40 GB). On miss, it fetches chunk 2432 from the cloud via a single HTTP range request. (3) **Stride detection** — the prefetcher maintains a sliding window of the last 10 chunk access patterns. If the user accesses chunk 2432, then 2433, then 2434 (sequential stride +1), the prefetcher predicts the next 10 chunks and fetches them in parallel. If the user jumps to chunk 500 (random access), the prefetcher resets and only fetches the requested chunk. (4) **Adaptive window** — the prefetch window size adjusts based on network bandwidth: if measured throughput > 50 MB/s (likely local network or fast internet), prefetch 20 chunks ahead (80 MB). If throughput < 5 MB/s (slow connection), prefetch only 2 chunks ahead (8 MB) to avoid wasting bandwidth on unused data. (5) **Cache eviction** — uses ARC (Adaptive Replacement Cache) with two pools: recent (recency-biased) and frequent (frequency-biased). Chunks for the 10 GB PSD file that are accessed once during a single session are evicted quickly. Chunks that are re-read (e.g., Photoshop reads the same layer data multiple times) are promoted to the frequent pool. The cache is scanned every 5 minutes for eviction candidates. (6) **SSD wear management** — the local cache is stored on the user's NVMe SSD, which has limited write endurance (e.g., 600 TBW for a 1 TB SSD). File Stream limits the write rate to the cache: max 10 GB/day of new cache writes. If the user routinely opens large files, the cache may become write-throttled, falling back to streaming from the cloud with local LRU eviction. (7) **User-perceived latency** — the critical metric is "time to first byte" for a seek operation. On a cache miss, the 4 MB chunk fetch over a typical 50 Mbps connection takes ~650 ms. The file system presents a "busy" indicator during this time. To mask latency, File Stream returns zero-filled data for the first 64 KB while the chunk is being fetched (optimistic zero-fill), then overwrites with real data once the chunk arrives. This makes interactive apps feel responsive.

## Q26: Google Drive's enterprise audit logging system must handle 10 million events per minute and support queries like "show me all files downloaded by user X in the last 90 days" in under 5 seconds. Design the storage and query pipeline that meets both the write throughput and the read latency requirements.

**Answer:** The audit pipeline uses a **lambda architecture** with separate write-optimized and read-optimized paths: (1) **Write path** — each Drive action (view, edit, download, print, share) generates an audit event: `{actor, action, resource_id, timestamp, ip, user_agent, org_unit_id}`. Events are published to Pub/Sub (partitioned by org_unit_id for locality). A Dataflow streaming pipeline consumes events and writes to two sinks simultaneously: (a) **BigQuery** — daily partitions, clustered by `(actor, resource_id)`. The table is `audit_logs$YYYYMMDD` with clustering on `actor`. Write throughput: BigQuery streaming inserts handle up to 100K rows/sec default. For 166K writes/sec (10M/min), shard across multiple tables (one per org_unit region) — 4 tables × 50K rows/sec each fits comfortably. (b) **Elasticsearch** — for real-time dashboards and alerting. Events are indexed with a 7-day retention. Elasticsearch supports 50K writes/sec per cluster node; use 4 data nodes. (2) **Read path for "files downloaded by user X in 90 days"** — the query is: `SELECT resource_id, timestamp FROM audit_logs WHERE actor = 'X' AND action = 'download' AND _PARTITIONDATE >= DATE_SUB(CURRENT_DATE(), INTERVAL 90 DAY)`. With BigQuery's partition pruning, this scans only 90 partitions (not the full table). With clustering on `actor`, each partition scan is a range scan on the clustered index — ~1000 rows per partition, 90K rows total. Query completes in 2-3 seconds. (3) **Optimization for common queries** — the most common audit query pattern is "show me recent activity for user X." Maintain a **hot index** in Redis: `user_activity:{actor_id}` is a sorted set of `(timestamp, action_summary)` with 7-day TTL. Recent queries hit Redis (sub-millisecond) instead of BigQuery. Only queries spanning > 7 days hit BigQuery. (4) **Retention tiering** — raw events in BigQuery: 90 days (Enterprise), 180 days (Enterprise Plus). After the raw retention window, events are aggregated into hourly rollups: `audit_hourly (actor, action, resource_count, hour)` — this is 1/3600th the data volume. Long-term queries (1 year+) query the aggregated table, which is 1000x smaller. (5) **Cost optimization** — BigQuery storage for 10M events/min × 90 days = 1.3 trillion rows. At $5/TB scanned, a query that scans 90 partitions × 10 GB per partition = 900 GB would cost $4.50 per query. Optimization: use **materialized views** pre-aggregated per (actor, day): `materialized_view = SELECT actor, action, ARRAY_AGG(resource_id) FROM audit_logs GROUP BY actor, action, _PARTITIONDATE`. Queries on this view scan only the aggregated rows (1M rows instead of 1.3T), reducing cost by 1000x and latency to < 2 seconds.

## Q27: Google Drive's third-party app OAuth scopes allow granular file access. Design the scoped token system that lets a user grant "read access only to files that Slack has been specifically opened from" while preventing the app from enumerating the entire Drive. How do you implement resource-bound tokens and handle scope creep detection?

**Answer:** The core mechanism is **resource-bound access tokens** with per-file consent granularity: (1) **Scope definition** — Drive defines three tiered scopes for third-party apps: (a) `drive.file` — access only to files created or explicitly opened by the app (narrowest). (b) `drive.metadata.readonly` — read file metadata only. (c) `drive.readonly` — full read access. The user-facing consent screen shows the scope in plain language: "Slack will be able to access only the files you explicitly open through Slack." (2) **Resource-bound token** — when the user authorizes Slack to access a specific file (via the Drive Picker UI), the OAuth token is issued with a **resource constraint** embedded in the token claims: `{scope: "drive.file", resource_ids: ["file:abc123", "file:def456"], exp: ..., sub: user_id, aud: slack_client_id}`. (3) **Token introspection** — the Drive API gateway, on every API call, validates: (a) The token's `resource_ids` must include the `file_id` in the request path or query parameter. (4) **Scope expansion detection** — when an app requests a scope upgrade (e.g., from `drive.file` to `drive.readonly`), the system shows a diff consent screen: "Slack currently has access to files you open. It is requesting access to ALL your files." The user must explicitly approve. Anomaly detection: if an app requests scope upgrade within 24 hours of initial consent, the request is flagged for manual review — this catches "bait and switch" patterns. (5) **Token rotation** — access tokens expire in 1 hour. Refresh tokens are bound to the initial device/user-agent via token binding (OAuth 2.0 Token Binding RFC 8473). If a refresh request comes from a different IP/user-agent than the original authorization, the refresh is denied and the user must re-consent. This prevents token theft across devices. (6) **Revocation** — the user can revoke an app's access from the Google Account "Third-party apps" page. Revocation invalidates all tokens for that app-user pair immediately via a blocklist in Redis. The blocklist is checked on every API call before token introspection. (7) **Audit** — all API calls made with resource-bound tokens are logged with `app_id`, `resource_id`, and `action` in the audit trail. If an app with `drive.file` scope suddenly reads 10K files per hour (suggesting it's iterating over resource_ids), anomaly detection flags it. The app is rate-limited and an alert is sent to the user: "Slack has accessed 10,000 of your files in the last hour. Review activity."

## Q29: Google Drive API serves millions of developers with per-user, per-method rate limits. Design the multi-tenant rate limiting system that enforces quotas for 10M+ API keys across thousands of methods, while keeping the authorization path under 5 ms P99 latency overhead.

## Q30: You are tasked with reducing Google Drive's cold read latency (time to open a file not accessed in 90+ days, stored on cold storage) from 5 minutes to under 30 seconds without increasing storage cost by more than 5%. Design the multi-tier caching and prefetching strategy.

**Answer:** The 5-minute cold read latency comes from recalling data from cold storage (tape/optical) to hot storage. To reduce to 30 seconds, we need a prediction-based prefetching layer: (1) **Prediction model** — a logistic regression or gradient-boosted model predicts `P(access_in_next_24h)` for each cold file. Features: `days_since_last_access`, `file_type`, `file_size`, `num_editors`, `calendar_events` (upcoming meeting with this file attached), `pattern_match` (if the user has accessed similar files before a recurring meeting — e.g., always opens the board deck on Monday morning). The model runs daily for all files that were demoted to cold in the last 90 days. (2) **Prefetch tier** — a new "lukewarm" storage tier is introduced: cost = 2× cold, 0.5× hot, recall latency = 10 seconds (from HDD with erasure coding, not tape). When the prediction model scores a file > 0.7, the file is automatically promoted from cold to lukewarm overnight. The prediction window is 24 hours — if the file is actually accessed, it's already warm; if not, the promotion cost is 2× cold storage for 1 day before it can be demoted back. (3) **Prediction accuracy budget** — at 5% storage cost increase, we can afford to keep 2.5% of all cold-eligible files in lukewarm tier. If the model's precision is 50% (half of promoted files are actually accessed), we get a 50% cold-read improvement for 2.5% storage cost. Target: the model achieves 60% precision and 30% recall (30% of cold reads are eliminated). (4) **User-initiated prefetch** — the Drive UI adds a "Pin for fast access" button. When the user pins a file, the file is immediately promoted to hot storage (cost within pinned quota). The user can pin up to 10 files per account. This is free for the user but signals usage intent. (5) **Blind recall optimization** — for files not predicted but requested (cache miss), the cold recall pipeline is optimized: (a) The recall request is priority-queued (bypasses bulk retrieval). (b) The tape robot dispatches to the robotic arm immediately (interrupts current bulk job within 100ms). (c) The file is read from tape at ~80 MB/s (sequential). A 100 MB file is fully read in ~1.25 seconds. (d) The file is written to an SSD staging area (100 MB at 500 MB/s = 0.2 seconds). (e) The client's GET request, which was waiting with a streaming response, starts receiving data after total ~15 seconds (tape seek + read + staging). Combined with prediction (30% hit rate), P50 cold read drops to 15 seconds, P95 (prediction miss) stays at 30 seconds. (6) **Cost containment** — the lukewarm tier uses a dedicated pool of HDDs with erasure coding (4+2), not SSDs. This keeps the per-GB cost at ~1.4× cold (vs. 10× for SSD hot). The 5% storage cost increase is absorbed by the 30% reduction in cold reads, which saves compute and egress costs. The net TCO impact is zero or slightly negative.
