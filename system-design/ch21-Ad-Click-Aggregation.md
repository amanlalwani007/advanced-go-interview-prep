# Chapter 21 — Ad Click Event Aggregation

## Q1: Design a real-time ad click aggregation pipeline that counts clicks per ad per minute across 10 million clicks per minute. How do you handle the write throughput problem?

**Answer:** At 10M clicks/min = ~166K clicks/second, the write path is the bottleneck. Architecture: (1) **Event ingestion** — clicks arrive as HTTP requests at a global API gateway. Each click event is `{ad_id, user_id, timestamp, ip, user_agent}`. The gateway publishes raw events to Kafka partitioned by `ad_id`. (2) **Stream processing** — a Flink/Spark Streaming job reads from Kafka with a 1-minute tumbling window, aggregating `count(*), sum(revenue)` per `ad_id` per minute. The aggregation is stored in a wide-column store (Cassandra or ScyllaDB) keyed by `(ad_id, minute_timestamp)`. (3) **Handling the write throughput** — the stream processor batches writes to the DB. Instead of one write per click, it writes one aggregated row per `(ad_id, minute)` — reducing write load from 166K/s to roughly (#unique ad_ids per minute) / (60 seconds). If there are 1M active ad_ids, that's ~16K writes/sec, which a well-tuned ScyllaDB cluster can handle. (4) **Read path** — dashboards query the aggregated table by `ad_id` and `timestamp` range. A Redis cache in front of the DB serves the latest minute's data for real-time dashboards.

## Q2: How do you ensure exactly-once counting of ad clicks in a distributed system where users might double-click or the browser might retry?

**Answer:** Exactly-once in ad counting has two concerns: (1) **System-level dedup** — the stream processor (Flink) uses exactly-once semantics via checkpointing and idempotent sinks. If a failure occurs, the processor restarts from the last checkpoint and replays the same events — the sink (e.g., Cassandra) uses upserts: `INSERT INTO ad_clicks (ad_id, minute, count) VALUES (?, ?, 1) ON CONFLICT (ad_id, minute) DO UPDATE SET count = count + 1`. This makes replay safe. (2) **Business-level dedup** — a single user clicking the same ad twice in 5 minutes might be two genuine clicks or a double-click. The ad platform must decide: treat both as valid (count both, charge the advertiser twice) or deduplicate (count only the first). This is a business decision, not a technical one. If deduplication is required, the stream processor maintains a sliding window (5 minutes) of `(user_id, ad_id)` pairs in a Redis set, filtering out duplicates before aggregation.

## Q3: Design the anti-fraud layer. How do you detect and filter click fraud (bots clicking ads, competitors draining budgets) in real-time?

**Answer:** A multi-layered fraud detection pipeline: (1) **IP-based filtering** — maintain a blocklist of known bot IPs (from shared blacklists). Also detect a single IP clicking > 10 unique ads per minute → flag for manual review. (2) **User-agent analysis** — headless browsers, missing user-agent, or known bot signatures → filter. (3) **Behavioral analysis** — compute per-user click patterns: time between clicks (too consistent → bot), scroll activity before click (no scroll → suspicious), mouse movement patterns. (4) **Device fingerprint** — detect virtual machines, emulators, or datacenter IP ranges. (5) **Rate-of-change detection** — if an ad's click-through rate jumps from 0.1% to 50% in 5 minutes, it's likely fraudulent. (6) **Machine learning model** — a real-time ML model (e.g., XGBoost) scores each click with a fraud probability. Clicks above a threshold (e.g., 0.95) are rejected before aggregation; those between 0.7-0.95 are accepted but marked for manual review. The model is trained on historical labeled data (confirmed fraud vs. legitimate clicks). At 166K clicks/sec, the fraud detection pipeline must process each event in <1 ms using a sidecar service with the model loaded in memory.

## Q4: How would you design the attribution system — "which ad campaign led to this conversion"? Handle the case where a user sees an ad, doesn't click, but converts later (view-through attribution)?

**Answer:** Attribution models: (1) **Last-click attribution** — the last ad clicked before conversion gets 100% credit. Simplest, but ignores upper-funnel influence. (2) **View-through attribution** — if a user views an ad (impression) but doesn't click, and converts within a time window (e.g., 7 days), the impression gets some credit. Implementation: maintain a per-user impression log in Redis: `{user_id: [{ad_id, timestamp, campaign_id}]}` with a TTL equal to the attribution window. When a conversion event arrives, look up the user's recent impressions. Credit is distributed using a time-decay model (impressions closer to conversion get more weight) or a linear model (equal credit to all touchpoints). (3) **Multi-touch attribution** — a ML model (Shapley value or Markov chain) assigns fractional credit to each touchpoint across the user journey. Storage: the attribution result is written to an OLAP store (ClickHouse) for reporting, with rows like `{user_id, ad_id, campaign_id, conversion_id, attribution_value, attribution_model}`. The attribution pipeline is a batch job (hourly) rather than real-time, because it's computationally expensive.

## Q5: Design the real-time bidding (RTB) system. An ad request comes in, and you have 100 ms to decide which ad to show. How does the system evaluate 10,000 potential ads in that time?

**Answer:** The RTB decision is a constraint satisfaction + optimization problem solved in <100 ms. Architecture: (1) **Pre-filtering** — on each ad request (`{user_id, page_url, device_type, location}`), the RTB server first filters the ad corpus using inverted indexes in memory: only ads whose targeting criteria match. This typically reduces 10M ads to 1K-10K candidates. (2) **Budget check** — each candidate's remaining daily budget is checked in a Redis cache (sharded by ad campaign ID). Ads with exhausted budgets are removed. (3) **Frequency capping** — check if the user has seen this ad too many times already. Redis sorted sets keyed by `(user_id, ad_id)` track impression counts. (4) **Pacing** — some campaigns want to spread their budget evenly across the day. The pacing controller adjusts the probability of selecting a given campaign based on budget consumption rate. (5) **CTR prediction** — a lightweight ML model (logistic regression or FTRL-Proximal) predicts the probability the user will click on each remaining ad. Features: user demographics, page category, ad category, time of day, device. (6) **Ranking** — rank candidates by `expected_revenue = bid_price * predicted_ctr`. Select the top ad. (7) **Execution** — all these steps happen in a single-threaded event loop on a high-performance server (e.g., C++ or Java with Netty) with all data structures in memory. The 100 ms window includes network round-trips, so the actual computation budget is ~50 ms.

## Q6: Design the attribution window — a user clicks an ad on their phone, then buys the product on their laptop 2 days later. How do you connect these two events as a single conversion funnel?

**Answer:** Cross-device attribution requires user identity resolution. (1) **Logged-in users** — if the user is logged in (Facebook, Google), the `user_id` is consistent across devices. The click event on phone and conversion on laptop both carry the same `user_id`. Trivially joinable. (2) **Deterministic matching** — for non-logged-in scenarios, match by email hash, phone number, or credit card token (at checkout time, the advertiser knows the user's identity). (3) **Probabilistic matching** — Google/Facebook use device graphs: they observe two devices using the same IP address, similar browsing patterns, and overlapping app installs to probabilistically link them. The probabilistic match is a score (0-100). Conversions are attributed only if the match score exceeds a threshold. (4) **User ID graph** — a dedicated identity service (e.g., Tapad, Liveramp) maintains a graph of device IDs → canonical user ID. This graph is built from deterministic + probabilistic signals and is stored as a sparse adjacency matrix in a graph DB (e.g., DGraph or Neo4j) with billions of edges. (5) **Time window** — even with solved identity, the click event might be 2+ days old. The attribution system maintains a 30-day sliding window of click events per user. When a conversion arrives, the system looks back up to 30 days for the most recent click from the same user on the target advertiser's ads.

## Q7: How do you handle the data volume of storing raw click events and impressions for years (regulatory compliance requires 2+ year retention)?

**Answer:** Tiered storage strategy: (1) **Hot tier (7 days)** — raw events in Kafka (configured 7-day retention) plus aggregated data in Cassandra/ScyllaDB for real-time dashboards. (2) **Warm tier (30-90 days)** — aggregated data only, in ScyllaDB or ClickHouse. Raw events are discarded. (3) **Cold tier (2+ years)** — aggregated data compressed and stored in object storage (S3/GCS) in Parquet format using Apache Hive/Spark partitioning by `(year, month, day)`. Queries on cold data are run via Presto/Trino or Athena. (4) **Cost optimization** — raw click events (unaggregated) are the biggest storage consumer. For regulatory compliance, you only need to be able to prove the total count was accurate, not replay individual events from 2 years ago. Store just the 1-minute rollups in the cold tier: `{ad_id, minute_ts, impressions, clicks, unique_users, revenue}` — this reduces storage by 1000× compared to raw events. (5) **Data pipeline** — a scheduled Spark job (daily) reads from Kafka cold storage (if needed) or the aggregated DB, transforms to Parquet, and writes to S3 partitioned by date. An Hive/Trino table on top of this Parquet data makes it queryable via SQL.

## Q8: Design the real-time dashboard that shows ad performance metrics (impressions, clicks, CTR, spend, CPA) with <5 second latency for the last 60 minutes of data.

**Answer:** For sub-5 second latency on 60 minutes of data: (1) **Streaming aggregation** — the Flink job writes 1-minute rolled-up data to both the long-term store (for history) and a fast Redis store keyed by `dashboard:{ad_id}:{yyyy-mm-dd:HH:MM}`. (2) **Dashboard query** — the frontend requests `GET /api/ad/{ad_id}/realtime?window=60m`. The API server fans out to: (a) Redis — fetch the latest 60 Redis keys (60 minutes × ad_id) — takes <2 ms. (b) The long-term store (ClickHouse) — for comparison data (e.g., same hour yesterday). (3) **Client-side** — the dashboard establishes a WebSocket connection to a push service. The Flink job publishes aggregated results to a Kafka topic consumed by a WebSocket relay that pushes the latest 1-minute bucket to all connected dashboards. This avoids polling. (4) **Refresh mechanism** — the WebSocket push keeps the displayed data real-time. Every new 1-minute bucket, the server pushes the update. The client appends it to the chart and removes the oldest bucket. (5) **Scalability** — each dashboard panel needs at most ~60 data points. The API server can shard by `ad_id` and cache results in Redis with a 5-second TTL. For 10K concurrent dashboard viewers, Redis handles this trivially.

## Q9: How do you design the budget pacing system? An advertiser sets a $10,000 daily budget, and the system should spend it evenly across 24 hours without spiking.

**Answer:** The pacing system ensures a campaign's budget is consumed at a target rate. (1) **Target rate** — $10,000 / 24 hours = $416.67/hour = ~$6.94/minute. (2) **Real-time budget tracking** — after each ad impression, the RTB system decrements the budget in Redis: `DECRBY campaign:1234:budget_spent 0.05`. (3) **Pacing PID controller** — the system uses a feedback loop. At each decision point, compute: `error = (expected_spent - actual_spent) / expected_spent`. If actual spent is below expected, increase the probability of selecting this campaign's ads (higher bid price multiplier). If above, decrease probability. (4) **Pacing formula** — `bid_modifier = base_bid × (1 + kP × error + kI × integral_error + kD × derivative_error)`. The PID constants are tuned per campaign. (5) **Spike prevention** — the system holds back some budget early in the day to account for traffic spikes. A common heuristic: do not spend more than 1.5× the target pace in any hour. (6) **Budget cap enforcement** — if `budget_spent >= daily_budget`, the campaign is paused (removed from all RTB consideration). This is checked before every ad selection.

## Q10: How does the system handle the "ad verification" requirement — ensuring ads appear on brand-safe content and are viewable by real users?

**Answer:** Ad verification is a parallel pipeline: (1) **Pre-bid verification** — before serving an ad, the RTB system checks the page URL against a blocklist (pornography, hate speech, illegal content). This uses a real-time URL categorization service (a classifier API returning the content category in <10 ms). (2) **Viewability measurement** — the ad creative includes a JavaScript tag (from IAS, DoubleVerify, or MOAT) that measures: what percentage of the ad was visible on screen, for how long, and whether the user was active (scrolling, clicking, typing). The MRC (Media Rating Council) standard: 50% of pixels visible for 1 continuous second (display) or 2 seconds (video). (3) **Post-bid verification** — the verification vendor's measurement data is sent back as a separate event stream. The ad server joins this with the original impression event to compute the "viewable impression" rate. (4) **Invalid traffic (IVT) filtering** — the verification vendor also detects sophisticated bots using: headless browser detection, mouse movement analysis, JavaScript engine fingerprinting, and canvas fingerprinting. Impressions flagged as IVT are excluded from the advertiser's bill. (5) **Reconciliation** — the advertiser is billed only for viewable, brand-safe, human impressions. The verification pipeline runs as a batch job (hourly) that reconciles the raw server logs with the verification vendor's data and adjusts the billing accordingly.

## Q11: Walk through the full Real-Time Bidding (RTB) protocol flow — from a user visiting a publisher page to an ad appearing in the browser. What are the critical latency budgets at each hop?

**Answer:** The RTB flow involves 6+ network round-trips, all must complete in under 300 ms total to avoid degrading page load time. Step-by-step: (1) **User visits publisher page** — publisher's ad tag (typically a JavaScript snippet from Google Ad Manager or Prebid.js) executes in the browser. This sends an ad request to the publisher's ad server (SSP). (2) **SSP sends bid request** — the SSP formats a bid request (OpenRTB 2.6 JSON) containing: impression details (ad slot size `{width:300, height:250}`), site/page data, device info (user-agent, IP, device type), and optionally a user ID (cookie or device ID). This is sent to the ad exchange. Budget: ~50 ms from page load to bid request sent. (3) **Ad exchange → DSPs** — the ad exchange sends the bid request to multiple DSPs (Demand-Side Platforms) in parallel via HTTP POST. Each DSP has 100 ms to respond with a bid (the `nbr` — "no bid reason" if not bidding). DSPs hold their models in memory (CTR prediction, budget pacing, frequency capping) and compute a bid price. (4) **Ad exchange selects winner** — the exchange collects all bids, runs a second-price auction (winner pays the second-highest bid price, or a floor price). The exchange notifies the winning DSP via a "win notice" (separate HTTP request). Budget: ~20 ms. (5) **Ad exchange → SSP** — the exchange returns the winning ad markup (a VAST XML for video or a JavaScript creative URL for display) to the SSP. The SSP renders this in the publisher's page. Budget: ~50 ms. (6) **Impression tracking** — the ad creative includes a tracking pixel (1×1 GIF). When the ad renders, the browser loads this pixel, registering an impression event. The tracking server logs the event for billing and analytics. (7) **Critical latency constraints**: The entire RTB chain must complete before the browser's pageload timeout (~300 ms for above-the-fold ads). If it takes longer, the ad slot collapses and no ad is shown. Each DSP's 100 ms window includes network latency — so effective compute time per DSP is ~50 ms. At scale (100K QPS), the exchange must handle 100s of concurrent DSP connections. Tier 1 exchanges (Google, Amazon) use C++ or Go with epoll/kqueue, running on hundreds of servers across multiple regions.

## Q12: Design the SSP (Supply-Side Platform) architecture. How does it decide which ad exchange to send a given impression to, and how does it optimize for publisher revenue?

**Answer:** The SSP acts as a yield optimization layer for the publisher. Its core decision is: "which demand source (ad exchange, ad network, or direct advertiser) will pay the most for this impression?" Architecture: (1) **Demand source registry**: The SSP maintains a list of connected demand sources, each with a bidder adapter (a plugin that translates the SSP's internal bid request format into the specific format required by that exchange, e.g., OpenRTB for exchanges, custom API for direct deals). Each adapter is isolated in a lightweight sandbox (Go plugin or Lua script) to prevent a single adapter crash from taking down the entire SSP. (2) **Waterfall vs header bidding**: The SSP supports both strategies. **Waterfall** (legacy): query demand sources in order of expected CPM (highest first). If the highest doesn't bid, go to the next. Problem: latency adds up linearly, and lower-tier sources rarely get a chance. **Header bidding** (modern): query all demand sources simultaneously via the browser (Prebid.js) and let them compete in a unified auction. The SSP runs a server-side auction (preferred) or lets the browser decide (client-side). (3) **Unified auction**: The SSP collects all bids (both real-time bids from exchanges and guaranteed bids from direct advertisers). It runs a **unified price floor** — a minimum CPM set by the publisher. Ads below the floor are excluded. Then the SSP selects the highest bid > floor. (4) **Dynamic floor optimization**: The SSP dynamically adjusts price floors per impression based on historical data. If a particular publisher's tech audience consistently attracts $5 CPM, the floor is set to $3.50 to balance fill rate and revenue. An ML model predicts the optimal floor for each (publisher, ad_slot, device, time_of_day) combination. (5) **Real-time analytics**: The SSP exposes a real-time dashboard showing: fill rate (impressions filled / total requests), average CPM, revenue by demand source, and latency by demand source. If a demand source is consistently slow (>200 ms), the SSP degrades it — sends fewer requests to it or removes it from the auction for latency-sensitive pages. (6) **Scalability**: An SSP handles 1M+ QPS. Each bid request is CPU-intensive (serialization, network I/O, auction computation). Use a thread-per-core model (e.g., NGINX + Lua or Vert.x on Java). Cache bidder adapters in memory. Pre-compute per-request features (user geo, device type) and pass them to all adapters to avoid redundant computation.

## Q13: Design an ad fraud detection system using ML that operates in real-time on the bid stream. What features and model architecture do you use to catch sophisticated click farms and botnets?

**Answer:** Real-time fraud detection requires a lightweight model (inference <1 ms) on the bidding path, with a deeper model running offline for batch investigation. (1) **Real-time scoring model**: Architecture — a gradient-boosted decision tree (LightGBM or XGBoost) with 200 trees, depth 6, quantized to 8-bit integers for fast inference. Features: (a) **User-level**: click-to-impression ratio (CTR) over 1-hour and 24-hour windows; inter-click interval (mean and std); scroll depth before click (bots don't scroll); mouse movement entropy (human vs bot pattern). (b) **Device-level**: device fingerprint entropy (too many unique fingerprints from one IP → farm); browser language mismatch (en-US browser with non-US IP); headless browser detection (missing `navigator.webdriver` flag check). (c) **Network-level**: IP reputation score (from shared blocklist); ASN (datacenter ASNs flagged); reverse DNS lookup (hostname matching known cloud providers). (d) **Pattern-level**: time-of-day uniformity (bots click evenly across 24 hours; humans peak at certain hours); click velocity (#clicks/second from one IP). (2) **Model training**: Training data: labeled clicks from confirmed fraud investigations + human-reviewed samples. Class imbalance: 99.9% legitimate, 0.1% fraudulent. Use weighted loss (fraud weight = 1000) and SMOTE oversampling. Features are computed from a 7-day click history in an OLAP store (ClickHouse). (3) **Threshold management**: The real-time model outputs a fraud probability score (0-1). Clicks with score > 0.95 are blocked immediately (rejected at the bid request). Scores 0.7-0.95: allowed but flagged for investigation. Scores < 0.7: accepted. The threshold is tuned weekly via precision-recall analysis. (4) **Offline deep investigation**: A separate pipeline processes flagged clicks with a more expensive model (neural network with attention over user session sequences — "is this session's click pattern similar to known fraud sessions?"). This re-scoring runs hourly with a 15-minute batch window. (5) **Adversarial robustness**: Fraudsters evolve. The system measures "model drift" — if the distribution of fraud scores shifts significantly (e.g., mean score drops from 0.1 to 0.05), retrain the model. A retraining pipeline runs weekly with the latest 30 days of labeled data. (6) **Feedback loop**: When a fraudster is confirmed (e.g., by manual investigation), all their historical clicks are added to the training set, and their IP/device fingerprint is added to the blocklist within seconds.

## Q14: Explain the MRC (Media Rating Council) viewability measurement standard in detail. How does an ad platform instrument viewability tracking, and what are the technical challenges with measuring "50% of pixels for 1 continuous second"?

**Answer:** The MRC viewability standard (current as of 2024) defines a viewable display impression as: ≥50% of pixels in the viewport for ≥1 continuous second. For video: ≥50% of pixels for ≥2 continuous seconds. Implementation and challenges: (1) **Measurement technique**: The ad creative includes a JavaScript tag (from an accredited measurement vendor like IAS, DoubleVerify, or Moat) that: (a) uses the Intersection Observer API to detect when the ad element enters the viewport; (b) uses `requestAnimationFrame` to check pixel visibility at ~60fps; (c) tracks cumulative time the ad is ≥50% visible. When the threshold is met, the tag fires a "viewable impression" beacon. (2) **Technical challenges**: (a) **iframe hell**: Ads are typically served in cross-domain iframes. The measurement tag inside the iframe cannot access the parent page's scroll position or viewport size without the publisher's cooperation. Solution: the publisher must include a small SDK (e.g., Google's "gpt.js" or Prebid.js) that passes viewport data into the iframe via `postMessage`. If the publisher doesn't include the SDK, viewability cannot be measured — the ad is reported as "not measurable". (b) **Friendly iframe vs. cross-domain iframe**: Some SSPs use "friendly iframes" (same domain) where measurement can access the parent DOM directly. But cross-domain is more common for security — needing postMessage for viewport data. (c) **Single continuous second**: The ad must be visible for a full second without interruption. Quick scrolling past the ad (visible for 0.8 seconds) does NOT count. This requires careful timestamp tracking at the animation frame level. (d) **Bots and automation**: Some publishers cheat by having invisible iframes that report themselves as always viewable. Verification vendors detect this by: checking if the iframe is in a hidden DOM subtree; measuring the actual pixel count by comparing with the known ad size; and using the parent page's visibility API (if tab is backgrounded, ad cannot be viewable). (3) **Viewability rate optimization**: A low viewability rate (<50%) means ads are below the fold or the page loads slowly. Publishers optimize by: moving ad slots above the fold, reducing page load time (faster rendering = higher chance ad loads before user scrolls), and reducing the number of ad slots (fewer ads per page = higher viewability per ad). (4) **Billing impact**: Most programmatic deals now use "viewable CPM" (vCPM) — the advertiser pays only for viewable impressions. The viewability rate directly impacts publisher revenue.

## Q15: Explain the difference between header bidding and waterfall ad serving. Why did the industry move to header bidding, and what are the remaining downsides?

**Answer:** Waterfall and header bidding are two strategies for sourcing demand, and the shift between them was the biggest tectonic shift in ad tech in the 2010s. (1) **Waterfall** (pre-2015): The publisher's ad server (e.g., Google Ad Manager) queries demand sources in a fixed order: first ask direct-sold advertisers (highest CPM), then premium ad networks, then exchanges, then remnant networks. Each source gets ~200 ms to respond. If the first source doesn't bid, the ad server falls through to the next. Problem: (a) later sources rarely get served, so they bid lower over time; (b) latency accumulates linearly — with 5 tiers, the waterfall can take 1+ second; (c) publishers cannot easily compare bids from different exchanges. (2) **Header bidding** (2015+): A small JavaScript library (Prebid.js or Amazon TAM) runs in the browser. It sends bid requests to multiple exchanges simultaneously. Each exchange returns a bid with a CPM. The header bidding wrapper selects the highest bid and passes it to the publisher's ad server. The ad server runs a final auction (header bid vs. direct sold vs. Google Adsense). This happens in parallel, so total latency is ~200 ms (max of all exchange response times, not sum). (3) **Key advantages of header bidding**: (a) **Increased competition** — all demand sources compete simultaneously, raising CPMs by 10-40%. (b) **Transparency** — publishers see bids from all sources. (c) **Reduced latency** — parallel requests are faster than waterfall. (d) **Faster adoption of new demand sources** — adding a new exchange is just adding a Prebid adapter. (4) **Remaining downsides**: (a) **Browser overhead** — Prebid.js adds 50-200 KB of JavaScript to the page, which can slow down page load (especially on mobile). (b) **Latency on the client** — all bid requests are made from the browser, which means the user's network is in the critical path. If the user has a slow connection, bids arrive late and ad slots collapse. (c) **Cookie deprecation** — header bidding relies on third-party cookies for user identification. Google's phase-out of third-party cookies (ongoing) is forcing a move to server-side header bidding or Google's "Privacy Sandbox" APIs (FLEDGE, Topics). (d) **Complexity** — managing 10+ exchange adapters, each with different configurations, timeouts, and error handling, is operationally complex. (5) **Modern trend: server-side header bidding** — Prebid Server moves the bidding logic to a server (the SSP). The browser sends a single request to the SSP, which fans out to all exchanges from the server side. This reduces browser latency but requires a robust SSP infrastructure.

## Q16: How does an ad platform ensure GDPR/GDPR-TCPA compliance in the real-time bidding flow? Walk through the consent management, data anonymization, and user rights enforcement.

**Answer:** GDPR compliance in RTB is particularly complex because bid requests contain personal data (IP, device ID, location) that is shared with dozens of DSPs. The IAB Europe Transparency & Consent Framework (TCF) v2.2 is the standard. Architecture: (1) **Consent Management Platform (CMP)**: When a user visits a publisher site, a CMP (e.g., OneTrust, Cookiebot) shows a consent banner. The user makes choices about: (a) Purpose 1: Store and/or access information on device (essential — no opt-out). (b) Purposes 2-10: Personalization, ad selection, measurement, etc. (all opt-in). (c) Legitimate interests for each purpose. The CMP stores the user's consent as a TC string (a Base64-encoded string encoding all consent signals) in a first-party cookie. (2) **Consent propagation in RTB**: The SSP includes the TC string in the bid request's `regs.ext.gdpr = 1` and `user.ext.consent = "TC_string"` fields. Each DSP must decode and enforce the TC string: if the user did not consent to Purpose 2 (ad personalization), the DSP must NOT use cookies or behavioral targeting — it can still bid using contextual targeting (page content only). (3) **Data minimization**: GDPR requires data minimization — only send necessary data. The SSP should: (a) truncate IP to /24 subnet (last octet removed); (b) round latitude/longitude to 1 km resolution; (c) NOT send the user's full name, email, or phone number (which should never have been collected without explicit opt-in). (4) **User rights enforcement**: When a user requests "delete my data" via the ad platform's privacy portal: (a) The platform must delete all stored user data (cookie sync IDs, behavioral profiles, impression logs keyed by user ID). (b) Implementation: a `DELETE /users/{user_id}` API that purges all rows from the user profile database (keyed by pseudonymous ID) and the user ID graph. (c) The platform must also propagate the deletion request to its partners (DSPs, data brokers) — this is the hardest part. Most platforms use a "data deletion request manifest" — a shared file that partners must process within 30 days. (5) **TCPA (US)**: The Telephone Consumer Protection Act applies to SMS ad campaigns. Enforcement: pre-check opt-in from the user, maintain an opt-out list (do-not-call list), and respect a 30-day processing window for opt-out requests. (6) **Audit trail**: Every bid request, consent string, and data access must be logged for 3 years. The audit log is immutable (stored in append-only storage like Amazon QLDB or a sled in S3 with WORM lock). Regulators can request audit logs to verify compliance.

## Q17: Design the attribution window system for a performance ad platform. How do you choose the attribution window length (1-day, 7-day, 28-day), and what technical problems arise with long attribution windows?

**Answer:** Attribution windows are surprisingly contentious in the ad industry — longer windows give more credit to upper-funnel ads but delay optimization signals. (1) **Window selection trade-offs**: Short window (1-day): fast feedback loops for ML model training (CTR prediction, budget pacing), but misses conversions that take longer (e.g., B2B SaaS with 30-day sales cycle). Long window (28-day): captures more conversions, but (a) attribution delay — a campaign's performance today depends on clicks from 28 days ago, making optimization harder; (b) overlapping windows — a user clicks on Ad A on day 1, sees Ad B on day 25, converts on day 30 — which ad gets credit? The default is last-click (Ad B), but the first click (Ad A) was equally important. (2) **Technical problems with long windows**: (a) **Click history storage**: For a platform with 10B clicks/month, storing 28 days of click history per user requires ~280B click events in an online DB (Redis or Cassandra). This is expensive. Solution: use a streaming architecture with event time windowing — the Flink job maintains a 28-day sliding window state keyed by `user_id`. When a conversion arrives, the job scans the user's click history within the 28-day window and applies the attribution model. The Flink state backend (RocksDB) handles this at scale. (b) **Latency**: Attribution is typically a batch job (hourly) for long windows because scanning 28 days of history per conversion in real-time is too slow. For 7-day windows, real-time attribution is feasible with a Redis-backed user profile store. (c) **Data consistency**: If a user clears cookies, their click history is lost, and the attribution window is effectively reset. Cross-device identity resolution mitigates this but adds complexity. (3) **Multi-touch attribution with varying windows**: A robust system applies different attribution models simultaneously: "last-click (7-day)", "linear (14-day)", "time-decay (28-day)". The advertiser selects which model to use for reporting. The attribution system stores all models in a single table: `{conversion_id, touchpoint_id, model_name, attribution_weight, window_days}`. (4) **Industry norms**: Facebook/Instagram: 1-day click, 1-day view. Google Ads: 30-day click, 30-day view (adjustable). LinkedIn: 30-day click, 7-day view. For e-commerce, 7-day click / 1-day view is the most common baseline. For SaaS, 90-day click windows are common.

## Q18: Design a click-through rate (CTR) prediction model that runs in real-time during a RTB auction. What features, model architecture, and serving infrastructure do you use?

**Answer:** CTR prediction is the core ML task in RTB — within 50 ms compute budget, the model must predict the probability of click for each candidate ad. (1) **Feature engineering**: Features are categorized: (a) **User features**: user ID (hashed), the user's historical CTR (CTR over last 1h, 24h, 7d), the user's most frequently clicked categories, the user's device type and OS. (b) **Ad features**: ad ID, campaign ID, advertiser ID, ad category, ad format (banner, video, native), creative size. (c) **Context features**: page URL category, time of day, day of week, device type, browser, language. (d) **Cross features**: user_category × ad_category (e.g., user who clicks "sports" ads on "news" pages). (2) **Model architecture**: The industry standard is a **wide-and-deep** model or a **deep factorization machine** (DeepFM). The "wide" part memorizes feature interactions (e.g., "user X clicks ad Y"), implemented as a logistic regression on cross-product features. The "deep" part generalizes to unseen combinations using embeddings: each categorical feature (user ID, ad ID) is mapped to a dense embedding vector (64-256 dimensions). The deep network is 3-4 layers of 256, 128, 64 neurons with ReLU activation. Output: a single sigmoid neuron — P(click). (3) **Training**: Training data: all historical impressions + clicks (imbalance: 0.1% click rate). Use negative sampling — downsample non-click events (e.g., keep 10% of non-clicks, weight them 10×). Optimizer: Adagrad or FTRL (Follow The Regularized Leader) for online learning. The model is trained on the last 30 days of data and retrained daily. (4) **Serving infrastructure**: The model must be available in under 1 ms inference time. Deployment: (a) Export the trained model to ONNX or a custom C++ serving library. (b) Load the model into each RTB server's memory (not a separate model server — the network round-trip to a model server would blow the 50 ms budget). (c) Feature extraction happens inline — the RTB server has the feature pipeline (feature store in Redis) on the same host. (d) The model is hot-reloaded every hour without restarting the server (gRPC reflection or file watch on updated model weights). (5) **Calibration**: The raw model output (sigmoid) may be miscalibrated — a predicted CTR of 0.001 may actually be 0.002 in practice. Apply **isotonic regression** or **Platt scaling** as a post-processing step to calibrate predictions. Calibration is critical because the bid price is computed as `bid = predicted_ctr * conversion_rate * conversion_value`. If CTR is systematically over- or under-estimated, the DSP overpays or underpays. (6) **Online learning**: The model updates incrementally as new clicks and non-clicks arrive. Use FTRL-Proximal (Google's algorithm) which updates per-coordinate learning rates. This captures concept drift (e.g., a new ad campaign suddenly becomes popular). Online learning has a 1-minute delay — new clicks update the model within 60 seconds.

## Q19: Design a budget pacing algorithm that maintains a campaign's target spend rate across a 24-hour day while reacting to traffic fluctuations. Compare PID control vs. bang-bang control vs. reinforcement learning approaches.

**Answer:** Budget pacing is a control problem: given a daily budget B and 24 hours, spend B evenly across time while traffic is irregular. (1) **Bang-bang control** (simplest): At the start of each hour, compute `expected_spend = (elapsed_hours / 24) × B`. If actual_spend < expected_spend, bid aggressively (no cap). If actual_spend >= expected_spend, stop spending entirely (set bid to 0). Problem: over-corrects — spend stops in the last few hours if budget is hit early, then restarts if traffic drops. Causes bursty spending which is bad for both advertisers (missed opportunities) and publishers (price suppression). (2) **PID control** (most common): Three components: P (proportional) — steady-state error correction. If you're 10% behind schedule, increase bid by 10%. I (integral) — accumulated error over time. If you've been behind for hours, the integral term ramps up the bid. D (derivative) — rate of change. If error is decreasing rapidly (you're catching up), dampen the response to avoid overshoot. Tuning: KP = 0.5, KI = 0.1, KD = 0.05 (typical starting values). The PID output is a `bid_multiplier` applied to the base bid from the CTR model. Max multiplier typically capped at 5× (never pay 5× the base bid) to limit overshoot. Min multiplier at 0.1× (never completely stop spending, to maintain model feedback). (3) **Reinforcement learning** (state-of-the-art): The pacing RL agent observes state: (time_of_day, remaining_budget, traffic_rate_forecast, remaining_time). Action: choose bid_multiplier from a discrete set {0.1, 0.5, 1.0, 2.0, 5.0}. Reward: spend target + maximize conversions. The RL policy (a small neural network or Q-table) is trained offline on historical data, then deployed to production. Advantage: RL handles non-stationarity (traffic changes across days, weekends, holidays) better than fixed PID constants. (4) **Industry practice**: Most DSPs use a modified PID with a "pacing curve" — the target spend is not linear (1/24 per hour) but a smooth S-curve: spend 40% in the first 50% of the day, 60% in the last 50%. This holds back budget for the evening hours when traffic and conversion rates are higher. The curve is pre-computed per campaign based on historical traffic patterns. (5) **Budget capping edge cases**: (a) If traffic on a given hour is 10× the forecast (viral event), the pacing system must NOT panic-spend the entire budget. Cap the spend rate at 5× the target rate per hour. (b) If traffic is 0 (e.g., targeting a small audience that goes offline), the system should not stay in "catch-up" mode forever. Implement a "max catch-up" — if you're behind by more than 30% of budget after 80% of the day has elapsed, accept the under-spend and refund the budget (this is better than bidding 5× on low-quality traffic). (6) **Real-world measurement**: Each campaign has a "pacing score" — `actual_spend / expected_spend` at the current time. A score of 1.0 is perfect pacing. Monitor the distribution of pacing scores across all campaigns. If >10% of campaigns have score <0.8 or >1.2, investigate the pacing model.

## Q20: Explain how an ad exchange computes the clearing price in a second-price auction with soft and hard floors. How does the seller set the optimal floor price to maximize revenue without reducing the fill rate?

**Answer:** The clearing price in an ad exchange auction determines what the winning bidder actually pays. (1) **Second-price auction**: The highest bidder wins but pays the second-highest bid price (or the floor, whichever is higher). Example: bids [$2.00, $1.50, $1.00], floor $0.50. Winner: $2.00 bidder pays $1.50. This encourages bidders to bid their true valuation (truthful bidding is the Nash equilibrium). (2) **Hard floor vs soft floor**: **Hard floor**: bids below the floor are excluded entirely (auction only considers bids ≥ floor). **Soft floor**: bids below the floor are allowed, but if the winning bid is below the floor, the impression is not sold (or is sold at a discount). Soft floors give the seller more flexibility: if there's only one bid at $0.80 and the floor is $1.00, the seller can choose to accept the $0.80 bid rather than leave the impression unfilled. (3) **Floor price optimization (the seller's problem)**: Set floor too high — fewer impressions sold (low fill rate). Set floor too low — revenue per impression is low. The optimal floor balances: `expected_revenue(floor) = fill_rate(floor) × average_clearing_price(floor)`. Compute optimal floor per (publisher, ad_slot, device) segment. An ML model predicts the optimal floor by estimating the bid distribution from historical data. If bid distribution is heavy-tailed (some bidders willing to pay $10 CPM), set floor high (e.g., $2-3). If bid distribution is concentrated near $0.50, set floor near $0.50. (4) **Dynamic floors with reserve price optimization**: Google's approach uses a "smart pricing" algorithm that adjusts floors in real-time based on the observed bid landscape for the current impression. For each request, the exchange computes a `floor = f(historical_bids_for_similar_impressions)`. This is a variant of the "Myerson optimal auction" — if you know the distribution of bidders' values, the optimal reserve price is independent of the number of bidders (it depends only on the distribution's hazard rate). (5) **Multi-impression clearing**: For video ads, a single impression might have multiple ad pods (slots within the same video). The exchange runs a "multi-unit auction": the highest bidders fill each slot, but they all pay the lowest accepted bid (uniform price) or their own bid (pay-as-bid). Google's YouTube uses a generalized second-price (GSP) auction for video ads, where each slot is filled by the next highest bidder but they pay the next bidder's price. (6) **Real-world constraints**: (a) **Budget pacing interaction**: A bidder close to their daily budget might not bid aggressively, affecting the floor price calculation. The exchange doesn't know each bidder's budget state. (b) **Latency**: The auction loop (collect bids, compute clearing price, notify winners) must complete in <20 ms total. (c) **Fraud**: Bidders can collude — Bidder A bids high on impressions Bidder B wants, and vice versa. The exchange monitors for collusion patterns using Herfindahl-Hirschman Index (HHI) on bidder competitiveness.

## Q21: Walk through the complete RTB protocol flow — from user visiting publisher page to ad appearing in browser. Critical latency budgets at each hop.

**Answer:** 6+ network round-trips must complete in <300ms total. (1) **User visits page** → publisher ad tag (Prebid.js) executes, sends request to SSP (~50ms). (2) **SSP → Ad Exchange**: bid request (OpenRTB JSON) with impression, site, device data. (3) **Exchange → DSPs**: bid request sent in parallel to multiple DSPs via HTTP POST. Each DSP has 100ms budget (includes network) to respond with bid. DSPs hold CTR prediction, budget pacing, freq capping in memory. (4) **Exchange selects winner**: runs second-price auction. ~20ms. Notifies winning DSP via win notice. (5) **Exchange → SSP**: returns winning ad markup (VAST/JavaScript) to SSP for rendering (~50ms). (6) **Impression tracking**: ad creative includes 1x1 tracking pixel — browser loads it, registering impression for billing. Total must be <300ms — above-the-fold ads timeout otherwise, slot collapses, no ad shown. DSP's 100ms window includes network — effective compute budget ~50ms.

## Q22: Design the SSP (Supply-Side Platform) unified auction. How does it decide which demand source to send an impression to and optimize publisher revenue?

## Q23: Design an ad fraud detection ML system that operates in real-time. What features and architecture catch sophisticated click farms and botnets?

## Q24: Explain the MRC viewability standard. How does an ad platform instrument "50% of pixels for 1 continuous second" tracking, and what are the technical challenges?

**Answer:** MRC standard: ≥50% pixels in viewport for ≥1 continuous second. (1) **Measurement**: JS tag in creative uses Intersection Observer API + requestAnimationFrame (~60fps). ## Q25: Compare header bidding vs. waterfall. Why did the industry shift, and what are the remaining downsides of header bidding?

**Answer:** (1) **Waterfall (pre-2015)**: query demand sources in fixed order. Later sources rarely served → bid lower over time. Latency accumulates linearly (5 tiers → 1+ second). (2) **Header bidding (2015+)**: Prebid.js sends bid requests to all exchanges simultaneously. Selects highest bid, passes to ad server for final auction. Latency ~200ms (max of response times, not sum). CPMs increased 10-40%. (3) **Remaining downsides**: (a) Browser overhead — Prebid.js adds 50-200KB JS, slows page load on mobile. (b) Client latency — user's network is in critical path. (c) Cookie deprecation — Chrome phasing out 3rd-party cookies breaks header bidding's user identification. (d) Complexity — debugging across 10+ exchange adapters is hard. (e) Server-side alternative: Open Bidding / EBDA (Exchange Bidding in Dynamic Allocation) moves competition to Google's server, reducing browser overhead but giving Google more control.
