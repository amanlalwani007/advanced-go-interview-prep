# Chapter 22 — Design a Hotel Reservation System

## Q1: Design the inventory and availability system. How do you track room availability across different room types, date ranges, and overbookings?

**Answer:** The core availability model: (1) **Inventory store** — a table `room_inventory (hotel_id, room_type_id, date, total_rooms, booked_rooms)`. For each hotel and room type, there's one row per date in the next 365+ days. (2) **Availability query** — for a stay from `check_in` to `check_out` (e.g., July 1-5), join the rows for each date in the range and check that `total_rooms - booked_rooms >= requested_rooms` for all dates. (3) **Reservation hold** — when a user searches for availability but hasn't booked yet, the system places a temporary hold (TTL-based) on the inventory to prevent double-booking during the checkout flow. This is done via Redis: `SETNX hotel:1234:room_type:deluxe:2024-07-01 1 EX 300` — decrements available inventory by 1, auto-expires in 5 minutes if not confirmed. (4) **Overbooking** — hotels purposely overbook (sell more than available rooms) expecting cancellations. The system maintains an `overbooking_limit` per hotel (e.g., 5% of total rooms). The availability check is `booked_rooms <= total_rooms + overbooking_limit`. (5) **Release on cancellation** — when a booking is cancelled, the inventory rows for the affected dates are decremented. This is done inside a transaction to ensure atomicity.

## Q2: How does the system handle high-concurrency scenarios like "10 users trying to book the last room at a popular hotel simultaneously"?

**Answer:** This is a classic hot-spot problem. Strategies: (1) **Pessimistic locking** — at the database level, use `SELECT ... FOR UPDATE` on the inventory rows for the requested dates when starting the booking transaction. This serializes conflicting bookings but limits throughput to ~1 booking per hotel per database round-trip. (2) **Optimistic locking** — add a `version` column to the inventory table. The update query: `UPDATE room_inventory SET booked_rooms = booked_rooms + 1, version = version + 1 WHERE hotel_id = X AND room_type = Y AND date = Z AND version = V`. If the version doesn't match (another transaction updated it), retry. (3) **Redis Lua script** — atomic inventory decrement in Redis: `local remaining = redis.call('DECR', key); if remaining < 0 then redis.call('INCR', key); return -1 else return remaining end`. This is atomic and can handle 100K ops/sec per Redis node. (4) **Queue-based booking** — instead of processing booking requests synchronously, enqueue them and process sequentially per hotel. This ensures FIFO ordering and no race conditions. (5) **Compensation** — if two bookings succeed due to a race condition (overbooked beyond limit), the system has a compensating transaction: cancel the later booking and notify the user with an apology + upgrade to a premium suite.

## Q3: Design the rate plan system. Hotels have complex pricing: different rates for weekdays vs. weekends, seasonal pricing, length-of-stay discounts, and promotional codes. How do you model this?

**Answer:** The rate plan model uses a rule engine: (1) **Base rate** — `room_type_rate (hotel_id, room_type_id, date, base_price)`. This is the default price for a given night. It varies by season (high/low season). (2) **Dynamic adjustments** — a rules table: `rate_rules (hotel_id, room_type_id, condition, adjustment_type, adjustment_value)`. Conditions: `day_of_week IN (Sat, Sun) → +20%`, `length_of_stay >= 7 → -15%`, `booking_advance_days > 30 → -10%`. (3) **Promotional codes** — a separate table: `promotions (code, discount_type, discount_value, valid_dates, min_stay, applicable_hotels)`. These are applied as a final modifier. (4) **Rate calculation at query time** — when a user searches, the system computes the nightly price for each date of their stay: start with base price, apply dynamic adjustments (in order), then apply promotional code. This is done server-side, not in the database, using a configurable rule engine (e.g., Drools or a custom rules config). (5) **Caching** — computed rates are cached in Redis with a 5-minute TTL: `RATE:{hotel}:{room}:{checkin}:{checkout}:{promo}`. (6) **Real-time updates** — when a hotel changes rates, the rules are updated and the affected Redis cache keys are invalidated via a publish message.

## Q4: Design the payment processing pipeline. How do you authorize a card at booking time and capture it at check-in (or no-show)?

**Answer:** Two-phase payment flow: (1) **Authorization** — at booking time, the system sends an authorization request to the payment gateway (e.g., Stripe, Adyen) for the total amount. The gateway returns an authorization ID and places a hold on the card for the authorized amount. This hold typically expires in 7-14 days. (2) **Capture** — at check-in (or if no-show after 6 PM of the arrival date), the system sends a capture request with the authorization ID. The gateway completes the transfer. (3) **Partial cancellation** — if the user modifies the booking (reducing stay), a void is sent for the original authorization, and a new authorization for the adjusted amount is made. (4) **No-show handling** — a cron job runs nightly and checks for no-shows: bookings where `check_in < now AND status = 'confirmed' AND NOT checked_in`. For no-shows, the system captures the first night's charge and cancels the rest. (5) **Payment retry** — if authorization fails (insufficient funds), the system retries with a 24-hour interval, up to 3 attempts. If all fail, the booking is cancelled. The payment pipeline is idempotent — each payment operation has a unique `idempotency_key` to prevent duplicate charges if the request is retried.

## Q5: How do you support the "search near landmark" feature — e.g., "find hotels within 5 km of the Eiffel Tower"? Design the geospatial index.

**Answer:** The hotel search uses a geo-index: (1) **Indexing** — each hotel has a (lat, lng). Use Elasticsearch with a geo-shape or geo-point field. The index is populated from the hotel master database via CDC (Debezium → Kafka → Elasticsearch). (2) **Query** — `GET /hotels/_search {"query": {"bool": {"filter": {"geo_distance": {"distance": "5km", "location": {"lat": 48.8584, "lon": 2.2945}}}}}}`. Elasticsearch's geo-distance filter uses geohash cell prefix filtering internally, making it fast. (3) **Aggregation** — return hotel details (name, rating, price range, availability count) along with distance from the search point. (4) **Fallback** — if Elasticsearch is unavailable, fall back to a MySQL geo-spatial index (`ST_Distance_Sphere`) on a read replica. (5) **Filtering by availability** — the geo query returns matching hotel IDs, then the availability service checks inventory for those hotels on the requested dates. This two-step flow (geo filter → availability check) avoids embedding availability data into Elasticsearch, which would change too frequently (every booking). For performance, hotels with zero availability (all room types sold out) are marked in Redis: `SADD hotel:soldout:2024-07-01 {hotel_id}`. The search service filters these out.

## Q6: Design the cancellation and refund workflow. A user cancels a non-refundable booking 3 days before check-in. How do you handle the partial refund and inventory release?

**Answer:** The cancellation policy is configured per hotel/rate plan: (1) **Policy evaluation** — when the user requests cancellation, the system evaluates the cancellation policy: `free_cancellation_before = check_in - 7 days`, `penalty_percentage = 50% for cancellations between 7-1 days`, `no_refund for cancellations within 24 hours`. Since the user cancels 3 days before (inside 7-1 day window), a 50% penalty applies. (2) **Refund calculation** — total charged = $1000. Refund = $1000 × 50% = $500. (3) **Payment reverse** — the system sends a refund request to the payment gateway using the original transaction ID. The gateway processes the refund (typically $0 fee for full refunds, may retain processing fee for partial refunds). (4) **Inventory release** — the inventory rows for the affected dates are decremented atomically in a DB transaction (same transaction that updates the booking status). (5) **Notification** — the user receives an email with the refund confirmation. (6) **Async processing** — refunds from credit cards take 5-10 business days to appear. The system logs the refund request and tracks its settlement status via a reconciliation job (runs daily, checks gateway reports). (7) **Grace handling** — if the payment gateway's refund API is down, the system queues the refund as "pending" and retries with exponential backoff.

## Q7: Design the "best rate guarantee" system. How does the system continuously monitor competitor prices and adjust its own pricing dynamically?

**Answer:** A competitive rate shopping system: (1) **Web scraper / API** — each hotel provides a feed of its competitor rates, or the system uses a third-party rate shopping service (OTA Insight, Lighthouse) that scrapes competitor OTAs (Booking.com, Expedia) and provides rate comparisons via API. (2) **Rate matching engine** — this is a batch process running every 15 minutes. For each hotel room type, it compares the hotel's current BAR (Best Available Rate) against competitor rates for the same room category and dates. (3) **Pricing rules** — `if competitor_rate < our_rate * 0.95 and occupancy < 80% → match competitor rate (floor at cost + margin)`. `if competitor_rate > our_rate * 1.2 and occupancy > 90% → increase our rate by 10%`. (4) **Execution** — the pricing engine writes updated rates to the rate plan database and invalidates the relevant Redis cache keys. (5) **Yielding** — advanced revenue management systems use a yield management algorithm (e.g., modified EMSR-b) that considers: historical booking patterns, current booking pace, competitor rates, and remaining inventory to dynamically set prices per room-night. This is the same algorithm used by airlines. (6) **A/B testing** — the system can run pricing experiments: assign a percentage of users to a control group (current price) and test group (new price) and measure conversion rate differences.

## Q8: How do you design the channel distribution system — the same inventory is sold on the hotel's own website, Booking.com, Expedia, and GDS. How do you avoid double-booking?

**Answer:** This is the central inventory management problem in hospitality. The industry solution is a **Channel Manager** with an **iHotelier / XML interface**: (1) **Central inventory** — the hotel maintains a single source of truth for inventory (PMS / Central Reservation System). All channels connect to this central system. (2) **API integration** — each OTA (Booking.com, Expedia) connects via their XML API or a third-party channel manager (Siteminder, Cloudbeds). Available room counts are pushed to each OTA every 5-15 minutes. (3) **Push vs. Pull** — the preferred model is **push**: the central system pushes inventory counts to each OTA. When a booking happens on any channel, the central system decrements inventory and immediately pushes updated counts to all other OTAs. (4) **Allocation-based** — alternatively, pre-allocate a fixed number of rooms per channel (e.g., 20 rooms to Booking.com, 15 to Expedia, 10 to direct). Each channel sells from its own allocation. When one channel sells out, it can request additional allocation from a shared pool (allotment). This reduces real-time syncing needs. (5) **Race condition handling** — if two bookings arrive within seconds from different channels, the central system uses a last-writer-wins or a distributed lock (Redis lock on hotel_id). The one that succeeds gets the room; the other receives an error response that the OTA handles (notifies user the room is no longer available).

## Q9: Design the property management system (PMS) integration — housekeeping status, room block, maintenance, and front desk operations.

**Answer:** The PMS is the operational backbone: (1) **Room status** — each room has a status (Vacant Clean, Vacant Dirty, Occupied Clean, Occupied Dirty, Out of Order, Out of Service). Status transitions: check-in (Vacant → Occupied), check-out (Occupied → Vacant Dirty), housekeeping cleaning (Vacant Dirty → Vacant Clean). (2) **Housekeeping module** — a dashboard shows all rooms requiring cleaning, prioritized by: expected check-in time (incoming guests need rooms first), VIP status, stayover cleaning requests. When a housekeeper finishes, they scan a QR code in the room to mark it clean in real-time. (3) **Maintenance** — rooms can be flagged as Out of Order (OOO) for maintenance (plumbing, AC repair). OOO rooms are excluded from the inventory. The maintenance team uses a mobile app to update status and log repairs. (4) **Front desk operations** — check-in: verify ID, collect payment (if not prepaid), assign room, issue key cards. Check-out: post any incidental charges (minibar, room service), process final payment, release room. (5) **Real-time sync** — all PMS operations (check-in, check-out, housekeeping complete) trigger events that update the central inventory system via an event bus (Kafka). For example, a housekeeping-complete event for room 305 triggers: `UPDATE room_status SET status='VACANT_CLEAN' WHERE room_id=305` → if status was Vacant Dirty before, this adds a room to the available inventory.

## Q10: How would you design the recommendation engine for a hotel booking site — "users who viewed this hotel also viewed X"?

**Answer:** Collaborative filtering for hotel recommendations: (1) **Data collection** — log every user action: `{user_id, hotel_id, action (view/click/book), timestamp}`. Store in a time-series DB (ClickHouse). (2) **Item-to-item similarity** — for each hotel, compute the top 10 most similar hotels using cosine similarity on co-viewing patterns: if users who viewed hotel A also viewed hotel B frequently, they are similar. The similarity matrix is computed offline (nightly Spark job): `SIM(A, B) = count(users who viewed both A and B) / sqrt(count(viewed A) * count(viewed B))`. (3) **Personalization** — for a logged-in user, retrieve their recent viewing history (last 10 hotels viewed). Aggregate the similar hotels for those 10, weighting by recency and similarity score. (4) **Business rules** — filter out hotels in different cities (unless the user has cross-city booking patterns), prioritize hotels with available inventory, exclude fully booked hotels. (5) **Real-time serving** — the precomputed similar hotels are stored in Redis as sorted sets: `ZADD similar:hotel:1234 0.95 hotel:456 0.90 hotel:789`. The recommendation API does a `ZREVRANGE` query, which takes <1 ms. (6) **Cold start** — for new hotels with no viewing history, use content-based recommendations: match by attributes (location, price range, amenities, star rating). For new users with no history, show trending/popular hotels in their searched city. (7) **A/B testing** — alternative recommendation algorithms (Matrix Factorization, Neural collaborative filtering) can be live-tested against the baseline.

## Q11: Design the Global Distribution System (GDS) integration layer for the hotel reservation system. You need to expose inventory and rates through Sabre, Amadeus, and Travelport. How do you handle protocol differences (OTA XML vs. EDIFACT vs. REST), and how do you manage the availability caching vs. real-time lookup trade-off?

**Answer:** GDS integration is historically the most complex part of hotel distribution. Architecture: (1) **Protocol abstraction layer**: build a GDS gateway that normalizes all protocols into an internal protobuf format. Sabre uses OTA XML (OpenTravel Alliance) via SOAP. Amadeus offers both OTA XML and its native Web Services API (REST/JSON). Travelport (via Galileo/Apollo) uses XML over HTTPS. EDIFACT is legacy but some regions still use it � maintain a COBOL-to-Java bridge or use a third-party connector (e.g., Travelsky). (2) **Operation mapping**: common operations (AvailabilityCheck, RateQuery, Book, Cancel, Modify) map across all GDSes with slight semantic differences. The core challenge is **availability source of truth** � GDS systems cache hotel inventory and do not query the hotel's CRS in real-time for every search. Trade-off: (a) **Pull model (real-time)**: the GDS queries the hotel's CRS on each search � latency is 2-5 seconds, 100% accurate but expensive in API calls. (b) **Push model (cached)**: the hotel pushes inventory snapshots to the GDS every 15-30 minutes � fast (<200ms), but stale inventory leads to "booking failures" (the GDS shows availability, but the hotel is actually sold out). (3) **Hybrid approach**: push a "capacity band" to the GDS every 5 minutes (e.g., 10+ rooms available = "many", 3-9 = "limited", 1-2 = "few"). When the user tries to book, the system does a real-time availability check against the hotel's CRS. This balances search speed (fast, from GDS cache) with booking accuracy (real-time at point of sale). (4) **Error handling**: GDS timeouts are common (Amadeus is particularly slow). Use a circuit breaker pattern: if a GDS endpoint fails 3 times in 60 seconds, degrade to cached data for 5 minutes. Use exponential backoff with jitter for retries. (5) **GDS-specific quirks**: Sabre requires comprehensive rate descriptions in specific currencies; Amadeus has a strict 7-day advance purchase requirement for certain rate codes; Travelport does not support all room type combinations. The GDS gateway maintains a rules engine per GDS for these transformations. (6) **Monitoring**: track GDS-specific KPIs: search latency per GDS, booking success rate per GDS, booking failure reason codes (GDS-015 = "Hotel not found", GDS-032 = "Rate not available"), and cost per booking (each GDS transaction costs .10-.50).

## Q12: Design the allotment management system. A hotel allocates 20 rooms to Booking.com, 15 to Expedia, 10 to direct, and 10 to a corporate account. When Booking.com sells out its 20, it can "borrow" from the direct allotment pool. How do you model allotments and handle the borrow/return mechanism without complex locking?

**Answer:** Allotment management is a nested inventory allocation problem: (1) **Allotment model**: each channel has a base allocation: allotments (channel_id, hotel_id, room_type_id, date, allocated_rooms, borrowed_rooms). The base allocation is hard-set by the revenue manager via a contract. (2) **Borrow mechanism**: when a channel runs out (booked_rooms >= allocated_rooms), it can borrow from a shared "pool" or from other channels' underutilized allotments. Rules: (a) borrow only if the channel's utilization is >80% (to prevent hoarding), (b) max borrow = 30% of base allocation, (c) return borrowed rooms if the lender channel needs them (with 24-hour notice). (3) **Implementation**: use a soft reservation with a priority system. The central inventory system tracks: total_inventory = sum(all base allocations) + pool. When a channel needs to borrow: a request is sent to a "borrow arbiter" service, which checks if pool rooms are available, decrements pool, increments the channel's borrowed_rooms. The key data structure: llotment_state (channel_id, date, total_allocated, total_booked, borrowed_from_pool). (4) **Lock-free borrow**: instead of distributed locks, use optimistic concurrency with a version field: UPDATE allotment SET borrowed_rooms = borrowed_rooms + 1, version = version + 1 WHERE channel_id = 'BKG' AND date = '2024-07-01' AND allocated_rooms - booked_rooms - borrowed_rooms > 0 AND version = V. If the update fails (version mismatch or no rooms available), the channel receives a "no borrow available" response and can either show the room as sold out or escalate to a waitlist. (5) **Return mechanism**: a nightly reconciliation job reviews borrows. If a channel's booking_velocity drops below its allocation (i.e., it won't sell out), the system returns borrowed rooms to the pool. The return is automatic unless the channel's forecasted demand exceeds its allocation. (6) **Borrow debt tracking**: if borrows cannot be returned before the lender needs them, the system holds a "debt" � the borrower must give back rooms at the same rate or compensate with a commission fee. This is settled as a financial transaction in the revenue reconciliation system.

## Q13: Design a rate parity enforcement system. You must ensure that the hotel's rate on Booking.com is not lower than on Expedia, and neither is lower than the hotel's own direct website. How do you detect violations programmatically and handle the case where a partner OTA offers an opaque discount that undercuts publicly available rates?

**Answer:** Rate parity enforcement is a constant cat-and-mouse game in hospitality: (1) **Automated rate shopping**: subscribe to a third-party rate shopping service (OTA Insight, Lighthouse, RateGain) that scrapes all OTAs every 30 minutes for the hotel's rates. The scraper reports (ota_name, room_type, date, rate, currency, cancellation_policy, rate_category). (2) **Violation detection**: a parity checker service compares rates across channels: IF min_rate_on_any_ota < direct_rate - margin AND NOT authorized_discount THEN flag_violation(ota_name, differential). "authorized_discount" includes: loyalty program member rates, corporate negotiated rates, opaque rates (Priceline/Hotwire, where the hotel brand is hidden until purchase). (3) **Opaque discount handling**: opaque channels (Priceline, Hotwire) show rates that appear lower but the consumer does not know the hotel brand until purchase, so they do not cannibalize direct bookings. The system exempts opaque rates from parity checks with a special flag. However, the exemption requires verification that the channel is truly opaque (brand hidden) and not offering identifiable rates. (4) **Enforcement actions**: on first violation, send an automated email to the hotel's revenue manager with details. Second violation: escalate to the OTA's account manager with a notice. Third violation: the OTA may receive a warning of contract termination. The severity escalates over a rolling 30-day window. (5) **False positive prevention**: a hotel may have multiple rate plans (BAR, package, promo) that legitimately differ. The parity checker must compare identical rate plans: if Booking.com's "Standard Rate" is  and Expedia's "Breakfast Included Rate" is , this is not a violation because the products differ. Match rate plans by a normalizing function: 
ormalized_rate_key(room_type, cancellation_policy, inclusions, prepayment) must match across channels before comparing. (6) **Automated rate adjustment**: when a violation is detected and confirmed, the system can automatically adjust the direct website rate to match the lowest OTA rate (maintaining parity by raising the OTA rate or lowering the direct rate, per the hotel's strategy). Rule: if ota_rate < direct_rate: set direct_rate = ota_rate (to comply with parity by being the lowest). This is audited and reversible. (7) **Parity clause in contracts**: the system maintains contract terms per OTA: max_discrepancy_percentage = 5%, grace_period_minutes = 120 (the OTA has 2 hours to fix the discrepancy before escalation). The parity enforcement system logs all events for audit and contract compliance reporting.

## Q14: Design a revenue management yield optimization system for a hotel chain. How do you implement an EMSR-b (Expected Marginal Seat Revenue-b) algorithm adapted for hotel rooms (which have length-of-stay constraints unlike airlines)? How do you balance occupancy vs. rate in real-time?

**Answer:** Hotel yield management extends airline EMSR-b with the dimension of length-of-stay (LOS): (1) **EMSR-b adapted for hotels**: EMSR-b sorts demand by rate class and computes protection levels. For hotels, the unit is "room-nights" not "seats." The algorithm: (a) forecast demand per rate class per date (using 90-day booking curve at 1-day granularity), (b) compute cumulative revenue: for rate classes sorted descending, EMSR(Class k) = WeightedAvg(Rev of classes 1...k), (c) find the protection level for class k: P(Class k) = F^{-1}(1 - EMSR(k) / Rev(k+1)) where F is the cumulative demand distribution. (2) **Length-of-stay adjustment**: the hotel version must ensure that a booking's LOS doesn't block higher-revenue bookings on the middle dates. Apply a "bid price" per room-night: the marginal opportunity cost of accepting a booking. If a booking's revenue > sum(bid_prices for each night of stay) -> accept. The bid price is computed from the dual values of the linear programming network revenue management optimization. (3) **Implementation**: run a nightly batch optimization that produces bid prices per room-night per rate class. The LP model: maximize(sum(rev * bookings)) subject to sum(bookings_per_night) <= capacity_per_night, bookings_per_rate <= demand_forecast. Solve with simplex or interior-point method. At 1000 hotels x 365 days = 365K variables, the LP solves in ~30 seconds on a single server. (4) **Overbooking optimization**: compute the optimal overbooking level: overbooking_level = floor(mean_no_show + z * std_no_show). where z is the service level parameter (z=1.65 for 95% confidence). The no-show forecast is per-season, per-rate-class (non-refundable rates have lower no-show rates). (5) **Real-time adjustments**: the nightly bid prices are cached in Redis. On each booking or cancellation, the system recomputes the bid prices incrementally for the affected dates. A delta optimization runs: if remaining capacity for date X changes by >5%, re-run bid price for that date only (takes <1 second). (6) **Competitive override**: if competitor rates drop significantly (>15%), the system applies a "competitive pressure" override: reduce bid prices by up to 10% for the affected dates, prioritizing occupancy over yield. The override decays over 48 hours. (7) **Performance metrics**: track RevPAR (Revenue per Available Room), Yield Stat = Actual Revenue / Potential Revenue at BAR, and Occupancy %. The system triggers an alert if RevPAR drops >10% YoY for any hotel, prompting human review.

## Q15: Design the group booking block management system. A corporate client requests 50 rooms for a conference (July 10-15). The hotel puts a "block" on 50 rooms per night. How do you handle the attrition clause (the group only uses 40 of 50), the cutoff date (unsold rooms released back to inventory), and the master bill (one invoice covers all rooms)?

**Answer:** Group booking blocks are fundamentally different from transient bookings � they involve contracts, attrition, and cutoffs: (1) **Block creation**: the sales team creates a group block in the system: block (block_id, hotel_id, room_type, dates[5], blocked_rooms=50, rate_agreed=/night, attrition_clause=80%, cutoff_date=June 10). The block is linked to a group contract stored as a PDF with key terms extracted into the block record. (2) **Inventory segregation**: the 50 rooms per night are subtracted from the hotel's total inventory when the block is active. They appear as "Blocked" in the inventory dashboard, not as "Booked." The system ensures that transient booking queries see total_inventory - blocked_rooms. (3) **Cutoff date handling**: before the cutoff, unbooked rooms in the block are not available for transient sale. On the cutoff date (June 10, 30 days before check-in), a cron job scans all blocks where cutoff_date <= TODAY. For each block: UPDATE inventory SET blocked_rooms = actual_bookings WHERE block_id = X. Rooms where blocked > actual_bookings are released back to the general inventory pool. (4) **Attrition clause enforcement**: after the stay completes, the system compares the contract's attrition clause against actual usage. If the block guaranteed 50 rooms but only 40 were booked (80% utilization), and the contract specifies minimum_usage = 80%, then no penalty. If only 30 were booked (60%), the attrition clause is triggered: penalty = (min_guaranteed_rooms - actual_rooms) * rate * nights. The system generates an attrition invoice automatically and adds it to the group's master bill. (5) **Master bill**: all group charges (room charges + incidentals + meeting room rental + attrition fees) are aggregated into a single invoice. The system maintains a group_master_bill (block_id, line_item, amount, is_attrition, is_room_charge) table. The front desk posts incidental charges to the master bill when group members present their group code at checkout. (6) **Pickup report**: the system generates a daily "pickup report" for the group contact, showing how many rooms are booked vs. the block. This is critical for groups � they add/drop rooms up to the cutoff date. The pickup report is sent automatically every morning via email with a real-time dashboard link. (7) **Sell and release (S&R)**: after cutoff, any remaining blocked rooms are released. If the group later calls to add more rooms, the system attempts to find available rooms in the general inventory, but the rate may differ from the contracted group rate. This is a common cause of friction � the system notifies the group that rooms are subject to availability at the prevailing rate.

## Q16: Design the waitlist and upgrade prioritization system. A guest wants a specific room type that is sold out. They join a waitlist. Another guest cancels. How do you determine which waitlisted guest gets the room, and what happens if a paid upgrade (e.g., suite upgrade for /night) becomes available?

**Answer:** Waitlist prioritization is an auction-like optimization problem: (1) **Waitlist model**: each waitlist entry has: (guest_id, hotel_id, requested_room_type, requested_dates, priority_score, max_price_willing_to_pay, is_upgrade_waitlist). The priority_score is a composite: priority = loyalty_tier_weight * 100 + waitlist_timestamp_weight * 50 + past_revenue_weight * 30 + is_vip * 200. The weights are configurable per hotel chain � a casino hotel might weight past_revenue higher than loyalty tier. (2) **Room release flow**: when a cancellation frees a room, the system evaluates all waitlist entries for that room type and dates, sorted by priority_score descending. The top candidate is offered the room. The offer has a TTL of 2 hours (mobile push notification + SMS + email). If the guest doesn't respond, the offer goes to the next in line. (3) **Upgrade waitlist**: a guest booked a standard room but waitlisted for a suite upgrade at /night. When a suite becomes available, the system checks: (a) is the upgrade profitable? upgrade_revenue = suite_rate - standard_rate_guest_is_paying. If  < upgrade_revenue, the upgrade is less profitable than selling the suite at full rate to a new guest. Decision: the system runs a mini-optimization: if the suite's full-rate booking probability in the next 24 hours is <30%, offer the upgrade (accept incremental  rather than risk ). (4) **Waitlist expiration**: waitlist entries expire if the guest has booked an alternative hotel (detected via OTA channel data). The system periodically (every 6 hours) checks: if the guest booked elsewhere and cancelled the waitlist request (via webhook or polling), the entry is removed. (5) **Upgrade bidding**: for high-demand periods, the system can run a silent second-price auction on upgrades. Guests set their max upgrade price (e.g., /night for a suite). When a suite becomes available, the winning bidder pays the second-highest bid (Vickrey auction). This maximizes revenue while encouraging truthful bidding. (6) **Guest communication**: the waitlist status is visible in the app. The guest can adjust their priority by paying a "priority fee" ( for 24h priority position). This generates additional revenue and reduces waitlist churn. (7) **Waitlist pool management**: if multiple guests waitlist for the same room type on overlapping dates, the release logic must handle conflicts: "Guest A waitlisted for July 1-3, Guest B for July 2-4. If a cancellation frees July 1-4, who gets it?" Solution: evaluate the maximizing total priority: max(sum(priority(A) for 1-3) + sum(priority(B) for 4-4), sum(priority(shorter) for some subset)). This is a bin-packing problem; at low volume (<100 waitlist entries), brute-force is fine.

## Q17: Design the multi-property search feature. A user searches "hotels in Manhattan with a pool, under /night, checking in July 1-5." The system must check availability across 500 hotels simultaneously and return results within 2 seconds. How do you parallelize the availability check and handle the variability in check-in/check-out date combinations?

**Answer:** Multi-property search is a distributed query with high fan-out: (1) **Pre-filter stage**: first, filter hotels by static attributes (location, amenities). Elasticsearch returns 500 matching hotel IDs from Manhattan with a pool. This is fast (<100ms). (2) **Rate/availability cache**: each hotel's upcoming availability is precomputed and stored in a compressed bitmap in Redis. Key: vail:{hotel_id}:{date} -> bitmap of room types with available rooms (5 bits per room type, allowing up to 32 room types). For a 7-night stay (July 1-5 = 4 nights), the system fetches 4 bitmaps per hotel and ANDs them to get room types available for the entire stay. This is O(4 bitmap AND operations per hotel) � very fast. Redis pipelining allows 500 hotels * 4 keys = 2000 keys fetched in ~20ms. (3) **Rate computation**: for each hotel with availability, compute the total rate for the stay: 	otal_rate = sum(nightly_rate(room_type, date) for date in stay). The nightly rate is cached in Redis as a sorted set: ZRANGE hotel:{id}:rate:{room_type}:{date} 0 0. For 500 hotels, this is 500 * up to 3 room types * 4 nights = 6000 Redis lookups in a pipeline � ~50ms total. (4) **Ranked results**: after availability and rate checks, sort by total_price ascending. Return top 50 results. The entire search is a scatter-gather: frontend -> search service -> parallel queries to Elasticsearch + availability/rate Redis -> merge -> sort. Target: <2 seconds. (5) **Handling variable check-in/check-out**: the length-of-stay filter is applied at the availability check stage. Some hotels have min-stay requirements (e.g., 3-night minimum for holiday weekends). These are stored as metadata and checked during the retrieve stage. For flexible date search ("check-in July 1-5, stay 3-5 nights"), the system runs multiple queries in parallel (check-in = July 1,2,3,4,5, each with stay = 3,4,5 � up to 15 combinations). The results are merged and deduplicated. (6) **Scale optimization**: for 500 hotels, the scatter-gather works on a single search service instance. At 10K+ hotels (a chain like Marriott), shard by region: NY hotels go to NY search shards, LA hotels to LA shards. The search service fans out to regional shards in parallel. Each shard returns its top 20 results, the search service merges 50 from 5 shards -> produces global top 50.

## Q18: Design the "last-room availability" (LRA) problem. A hotel has 1 room left for July 1. It's listed on 5 channels (direct, Booking.com, Expedia, GDS, phone). How does the system ensure the room is not double-booked, and how does it handle the case where the direct channel holds the room for a browsing user vs. an OTA that requires instant confirmation?

**Answer:** The LRA problem is the hardest consistency challenge in hotel reservations: (1) **Last-room state management**: maintain a real-time last-room indicator in Redis: SET last_room:{hotel}:{date}:{room_type} "available" EX 1. This key auto-expires after 1 second, meaning the "last room" status is volatile � the next booking may sell it. The booking flow reads this key before attempting a booking: if GET last_room:... == "available": attempt_booking(). (2) **Booking hold vs. instant confirm**: the direct channel allows a 5-minute hold for the last room (during checkout flow). For OTA channels, the contract typically requires instant confirmation (no hold). Reconciliation: when the direct channel places a hold, the inventory for that date is decremented but with a special "held, awaiting payment" marker. If another channel requests the room during the hold window, the system responds: "available" (not revealing the hold) and accepts the booking if it arrives first. If the OTA booking arrives before the direct hold converts, the OTA wins and the direct user is notified "SOLD OUT." (3) **Two-phase commit for OTA**: when an OTA sends a Book request, the system must respond within 2 seconds (per GDS SLA). Flow: (a) SELECT ... FOR UPDATE on the inventory row for the specific date/room_type, (b) check remaining rooms > 0, (c) decrement, (d) send confirmation to OTA, (e) respond to GDS, (f) if subsequent payment fails (rare for OTAs), the booking is cancelled and inventory restored. The SELECT ... FOR UPDATE serializes all booking attempts for the last room, preventing double-booking. (4) **Race condition in microservices**: the inventory update and the charge request are separate microservices. If the inventory decrement succeeds but the payment service is down, the room is gone but not paid. Solution: use a saga pattern � inventory decrement is the first step. If payment fails, the compensating step is inventory increment. The saga is orchestrated by a booking event bus. (5) **Over-sell limit for LRA**: allow last-room oversell by 1 with a compensating upgrade promise. If two bookings land simultaneously (race condition in SELECT FOR UPDATE at different DB shards), one guest gets the room, the other gets an automatic upgrade to the next room category at no charge. The cost of the upgrade is < 0.1% of revenue � cheaper than the complexity of perfect synchronization. (6) **Monitoring and alerting**: track last-room booking failure rate (when a guest arrives and no room is available despite a confirmed booking). Target: <0.01% of all bookings. If exceeded, the over-sell limit is reduced. A "walked guest" event triggers: walked_guest (booking_id, original_hotel, walked_to_hotel, compensation_cost). The revenue management team reviews walked guest events daily.

## Q20: Design the GDPR/CCPA data retention and right-to-erasure system for booking history. A user requests deletion of all their personal data. How do you handle the conflict between legal retention requirements (invoices must be kept for 7 years for tax purposes) and the right to be forgotten? How do you implement partial anonymization vs. full deletion?

(b) **Booking transaction data** � dates, room type, rate paid, payment method, booking reference. (c) **Derived/analytics data** � search history, click stream, preference tags. (2) **Retention periods per regulation**: GDPR requires deletion of PII upon request. Tax law (varies by country) requires retaining financial transaction records for 5-10 years (7 years in the US, 10 years in Germany). The reconciled approach: **anonymization, not deletion**. The booking transaction record is retained (tax compliance), but all PII fields are overwritten with anonymized values. (3) **Anonymization procedure**: on a right-to-erasure request: (a) UPDATE bookings SET name = 'ANONYMIZED', email = SHA256(email) || '@anonymized.com', phone = NULL, ip_address = NULL, device_id = NULL, address_street = NULL, address_city = NULL, address_postal_code = NULL WHERE user_id = X. (b) The user_id is mapped to a new anonymous_id via a one-way mapping. The original mapping is deleted. (c) The booking reference (e.g., H123456789) is retained but no longer linkable to a natural person. (4) **Guest profile deletion**: the user's profile record in the CRM system (guest_profiles (user_id, name, email, preferences, total_stays, lifetime_value)) is either deleted entirely or anonymized. However, 	otal_stays and lifetime_value may be needed for aggregated reporting. Solution: the profile is deleted, but the booking records (on which those stats are computed) are anonymized rather than deleted, so queries like SELECT COUNT(*) FROM bookings WHERE hotel_id = X still work. (5) **Right to object (marketing)**: GDPR Article 21 gives users the right to object to processing for direct marketing. This is distinct from right to erasure. The system must maintain a suppression list: marketing_suppressions (email_hash, suppressed_at, channel). (8) **Testing and compliance**: a daily compliance scanner checks: for any user who requested erasure >30 days ago, verify that no PII fields in any table contain recognizable data. This is a data quality rule running against the DB with regex patterns (email regex, phone regex). Violations trigger a critical alert to the DPO (Data Protection Officer).

## Q21: Design the payment processing pipeline for hotel bookings. How do you handle credit card authorization, capture, refunds, and chargebacks while ensuring exactly-once semantics?

**Answer:** Hotel payment processing has a multi-step lifecycle. (1) **Authorization**: when a user books, the system sends an authorization request to the payment gateway (Stripe/Adyen). The gateway holds the amount (e.g., $500) but doesn't transfer funds yet. The auth is valid for 7-10 days (Visa/MC policy). Store auth details: `payment_auths (booking_id, gateway, auth_id, amount, status, expires_at)`. (2) **Capture**: at check-in (or no-show), the hotel captures the authorized amount. The gateway completes the transfer. Capture must occur before the auth expires (7 days). For bookings with later check-in dates, the system re-auths 48 hours before check-in to refresh the auth window. (3) **Exactly-once semantics via idempotency keys**: (a) Every payment operation includes an `idempotency_key = SHA256(booking_id + operation_type + attempt_number)`. (b) The gateway deduplicates requests with the same idempotency key within 24 hours. (c) On network timeout, the system retries with the same key — no double charge risk. (d) State machine: `AUTHORIZED → CAPTURED → SETTLED`. Each transition is idempotent. (4) **Refunds**: (a) Full cancellation before check-in: release auth (void, no capture needed). (b) Partial cancellation (e.g., cut stay short by 1 night): refund the unused nights. The system computes the refund amount from the cancellation policy. (c) Refund flow: create a refund request with idempotency key via the gateway API. Funds return to the customer's card in 5-10 business days. (5) **Chargebacks**: a customer disputes a charge with their bank. The system must respond within the card network's window (typically 30-45 days). (a) Automated response: gather evidence (signed booking confirmation, check-in records, folio). (b) Submit via the gateway's dispute response API. (c) Track chargeback rate: if >1% of transactions, the processor may terminate the account. (6) **Settlement and reconciliation**: (a) Daily settlement file from the gateway lists all captured transactions. (b) The system reconciles: match gateway transactions against booking records. Flag mismatches for manual review. (c) Payouts: the gateway deposits funds (net of fees) to the merchant account on a T+2 schedule.

## Q28: Design the fraud detection system for hotel bookings. How do you identify and prevent fraudulent reservations, payment fraud, and loyalty point abuse in real-time?

**Answer:** Hotel booking fraud includes stolen cards, chargeback fraud, loyalty theft, and phishing. (1) **Real-time scoring engine**: (a) Every booking triggers a fraud check at the API gateway before the booking is confirmed. (b) Features: device fingerprint, IP geolocation vs. billing address (distance), email domain reputation (temporary email?), phone number validity, booking velocity (same card used for 3+ bookings in 1 hour), guest history (new account?), payment method (gift card/virtual card higher risk). (c) ML model (XGBoost) scores in <10ms: output = fraud_probability [0,1]. Thresholds: <0.3 → auto-approve. 0.3-0.7 → manual review queue. >0.7 → auto-decline. (2) **Rule-based pre-filter**: (a) Rules evaluated before the ML model (cheaper, faster). (b) Rules are maintained by the fraud team in a rule engine. (3) **Device fingerprinting**: (a) Embed a JavaScript snippet that generates a device fingerprint (canvas, WebGL, fonts, screen resolution, timezone). (b) Store fingerprint hash with booking. If the same fingerprint appears in 5 declined bookings in 1 hour → ban the device. (4) **Chargeback prediction**: (a) Post-booking, the system monitors for chargeback signals: customer disputes the charge with their bank. (b) Pre-chargeback signals: guest no-shows, multiple calls to customer support about the booking, email bounces after booking. (c) When risk is elevated, the system may request additional verification (CVV re-entry, 3D Secure). (5) **Loyalty fraud detection**: (a) Monitor points redemption patterns: multiple redemptions to different hotels from same account in 1 day → suspicious. (b) Login from multiple IPs in a short time → possible account takeover. (c) Points balance checking API rate-limited to 10 requests/minute per account. (6) **Human review queue**: (a) Bookings flagged for review appear in a dashboard with: booking details, risk score, reason codes, and one-click actions: approve, decline, request ID verification. (b) SLA: review within 5 minutes (booking flow is real-time). (c) Automated email to the customer: "Please verify your booking by clicking this link" (ID verification via government ID upload). (7) **Feedback loop**: declined bookings that later turn out legitimate → reduce model sensitivity. Approved bookings that resulted in chargebacks → increase the weight of those features.

## Q29: Design the channel manager — how do you synchronize room inventory, rates, and availability across multiple OTAs (Booking.com, Expedia, Agoda) in real-time? Handle the challenge of differing data models, API rate limits, and connectivity issues.

**Answer:** A channel manager is the bridge between the hotel's central inventory and multiple OTAs. (1) **Channel abstraction layer**: each OTA has a unique API and data model. Define a canonical schema for: inventory (room_type_id, date, total, available), rates (room_type_id, date, rate_plan_code, amount, currency), reservations (external_id, room_type_id, dates, guest_name, status). Each OTA adapter maps between canonical and OTA-native formats. (2) **Inventory push**: (a) When a direct booking is made, the channel manager decrements inventory for the booked room type and dates. (b) It pushes the updated availability to all connected OTAs via their APIs (or pull-based — some OTAs read from an XML feed). (c) Real-time push target: <30 seconds from booking to OTA update. (3) **Rate push**: (a) When dynamic pricing updates rates, the channel manager pushes rate changes to OTAs. (b) Rate push frequency: every 15 minutes or on significant change (>10% change). (c) Rate codes mapping: each OTA may have different rate plan names — the channel manager maintains a mapping table. (4) **OTA reservation ingestion**: (a) OTAs push reservation confirmations via webhook or the channel manager polls. (b) The channel manager creates a reservation in the central reservation system (CRS). (c) Deduplication: OTA sometimes sends the same reservation twice. Use OTA's reservation ID for idempotency. (5) **Concurrency and race conditions**: (a) Two bookings (one direct, one via OTA) for the same room type and date arrive simultaneously. (b) Use optimistic locking on the inventory row: `UPDATE inventory SET available = available - 1 WHERE room_type = X AND date = Y AND available > 0`. (c) If the update fails (available was 0), reject the second booking and push a "sold out" update to the OTA. (6) **API rate limits**: (a) Booking.com: 1000 requests/hour. Expedia: 500 requests/5 minutes. (b) The channel manager implements a token-bucket rate limiter per OTA. (c) Batched updates: when pushing inventory for 365 days, batch into chunks of 31 days per API call. (7) **Connectivity failure handling**: (a) If an OTA API is down, queue updates (Kafka) for retry. (b) Retry with exponential backoff (1s, 2s, 4s, ..., max 5 min). (c) If the OTA is down >30 minutes, send alert and switch to "pull mode" — the OTA will fetch the XML feed when it recovers. (d) Daily reconciliation: compare inventory counts between CRS and each OTA. Report discrepancies >5 rooms for investigation.

## Q30: Design the customer support system for handling booking issues. How do you manage support tickets (booking modifications, cancellation requests, complaints), integrate with the booking system, and provide agents with a 360° guest view?

**Answer:** Hotel customer support requires deep integration with the booking engine, CRM, and payment systems. (1) **Ticket lifecycle**: (a) A ticket is created via: web form, email (automatically parsed), phone call (agent creates on behalf), or chatbot escalation. (b) Each ticket has: category (booking_modification, cancellation, payment_issue, complaint, general_inquiry), priority (low/medium/high/critical), and status (open, in_progress, waiting_on_customer, resolved, closed). (c) SLA rules: critical (guest stranded) → respond within 5 minutes. High (payment issue pre-check-in) → 1 hour. Medium → 4 hours. Low → 24 hours. (2) **Triage and routing**: (a) Rule-based assignment: cancellation request → available agent with cancellation queue. Payment issue → finance team. Platinum tier guest → senior agent. (b) Load balancing: round-robin across available agents per skill group. (c) If queue exceeds 20 unassigned tickets, trigger overflow to the backup team. (3) **360° guest view**: when an agent opens a ticket, they see: (a) Guest profile: name, tier, lifetime value, total stays, last stay. (b) Active and past bookings: dates, hotels, room types, rate paid. (c) Payment history: auth attempts, capture status, refunds. (d) Interaction history: past tickets, chat transcripts, email threads, call recordings (transcript via speech-to-text). (e) Preferences: room type preference, special requests, past complaints. (4) **Agent actions within the ticket**: (a) Modify booking: change dates, room type, add-ons. The agent's UI calls the booking API directly with agent credentials. (b) Issue refund: the agent can initiate a refund up to their authorization limit ($200 for frontline, $1000 for supervisors). Above limit → requires manager approval. (c) Add complimentary nights: within loyalty program rules. (5) **Chatbot + human handoff**: (a) Chatbot handles common requests: "change my check-out date to July 15" (if within cancellation policy). (b) If the request is complex or the customer escalates (types "speak to agent"), the chatbot creates a ticket and transfers to the appropriate queue. (c) Context: the agent sees the full chat transcript and any actions already taken by the bot. (6) **Post-resolution survey**: (a) After ticket resolution, trigger an automated satisfaction survey (CSAT). (b) Score 1-5. If <3, the ticket is reopened and assigned to a senior agent for follow-up. (c) Track agent performance: avg resolution time, CSAT score, tickets resolved per shift.
